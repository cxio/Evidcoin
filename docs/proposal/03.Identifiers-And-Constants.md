# Identifiers And Constants（标识符与常量）

## 来源构想

本提案来源于以下基础 Proposal 与构想文档：

- `docs/proposal/00.Project-Scope.md`：定义第一阶段只生成基础层共享规格，允许规划基础类型、固定长度 `Hash`、各类 `ID` 与跨模块共享常量，但不实现区块、交易、状态、脚本或共识逻辑。
- `docs/proposal/01.Types-And-Encoding.md`：定义固定长度字节、规范化编码、整数、字节串、列表和可选字段规则。所有标识符作为固定长度字节序列参与编码时，必须使用精确长度，不额外携带长度前缀。
- `docs/proposal/02.Cryptography-And-Hashing.md`：定义 `Block header`、`CheckRoot`、`TxID`、树结构、附件指纹、地址哈希和铸凭哈希的算法分配与输出长度。
- `docs/conception/blockchain.md`：提出区块头 Hash、`CheckRoot`、年块高度、区块间隔、主链和分叉标识、Hash 策略，以及 `87661` 块/年的时间高度模型。
- `docs/conception/附.交易.md`：提出 `TxID = SHA3-384(TxHeader)`、交易输入中的 48 字节完整交易 ID、20 字节局部交易引用、地址哈希和地址文本编码构想。
- `docs/conception/5.信用结构.md`：提出 `Coin`、`Credit`、`Proof` 三类信元，附件 ID 中的 `SHA3-512` 附件指纹、片组哈希和若干字段长度限制。
- `docs/conception/6.脚本系统.md`：提出脚本资源上限，包括数据栈高度、栈项大小、锁定脚本长度、解锁脚本长度和单笔交易大小限制。
- `docs/conception/1.共识-历史证明（PoH）.md`：提出 PoH 的基础时间窗口、铸凭哈希、评参区块、币权因子和择优池参数。本提案只记录可被基础层引用的常量边界，不定义完整 PoH 流程。

本文件是基础层 Proposal 的标识符与常量文档。后续 Plan 应引用本文件定义 `pkg/types/` 中的固定长度 ID、Hash 类型别名或包装类型，以及跨模块共享的常量。

## 目标

本提案定义 Evidcoin 基础层可共享的标识符长度与核心常量，为后续 `pkg/types/`、`pkg/crypto/` 和高层模块提供统一引用点。

具体目标：

- 固定主要协议标识符的名称、长度和来源算法。
- 区分 `Hash` 输出类型与语义化 `ID` 类型，避免不同用途的 32B、48B、64B 字节序列被误用。
- 记录时间、高度、脚本、交易和共识基础常量，使后续模块不重复定义不一致的数值。
- 说明地址文本编码的构想边界，同时避免在基础层提前固定最终文本前缀、校验算法细节或第三方库。
- 为后续 Plan 的类型定义、边界测试和文档追溯提供依据。

## 非目标

本提案不完整定义以下内容：

- 不定义完整区块结构、区块头字段顺序、区块验证流程、主链切换或年块存储策略。
- 不定义完整交易结构、交易状态机、签名消息结构、输入输出树构造或交易验证流程。
- 不定义脚本 VM、脚本指令集、脚本成本模型或执行语义。
- 不定义 PoH 完整铸造竞争流程、择优池同步、铸造者资格验证或 Coinbase 交易格式。
- 不定义 UTXO/UTCO 状态集合、集合指纹计算流程或状态存储格式。
- 不固定地址文本编码的最终前缀、最终 Base 编码库、校验码 Hash 输入格式或多签地址排序规则。
- 不选择最终 Go 类型实现方式，例如数组类型、别名类型、结构包装类型或泛型固定字节类型。

这些内容应由后续专门 Proposal 和 Plan 承接。本文件只固化基础层共享的命名、长度和常量边界。

## 标识符类型

标识符应当被视为带语义的固定长度字节序列。即使两个标识符长度相同，也不能在 API、测试向量或文档中随意互换。

基础标识符长度如下：

| 标识符 | 长度 | 来源算法或用途 | 说明 |
|--------|------|----------------|------|
| `BlockID` | 48 bytes | `SHA3-384(BlockHeader)` | 区块 ID 来源于区块头 Hash。完整区块头结构由后续区块 Proposal 定义。 |
| `TxID` | 48 bytes | `SHA3-384(TxHeader)` | 交易 ID 来源于交易头 Hash。交易头字段顺序由后续交易 Proposal 定义。 |
| `CheckRoot` | 48 bytes | `SHA3-384` | 校验根用于约束交易树根和 UTXO/UTCO 指纹等状态摘要。精确组合方式由后续区块或状态 Proposal 定义。 |
| `AddressHash` | 32 bytes | `SHA3-256(BLAKE2b-512(pubkey material))` | 账户地址、公钥哈希和复合公钥哈希的基础摘要。文本编码另见本文件地址章节。 |
| `TreeHash` | 32 bytes | `BLAKE3-256` | 树内部节点、树根、交易输入/输出树或附件片组树中高频计算的 Hash。具体树节点格式由后续 Proposal 定义。 |
| `AttachmentHash` | 64 bytes | `SHA3-512` | 附件整体指纹，用于超长期完整性保护。不同于附件片组哈希。 |
| `MintHash` | 64 bytes | `BLAKE3-512` with explicit XOF output length | PoH 铸凭哈希，必须显式输出 64 字节。完整铸凭输入和比较规则由后续 PoH Proposal 定义。 |

补充约束：

- `BlockID`、`TxID` 和 `CheckRoot` 均为 48 bytes，但语义不同，应定义为不同命名类型或具备等价的 API 隔离。
- `AddressHash` 和 `TreeHash` 均为 32 bytes，但前者用于地址与公钥材料，后者用于树结构，不能相互替代。
- `AttachmentHash` 和 `MintHash` 均为 64 bytes，但算法、用途和安全语义不同，不能相互替代。
- 固定长度标识符在规范化编码中直接写入原始字节，不带长度前缀；若被放入可变长度容器，必须由外层容器按 `01.Types-And-Encoding.md` 编码。
- 构想层提到交易输入中可使用 `TxID` 前 20 字节作为局部引用。本提案不把该片段提升为通用 `ID` 类型；若后续交易 Proposal 需要，应单独定义其名称、碰撞处理和适用范围。
- `Protocol-ID`、`Chain-ID`、`Genesis-ID` 和 `Bound-ID` 属于链识别与签名消息前置识别信息。`Genesis-ID` 可引用创世区块的 `BlockID`，但是否定义为独立语义类型由后续链标识 Proposal 决定；其余文本或截断标识的编码规则由后续链标识或交易签名 Proposal 定义。

## 时间与高度

Evidcoin 构想层倾向用区块高度和固定区块间隔表达链上确定性时间，区块头本身不包含普通时间戳字段。

基础时间和高度常量：

```text
BlockInterval = 6 minutes = 360 seconds
BlocksPerYear = 87661
```

约束：

- `BlockInterval` 表示目标区块间隔为 6 分钟，即 360 秒，用于从创世块时间和高度推导确定性链上时间。Go 实现中的 `time.Duration` 表达由后续 Plan 或代码定义。
- `BlocksPerYear` 表示按恒星年折算的年度区块数，用于年块引用、年度检索和按年增长的容量规则。
- 区块高度应作为确定性协议输入；涉及有效期、高度截止、年块引用或 PoH 窗口时，后续 Proposal 应明确使用相对高度还是绝对高度。
- 构想层中交易头示意包含 `Timestamp`，而区块头构想强调通过高度和间隔推导时间。交易时间戳的最终语义、精度和是否参与签名消息，必须由后续交易 Proposal 单独确认。

## 脚本限制常量

基础脚本限制常量：

```go
MaxStackHeight = 256
MaxStackItem = 1024
MaxLockScript = 1024
MaxUnlockScript = 4096
```

约束：

- `MaxStackHeight` 表示脚本执行期间数据栈高度上限。构想文档描述为 `< 256`，本提案将实现常量记录为 `256`，后续 Plan 必须明确检查语义是 `len(stack) <= 256` 还是入栈前拒绝达到第 256 项以上。
- `MaxStackItem` 表示单个栈项最大字节数，单位为 bytes，对应构想中的 `< 1KB`。
- `MaxLockScript` 表示锁定脚本最大字节数，单位为 bytes，对应构想中的 `< 1KB`。
- `MaxUnlockScript` 表示解锁脚本最大字节数，单位为 bytes，对应构想中的 `< 4KB`，不含构想中提到的标准签名数据例外；例外边界由后续 Script 或 Transaction Proposal 固定。
- 这些常量只定义硬资源上限，不定义脚本指令、成本函数、可用 opcode 区间、`EMBED`/`GOTO` 运行时限制或公共验证规则。

## 交易限制常量

基础交易限制候选常量：

```text
MaxTxSizeCandidate = 65535 bytes
```

约束：

- `MaxTxSizeCandidate` 表示单笔交易大小上限的候选精确字节值，单位为 bytes，尚未作为最终协议常量固定。
- `docs/conception/6.脚本系统.md` 同时描述单笔交易大小为 `< 64KB`，且不含解锁部分；`65535` 是接近 64KB 的精确字节上限。实现前必须固定最终检查语义，包括是否使用 `<= 65535`、是否排除标准签名数据、是否排除全部解锁部分，以及序列化外层长度是否计入。
- 本提案不定义交易字段集合、输入输出数量上限、附件字段、签名消息、交易费计算或交易状态机。
- 构想层提到输出项数量 `< 1024`，但该值暂不纳入本基础常量列表。若后续交易 Proposal 需要，应在交易结构上下文中定义并解释序列化和验证边界。

## 共识基础常量

本节只记录构想层中可能影响基础常量或后续 Proposal 的 PoH 参数，不定义完整共识流程。

已固定为基础共享常量的共识相关值：

- `BlockInterval = 6 minutes = 360 seconds`：PoH 窗口、评参区块时间估算和同步阶段均建立在 6 分钟区块间隔之上。
- `BlocksPerYear = 87661`：用于年块引用和按年度估算的链上时间。
- `MintHash` 长度为 64 bytes：铸凭哈希由显式 64 字节 `BLAKE3-512` 输出表示。

构想层中已出现但不在本提案固定为基础常量的 PoH 参数：

- 铸凭交易有效范围 `[-80000, -28]`。
- 评参区块为链末端 `-9` 号区块。
- 币权总值取链末端 `-27` 号区块。
- 择优池容量为 `20`，后 `15` 名有权发起同步。
- 铸造者最低币权门槛为 `4M` 聪时。

这些值属于共识层规则，后续 PoH Proposal 必须重新确认、命名并给出验证流程、升级策略和测试向量。本基础提案不把它们直接放入 `pkg/types/` 的通用常量，避免在未定义流程前冻结共识实现细节。

## 地址文本编码

构想层提出将公钥哈希编码为人类可读的账户地址。基础 Hash 规格已经固定：

```text
AddressHash = SHA3-256(BLAKE2b-512(pubkey material))
AddressHash length = 32 bytes
```

地址文本编码的构想规则：

- 对 `AddressHash` 添加识别前缀，形成用于校验码计算的输入。
- 对前缀和 `AddressHash` 执行 Hash 运算，生成固定长度 checksum；checksum 长度和截取规则由后续地址 Proposal 固定。
- 将 `AddressHash || checksum` 编码为文本序列。
- 在文本序列前附加识别前缀，形成最终人类可读地址。
- 解码时提取前缀和文本序列，恢复 `AddressHash` 与校验码，并重新计算校验码进行比较。

本提案不固定以下事项：

- 不固定最终 Base 编码方案或具体 Base58 库。
- 不固定主网、测试网、私有链或应用链的最终地址前缀。
- 不固定校验码使用的 Hash 算法和 domain tag 字节。
- 不固定多签复合地址中 `M-Of-N`、公钥哈希排序和复合哈希输入格式。

后续地址或交易 Proposal 应在保持单签地址与多签地址外观一致的前提下，固定文本编码、前缀、校验码、错误处理和测试向量。

## 未决问题

- `MaxStackHeight = 256` 与构想层 `< 256` 的自然语言描述需要在实现前统一为精确检查语义。
- `MaxStackItem = 1024`、`MaxLockScript = 1024`、`MaxUnlockScript = 4096` 与构想层 `< 1KB`、`< 4KB` 的关系需要在实现前统一为 `<= constant` 还是 `< constant`。
- `MaxTxSizeCandidate = 65535 bytes` 与构想层 `< 64KB`、不含解锁部分的描述需要在交易 Proposal 中固定精确计数边界。确认前不得把 `65535` 作为最终硬常量实现。
- `Protocol-ID`、`Chain-ID`、`Genesis-ID` 和 `Bound-ID` 的类型、编码、是否进入签名消息以及 `Bound-ID` 截断长度需要由后续链标识或交易签名 Proposal 定义；`Genesis-ID` 可引用创世区块的 `BlockID`，但是否定义为独立语义类型由后续链标识 Proposal 决定。
- 地址文本编码的最终前缀、Base 编码、校验码算法和 domain tag 尚未固定。
- PoH 构想中的评参区块、铸凭交易有效范围、择优池容量、授权节点数量和币权门槛是否作为可升级共识参数，需要由后续 PoH Proposal 确认。

## 对后续 Plan 的影响

后续 Plan 必须遵守以下影响：

- `pkg/types/` 应定义或等价表达 `BlockID`、`TxID`、`CheckRoot`、`AddressHash`、`TreeHash`、`AttachmentHash` 和 `MintHash` 的固定长度边界。
- 后续实现应避免仅用裸 `[]byte` 传递协议标识符；即使底层表示相同，也应通过命名类型、构造函数或验证函数防止长度和用途混淆。
- `pkg/types/` 或基础常量包应提供 `BlockInterval`、`BlocksPerYear`、`MaxStackHeight`、`MaxStackItem`、`MaxLockScript` 和 `MaxUnlockScript` 的单一来源；`MaxTxSize` 需等待交易 Proposal 确认后再进入最终常量。
- `pkg/crypto/` 应继续引用 `02.Cryptography-And-Hashing.md` 的算法分配，确保本文件中的标识符长度与 Hash 输出长度一致。
- 后续区块 Plan 应复用 `BlockID`、`CheckRoot`、`BlockInterval` 和 `BlocksPerYear`，不得重复定义不一致的区块 ID 长度或年块常量。
- 后续交易 Plan 应复用 `TxID`、`AddressHash`，并根据交易 Proposal 确认 `MaxTxSize` 是否采用 `65535 bytes` 候选值，以及交易大小是否包含解锁脚本、标准签名数据和外层长度。
- 后续 Script Plan 应复用脚本限制常量，但脚本 VM、opcode、成本函数和执行规则必须在脚本层定义。
- 后续 PoH Plan 应复用 `MintHash` 长度、`BlockInterval` 和 `BlocksPerYear`，但完整铸造竞争、择优池同步和币权门槛规则必须在共识层定义。
- 后续地址 Plan 应以 `AddressHash` 32 bytes 为基础，固定文本编码、前缀、校验码和测试向量；在确认前不得把某个 Base58 库作为协议事实写入基础层。
