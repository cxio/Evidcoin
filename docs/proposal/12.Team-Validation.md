# Team Validation（组队校验）

## 来源构想

- `docs/conception/附.组队校验.md`：定义校验组、管理层、守卫者、校验员、交易校验、铸造协作和区块发布。
- `docs/conception/2.共识-端点约定.md`：定义冗余出块、分叉和交易回收。
- `docs/conception/1.共识-历史证明（PoH）.md`：定义择优池和铸造候选者。
- `docs/proposal/04.Hash-Trees.md`：定义 Coinbase 纳入证明和交易树路径。
- `docs/proposal/10.PoH-Consensus.md`：定义铸造候选者和择优凭证边界。

## 目标

本提案定义校验组作为自由协作实体的技术边界。校验组提高交易验证和区块构造效率，但不成为新的共识权主体。

## 非目标

- 不实现组内 RPC、任务调度或数据库。
- 不定义校验员信誉系统。
- 不定义组内收益分配私约。
- 不实现交易池或区块打包代码。
- 不定义 P2P 拓扑。

## 校验组模型

- 校验组是协同工作的节点群，可视为逻辑单体。
- 校验组创建自由，无协议许可。
- 节点可自由加入校验组；强节点可加入多个校验组。
- 校验组之间保持 P2P 关系。
- 最终区块仍由 PoH 择优池中的铸造者签名。

## 角色

管理层：

- 分发交易校验任务。
- 控制冗余校验。
- 汇总合法交易。
- 记录组内工作绩效。
- 与外部校验组广播者交互。
- 接收铸造申请并协助打包区块。

守卫者：

- 接收外部交易。
- 执行首领校验。
- 将初步合法交易提交管理层。
- 与其它组守卫者和校验员传播交易。
- 维护首领输入黑名单。

校验员：

- 向管理层请求交易任务。
- 执行完整交易验证。
- 返回合法或非法结果。
- 将验证合法交易传递给其它组守卫者。
- 可在高负载时拒绝任务。

## 组内缓存服务

校验组可部署：

- UTXO Cache。
- UTCO Cache。
- 外部脚本引用缓存。

缓存只提升效率，不改变状态合法性来源。缓存结果必须可由区块头、状态指纹和交易证明验证。

## 首领校验

- 首领校验仅验证首笔输入。
- 首笔输入必须是 Coin 输入。
- 首笔输入应为全部 Coin 输入中币权最大者。
- 首领校验通过只表示准合法，不表示完整交易合法。
- 完整验证失败后，首领输入进入临时黑名单。
- 黑名单冻结建议为 24 小时或等价区块数；精确协议地位未决。

## 冗余与复核

基础参数：

```text
MinValidationRedundancy = 2
```

规则：

- 每笔交易至少由两名校验员完整验证。
- 全部反馈合法，则交易可进入合法池。
- 任一反馈非法，则进入扩展复核。

一级复核：

- 零报错：交易合法。
- 超过半数报错：交易非法。
- 低于半数报错：进入二级复核。

二级复核：

- 只要有报错，即非法。

复核任务不应向校验员透露其复核身份，避免针对性行为。

## 组间反馈

- 校验员可向其它组守卫者传播已验证合法交易。
- 接收组若完整验证后判定非法，应反馈给发送者。
- 发送者可向本组管理层请求扩展复核。
- 管理层需要记录交易递送来源，以支持反馈路径。

## 交易收录策略

推荐优先级：

```text
coin-age destroyed more > transaction fee more > Credit early destruction
```

该优先级是区块构建策略，不是交易合法性规则，除非后续共识 Proposal 明确升级为协议规则。

## 碰撞拒绝规则

构想层提出两类不收录规则：

- 输入项 `TxID` 20 bytes 局部引用碰撞的交易不收录。
- 进入 UTXO/UTCO 的新交易，若末端叶子层 `TxID` 前 8 bytes 碰撞，不收录。

本提案将其记录为待精确定义的协议候选规则。必须明确检查范围后才能实现。

## 铸造者协作

流程：

1. 铸造候选者确认自己在择优池。
2. 向管理层提交择优证明。
3. 管理层验证资格。
4. 管理层提供交易费、校验组收益地址、公共服务推荐地址、铸币量、兑奖截留和兑奖槽推荐。
5. 铸造者构造 Coinbase。
6. 管理层验证 Coinbase 并打包区块。
7. 管理层返回 Coinbase 纳入证明、交易树根、UTXO/UTCO 指纹和 `CheckRoot` 数据。
8. 铸造者验证 Coinbase 未被篡改。
9. 铸造者签署区块签名消息。
10. 管理层验证签名，构造区块头并发布。

## 区块发布三段

1. 区块证明：Coinbase、Coinbase 路径、铸造者签名、区块头。
2. 区块概要：完整交易 ID 序列，可用前 16 bytes 优化传输。
3. 交易数据同步：补足缺失交易后完整验证。

交易 ID 前 16 bytes 只是同步优化，最终验证必须使用完整 48 bytes `TxID`。

## 信息分离

- 管理层掌握完整交易集和状态指纹。
- 铸造者掌握私钥和 Coinbase 构造权。
- 管理层不能伪造铸造者签名。
- 铸造者不能单方面伪造完整区块状态。
- 铸造者必须能验证自己的 Coinbase 被包含。

## 未决问题

- 管理层是否多节点化，以及广播者/调度者是否有签名身份。
- 首领输入黑名单是本地策略、共约还是协议规则。
- “首笔输入币权最大”是否纳入交易合法性。
- 20B 和 8B 碰撞拒绝规则的检查范围。
- 校验组收益地址格式和组内收益拆分是否上链。
- 区块签名消息是只签 `CheckRoot` 还是签完整区块头上下文。
- 管理层公共服务推荐是否需要签名。

## 对后续 Plan 的影响

- 校验组接口应围绕任务、结果、证明和签名消息设计，不依赖具体网络实现。
- 首领校验和完整校验必须有不同状态。
- 区块证明发布必须能在缺失交易时进行临时转播，但最终必须完整验证。
