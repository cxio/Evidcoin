# UTXO UTCO State（UTXO/UTCO 状态）

## 来源构想

- `docs/conception/附.交易.md`：定义 UTXO/UTCO 来源、未确认输出不可消费和输入解析规则。
- `docs/conception/5.信用结构.md`：定义 Coin/Credit/Proof 进入状态集的差异、Credit 有效期和清理规则。
- `docs/conception/blockchain.md`：定义 UTXO/UTCO 指纹参与 `CheckRoot`。
- `docs/proposal/04.Hash-Trees.md`：定义四层宽成员状态指纹树和末端叶子摘要。
- `docs/proposal/06.Transaction-Model.md`：定义输入项、输出 envelope 和交易验证边界。
- `docs/proposal/07.Coin-Credit-Proof-Units.md`：定义三类信元状态语义。

## 目标

本提案定义状态层规格：

- UTXO 表示已确认且未花费的 Coin 输出。
- UTCO 表示已确认且未转移的 Credit 输出。
- Proof 不进入状态集。
- 状态转移必须确定性执行，并生成可参与 `CheckRoot` 的状态指纹。
- 状态层为交易验证提供输入解析、可用性检查、重复消费检查和输出插入/清理规则。

## 非目标

- 不实现数据库、缓存、快照或 undo log。
- 不执行脚本 VM，只调用脚本验证结果或接口。
- 不定义完整交易结构、Coinbase 结构或 PoH 规则。
- 不实现 Blockqs 状态查询服务。

## UTXO 模型

UTXO entry 应至少承诺：

- 来源年度。
- 完整 `TxID`。
- `OutIndex`。
- Coin payload 摘要或完整 payload。
- 金额。
- 接收者。
- 锁定脚本。
- 创建高度。
- 有效状态位。

规则：

- 只有已确认 Coin 输出可进入 UTXO。
- 销毁 Coin 输出不进入 UTXO。
- 输入消费后，对应 UTXO 从有效集合移除或标记为无效。
- 同一交易、同一区块或同一验证批次内不得重复消费同一 UTXO。
- 未确认交易输出不得作为 UTXO。
- 状态更新必须与交易顺序、区块顺序和输入解析规则一致。

## UTCO 模型

UTCO entry 应至少承诺：

- 来源年度。
- 完整 `TxID`。
- `OutIndex`。
- Credit payload 摘要或完整 payload。
- 接收者。
- 创建者或创建者引用。
- 配置。
- 标题。
- 描述。
- 附件 ID。
- 锁定脚本。
- 创建高度或最近激活高度。
- 截止高度。
- 剩余转移次数。
- 有效状态位。

规则：

- 只有已确认且仍可转移的 Credit 输出进入 UTCO。
- Proof 不进入 UTCO。
- 销毁或终止转移能力的 Credit 不进入 UTCO。
- 转移 Credit 时旧 UTCO 失效，新 Credit 输出按规则进入 UTCO。
- 转移次数从 1 降到 0 后，新输出不再进入 UTCO。
- 到达截止高度后应清理。
- 无期限 Credit 超过 11 年未激活应失效。
- 有期限 Credit 应维护高度索引或可重建索引，以便确定性清理。

## 状态转移

按区块应用交易的逻辑流程：

1. 从上一区块状态开始。
2. 按区块交易序列验证交易。
3. Coinbase 创建 Coin 输出，不引用输入。
4. 普通交易解析输入，首领输入必须使用完整 `TxID`。
5. 验证输入存在、有效且未在当前批次重复消费。
6. 调用脚本验证或接收外部脚本验证结果。
7. Coin 输入消费 UTXO。
8. Credit 输入消费 UTCO。
9. Proof 输入引用必须拒绝。
10. 输出按类型、销毁标记、有效期和转移次数插入 UTXO/UTCO 或忽略。
11. 区块结束后执行到期 Credit 清理。
12. 计算新的 `UTXORoot` 和 `UTCORoot`。
13. 与交易树根组合得到 `CheckRoot`，并与区块头对比。

状态根应表示区块后状态。本规则若被修订，必须同步修订 `05.Blockchain-Core.md` 和测试向量。

## 局部 TxID 引用解析

非首领输入使用 `TxID` 前 20 bytes：

- 查找键至少包含 `Year + TxIDPart + OutIndex`。
- 状态层可维护 `TxIDPart => full TxID` 索引以提高解析效率。
- 若匹配多个有效状态项，交易必须拒绝。
- 20 bytes 局部引用只在已由状态指纹约束的 UTXO/UTCO 集内使用，不可替代全局 `TxID`。
- 首领输入不得使用局部引用。

## 状态指纹树

UTXO/UTCO 指纹树引用 `04.Hash-Trees.md`：

- UTXO 和 UTCO 使用同结构但必须语义隔离。
- 四层宽成员哈希校验树。
- 顶层按年度分级。
- 后三级使用 `TxID` 字节序列的 `[8,13,18]` 字节值分层；字节索引是否 0-based 需确认，Plan 中不得默默假定。
- 末端叶子摘要：

```text
LeafHash = SHA3-384(TxID || DataID || FlagOutputs)
DataID = SHA3-384(OutputsPayloadsSortedByOutIndex)
```

规则：

- `FlagOutputs` 中 `1` 表示未花费或未转移，`0` 表示已无效。
- 输出 payload 必须按 `OutIndex` 升序。
- UTXO 与 UTCO 即使结构相同，也必须通过 domain tag、类型字段或 API 类型隔离。
- 年度分区编码、空分支、成员排序和节点组合规则仍需固定。

## 状态同步与快照

状态可通过以下方式得到：

- 从创世块重放完整历史。
- 从可验证快照加载，并验证 `UTXORoot` 和 `UTCORoot`。
- 从公共服务获取状态片段或完整集合，再用状态指纹验证。
- 从最近可信状态开始应用后续区块。

状态快照必须绑定高度、`BlockID`、`UTXORoot`、`UTCORoot` 和链识别信息。公共服务返回的状态数据不能被信任，只能验证后使用。

## 回滚与分叉

状态层必须支持分叉处理边界：

- 短分叉重组需要状态回滚或从快照重建。
- 长期手动切换主链需要显式替换状态。
- 是否使用 undo log、周期快照或外部重建是实现选择。
- 状态层不得自动决定超过竞争窗口的主链归属。

## 未决问题

- UTXO/UTCO entry 的最终字段集合和 payload 摘要字段。
- `FlagOutputs` 位序、长度和补齐规则。
- 四层宽成员树的节点组合算法、年度分区编码和空根。
- `TxID` 字节 `[8,13,18]` 的 0-based/1-based 索引语义。
- 同一区块内是否允许消费前序交易输出；本提案倾向禁止。
- Credit 到期边界是到达截止高度立即失效，还是下一区块失效。
- 无期限 Credit 11 年激活窗口的精确高度换算。
- 状态快照格式和状态证明格式。

## 对后续 Plan 的影响

- 状态实现必须先定义可测试的 UTXO/UTCO entry 规范化编码。
- 状态转移测试必须覆盖重复消费、局部 TxID 歧义、Proof 输入拒绝、Credit 到期清理和状态根变更。
- 状态根实现不得复用普通二元 Merkle 树替代四层宽成员树。
- 在空根、节点组合和 domain tag 未决前，只能实现结构化占位或拒绝最终协议测试向量。
