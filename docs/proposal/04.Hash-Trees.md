# Hash Trees（哈希树）

## 来源构想

本提案来源于以下基础 Proposal 与构想文档：

- `docs/proposal/00.Project-Scope.md`：定义第一阶段只生成基础层共享规格，不实现交易、状态、区块或共识逻辑。
- `docs/proposal/01.Types-And-Encoding.md`：定义规范化编码、固定长度字节、固定宽度整数、列表和结构体序列化规则，Hash 树输入必须引用这些规则。
- `docs/proposal/02.Cryptography-And-Hashing.md`：定义树内部节点使用 `BLAKE3-256` 的基础分配，树叶子在构想明确时可使用 `SHA3-384`，并要求 Hash 输入具备用途隔离语义。
- `docs/proposal/03.Identifiers-And-Constants.md`：定义 `TreeHash` 为 32 bytes，`TxID` 为 48 bytes，`CheckRoot` 为 48 bytes，`AttachmentHash` 为 64 bytes，并要求不同语义的同长字节序列不得混用。
- `docs/conception/blockchain.md`：提出区块交易集的含序二元哈希校验树、`CheckRoot = Hash384(TreeRoot || UTXORoot || UTCORoot)` 的构想，以及 UTXO/UTCO 指纹、交易输入树和附件片组树等树结构。
- `docs/conception/附.交易.md`：提出交易输入集、输出集和区块内交易验证需要哈希校验树；区块内交易数据的检索和验证依赖交易 ID 清单与验证路径序列。
- `docs/conception/5.信用结构.md`：提出附件 ID 包含附件整体指纹、分片数量和片组哈希；附件片组树叶子为分片数据哈希并前置 2 字节序号。
- `docs/conception/附.组队校验.md`：提出铸造者需要通过哈希树路径验证 Coinbase 交易已被纳入区块，且 UTXO/UTCO 指纹是独立的四层宽成员哈希校验树。

本文件是基础层 Proposal 的 Hash 树文档。后续区块、交易、状态、附件和共识 Proposal 可以引用本文件中的共享树规则，但必须在各自领域内继续固定具体字段、验证流程和测试向量。

## 目标

本提案定义 Evidcoin Hash 树的共享规则和边界，为后续交易树、状态指纹树、附件片组树和 `CheckRoot` 组合提供统一引用点。

具体目标：

- 定义通用节点类型、二元哈希树、含序叶子、空树与单叶树、证明路径的基础约束。
- 固定共享规则：内部分支 Hash 默认使用 `BLAKE3-256`，含序叶子在 Hash 前包含文档指定的序号前缀，证明路径包含重建根所需的位置数据。
- 记录交易集合树、附件片组树和 UTXO/UTCO 指纹树的差异，避免把状态树误当作交易类 Merkle 树。
- 为后续 Plan 的测试向量、类型边界和跨实现一致性提供规则来源。

## 非目标

本提案不完整定义以下内容：

- 不定义完整交易树实现、交易输入树实现、交易输出树实现或区块交易集合的最终二进制节点格式。
- 不定义 UTXO/UTCO 状态存储实现、缓存结构、更新流程、逆向推导流程或四层宽成员树的完整平衡策略。
- 不定义树平衡策略、奇数叶处理策略、批量更新算法、节点持久化格式或缓存淘汰策略。
- 不定义证明路径的最终二进制格式、网络传输格式、错误码或压缩策略。
- 不定义 `CheckRoot` 的最终字段顺序、domain tag 字节、区块头签名消息或铸造交互协议。
- 不定义附件分片大小、数据源协议、Depots 存储策略或 P2P 分片传输流程。

这些内容应由后续专门 Proposal 和 Plan 承接。本文件只固化跨树共享的 Hash 约束和文档边界。

## 通用节点类型

Hash 树的节点按用途分为叶子节点、内部分支节点和根节点。

- `Leaf`：由领域对象或领域对象摘要构成。叶子输入必须使用规范化编码；若该树要求含序叶子，序号前缀必须先进入叶子 Hash 输入。
- `Branch`：由一个或多个子节点 Hash 按固定顺序组合得到。除后续 Proposal 对某一具体树另有收窄外，内部分支 Hash 使用 `BLAKE3-256`，输出为 32 bytes `TreeHash`。
- `Root`：代表整棵树的最终 Hash。若根由分支节点计算得到，默认同属 `BLAKE3-256` 的 32 bytes `TreeHash`；若后续 Proposal 对某一树的根另有外层组合或转换，必须明确记录。

通用约束：

- Hash 输入必须引用 `01.Types-And-Encoding.md` 的规范化编码，不能使用 JSON、map 遍历顺序、反射序列化或平台相关字节序。
- 节点 Hash 必须具备用途隔离语义。最终 domain tag 字节表仍未固定，但测试向量必须能区分叶子、分支、交易集合树、附件片组树和状态指纹树。
- 固定长度 Hash 或 ID 作为节点输入时直接使用原始字节，不额外添加长度前缀，除非外层结构明确把它们放入可变长度字节容器。
- 不同语义的同长 Hash 不得混用。例如 `AddressHash` 与 `TreeHash` 均为 32 bytes，但不能相互替代。

## 二元哈希树

二元哈希树用于构想层明确要求类 Merkle 结构的场景，例如区块交易集合、交易输出集合和附件片组集合。

共享规则：

- 内部分支节点的 Hash 算法默认为 `BLAKE3-256`，除非后续 Proposal 对具体树收窄为其它算法或外层组合方式。
- 分支输入必须保留左右顺序，不能按 Hash 值排序后再组合。
- 分支 Hash 的输入应包含足以区分左子节点和右子节点的规范化结构。具体字段、domain tag 是否进入协议字节和单子节点处理方式由后续 Proposal 固定。
- 树根通常为最终分支 Hash；若只有单叶或空树，按本文件“空树与单叶树”章节处理。

本提案不固定奇数叶提升、复制末叶、稀疏补空、完全二叉树填充等策略。后续交易、附件或状态 Proposal 必须在需要验证路径互操作前固定这些策略。

## 含序叶子

含序叶子用于需要随机定位、快速同步或分片验证的树。构想层已经明确两类含序叶子：区块交易集合树和附件片组树。

共享规则：

- 含序叶子（`ordered leaves`）必须在叶子 Hash 输入中包含文档指定的序号前缀。
- 序号前缀是叶子内容的一部分，不是证明路径的外部提示；验证者必须用相同序号前缀重算叶子 Hash。
- 固定宽度序号使用 big-endian byte order，除非对应 Proposal 明确列出协议例外。
- 序号从 `0` 开始还是从 `1` 开始由对应具体树 Proposal 固定。构想层的交易输出序位已出现“从0开始”，但本文件不把所有树统一推导为从 `0` 开始。

已记录的含序叶子约束：

- 区块交易集合树使用交易 ID 作为叶子主体，并使用 3-byte sequence prefix。
- 附件片组树使用 shard hash 作为叶子主体，并使用 2-byte sequence prefix。

## 空树与单叶树

空树根尚未由构想层固定，因此本提案将 `Empty tree root` 记录为未决问题。

约束：

- 在空树根未固定前，后续 Plan 不得为任何协议级树自行选择全零、空输入 Hash、常量标签 Hash 或其它替代表达并声称为最终协议规则。
- 若某一具体树不允许为空，应由该树的 Proposal 明确拒绝空输入，并在测试向量中覆盖。
- 单叶树是否直接使用叶子 Hash 作为根，还是对叶子 Hash 再执行一次根节点 Hash，尚未在基础层统一固定。后续 Proposal 必须按树类型明确。
- 附件片组中分片数量为 `1` 时，构想层要求“无树结构，但片组哈希正常计算”；该场景的精确单叶计算方式应由后续附件 Proposal 固定。

## 证明路径

证明路径用于验证某个叶子是否属于给定树根。构想层中的交易验证、Coinbase 纳入验证和附件分片验证都依赖证明路径。

共享规则：

- 证明路径（`proof path`）必须包含重建根所需的兄弟节点摘要和位置数据。
- 二元树路径项必须包含 `sibling hash` 和 `sibling side`。`sibling side` 用于说明兄弟节点位于当前节点左侧还是右侧。
- 非二元树或宽成员树不得套用二元 `sibling side` 语义，必须提供等价的位置、成员索引、分组键或层级信息，使验证者能按同一树构造规则重建根。
- 证明路径必须包含足够的位置数据，例如叶子序号、层级位置、必要的总叶子数或同等信息。
- 对含序叶子，验证者必须先用叶子主体和序号前缀重算叶子 Hash，再沿路径逐层重建根。
- 证明路径不得依赖接收方本地排序、隐式交易池顺序或外部数据库状态来补足关键位置语义。

本提案不固定证明路径的二进制格式。后续 Plan 可以先以结构化测试向量表达路径项，但不能把临时编码格式写成最终协议事实。

## 交易相关哈希树

交易相关树包含区块交易集合树、交易输入集 Hash 结构和交易输出集树。它们服务于交易定位、交易头 Hash、区块 `CheckRoot` 和轻量验证。

区块交易集合树约束：

- 叶子主体为 `TxID`，长度为 48 bytes，来源为 `SHA3-384(TxHeader)`。
- 叶子 Hash 输入必须包含 3-byte sequence prefix，以支持区块内交易的快速定位、同步和验证。
- Coinbase 交易按构想应始终位于区块交易序列首个位置 `[0]`，但 Coinbase 交易结构和验证流程由后续交易或共识 Proposal 定义。
- 区块交易集合树根是 `CheckRoot` 的组成部分之一，但 `CheckRoot` 的精确字段顺序、domain tag 和是否包含其它元数据由后续区块 Proposal 固定。

交易输入和输出相关约束：

- 构想层提出交易输入集可由 `InputHash = BLAKE3-256(LeadHash || RestHash)` 表达，其中首领输入单列，其余输入作为列表 Hash。
- 构想层提出交易输出项按二元树结构组织为 `OutputHash`，内部节点与根均使用 `BLAKE3-256`。
- 本提案不完整定义输入项、输出项、签名消息、交易状态机或交易头字段顺序；这些内容由后续交易 Proposal 固定。

## UTXO/UTCO 指纹树

UTXO/UTCO 指纹树是独立的状态树，不应与交易集合的类 Merkle 树混同。

约束：

- UTXO/UTCO 指纹树是独立状态树，不应与交易集合的类 Merkle 树混同。
- UTXO 集和 UTCO 集使用相同结构和计算方式，但代表不同状态语义：未花费币金输出与未转移凭信输出。
- 构想层描述其为四层宽成员哈希校验树，顶层按年度分级，后续三级使用 TxID 序列的 `[8,13,18]` 字节值分层，以获得更好的随机性保证。
- 末端叶子节点数据摘要为 `SHA3-384(TxID || DataID || FlagOutputs)`。其中 `DataID` 是该 `TxID` 下所有未花费或未转出输出项有效载荷数据按 `OutIndex` 升序序列化后的 `SHA3-384` 摘要。
- `FlagOutputs` 表达输出项有效状态位，`1` 表示未花费或未转移，`0` 表示已无效。
- UTXO/UTCO 指纹参与 `CheckRoot`，与区块交易树根共同约束区块合法性。

本提案不定义状态存储格式、缓存器接口、年度分片持久化、更新算法、逆向推导流程或四层宽成员树的完整节点平衡策略。这些属于后续状态 Proposal 和 Plan。

## 附件片组树

附件片组树用于验证附件分片数据，与附件整体指纹用途不同。

约束：

- 附件整体指纹使用 `SHA3-512`，长度为 64 bytes，用于对完整附件数据提供超长期完整性保护。
- 附件片组树使用 `BLAKE3-256`，长度为 32 bytes，用于分片验算和随机分片验证。
- 附件片组树叶子主体为 shard hash，叶子 Hash 输入包含 2-byte sequence prefix。
- 分片数量为 `0` 时，构想层表示无分片，后续片组哈希字段取消。
- 分片数量为 `1` 时，构想层表示无树结构，但片组哈希正常计算；具体单叶计算方式由后续附件 Proposal 固定。
- 分片数量大于 `1` 时，片组哈希由正常的含序哈希校验树计算得到。
- 验证分片时，验证者必须先计算分片数据的 Hash，再前置序号，结合路径 Hash 重建并验证片组哈希。

本提案不定义分片大小、附件类型表、Depots 请求协议、分片缓存策略或 P2P 传输策略。

## 测试向量要求

后续 Plan 实现 Hash 树相关基础工具或具体树时，必须提供跨实现可复现的测试向量。

最低测试向量要求：

- 内部分支 Hash：覆盖左右子节点顺序不同导致根不同，输出长度必须为 32 bytes。
- 含序叶子：覆盖同一叶子主体使用不同序号前缀得到不同叶子 Hash，交易集合树使用 3-byte prefix，附件片组树使用 2-byte prefix。
- 证明路径：二元树覆盖 `sibling hash`、`sibling side`、叶子位置和重建根成功；宽成员树覆盖成员索引、分组键或等价位置数据；覆盖兄弟方向反转、位置错误或缺少位置数据时验证失败。
- 空树：在空树根未决前，只允许测试具体树是否拒绝空输入或标记为未实现，不能固化最终空树根字节。
- 单叶树：在具体 Proposal 固定规则后覆盖单叶根计算；附件分片数量为 `1` 的片组哈希必须单独覆盖。
- 区块交易集合树：覆盖交易 ID 加 3-byte sequence prefix 的叶子构造、路径验证和顺序变化导致根变化。
- 附件片组树：覆盖 shard hash 加 2-byte sequence prefix、分片数据重算和路径验证。
- UTXO/UTCO 指纹树：覆盖末端叶子 `SHA3-384(TxID || DataID || FlagOutputs)` 的规范化输入、`OutIndex` 升序约束和 UTXO/UTCO 语义隔离。

测试向量应记录算法名、用途标签或 domain tag、输入字段、规范化字节、输出 Hash、路径项、规则来源和期望错误。若 domain tag 或 varint 算法尚未固定，不得生成依赖其最终字节的测试向量。

## 未决问题

- 空树根尚未由构想层固定。需要决定具体树是否拒绝空输入，或使用统一空树根常量、domain tag Hash、全零值或其它规则。
- 单叶树根是否直接等于叶子 Hash，还是使用根节点 domain tag 再 Hash，需由具体树 Proposal 固定。
- 二元树奇数叶处理策略尚未固定，包括提升末叶、复制末叶、补空节点或其它结构。
- 证明路径的最终二进制格式、路径项字段顺序、是否包含总叶子数、二元树是否压缩 `sibling side`、宽成员树如何表达成员索引或分组键尚未固定。
- 树节点和叶子节点的最终 domain tag 字节表尚未固定。
- 区块交易集合树的 3-byte sequence prefix 的最大交易数约束、溢出处理和序号起点需在交易或区块 Proposal 中确认。
- 附件片组树中 shard hash 的精确算法输入、分片数量为 `1` 时的片组哈希计算方式，以及分片大小建议需在附件 Proposal 中确认。
- UTXO/UTCO 指纹树的完整节点组合、宽成员分组排序、年度分区编码和持久化格式需由状态 Proposal 固定。

## 对后续 Plan 的影响

后续 Plan 必须遵守以下影响：

- `pkg/types/` 应为 `TreeHash`、`TxID`、`CheckRoot`、`AttachmentHash` 和 UTXO/UTCO 指纹相关 Hash 提供固定长度边界，避免裸 `[]byte` 混用。
- `pkg/crypto/` 应提供明确输出 32 bytes 的 `BLAKE3-256`，供树内部节点和片组树使用。
- 后续交易 Plan 应引用本文件定义区块交易集合树的 3-byte sequence prefix、交易输出树的 `BLAKE3-256` 边界和交易输入 Hash 的构想来源，但交易结构和签名消息由交易 Proposal 固定。
- 后续区块 Plan 应引用本文件定义交易树根与 UTXO/UTCO 指纹作为 `CheckRoot` 组成部分，但 `CheckRoot` 精确组合方式由区块 Proposal 固定。
- 后续状态 Plan 应把 UTXO/UTCO 指纹树作为独立状态树实现，不得复用交易 Merkle-like 树假设替代四层宽成员结构。
- 后续附件 Plan 应引用本文件定义附件片组树的 2-byte sequence prefix、分片数量边界和附件整体指纹与片组 Hash 的用途隔离。
- 在空树、单叶树、奇数叶处理和证明路径二进制格式未固定前，Plan 只能保留接口、结构化测试占位或具体树的拒绝规则，不得声称最终协议格式已经确定。
