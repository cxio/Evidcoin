# Public Service Interfaces（公共服务接口）

## 来源构想

- `docs/conception/README.md`：提出节点发现、Depots、Blockqs、stun2p 四类公共服务网络。
- `docs/conception/3.公共服务.md`：定义服务职责、数据查询、数据心跳和紧缺性感知。
- `docs/conception/4.激励机制.md`：定义公共服务奖励与兑奖。
- `docs/conception/附.组队校验.md`：定义 Blockqs、Depots 在区块同步和校验中的角色。
- `docs/proposal/02.Cryptography-And-Hashing.md`：定义附件指纹、地址哈希和签名边界。
- `docs/proposal/04.Hash-Trees.md`：定义附件片组树、交易树和证明路径。

## 目标

本提案定义 Evidcoin 与公共服务网络的接口边界。公共服务提供数据、查询和网络辅助能力，但不成为共识信任锚。

## 非目标

- 不实现 `cxio/p2p`、Depots、Blockqs 或 stun2p。
- 不定义 HTTP/gRPC/P2P 包格式。
- 不定义数据库 schema、缓存策略或服务评分算法。
- 不固定附件分片大小和小附件阈值的协议地位。

## 总边界

- 公共服务不参与区块合法性裁决。
- 公共服务返回的数据必须被验证。
- 公共服务失败不改变区块或交易合法性。
- 公共服务可被 Coinbase 奖励，但奖励选择与服务真实性通过后续兑奖确认形成弱约束。

## 服务类型

- 基网 / 节点发现：应用节点和服务节点发现。
- `Depots`：完整区块、大附件和附件分片存储分享。
- `Blockqs`：交易、区块、脚本、状态和小附件查询。
- `stun2p`：NAT 探测和 UDP 打洞协助。

## 节点发现

- 由外部 P2P 项目承担。
- 服务节点可向应用节点提供自己的区块链奖励地址。
- Evidcoin 不固定其 DHT、广播、路由或 NAT 策略。
- 链内只关心服务地址能否被 Coinbase 引用和后续兑奖确认。

## Depots

服务能力：

- 存储和分享完整区块数据。
- 存储和分享大附件、附件分片和分片索引。
- 响应数据查询和上传引导。
- 支持数据紧缺性感知。
- 支持数据心跳。

验证要求：

- 附件整体必须用 `SHA3-512` 附件指纹验证。
- 分片必须用片组 Hash 和证明路径验证。
- 完整区块数据必须通过区块头、交易树和 `CheckRoot` 验证。
- Depots 返回数据不可被信任，只能被验证后使用。

## 紧缺性感知与心跳

紧缺性感知：

- 查询请求包含 hop count。
- 拥有数据的节点回应并终止转播。
- 无数据节点递增 hop count 后转播。
- 较大 hop count 表示数据较稀缺。
- 起跳数可非零，用于隐藏起点或调试。

数据心跳：

- 节点可周期性探查链上附件或区块数据。
- 心跳类似查询，但不要求回复。
- 心跳用于发现冷门数据紧缺性，不是协议义务。

## Blockqs

查询能力：

- 脚本和输出项检索。
- 目标交易全部输出项。
- 完整交易数据。
- UTXO/UTCO 集或片段。
- 账号关联交易 ID 组。
- 小附件或大附件分片索引文件。

Blockqs 边界：

- 小于 10MB 附件可由 Blockqs 提供；该阈值协议地位未决。
- 完整区块和大附件由 Depots 提供。
- Blockqs 可通过 Depots 获取历史数据。
- Blockqs 返回的交易、输出项、状态数据必须可验证。

PoH 验证可能需要 Blockqs 返回：

- 铸凭交易头。
- 首领输入。
- 剩余输入段 Hash。
- 输出段根。
- 来源输出项。
- 哈希证明路径。

## stun2p

- 提供 NAT 探测。
- 提供 UDP 打洞协助。
- 服务于通用 P2P 网络连接。
- 不参与区块、交易、PoH 或脚本验证。
- 服务质量可被铸造者观察并用于公共服务兑奖确认。

## 服务奖励地址

- 服务节点可向应用节点声明奖励地址。
- 地址可以是 32 bytes `AddressHash` 或人类可读地址；文本编码由地址 Proposal 固定。
- Coinbase 中公共服务奖励引用服务地址。
- 服务地址是否必须签名绑定服务身份仍未决。

## 未决问题

- Blockqs 查询接口是否标准化到字节级。
- 小附件 `10MB` 阈值是否为协议常量、服务建议或默认配置。
- 附件分片大小和分片索引文件格式。
- 服务地址是否必须签名绑定服务身份。
- 服务观察记录是否可导出或共享。
- Blockqs 提供状态数据时如何分块、证明和绑定状态根。
- 未来新增公共服务类型如何进入奖励系统。

## 对后续 Plan 的影响

- 所有公共服务接口必须以“返回可验证数据”为核心设计。
- 实现不得把 Blockqs 或 Depots 当作信任源。
- 公共服务奖励实现必须引用激励 Proposal，而不是服务接口自行计算收益。
