# Blockchain Core（区块链核心）

## 来源构想

- `docs/conception/blockchain.md`：定义极简核心、区块头链、`CheckRoot`、年块、轻量存储、初始主链验证和主链/分叉标识。
- `docs/conception/附.交易.md`：定义交易树、Coinbase 位置、交易数据检索和签名消息中的链识别信息。
- `docs/proposal/01.Types-And-Encoding.md`：提供规范化编码、字段顺序、整数和固定长度字节规则。
- `docs/proposal/02.Cryptography-And-Hashing.md`：提供 `BlockID`、`CheckRoot`、交易树和签名相关 Hash 边界。
- `docs/proposal/03.Identifiers-And-Constants.md`：提供 `BlockID`、`CheckRoot`、`BlockInterval`、`BlocksPerYear` 等常量。
- `docs/proposal/04.Hash-Trees.md`：提供区块交易集合树、状态指纹树和证明路径边界。

## 目标

本提案定义区块链核心层的技术规格。核心层只维护区块头链和可验证的链身份，不负责交易执行、状态转移、脚本执行、PoH 竞争或 P2P 网络。

核心层必须提供：

- 区块头结构和 `BlockID` 计算边界。
- 入块时的最小头链连续性验证。
- `CheckRoot` 的组合语义和验证责任边界。
- 年块机制和轻量区块头存储策略。
- 区块头数据访问、缺失数据恢复和初始主链验证接口边界。
- 手动主链切换时核心层应支持的状态替换边界。

## 非目标

- 不验证完整交易合法性。
- 不执行脚本。
- 不更新 UTXO/UTCO 状态。
- 不实现 PoH 择优池、铸凭哈希、分叉竞争或校验组协作。
- 不实现 `Blockqs`、`Depots`、`stun2p` 或节点发现网络。
- 不定义数据库 schema、RPC 协议或 Go 结构体实现。

## 区块头

区块头的逻辑字段为：

```text
BlockHeader {
    Version
    Height
    PrevBlock BlockID
    CheckRoot CheckRoot
    Stakes
    YearBlock BlockID
}
```

规格要求：

- `BlockID = SHA3-384(BlockHeaderCanonicalBytes)`，输出 48 bytes。
- 字段顺序必须固定为 `Version, Height, PrevBlock, CheckRoot, Stakes, YearBlock`，除非后续区块头版本 Proposal 明确升级。
- 固定宽度整数使用 big-endian；若字段使用 varint，必须等待 `01.Types-And-Encoding.md` 的 canonical varint 未决问题关闭。
- `Height` 表示区块高度，并代替普通区块时间戳。
- `PrevBlock` 必须等于前一区块头的 `BlockID`。
- `Stakes` 表示该区块交易消耗的币权总量，单位为聪时；完整计算由交易/状态/共识 Proposal 固定。
- `YearBlock` 表示年块衔接引用；年度边界和非年度边界的精确取值仍为未决问题。

创世区块的 `PrevBlock`、`YearBlock`、`Stakes` 和 `CheckRoot` 必须由创世配置固定，并进入创世区块测试向量。

## CheckRoot

`CheckRoot` 是区块交易集与状态集的承诺值：

```text
CheckRoot = SHA3-384(TransactionTreeRoot || UTXORoot || UTCORoot)
```

约束：

- `TransactionTreeRoot` 来源于区块交易集合树，通常为 32 bytes `TreeHash`。
- `UTXORoot` 与 `UTCORoot` 来源于处理完该区块后的状态指纹，通常为 48 bytes 摘要。
- 字段顺序暂按构想层记录为 `TransactionTreeRoot, UTXORoot, UTCORoot`。
- domain tag 是否进入协议 Hash 输入仍未固定；实现前必须与 `02.Cryptography-And-Hashing.md` 和 `04.Hash-Trees.md` 统一。
- 空交易树、单叶树、奇数叶和空状态根规则必须由对应交易/状态 Proposal 固定后才能生成最终测试向量。

## 入块验证

核心层的入块验证只检查头链最小合法性：

- 区块头必须能按规范化编码解码。
- 重新计算 `BlockID` 必须匹配外部声明值。
- 新区块高度必须为当前 tip 高度加一。
- `PrevBlock` 必须等于当前 tip 的 `BlockID`。
- 同高度重复提交不同区块时返回冲突错误，不自动替换。
- 显式重组或手动切换流程之外，不得隐式改写已接收主链。

核心层不得在入块验证中直接执行完整交易验证、脚本执行、PoH 胜出者验证或奖励分配验证。这些验证结果由外部校验组件提供，并通过 `CheckRoot`、铸造者签名和可验证数据交叉约束。

## 年块与轻量存储

基础常量：

```text
BlocksPerYear = 87661
```

年块用于降低长期头链保存成本：

- 年度边界按 `height % BlocksPerYear == 0` 判断。
- 年块必须维持从创世块开始的年度衔接。
- 轻量节点可省略某些年度内部完整区块头，但必须保留足以验证年度衔接的数据。
- 缺失区块头可从 `Blockqs` 或校验组获取，但所有返回数据必须通过区块头 Hash 与年块衔接验证。
- 从多个公共服务获取头数据时，核心层应能比对同高度、同 `BlockID` 的一致性。

## 数据访问边界

核心层应提供以下能力边界，具体 API 由 Plan 定义：

- 按高度查询区块头。
- 按 `BlockID` 查询区块头。
- 查询当前 tip。
- 查询年度边界和年度内可用头数据。
- 判断本地是否缺失某高度或某年度头数据。
- 接收外部恢复的区块头并验证其与当前头链的衔接。
- 暴露链识别信息：`Protocol-ID`、`Chain-ID`、`Genesis-ID`、可选 `Bound-ID`。

核心层不定义 `Blockqs` 网络协议，只定义自身需要哪些可验证数据。

## 初始主链验证

初始节点上线时应按以下数据建立信任：

- 客户端内置创世区块头。
- 从公共服务或校验组获取当前头链，可使用年块衔接压缩。
- 获取末端区块的 Coinbase 交易、交易树证明路径、UTXO/UTCO 指纹和铸造者对 `CheckRoot` 的签名。
- 可选获取当前 UTXO/UTCO 集快照和末端区块数据，进行逆推式严格验证。
- 建议末端验证窗口至少覆盖 29 个区块，以覆盖分叉竞争区间。
- 不同数据源返回冲突结果时，核心层应暴露冲突，不自行进行社会性主链裁决。

## 主链和分叉标识

链身份信息包括：

- `Protocol-ID`：区分 Evidcoin 协议版本，例如 `Evidcoin@V1`。
- `Chain-ID`：运行态，例如 `mainnet`。
- `Genesis-ID`：创世区块 `BlockID`。
- `Bound-ID`：可选主链绑定标识，构想为相对交易时间的 `-29` 区块 ID 前 20 bytes。

这些信息不进入普通区块头 Hash。它们用于交易签名消息前置混合、节点握手和分叉后的重放风险控制。具体编码由交易签名或链标识 Proposal 固定。

## 手动切换主链

长期网络分区导致分叉超过协议竞争窗口后，主链选择属于社会性决策。核心层应支持：

- 用户或上层组件显式选择目标分叉数据源。
- 从分叉点重新获取目标头链。
- 验证目标头链与创世区块、年块和区块头 Hash 衔接一致。
- 要求状态层从快照、回滚日志或外部可验证状态重建到目标 tip。
- 在切换前暴露当前链和目标链差异，避免静默切换。

核心层不得自动选择超过竞争窗口的长期分叉。

## 未决问题

- `Version`、`Height`、`Stakes` 的精确字段宽度。
- `YearBlock` 在创世块、年度边界块和非年度边界块中的取值。
- `CheckRoot` 是否包含 domain tag、版本、高度或其它外层字段。
- `CheckRoot` 最终是否承诺区块后状态；本提案倾向使用区块后状态。
- 铸造者签名消息是只签 `CheckRoot`，还是签包含高度、前块、交易根和状态根的完整区块签名消息。
- 手动切换主链时状态回滚、快照替换和用户确认的具体接口。

## 对后续 Plan 的影响

- 区块核心 Plan 必须优先实现区块头规范化编码、`BlockID` 计算和最小入块验证。
- 不得在核心层引入交易验证、脚本 VM、PoH 择优池或公共服务网络实现。
- 初始主链验证和缺失头恢复应设计为接口边界，所有外部数据必须可验证。
- 任何涉及 `CheckRoot` 的实现必须等待交易树、状态指纹树和 domain tag 未决问题关闭，或以结构化占位和拒绝规则实现。
