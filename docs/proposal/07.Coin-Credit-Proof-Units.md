## 目标

本提案定义 Evidcoin 三种基础信元的输出 payload 规格：

- `Coin`：可分割、可合并、可花费的货币性信用单元。
- `Credit`：不可分割、可转移、可带附件和可选修改能力的凭证性信用单元。
- `Proof`：不可转让、不可作为输入源的存在性证明单元。

本提案还定义附件 ID、Mediator 和 Custom 输出的边界。

## 非目标

- 不定义交易头、输入 Hash、签名消息或 Coinbase 奖励。
- 不实现 UTXO/UTCO 状态转移。
- 不实现脚本 VM 或附件传输协议。
- 不固定地址文本编码、附件类型表、附件分片大小或最终 varint 算法。

## 通用输出规则

- 输出类型由交易输出配置低 4 bits 决定。
- 输出序位从 0 开始，并参与输出 Hash 或由输出树顺序承诺。
- 标准接收者使用 32 bytes `AddressHash`。
- 非标准接收者或空接收者必须由输出配置和脚本语义明确允许。
- 锁定脚本控制未来花费或转移；最大长度引用 `MaxLockScript`。
- 输出是否进入 UTXO/UTCO 由信元类型、销毁标记、有效期和转移次数共同决定。
- 进入状态集的输出必须有可规范化的 payload 摘要，以支持状态指纹树。

## Coin

Coin payload 逻辑字段：

```text
CoinOutput {
    Receiver AddressHash or null/custom bytes
    Amount
    Memo optional Bytes
    LockScript Bytes
}
```

规则：

- Coin 表达可拆分数值，最小单位为 `chx`。
- `Amount` 必须为非负整数；是否允许 0 金额输出仍为未决问题。
- 金额编码必须在交易/货币单位 Proposal 中固定，构想层同时出现变长整数和 `int64` 示意。
- 未花费 Coin 输出进入 UTXO。
- 被输入消费后从 UTXO 有效集合移除。
- 设置销毁标记的 Coin 输出不进入 UTXO，接收者可为 `null`。
- Coin 可包含短附言，建议最大 255 bytes。
- Coin 暂不支持附件 ID；若未来支持，必须修订本 Proposal。
- 交易费由 Coin 输入总额减输出总额得到，精确计算由交易/激励 Proposal 固定。

## Credit

Credit payload 逻辑字段：

```text
CreditOutput {
    Receiver AddressHash or custom bytes
    Creator or CreatorRef
    Config uint16
    Title Bytes
    Description Bytes
    AttachmentID optional
    LockScript Bytes
}
```

`Config` 位含义：

```text
bit 15: 新建标记
bit 14: 可否修改
bit 13: 是否修改
bit 12: Reserved
bit 11: 转移计次存在
bit 10: 高度截止存在
low 10 bits: 描述内容长度
```

规则：

- Credit 不可拆分。
- Credit 可转移；转移消费旧 UTCO，并创建新 Credit 输出。
- 任意人可创建 Credit，但仍需 Coin 输入支付上链成本。
- 新建 Credit 无 Credit 输入；转移 Credit 必须引用旧 UTCO。
- 未转移且仍有效的 Credit 输出进入 UTCO。
- 销毁、终止转移能力、转移次数归零或到期后，Credit 不再进入 UTCO。
- 创建者、标题和附件 ID 原则上不可变。
- 不可修改 Credit 只允许接收者和锁定脚本变化。
- 可修改 Credit 允许描述变化，并应设置“是否修改”标记。
- 可修改性可以从可修改降为不可修改，不得反向恢复。
- 转移次数和高度截止可以同时存在，以先达到者为准。
- 无期限 Credit 至少应 11 年激活一次，否则失效。
- 高度截止不应超过相对 100 年。

## Proof

Proof payload 逻辑字段：

```text
ProofOutput {
    Creator Bytes
    Title Bytes
    Content Bytes
    AttachmentID optional
    IdentifyScript Bytes
}
```

规则：

- Proof 表达存在性证明，不可转让。
- Proof 不可作为交易输入。
- Proof 不进入 UTXO 或 UTCO。
- Proof 可被脚本、Blockqs 或第三方客户端检索与识别。
- `Content` 最大候选值为 4KB，精确边界需与字段长度和交易大小共同确认。
- `IdentifyScript` 用于程序化识别，不是未来花费脚本。
- Proof 的链上价值在于被交易树和区块头约束，而不是被状态集维护。

## AttachmentID

附件 ID 逻辑结构：

```text
AttachmentID {
    Length       uint8
    Type         uint16
    Fingerprint  AttachmentHash
    ShardCount   uint16
    ShardRoot    optional TreeHash
    Size         unsigned integer
}
```

规则：

- `Length` 小于 256 bytes，表示附件 ID 总长。
- `Type` 表示附件大类和小类；类型表未固定。
- `Fingerprint` 使用 `SHA3-512`，长度 64 bytes。
- `ShardCount < 65536`。
- `ShardCount = 0` 表示无分片，`ShardRoot` 字段取消。
- `ShardCount = 1` 表示无树结构但片组哈希正常计算，具体单叶规则未决。
- `ShardCount > 1` 时使用含序附件片组树，叶子前置 2 bytes 序号。
- `Size` 是用户声明的附件大小，链上不验证附件真实大小。
- 附件传输、存储和分片索引由 Depots 或外部服务承担。

## Mediator 和 Custom

Mediator 输出：

- 只能被脚本跳转或嵌入引用。
- 不能作为交易输入源。
- 可用于监管、监听、链间交互或第三方中间件逻辑。

Custom 输出：

- 使用配置字节 bit 7 标记。
- 可携带最长 127 bytes 私有类 ID。
- 可包含私有扩展脚本。
- 公共客户端无法理解其私有语义时，不应把它作为公共输入源。
- Custom 私有逻辑不得影响公共共识验证结果。

## 未决问题

- Coin 与 `chx` 的换算关系、金额编码宽度和是否允许 0 金额。
- Coin 是否未来支持附件。
- Credit `CreatorRef` 的具体格式。
- Credit 描述长度低 10 bits 与可变字节串编码的关系。
- Credit 截止高度字段编码、11 年与 100 年的精确高度换算。
- Credit 转给自己激活时是否可修改描述、是否重置活跃高度。
- Proof 内容长度字段高位是否必须为 0。
- Proof `IdentifyScript` 的最大长度。
- 附件类型表、分片大小、分片数量为 1 时的片组哈希。
- Mediator 是否需要独立索引。
- Custom 输出是否完全禁止进入公共状态集。

## 对后续 Plan 的影响

- Coin/Credit/Proof 实现必须和交易输出 envelope 分离，避免把类型字段重复编码。
- 状态层必须只把 Coin 纳入 UTXO、把可转移 Credit 纳入 UTCO，Proof 不进入状态集。
- 附件验证必须区分整体 `AttachmentHash` 和片组 `TreeHash`。
- Credit 转移验证必须能比较不可变字段和状态字段，不能只验证签名。
