# Transaction Design Proposal（交易设计提案）

> 本提案定义 Evidcoin 区块链的交易模型，涵盖结构、哈希、UTXO/UTCO 指纹、Coinbase 交易、存储与大小/费用约束。


## 1. Overview（概述）

交易是 Evidcoin 信用转移的基本载体。不同于比特币仅将交易历史作为存储，Evidcoin 的交易历史必须支持下游需求：脚本应用识别、中介脚本引用与第三方开发。

设计重点：

- **Verifiability**：使用哈希树对输入与输出进行独立验证。
- **Retrievability**：按年度与 TxID 的结构化索引，便于高效检索。
- **Signature separation**：签名不参与 TxID 计算，支持灵活的授权模型。
- **Compactness**：单笔交易最大尺寸为 64 KB，不含解锁部分。

## 2. Transaction Header（交易头）

交易头是完整交易的紧凑抽象，类似区块头对区块的抽象。

```go
// TxHeader 交易头结构。
// 交易头的哈希即为交易 ID（TxID）。
type TxHeader struct {
    Version   int       // 版本号
    Timestamp int64     // 交易时间戳（Unix 毫秒）
    HashInputs Hash384  // 输入项根哈希
    HashOutputs Hash384 // 输出项根哈希
    ...                 // 其它可能需要的字段（如果有）
}
```

### 2.1 TxID Computation（交易 ID 计算）

```
TxID = SHA3-384( TxHeader )
```

关键规则：**Signatures are excluded from TxID computation.** TxID 仅由交易头派生，而交易头包含的 `HashInputs` 和 `HashOutputs` 仅由输入与输出计算而来。该分离防止交易可塑性，并允许签名独立验证。


## 3. Transaction Body（交易体）

交易体即为交易数据，由两部分组成：**inputs**（信用来源）与 **outputs**（信用去向）。

### 3.1 Inputs（输入项）

每个输入引用已确认交易中的未花费输出（UTXO）或未转移的凭信输出（UTCO）。

> **Rule**：未确认交易的输出不能作为输入。这保证了组队校验下的交易独立性。

#### 3.1.1 Input Structure（输入项结构）

一个输入由 3–4 个字段组成：

| 字段 | 大小 | 描述 |
|-------|------|-------------|
| Year | varint | 被引用交易的年度（粗粒度时间约束） |
| TxIDRef | 48 字节（首领）/ 20 字节（其余） | 交易 ID 引用 |
| OutIndex | varint | 被引用交易中的输出序位 |
| TransferIndex | varint（可选） | 用于凭信转移：当前输入项的目标输出位置 |

```go
// LeadInput 首领输入，使用完整的交易 ID 引用。
type LeadInput struct {
    Year          int       // 交易所在年度
    TxID          Hash384   // 完整交易 ID（48 字节）
    OutIndex      int       // 输出序位
}

// RestInput 非首领输入，使用截断的交易 ID 引用以节省空间。
type RestInput struct {
    Year          int       // 交易所在年度
    TxIDPart      [20]byte  // 交易 ID 前 20 字节
    OutIndex      int       // 输出序位
    TransferIndex int       // 凭信转出序位（可选）
}
```

**Design rationale**：

- **lead input** 使用完整 48 字节 TxID，便于铸造竞争者（铸凭交易）所有权检索和验证。
- **remaining inputs** 使用 20 字节截断引用以节省空间。由于所有输入都会针对当前 UTXO 集或 UTCO 集解析（由 UTXO/UTCO 指纹保证安全），单年度 20 字节前缀已足够，实际碰撞可忽略（且会被排除）。

#### 3.1.2 Lead Input Constraints（首领输入约束）

为抗洪泛保护（参见 “Team-based Verification” 设计）：

- 首领输入必须为 **coin** 类型，且在所有 coin 输入中具有 **largest coin-age**。
- 若交易最终未通过完整校验，会被视为攻击行为，首领输入将进入 **blacklist**，冻结 24 小时。

#### 3.1.3 Input Hash Tree（输入哈希树）

输入集被哈希成一个两层树状结构，其中首领输入被单独分离出来：

```
InputHash = SHA3-384( LeadHash || RestHash )

LeadHash  = SHA3-384( Year || TxID[48] || OutIndex )

RestHash  = SHA3-384( RestInput_1 || RestInput_2 || ... || RestInput_N )
```

分离首领输入使得节点对铸造竞争者的验证（择优池优选成员期间）更高效——只需要提供剩余输入序列的总哈希、首领输入和交易头信息，即可验证首领输入是否属于该交易。

#### 3.1.4 Input Validation（输入验证）

每个输入都通过脚本执行进行验证：

```
UnlockScript + LockScript → Execute → Success = Valid
```

解锁脚本提供所有权证明；锁定脚本（来自被引用输出）定义花费条件。

> **注：**
> 如果有签名数据，通过另外的方式载入系统，由 `SYS_CHKPASS` 指令使用。


### 3.2 Outputs（输出项）

每个输出定义信用去向——接收者获得币金、凭信或存证。

#### 3.2.1 Output Structure（输出项结构）

```go
// Output 交易输出项。
type Output struct {
    Serial     int        // 输出序位（从 0 开始）
    Config     byte       // 配置字节（类型 + 标记位）
    Amount     int64      // 币金数量（按最小单位 chx，同 sat）
    Address    [32]byte   // 接收地址（公钥哈希，SHA3-256(BLAKE2b-512)，32 字节）
    LockScript []byte     // 锁定脚本（最大 1024 字节）
    // 以下字段按信元类型选择性存在
    Memo       []byte     // 附言（币金，最大 255 字节）
    Creator    []byte     // 创建者（凭信/存证）
    Title      []byte     // 标题（凭信/存证，最大 255 字节）
    Desc       []byte     // 描述（凭信/存证）
    AttachID   []byte     // 附件 ID（可选）
    CredConfig uint16     // 凭信配置（2 字节）
    ProofConfig uint16    // 存证配置（2 字节）
}
```

#### 3.2.2 Output Config Byte（输出配置字节）

用1字节配置编码类型和标志：

```graph
Bit 7: Custom class    — When set, remaining 7 bits encode class ID length.
Bit 6: Has attachment  — Declares attachment presence (repurposed if Bit 7 set).
Bit 5: Burn flag       — Marks coin destruction or credential termination.
Bit 4: (Reserved)
Bits [3:0]: Type value:
    0 = Reserved
    1 = Coin         — Value transfer, can be UTXO input source.
    2 = Credential   — Credit certificate, transferable, can be UTCO input source.
    3 = Attestation  — Existence proof, NOT transferable, cannot be input source.
    4 = Intermediary — Receives cross-transaction script jumps (GOTO/EMBED).
```

#### 3.2.3 Address Format（地址格式）

地址由公钥哈希派生：

- **Hash algorithm**：`SHA3-256( BLAKE2b-512 )`，生成 32 字节公钥哈希。
- **Single-signature**：`Address = SHA3-256( BLAKE2b-512( pubKey ) )`
- **Multi-signature**：`Address = SHA3-256( BLAKE2b-512( <m/N> || sort(PKH_1 || PKH_2 || ... || PKH_N) ) )`
    - `m/N` 前缀为 2 字节（各 1 字节），支持最高 255-of-255。
    - 多签与单签地址外观不可区分（隐私）。

`null` 地址通常用于销毁目的地——但需配合输出项的销毁标记置位。非标准地址（非公钥哈希格式）通常意味着自定义验证逻辑（不使用 `SYS_CHKPASS` 指令）。

**Address encoding**（人类可读形式）：

1. 在公钥哈希前添加识别前缀。
2. 对带前缀数据哈希，取最后 4 字节作为 **checksum**。
3. 将校验码拼接到公钥哈希（不含前缀），编码为文本（Base58）。
4. 再加识别前缀得到最终地址字符串。

#### 3.2.4 Output Hash Tree（输出哈希树）

输出形成二元类 Merkle 哈希树：

```
OutputHash = BLAKE3-256( BinaryTree( Output_0, Output_1, ..., Output_N ) )
```

每个输出作为叶子节点采用 SHA3-384 单独哈希（48 字节）。树内部节点采用 BLAKE3-256 计算（32 字节）。二元树结构可在不处理全集的情况下，对任一输出进行独立验证与位置检索。

> **注：**
> 二元哈希校验树逻辑上类似 Merkle，但每一层级的节点为链表或数组结构。

#### 3.2.5 Special Output Types（特殊输出类型）

**Burn outputs**：当燃烧标记（Bit 5）置位时，输出不再进入 UTXO/UTCO 集。地址可为 `null`，锁定脚本任意（无限制）。

**Intermediary scripts**：类型 4 输出作为跨交易跳转目标。它们不能作为 UTXO/UTCO 输入源——仅能通过脚本指令（`GOTO`、`EMBED`）引用。连续跨交易跳转会被限频以保证安全。

**Custom classes**：Bit 7 置位的类型。包含专用客户端的私有扩展指令。类 ID 最多 127 字节用于客户端识别。不能作为 UTXO/UTCO 输入源。


## 4. Signature Model（签名模型）

签名证明对输入项的所有权，但与决定 TxID 的交易数据 **structurally separated**。

### 4.1 Single Signature（单签名）

标准情形：由一个或多个简单私钥签名交易（视输入项的数量而定），足以满足大多数个人转账或凭信转移。

> **注意：**<br>
> 输入项的单一私钥签名交易，只是证明该输入项合法。<br>
> 一笔交易里可以包含多笔输入，每笔输入项都需要其私钥签名交易，以证明合法性。

### 4.2 Multi-Signature（多重签名）

还存在另一种公钥哈希，它由多个私钥的公钥组合而成。其签名方案：M-of-N scheme where M out of N key holders must sign.

**Unlock data for multi-sig verification**：

| 组件 | 描述 |
|-----------|-------------|
| Signature set | 实际签名集（M 个私钥的签名） |
| Public key set | 对应的 M 个公钥 |
| Complement set | 未签名参与者的公钥哈希集（N-M 个） |

**Verification process**：

1. 由公钥集与补全集规模计算 `m/N`。
2. 对公钥集中的每个公钥进行哈希。
3. 将公钥哈希与补全哈希合并、排序、前置 `m/N` 并计算复合哈希。
4. 与输出地址哈希对比——匹配即身份有效。
5. 逐一验证签名集中的签名。

### 4.3 Signature Authorization Types（签名授权种类）

由**1字节**配置控制签名覆盖范围：

```
Bit 7: SIGIN_ALL     — All inputs
Bit 6: SIGIN_SELF    — Current input only
Bit 5: SIGOUT_ALL    — All outputs (Coin, Credit, Proof)
Bit 4: SIGOUT_SELF   — Output at same index as current input
Bit 3: SIGOUTPUT     — Full output entry (receiver + content + script)
Bit 2: SIGSCRIPT     — Output's lock script only
Bit 1: SIGCONTENT    — Output's content only (amount/memo for Coin; creator/config/title/desc/attachID for Credit/Proof)
Bit 0: SIGRECEIVER   — Output's receiver only
```

**Usage rules**：

- **Independent bits**（7、6）：可单独使用。例如 `SIGIN_ALL` 覆盖所有输入。
- **Primary bits**（5、4）：必须与辅项（3–0）组合。例如 `SIGOUT_ALL | SIGRECEIVER`。
- **Auxiliary bits**（3–0）：必须配合主项。

**Most common configuration**：`SIGIN_ALL | SIGOUT_ALL | SIGOUTPUT` ——覆盖所有输入与全部完整输出条目。


## 5. Coinbase Transaction（铸币交易）

Coinbase 交易是每个区块的第一笔交易（索引 `[0]`）。它没有输入——从无到有创建新币。

### 5.1 Coinbase Structure（铸币交易结构）

```go
// CoinbaseTx 铸币交易特殊结构。
type CoinbaseTx struct {
    Header      TxHeader   // 标准交易头
    BlockHeight int        // 区块高度
    MeritProof  []byte     // 择优凭证（铸造者证明）
    TotalReward int64      // 收益总额
    SelfData    []byte     // 自由数据（最大 255 字节）
    Outputs     []Output   // 收益分配输出
}
```

> **注：**
> Coinbase 交易的时间戳就是铸造者实际时间，与高度无严格相关性。

### 5.2 Revenue Composition（收益构成）

```
TotalReward = BlockReward + TransactionFees + LotteryRetention
```

- **BlockReward**：每区块铸造的新币（递减计划）。
- **TransactionFees**：总手续费仅 **50%** 计入 Coinbase 收入——其余 50% 永久销毁，形成通缩压力。销毁部分在 Coinbase 交易中以一笔特殊输出体现：设置销毁标记（Bit 5），接收者为 `null`。
- **LotteryRetention**：来自未能完整兑奖（确认数不足）的截留部分。截留发生在铸造者花费所铸新币时——其锁定脚本中的 `SYS_AWARD` 指令会检查后续 48 个区块的兑奖槽确认数，确认数不足 2 个则无法全额花费，截留部分计入第 49 个区块的铸造者收益。

### 5.3 Revenue Distribution（收益分成）

Coinbase 的收益将作为多个输出，奖励分配给多方。每个 Coinbase 输出使用专用的 **1 字节配置**（与普通交易输出不同）：

```
Bit 7: (Reserved)
Bit 6: (Reserved)
Bit 5: (Reserved)
Bit 4: (Reserved)
Bits [3:0]: Reward target（奖励/分成目标）:
    0 = Reserved
    1 = Credential minter  — 铸凭者（10%）
    2 = Check team         — 校验组（40%）
    3 = Blockqs            — 区块查询服务（20%）
    4 = Depots             — 数据驿站服务（20%）
    5 = STUN services      — NAT 穿透服务（10%）
```

| 接收方 | 配置值 | 比例 | 描述 |
|-----------|--------|------|-------------|
| Credential minter | 1 | 10% | 铸凭交易提供者奖励 |
| Check team (minter) | 2 | 40% | 区块铸造校验组奖励 |
| Blockqs | 3 | 20% | 区块查询服务节点 |
| Depots | 4 | 20% | 数据驿站服务节点 |
| STUN services | 5 | 10% | NAT 穿透服务提供者 |

> **注：**
> 分配比例按除法逻辑，不能整除的余数留给最终分配者。
> 铸造者花费所铸新币时，锁定脚本中的 `SYS_AWARD` 指令根据后续 48 个区块 Coinbase 兑奖槽中的确认标记，计算可提取金额。若确认数不足 2 个，则无法全额花费——未兑现部分截留给第 49 个区块的铸造者。

### 5.4 Minting Process（铸造流程）

1. **Application**：合格铸造者（择优池成员）向管理层提交择优凭证。
2. **Information exchange**：管理层返回手续费总额、团队奖励地址、服务地址、铸币金额以及兑奖截留等必要信息。
3. **Construction**：铸造者按分配规则构造 Coinbase 交易并提交。
4. **Verification**：管理层验证 Coinbase、打包区块，并返回哈希树验证路径（含 TreeRoot 与 UTXO/UTCO 指纹）。
5. **Signing**：铸造者通过哈希路径确认其 Coinbase 被包含后签名 CheckRoot 并提交签名。
6. **Publication**：管理层构造区块头并广播。

### 5.5 Low-Revenue Principle（低收益原则）

若同一铸造者在同一高度签名多个区块，**only the block with the lowest revenue is valid**。该规则防止多签并鼓励及时出块。


## 6. UTXO/UTCO Model and Fingerprint（UTXO/UTCO 模型与指纹）

### 6.1 UTXO/UTCO Set Definition（UTXO/UTCO 集定义）

UTXO 集由已确认交易的所有未花费输出组成，UTCO 集由已确认交易的未转移凭信输出组成。它们是各自独立但结构相似的集合，都由前一区块决定。

**Exclusions from UTXO/UTCO**：

- 未确认交易的输出（不支持交易池级别的链式花费）。
- 置位燃烧标记的输出。
- 存证输出（type 3，不可转移）。
- 介管脚本输出（type 4）。
- 自定义类输出（Bit 7 置位）。
- 过期凭信（达到转移次数或时限——适用 UTCO 集）。

### 6.2 UTXO/UTCO Fingerprint — 4-Level Hierarchical Hash（UTXO/UTCO 指纹 — 4 级层次哈希）

UTXO/UTCO 指纹是具有 **4 hierarchical levels** 的宽成员哈希校验树，旨在最小化输出变化时的重算成本。

> **注：**
> 下面提及 UTXO 的地方也基本适用于 UTCO，除非特别说明。

```graph
Level 0 (Root):    UTXOFingerprint = BLAKE3-256( Year_1 || Year_2 || ... || Year_N )
Level 1 (Year):    YearHash_y      = BLAKE3-256( Tx8Hash_1 || Tx8Hash_2 || ... )
Level 2 (Tx8):     Tx8Hash_t       = BLAKE3-256( Tx13Hash_1 || Tx13Hash_2 || ... )
Level 3 (Tx13):    Tx13Hash_t      = BLAKE3-256( Tx18Hash_1 || Tx18Hash_2 || ... )
Level 4 (Tx18):    Tx18Hash_t      = BLAKE3-256( infoHash_1 || infoHash_2 || ... )

infoHash = SHA3-384( TxID || BitBytes || FlagOutputs... )   // 叶子节点使用 SHA3-384
```

```go
// UTXOFingerprint 4 级层次哈希结构。
//
// Level 1: 年度级 — 按交易时间戳的 UTC 年度分组
// Level 2: 交易级 — 同一年度内按交易 ID 的第8字节值分组
// Level 3: 交易级 — 继上层分组内按交易 ID 的第13字节值分子组
// Level 4: 交易级 — 继上一子组内按交易 ID 的第18字节值分组
// 末端叶子节点：
// - 在上面4层分组后，海量交易ID已经被分散到不同的分支中。
// - 叶子节点的结构：TxID + 标记位字节数 + 标记位字节序列。
type UTXOFingerprint struct {
    Root      Hash256                // 指纹根哈希（BLAKE3-256，32 字节）
    YearNodes map[int]*YearNode      // 年度 → 年度节点
}

type YearNode struct {
    Year     int                    // UTC 年度
    Hash     Hash256                // 年度哈希（BLAKE3-256，32 字节）
    Tx8Nodes [256]*Tx8Node          // TxID 第8字节 → 子分组序列
}

type Tx8Node struct {
    Index     byte                  // 分组序位
    Hash      Hash256               // 分组哈希（BLAKE3-256，32 字节）
    Tx13Nodes [256]*Tx13Node        // TxID 第13字节 → 子分组序列
}

type Tx13Node struct {
    Index     byte                  // 分组序位
    Hash      Hash256               // 分组哈希（BLAKE3-256，32 字节）
    Tx18Nodes [256]*Tx18Node        // TxID 第18字节 → 子分组序列
}

type Tx18Node struct {
    Index      byte                 // 分组序位
    Hash       Hash256              // 分组哈希（BLAKE3-256，32 字节）
    InfoHashes []Hash384            // 叶子节点列表（有序）：每个交易对应一个 InfoHash（SHA3-384，48 字节）
}

InfoHash = SHA3-384( TxID || BitBytes || FlagOutputs... )
```

**Design benefits**：

- **Incremental update**：UTXO 被花费时，仅需重算受影响分支（output → transaction → year → root）。
- **Year-based partitioning**：顶层按交易时间戳年度（UTC）分组，提供自然分片且粒度适中。
- **Multi-path history protection**：配合区块 ID 链，UTXO/UTCO 指纹提供额外的两条独立的历史完整性证明。
- **Fast node bootstrap**：新节点可通过最新区块头中的指纹验证当前 UTXO/UTCO 集，无需重放全历史。

### 6.3 CheckRoot（校验根）

CheckRoot 将交易哈希树与 UTXO/UTCO 指纹链接为区块头中的单一可验证值：

```
CheckRoot = SHA3-384( TxTreeRoot || UTXOFingerprint || UTCOFingerprint )
```

Where:
- `TxTreeRoot` is the Merkle-like binary hash tree root of all transaction IDs in the block (leaves are TxIDs with 3-byte sequence prefix).
- `UTXOFingerprint` is the root hash of the 4-level UTXO fingerprint structure.
- `UTCOFingerprint` is the root hash of the 4-level UTCO fingerprint structure.

管理层构造包含 `CheckRoot` 的区块头，铸造者对 `CheckRoot` 进行签名。这形成了 **information separation** 机制：铸造者无需知道完整区块的交易组成，仅通过哈希路径即可验证其 Coinbase 是否被包含。

由此，铸造者的验证工作量与区块内交易的数量无关，极大降低了对铸造者计算资源的需求，因此可以广纳参与者。

### 6.4 UTXO Set Transition（UTXO 集转换）

For block at height `H`:

```graph
UTXO_current(H)   = The UTXO set for block H (before applying block H's transactions)
UTXO_result(H)    = UTXO_current(H) − spent_by(H) + new_outputs(H) + coinbase(H)
UTXO_current(H+1) = UTXO_result(H)
```

区块 `H` 的 UTXO 指纹由 `UTXO_current(H)` 计算——即在应用区块 H 交易之前。这提供充分的计算时间，并形成可逆验证的链式约束。

**Reverse verification**：给定 `UTXO_current(H+1)` 与区块 `H` 的交易，可反推 `UTXO_current(H)` 并与区块 H 的指纹比对。该过程可向历史任意点迭代。

> **注：**
> UTCO 集的转换逻辑与 UTXO 类似，但针对的是凭信的转移次数与过期规则等因素。


## 7. Transaction Storage and Retrieval（交易存储与检索）

完整交易的存储由外部第三方公共服务提供，不由区块链核心验证的逻辑负责。

### 7.1 Storage Architecture（存储架构）

| 方面 | 服务 | 索引 | 目的 |
|--------|---------|-------|---------|
| 数据 | Blockqs | TxYear + TxID | 为用户提供完整交易数据 |
| 校验 | Blockqs | BlockYear + BlockHeight | 区块内 TxID 列表与哈希树验证路径 |
| 附件 | Depots | AttachID | 通过 P2P 文件共享的附件数据 |

> **注：**
> **TxYear**（交易年度）按交易时间戳计算，**BlockYear**（区块年度）按区块时间戳计算。两者可能不一致，因为区块可以收录有效期内较早的交易。

### 7.2 Query and Verification Flow（查询与验证流程）

给定 `TxID` 与时间戳（或年度）：

1. **Locate**：通过 year + TxID 查询 Blockqs，获取完整交易数据与元信息。
2. **Position**：元信息包含交易所在区块高度与区块内位置索引。
3. **Verify**：从 Blockqs 获取哈希树验证路径（含 UTXO/UTCO 双指纹），并结合用户本地区块头链进行验证：
    - 由交易数据重算 TxID。
    - 通过验证路径确认 TxID 被包含在区块交易哈希树中。
    - 确认树根与区块头中的 `CheckRoot` 一致。

用户的 **local block header chain** 是所有交易验证的信任锚。

### 7.3 Attachment Storage（附件存储）

附件存储在外部第三方公共服务（Depots）网络，通过 P2P 传输。

**Attachment ID structure** (variable length, < 256 bytes total):

```go
// AttachmentID 附件标识结构。
// 注意：ShardTreeHash 字段的存在取决于 ShardCount 值（见下方规则）。
type AttachmentID struct {
    TotalLen      byte       // ID 总长
    MajorType     byte       // 附件大类
    MinorType     byte       // 附件小类
    FingerprintLv byte       // 指纹强度等级（0=16B, 1=20B, ... 12=64B）
    Fingerprint   []byte     // BLAKE3 哈希指纹（16–64 字节）
    ShardCount    uint16     // 分片数量（最大 65535）
    ShardTreeHash [48]byte   // 片组哈希树根（SHA3-384）——条件存在
    DataSize      int        // 附件大小（变长整数，单位：字节）
}
```

**Shard count rules（分片数量规则）**：

| ShardCount | ShardTreeHash 字段 | 说明 |
|------------|-------------------|------|
| 0 | **不存在**（字段取消） | 无分片，附件为单体数据 |
| 1 | 存在，值为附件数据本身的 SHA3-384 哈希 | 逻辑上无分片，但提供数据哈希 |
| > 1 | 存在，值为含序号前缀的叶子节点哈希树根 | 正常分片，叶子节点为 2 字节序号 + 48 字节分片哈希 |

**Shard verification（分片验证）**：在部分数据检索时，每个分片可通过含 **2 字节序号前缀**的分片哈希（共 50 字节叶子节点）与至分片树根的 Merkle 路径独立验证。序号由验证者前置，源节点无法伪造。


## 8. Transaction Size and Fee Constraints（交易大小与费用约束）

### 8.1 Size Limits（大小限制）

| 常量 | 值 | 描述 |
|----------|-------|-------------|
| MaxTxSize | 65535 bytes | 单笔交易最大尺寸（< 65536） |
| MaxLockScript | 1024 bytes | 锁定脚本最大长度 |
| MaxUnlockScript | 4096 bytes | 解锁脚本最大长度 |
| MaxMemo | 255 bytes | 币金附言最大长度 |
| MaxTitle | 255 bytes | 凭信/存证标题最大长度 |
| MaxCredDesc | 1023 bytes | 凭信描述最大长度（10 位长度） |
| MaxAttestContent | 4095 bytes | 存证内容最大长度（12 位长度） |
| MaxSelfData | 255 bytes | Coinbase 自由数据最大长度 |

### 8.2 Fee Calculation（费用计算）

交易手续费采用隐式计算：

```
Fee = Sum(input coin values) − Sum(output coin values)
```

手续费必须非负。区块内收取的总手续费：

- **50% is burned** ——永久移除流通，形成通缩压力。
- **50% is distributed** ——作为铸造团队与公共服务的收入进入 Coinbase 交易。

### 8.3 Transaction Priority（交易优先级）

区块收录遵循以下优先级规则：

1. **Higher coin-age burn** ——销毁更多币龄的交易优先。
2. **Higher fees** ——次级裁决。
3. **Premature destroy credit** ——提前销毁凭信的交易优先。


## 9. Hash Tree in Block（区块内的哈希校验树）

区块内所有交易组织为类 Merkle 的二元哈希树：

```
TxTreeRoot = BinaryTree( [3B_seq || TxID_0], [3B_seq || TxID_1], ..., [3B_seq || TxID_N] )
```

- 每个叶子是带 3 字节序号前缀的 TxID。
- 序号前缀用于支持按位置随机提取并验证的逻辑。
- 树根与 UTXO/UTCO 双指纹共同参与 `CheckRoot` 计算。

该设计支持区块发布优化：在区块同步时，初始仅广播 16 字节 TxID 前缀。接收者与本地交易池比对，仅请求缺失交易。


## 10. Summary of Hash Relationships（哈希关系总结）

```graph
                                       ┌─────────────────────────┐
                                       │      Block Header       │
                                       │  ┌───────────────────┐  │
                                       │  │     CheckRoot     │  │
                                       │  └─────────┬─────────┘  │
                                       └────────────┼────────────┘
                                                    │
                          ┌─────────────────────────┼─────────────────────────┐
                          │                         │                         │
                    ┌─────┴──────┐        ┌─────────┴────────┐      ┌─────────┴────────┐
                    │ TxTreeRoot │        │ UTXO Fingerprint │      │ UTCO Fingerprint │
                    └─────┬──────┘        └─────────┬────────┘      └─────────┬────────┘
                          │                         │                         │
                 ┌───────┬┴───────┐       ┌─────────┴─────────┐     ┌─────────┴─────────┐
                  │       │        │       │  4-Level:         │     │  4-Level:         │
               TxID_0   TxID_1    ...      │  Year ->          │     │  Year ->          │
                  │                        │  Tx8 ->           │     │  Tx8 ->           │
                  │                        │  Tx13 ->          │     │  Tx13 ->          │
         ┌────────┴────────┐               │  Tx18 -> infoHash │     │  Tx18 -> infoHash │
        │     TxHeader    │               └───────────────────┘     └───────────────────┘
        │ ┌─────────────┐ │
        │ │ HashInputs  │ │
        │ │ HashOutputs │ │
        │ └─────┬───────┘ │
        └───────┼─────────┘
                │
        ┌───────┴────────────┐
        │                    │
    InputHash            OutputHash
        │                    │
   ┌────┴──────┐         BinaryTree
   │           │         (outputs)
LeadHash    RestHash
```

**Key invariants**：

- `TxID = SHA3-384( TxHeader )` ——签名不参与。
- `HashInputs = SHA3-384( LeadHash || RestHash )` ——输入完整性。
- `HashOutputs = BLAKE3-256( BinaryTree( Output_0, Output_1, ... ) )` ——输出完整性（叶子 SHA3-384）。
- `CheckRoot = SHA3-384( TxTreeRoot || UTXO Fingerprint || UTCO Fingerprint)` ——区块完整性。
- 交易 ID、输入/输出根、CheckRoot 均采用 **SHA3-384**（48 字节）。
- 树结构内部节点采用 **BLAKE3-256**（32 字节），末端叶子节点采用 **SHA3-384**（48 字节）。
- UTXO/UTCO 指纹基于应用当前区块交易 **before** 的 UTXO/UTCO 集，叶子 infoHash 采用 SHA3-384，内部节点采用 BLAKE3-256。
- 附件指纹（对数据本体）采用 BLAKE3 可变长度哈希算法，以平衡速度与安全性需求。
- 附件的分片哈希树采用 SHA3-384，以提供高效且安全的分片验证。
- 公钥哈希采用 SHA3-384，以平衡空间占用和安全强度。
