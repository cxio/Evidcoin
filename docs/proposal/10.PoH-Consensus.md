## 目标

本提案定义 PoH 共识的核心规格：铸造资格、铸凭哈希、择优池、同步池和候选者验证边界。PoH 以历史交易 ID 和铸造者签名构造竞争凭证，值小者胜出。

## 非目标

- 不实现 P2P 广播、择优池缓存或网络重试。
- 不定义完整 Coinbase 交易结构和奖励拆分。
- 不实现交易查询服务。
- 不固定 ML-DSA-65 具体库。
- 不把初始部署策略、百日扩张或运营方案写成代码实现。

## 术语

- `Minting Transaction`：铸凭交易，有资格参与铸造评选的历史交易。
- `MintHash`：铸凭哈希，64 bytes，按字节序从小到大比较。
- `Reference Block`：评参区块，正常阶段为当前待铸高度向前 9 个区块。
- `Stake Source Block`：币权因子区块，正常阶段为当前待铸高度向前 27 个区块。
- `Eligibility Window`：铸凭交易有效窗口，正常阶段为相对当前高度 `28..80000`。
- `Best Pool`：择优池，按 `MintHash` 排序的候选集合。
- `Sync Pool`：授权节点签名广播的池快照。
- `Merge Pool`：接收多个同步池后本地合并得到的池。

## 铸凭交易有效性

正常阶段铸凭交易高度必须满足：

```text
28 <= currentHeight - txHeight <= 80000
```

构想伪代码等价为：

```text
h := currentHeight - txHeight
valid := h > 27 && h <= 80000
```

初段特殊规则：

- `currentHeight < 28` 时铸凭交易高度判断可宽松。
- `currentHeight < 9` 时评参区块使用创世块或创世锚点。
- 初段规则只用于链启动，不得泛化为正常阶段规则。

## 铸造者来源

- 铸造者取自铸凭交易首领输入来源输出项的接收者公钥地址。
- 首领输入必须为完整 48 bytes `TxID` 引用，且必须是 Coin 输入。
- 择优凭证中的公钥材料计算出的地址哈希必须等于来源输出接收者。
- 铸造者必须能用对应私钥签名 PoH 演算数据。
- 首领输入应满足最低币权销毁门槛 `4,000,000` 聪时；该数值是否可升级仍未决。

## 铸凭哈希

构想中的演算结构为：

```text
Mix = 0x517cc1b727220a95
Stakes = stakeDestroyedTotal of block(currentHeight - 27)
timeStamp = deterministic timestamp of currentHeight
X = Encode(timeStamp * Stakes * Mix)

hashData = Hash(mintTxID || referenceBlockMintHash || X)
signData = Sign(hashData)
MintHash = BLAKE3-512-XOF(signData, 64 bytes)
```

规范约束：

- `mintTxID` 为 48 bytes `TxID`。
- `referenceBlockMintHash` 为 64 bytes `MintHash`。
- `timeStamp` 应由创世时间和高度确定性推导，不能使用本机时间。
- `Stakes` 的单位、宽度和溢出行为必须由实现前确认。
- `X` 不得由实现语言自然整数随意转字节，必须使用规范化编码。
- 内层 `Hash(...)` 算法未固定，不能擅自选择最终算法。
- `signData` 是签名字节本身。
- 最终 `MintHash` 必须使用显式 64 bytes BLAKE3 XOF 输出。

## 排序与胜出

- `MintHash` 按 unsigned lexicographic byte order 比较。
- 值小者排序更优。
- 禁止按十六进制字符串、本地大整数或平台字节序比较。
- 完全相等时的 tie-breaker 未决；实现前必须固定。

## 择优凭证

逻辑字段：

```text
MintProof {
    txLocator
    minterPublicKey
    signData
    mintHash optional
}
```

验证流程：

1. 解析并定位铸凭交易。
2. 重算交易头和 `TxID`。
3. 验证交易在有效高度窗口内。
4. 根据首领输入定位来源输出项。
5. 验证来源输出项的交易证明路径。
6. 提取来源输出接收者地址哈希。
7. 规范化铸造者公钥材料并计算地址哈希。
8. 比对地址哈希。
9. 验证铸造者签名。
10. 重算 `MintHash`。
11. 验证币权门槛和评参区块一致性。

所有外部数据可由 `Blockqs` 或校验组提供，但必须可独立验证。

## 择优池

基础参数：

```text
BestPoolCapacity = 20
AuthorizedSyncMembers = 15
AuthorizedSyncRange = ranks 6..20
```

规则：

- 择优池最多保留 20 名候选者。
- 按 `MintHash` 从小到大排序。
- 新候选者优于池中最差者时插入，否则忽略。
- 前 5 名不发起同步。
- 后 15 名有权发起同步。
- 每个授权节点对同一目标池仅有一次同步权。

## 择优池生命周期

- 新区块创建后，针对其未来成为评参区块的择优池开始形成。
- 广播收集阶段持续至该块成为 `-7` 前。
- 同步优化阶段从 `-7` 开始，持续 2 个区块时段。
- 当该块成为 `-9` 评参区块后，择优池确定，用于当前待创建区块。

## 同步池

同步池必须由授权节点签名。接收方检查：

- 签名者在本地择优池或合并池中，新上线首次同步可例外。
- 签名者在其发布的同步池中位于后 15 名。
- 签名者未超过一次同步权。
- 同步池签名消息绑定目标评参区块和池内容。

通过后，将同步池合并到本地合并池，并用合并池替换目标块对应择优池。

## 未决问题

- PoH 内层 `Hash(...)` 的算法和 domain tag。
- `timeStamp * Stakes * Mix` 的整数宽度、溢出语义，是否应改为直接编码三字段。
- `Stakes` 的精确定义、单位和区块头字段含义。
- 最低 `4M` 聪时门槛是否最终固定及升级方式。
- `MintHash` 完全相等时的 tie-breaker。
- `txLocator` 是否只需年度和 `TxID`，还是必须包含高度或 `BlockID`。
- 同步池消息格式、防重放字段和首次同步例外条件。
- 百日扩张属于协议规则、启动策略还是创世计划。

## 对后续 Plan 的影响

- PoH 实现必须先实现可复现的 `MintHash` 输入编码和排序测试。
- 签名验证必须通过 `pkg/crypto` 抽象，不能绑定具体库。
- PoH 验证依赖交易、状态和查询接口，不能在共识包内伪造交易来源。
- 择优池同步必须有签名消息和 replay protection 测试。
