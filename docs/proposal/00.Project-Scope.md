# Project Scope（项目范围）

## 来源构想

本提案来源于以下构想文档与 `docs` 目录规则，并作为后续 `Plan` 的基础约束文档：

- `docs/conception/README.md`：提出 Evidcoin 的总体目标，即在遵循区块链核心价值的前提下，构建开放式、通用、可集成的信用载体系统；信用由 `Coin`、`Credit`、`Proof` 三类实体单元表达；区块链生态应与独立公共服务网络协作。
- `docs/conception/blockchain.md`：提出 `Blockchain Core` 应极致简化、专注核心、便于任意场景自由集成；共识构造应从核心中剥离；长期历史安全主要由哈希约束保证；不同用途的 Hash 算法、长度与安全权衡应被明确记录。
- `docs/AGENTS.md`：规定 `docs` 目录内的文档链路为 `Conception => Proposal => Plan`，开发提案必须可追溯到构想层，并为实施方案提供明确边界。

## 目标

本提案定义 Evidcoin 第一阶段的基础 Proposal 范围，只生成共享规格，不编写 Go 代码。

第一阶段目标是为后续区块链核心、交易、状态、脚本与共识模块提供稳定的基础层约定：

- 在 `pkg/types/` 中规划基础类型、固定长度 `Hash`、各类 `ID` 与跨模块共享常量。
- 在 `pkg/crypto/` 中规划 Hash 函数、地址哈希与签名接口抽象。
- 将 Hash 算法分配、长度、用途与安全权衡记录为可被后续实现引用的共享规格。
- 建立基础层的包边界，使后续 `internal/*` 模块只能依赖基础层，而基础层不反向依赖 `internal` 包。

这些目标对应 `blockchain.md` 中“核心应当精简并独立运行”“长期历史安全主要由哈希约束保证”的设计意图，也对应 `README.md` 中开放式通用架构与信用实体单元的总体定位。

## 非目标

第一阶段不是完整区块链功能实现，不包含以下内容：

- 不实现区块头链管理、入块验证、主链切换或区块数据访问。
- 不实现交易结构、交易状态机、交易签名消息或交易验证流程。
- 不实现 `UTXO` / `UTCO` 集、状态存储、状态转移或集合指纹计算流程。
- 不实现脚本语言、脚本 VM、脚本指令集或脚本执行语义。
- 不实现 PoH 共识、铸币权竞争、铸凭哈希计算流程或校验组逻辑。
- 不实现 P2P、`Blockqs`、`Depots`、`stun2p` 等公共服务。
- 不选择最终第三方库版本，不落地具体 Go 源码。

这些非目标来自 `blockchain.md` 对核心简化与共识剥离的要求，也来自 `README.md` 对公共服务网络独立化的描述。第一阶段只沉淀共享规格（`Shared specs`），不提前进入高层模块实现。

## 文档层级

Evidcoin 文档采用三层结构：

- `Conception`：位于 `docs/conception/`，记录作者原始设计构想，是所有后续文档的来源。
- `Proposal`：位于 `docs/proposal/`，由构想层生成，负责把构想转化为可实施的技术规格和边界。
- `Plan`：位于 `docs/plan/`，由提案层生成，负责拆解具体实施任务、代码骨架、测试和验收步骤。

本文件是基础层 Proposal 的范围文档。后续 Plan 必须引用本文件，并继续追溯到 `docs/conception/README.md` 与 `docs/conception/blockchain.md`。

## 第一阶段实现边界

第一阶段边界为基础层 Proposal，不写 Go 代码，只产出可供后续实现使用的共享规格。

第一阶段允许定义：

- `pkg/types/` 的基础类型范围。
- 固定长度 Hash 类型及其命名边界，例如 32B、48B、64B Hash 的用途划分。
- 区块、交易、创世块、协议、链、分叉绑定等 `ID` 的抽象命名边界。
- 系统级常量的归属原则，例如区块间隔、每年区块数、脚本大小上限、交易大小上限等。
- `pkg/crypto/` 中 Hash 函数族的职责边界，包括 `SHA3-384`、`SHA3-512`、`BLAKE3-256`、`BLAKE3-512` 与地址哈希。
- 签名接口抽象的职责边界，例如 `Signer` / `Verifier` 只表达签名能力，不绑定交易、区块或共识语义。

第一阶段禁止定义：

- 区块结构体完整字段实现。
- 交易输入、输出、附件、状态机或序列化格式。
- `UTXO` / `UTCO` 状态存储与验证流程。
- 脚本 VM 指令、栈语义或执行器。
- PoH 共识、铸造竞争、校验组协议或网络交互流程。

边界依据是 `blockchain.md` 中“Blockchain Core 应精简并独立运行”“共识构造应当被剥离出来”的原则。基础层只承载跨模块共享的稳定规格，避免把高层业务语义提前固化到底层包中。

## 包分层建议

基础层建议分为两个包：

- `pkg/types/`：承载与业务语义弱耦合的基础类型、固定长度 Hash、ID 类型别名或包装类型、共享常量。
- `pkg/crypto/`：承载 Hash 函数、地址哈希、签名接口抽象以及后量子签名算法的适配边界。

分层依赖规则：

- `pkg/types/` 不依赖项目内 `internal/*` 包。
- `pkg/crypto/` 不依赖项目内 `internal/*` 包。
- `pkg/crypto/` 可以在必要时依赖 `pkg/types/` 中的固定长度 Hash 或字节类型约定。
- `internal/blockchain/`、`internal/tx/`、`internal/utxo/`、`internal/utco/`、`internal/script/`、`internal/consensus/` 等后续模块可以依赖 `pkg/types/` 与 `pkg/crypto/`。
- 禁止基础层反向 import 高层模块，禁止为了某个高层实现细节在基础层引入专用状态机逻辑。

该分层继承 `blockchain.md` 对核心独立性的要求，并服务于 `README.md` 所描述的开放式集成场景。

## 术语约定

- `Conception`：构想层文档，记录原始设计思想。
- `Proposal`：提案层文档，记录可实施的技术规格。
- `Plan`：方案层文档，记录具体实施任务。
- `Shared specs`：跨模块共享规格，只描述类型、常量、算法用途、接口边界，不包含高层业务流程实现。
- `Coin`：币金，可分割的货币性信用单元。
- `Credit`：凭信，可转移但不可分割的凭证、证书或合约类信用单元。
- `Proof`：存证，不可转让的静态证明类信用单元。
- `Hash`：固定长度摘要值；不同用途必须明确算法、长度与安全权衡。
- `ID`：由 Hash 或其它稳定识别信息派生的标识，用于区分区块、交易、链、协议、创世块或分叉绑定等对象。
- `Address Hash`：账户地址哈希，来源于公钥哈希策略，当前构想为 `SHA3-256(BLAKE2b-512)`。
- `PoH`：历史证明共识，属于后续共识层，不在第一阶段实现。
- `UTXO` / `UTCO`：未花费或未转出数据集，属于后续状态层，不在第一阶段实现。

## 未决问题

- `pkg/types/` 中 Hash 类型应使用独立命名类型，还是以长度命名的通用数组类型为主，需在后续基础类型 Proposal 中确定。
- `ID` 类型是否全部直接基于 Hash，还是为 `Protocol-ID`、`Chain-ID` 等文本标识保留独立类型，需要在后续 Proposal 中确定。
- `Address Hash` 是否在第一阶段只定义函数规格，还是同时定义地址编码边界，需要在后续 Proposal 中确定。
- ML-DSA-65 的具体实现来源需要在实现阶段结合 Go 标准库与第三方库成熟度确认。
- `BLAKE3-512` 是否统一暴露为显式 64 字节 XOF 输出接口，需要在后续 Crypto Proposal 中固定。

## 对后续 Plan 的影响

后续 Plan 必须遵守以下影响：

- 本 Proposal 编写阶段只生成基础层文档，不写 Go 代码；后续第一阶段实现 Plan 若进入代码实现，也只能触及 `pkg/types/` 与 `pkg/crypto/`。
- 后续实现 `internal/blockchain/` 时，应复用基础层 Hash、ID 与常量，不重复定义区块头 Hash 长度或区块时间常量。
- 后续实现 `internal/tx/` 时，应复用基础层交易 ID 与签名接口抽象，但交易状态机不得回流到基础层。
- 后续实现 `internal/utxo/` 与 `internal/utco/` 时，应复用基础层集合指纹 Hash 规格，但状态存储和转移逻辑保持在状态层。
- 后续实现 `internal/script/` 时，应复用基础层脚本大小常量，但脚本 VM、指令集和执行语义保持在脚本层。
- 后续实现 `internal/consensus/` 时，应复用基础层铸凭哈希规格与签名接口，但 PoH 竞争规则和校验组协议保持在共识层。
- 任何后续 Plan 都必须能追溯到本 Proposal 以及对应 Conception 文档，不能绕过文档层级直接实现未被 Proposal 固化的规则。
