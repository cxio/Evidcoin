# Transaction Model（交易模型）

## 来源构想

- `docs/conception/附.交易.md`：定义交易头、输入项、输出项、签名、多签和 Coinbase。
- `docs/conception/blockchain.md`：定义交易树、`CheckRoot`、链识别信息和长期 Hash 安全约束。
- `docs/conception/5.信用结构.md`：定义 Coin、Credit、Proof 输出载荷和附件字段。
- `docs/proposal/01.Types-And-Encoding.md`：定义交易字段、列表、可选字段和签名消息的规范化编码规则。
- `docs/proposal/02.Cryptography-And-Hashing.md`：定义 `TxID`、树 Hash、地址哈希和签名接口边界。
- `docs/proposal/03.Identifiers-And-Constants.md`：定义 `TxID`、`AddressHash`、`TreeHash`、交易大小候选常量和链识别未决项。
- `docs/proposal/04.Hash-Trees.md`：定义输入/输出树、区块交易集合树和证明路径边界。

## 目标

本提案定义交易层的技术规格，包括交易头、交易体、输入模型、输出 envelope、签名消息、单签/多签验证边界和 Coinbase 交易语义。

交易层负责表达信用转移意图。交易是否能入块，还依赖状态层、脚本层、共识层和校验组完成验证。

## 非目标

- 不实现 UTXO/UTCO 存储和状态转移。
- 不定义 Coin/Credit/Proof 的完整 payload 字段；该内容由 `07.Coin-Credit-Proof-Units.md` 承接。
- 不定义脚本 VM 指令语义。
- 不定义 PoH 铸造竞争或奖励公式。
- 不实现 Blockqs、Depots 或网络广播。
- 不固定地址文本编码、最终 varint 算法或 domain tag 字节表。

## 交易头

交易头逻辑字段为：

```text
TxHeader {
    Version
    HashInputs  TreeHash
    HashOutputs TreeHash
    Timestamp
}
```

规格要求：

- `TxID = SHA3-384(TxHeaderCanonicalBytes)`，输出 48 bytes。
- `HashInputs` 和 `HashOutputs` 均为 32 bytes `TreeHash`。
- 字段顺序必须固定。
- `Timestamp` 构想为 Unix milliseconds，但精度、宽度和过期规则仍需在实现前确认。
- 交易头不包含完整输入/输出内容，只承诺输入/输出根。
- `TxID` 的长期安全性受 `HashInputs` 与 `HashOutputs` 中 `BLAKE3-256` 安全性传递影响；这是设计上的性能权衡。

## 交易体

交易体由输入项列表和输出项列表构成：

- 普通交易必须至少包含一个 Coin 输入，用于支付交易费并支持首领校验。
- 创建 Credit 或 Proof 的交易也必须包含 Coin 输入支付上链成本。
- 未确认交易输出不得作为输入，即同一区块内前序交易输出也不应被后序交易消费。
- Proof 不可作为输入。
- 一笔交易可以混合 Coin、Credit、Proof、Mediator 和 Custom 输出。
- 完整交易数据可由外部服务保存，但必须能通过 `TxID`、区块交易树证明和状态指纹验证。

## 输入项

输入项分为首领输入和其余输入。

```text
LeadInput {
    Year
    TxID TxID
    OutIndex
}

RestInput {
    Year
    TxIDPart [20]byte
    OutIndex
    TransferIndex optional
}
```

规则：

- 首领输入必须使用完整 48 bytes `TxID`。
- 首领输入必须是 Coin 输入，用于快速首领校验和 PoH 铸凭交易属主检索。
- 其余输入使用 `TxID` 前 20 bytes 作为局部引用，以节省空间。
- `Year` 是概略时间约束，用于定位交易数据和状态集合。
- `OutIndex` 指向来源交易输出序位。
- `TransferIndex` 仅用于 Credit 转移合法性检查；Coin 输入不应携带该字段。
- Coin 输入从 UTXO 集解析，Credit 输入从 UTCO 集解析。
- 20 bytes 局部引用出现歧义时交易必须拒绝，除非状态 Proposal 固定了可验证的唯一解析规则。

## 输入 Hash

交易输入根按构想分为首领输入与其余输入：

```text
HashInputs = BLAKE3-256(LeadHash || RestHash)
LeadHash  = BLAKE3-256(LeadInputCanonicalBytes)
RestHash  = BLAKE3-256(RestInputsCanonicalListBytes)
```

约束：

- 所有字段使用规范化编码。
- `LeadHash`、`RestHash` 和 `HashInputs` 必须具备用途隔离语义。
- 普通交易不得缺失首领输入。
- Coinbase 无输入，`HashInputs` 的特殊值或计算规则必须由 Coinbase 章节进一步固定；在未固定前不得生成最终测试向量。

## 输出 envelope

每个输出项必须包含输出序位和配置字节。配置字节高 4 位为标记，低 4 位为类型：

```text
bit 7: Custom class flag
bit 6: Has attachment, or custom-class-specific meaning when bit 7 is set
bit 5: Burn / terminate flag
bit 4: Reserved
low 4 bits:
  0: Reserved
  1: Coin
  2: Credit
  3: Proof
  4: Mediator
```

规则：

- `Serial` 从 0 开始，必须与输出列表顺序一致。
- Coin 和 Credit 可作为输入源，除非设置销毁/终止标记或状态规则使其不进入状态集。
- Proof 不可作为输入源。
- Mediator 不可作为输入源，只能被脚本 `GOTO`/`EMBED` 等引用。
- Custom 类通常不能作为公共输入源；其私有语义不得影响公共验证。
- Custom 类 ID 最长 127 bytes。
- 设置销毁标记时，Coin 不进入 UTXO，Credit 不进入 UTCO；接收者可为空，脚本任意。

## 输出 Hash

```text
HashOutputs = BLAKE3-256(Tree<Outputs>)
```

约束：

- 输出项作为二元哈希树叶子。
- 输出序位必须进入叶子 Hash 输入，或由含序树规则等价承诺；具体方式需由实现 Plan 固定。
- 输出列表通常不得为空；普通交易和 Coinbase 都应至少有一个输出。
- 空树、单叶树和奇数叶处理规则继承 `04.Hash-Trees.md` 的未决问题。

## 签名消息

签名消息由 1 byte flag 决定覆盖范围：

```text
bit 7: SIGIN_ALL
bit 6: SIGIN_SELF
bit 5: SIGOUT_ALL
bit 4: SIGOUT_SELF
bit 0: SIGRECEIVER
bit 1: SIGCONTENT
bit 2: SIGSCRIPT
bit 3: SIGOUTPUT
```

规则：

- `SIGIN_ALL` 与 `SIGIN_SELF` 是输入约束，可独立使用。
- `SIGOUT_ALL` 与 `SIGOUT_SELF` 是输出主项，必须配合至少一个输出辅项。
- `SIGOUTPUT` 表示完整输出条目，覆盖接收者、内容和脚本。
- 推荐默认模式为 `SIGIN_ALL | SIGOUT_ALL | SIGOUTPUT`。
- 非法 flag 组合必须拒绝，不得降级为宽松签名。
- 签名消息必须使用规范化编码，不能由 JSON、map 或反射序列化生成。

签名消息应混入链识别信息：

```text
MixData = (Protocol-ID || Chain-ID || Genesis-ID || Bound-ID) || TxMSG
signData = Sign(MixData)
```

`Protocol-ID`、`Chain-ID`、`Genesis-ID` 和 `Bound-ID` 的最终编码仍未固定。

## 单签与多签

单签验证：

- 解锁数据提供签名、公钥和授权 flag。
- 计算规范化公钥材料的 `AddressHash`。
- 对比来源输出接收者。
- 使用签名接口验证签名消息。

多签验证：

- 解锁数据提供签名集、公钥集和补全集。
- `M-Of-N` 配比由参与公钥数量与补全集数量计算，编码为两个字节。
- 对公钥集逐个计算公钥哈希，与补全集混合排序，前置 `M/N` 后计算复合公钥哈希。
- 复合公钥哈希必须等于来源输出接收者。
- 每个签名必须对应一个参与公钥并验证通过。

未决边界包括排序规则、重复公钥、重复签名、`M=0`、`N=0`、`M>N` 和补全集成员格式。

## Coinbase 交易

Coinbase 是无输入源的特殊交易，必须位于区块交易序列第 0 项。

Coinbase 必须承载：

- 区块高度。
- PoH 择优凭证。
- 铸币补贴。
- 交易费回收部分。
- 交易费销毁部分。
- 兑奖截留回收部分。
- 奖励分成目标。
- 自由数据，长度小于 256 bytes。

Coinbase 输出配置低 4 位用于区分奖励目标：

```text
1: 铸凭者 / 铸造者收益
2: 校验组收益
3: Blockqs 奖励
4: Depots 奖励
5: STUN 服务奖励
```

Coinbase 的完整结构、无输入 `HashInputs` 规则、奖励拆分、兑奖槽和销毁输出由经济与 Coinbase 实现 Plan 共同固定。

## 存储与查询边界

- 完整交易数据可由 `Blockqs` 或其它服务保存。
- 查询服务必须能返回交易、交易所在区块、序位和哈希证明路径。
- 用户或节点应通过区块头、交易树路径和 `CheckRoot` 验证交易。
- 交易层只定义需要的数据和验证边界，不实现查询服务。

## 未决问题

- `Version`、`Timestamp`、`Year`、`OutIndex`、`TransferIndex` 的精确编码宽度。
- Coinbase `HashInputs` 的特殊计算规则。
- 交易大小上限是否采用 `65535 bytes`，以及是否排除解锁脚本和标准签名数据。
- 输出数量上限 `<1024` 的精确检查语义。
- 20 bytes `TxIDPart` 碰撞或歧义的协议处理范围。
- 签名消息 `TxMSG` 的完整字段顺序和测试向量。
- 多签排序、重复处理和异常边界。
- 交易费、找零、销毁输出和奖励输出的精确金额规则。

## 对后续 Plan 的影响

- 交易 Plan 必须先实现规范化交易头和 `TxID` 计算，再实现输入/输出 Hash。
- 所有签名消息构造必须通过专用结构，不得由调用方拼接任意字节。
- 交易验证必须调用状态层解析 UTXO/UTCO，调用脚本层验证解锁/锁定脚本。
- Coinbase 实现必须同时引用 PoH、校验组和激励 Proposal，不能只在交易层内独立完成。
