# Types And Encoding（类型与编码）

## 来源构想

本提案来源于以下构想与范围文档：

- `docs/proposal/00.Project-Scope.md`：定义第一阶段只生成基础层共享规格，不实现区块、交易、状态、脚本或共识逻辑；基础层 Proposal 应服务于后续 `pkg/types/` 与 `pkg/crypto/` 的 Plan。
- `docs/conception/blockchain.md`：提出区块头、交易约束、主链标识与 Hash 策略；其中区块头、CheckRoot、TxID、UTXO/UTCO 指纹等均依赖稳定的字节级表达。
- `docs/conception/附.交易.md`：提出交易头、输入项、输出项、签名消息、地址编码和 Coinbase 交易等结构草案；这些结构包含固定长度 Hash、整数、字节串、列表和可选字段。
- `docs/conception/5.信用结构.md`：提出 `Coin`、`Credit`、`Proof` 三类信元，以及附件 ID、附言、标题、描述、锁定脚本等字段；这些字段需要统一长度、字节串和结构体序列化规则。
- `docs/conception/6.脚本系统.md`：提出链上脚本是已编译的指令值序列，包含 `Bytes`、`String`、`Int` 等运行时值和脚本长度限制；本提案只定义基础编码边界，不定义脚本 VM。

本文件是基础层 Proposal 的类型与编码文档。后续交易、区块、状态、脚本、Hash 树和共识 Proposal 可以引用本文件中的规范化编码规则，但不得把尚未定义的高层结构提前固化到本文件中。

## 目标

本提案定义 Evidcoin 基础层可共享的初始类型与规范化编码原则，用于后续二进制序列化、Hash 输入、签名消息和测试向量设计。

具体目标：

- 为固定长度字节、整数、可变长度字节串、列表、可选字段和结构体序列化建立统一规则。
- 确保参与 Hash、ID、签名消息和集合指纹计算的数据只有一种规范化字节表示。
- 为 `pkg/types/` 后续实现提供基础类型边界，而不绑定交易、区块、状态或脚本执行语义。
- 为后续 Proposal 和 Plan 提供统一引用点，避免各模块重复定义不兼容的编码规则。

## 非目标

本提案不完整定义以下内容：

- 不定义完整区块结构、区块头验证流程、主链切换或区块存储格式。
- 不定义完整交易结构、交易状态机、签名消息细节或交易验证流程。
- 不定义 `UTXO` / `UTCO` 状态机、状态存储、状态转移或集合指纹计算流程。
- 不定义脚本 VM、脚本源码语法、指令集、执行语义或运行时值的完整编码。
- 不定义 Hash 树节点结构、树构造算法或验证路径格式。
- 不选择最终 varint 算法实现，不固定所有字段宽度。

本文件只定义基础编码约束。高层结构的字段集合、字段顺序和业务含义应由后续对应 Proposal 定义。

## 基础类型

基础类型用于表达跨模块共享的字节级数据，不携带高层状态机语义。

- `FixedBytes[N]`：固定长度字节序列，长度必须精确等于 `N`。编码和解码时必须使用精确长度，任何长度不匹配都必须拒绝。
- `Hash32`：32 字节 Hash 值，适用于 `BLAKE3-256`、地址哈希等需要 32B 输出的场景。
- `Hash48`：48 字节 Hash 值，适用于 `SHA3-384` 输出，例如区块头 Hash、TxID、CheckRoot、UTXO/UTCO 指纹。
- `Hash64`：64 字节 Hash 值，适用于 `SHA3-512` 附件指纹或显式 64B 的 `BLAKE3-512` 输出。
- `Bytes`：可变长度字节串，使用规范化长度前缀加原始字节内容编码。
- `String`：人类可读文本字段在链上按 UTF-8 字节串处理；本提案不定义文本规范化、大小写折叠或语言规则。
- `Bool`：布尔值如需作为基础字段编码，应使用单字节 `0x00` 表示 false，`0x01` 表示 true，其它值必须拒绝。
- `Unsigned Integer`：无符号整数。数值字段默认使用无符号整数，除非业务域明确需要负数。
- `Signed Integer`：有符号整数。仅在明确需要负值的领域使用；脚本运行时的 `Int`、`DATE` 等 signed 表达必须由后续 Script Proposal 专门定义，不能直接复用本文件的 unsigned varint。

基础类型命名可以在后续 `pkg/types/` Plan 中细化为 Go 类型、数组类型或包装类型。本提案只固定编码语义。

## 整数编码

数值字段默认使用无符号整数。只有领域语义明确需要负值时，才允许使用有符号整数，并且必须在对应高层 Proposal 中说明原因和编码方式。

整数分为两类：

- 固定宽度整数：字段宽度由结构定义明确给出，例如 1 字节配置、2 字节计数、4 字节高度或 8 字节数值。
- 可变宽度整数：用于长度前缀、列表计数或尚未固定宽度的自然数值，使用规范化无符号 varint 编码。本文件只约束 unsigned varint 的用途和规范化要求，不定义 signed varint。

固定宽度整数的规范化要求：

- 使用字段定义指定的精确字节数。
- 不允许更短或更长的替代表达。
- 不允许超出字段宽度可表达范围的值。
- 编码结果必须可唯一还原为同一数值。

可变宽度整数的规范化要求：

- 使用 canonical unsigned varint length prefix 或 canonical unsigned varint count。
- 精确 varint 算法仍为未决问题，在作者确认前，后续实现不得假定某一种具体算法已最终批准。
- varint 算法一经确认，必须拒绝该算法下的非最短表达、冗余表达、溢出和无法终止的编码。
- 需要负值的 signed varint 不得隐式套用 unsigned varint；其符号位、补码、ZigZag 或其它表达方式必须由对应 Proposal 独立确认。

## 字节序

固定宽度整数在规范化序列化中使用 big-endian byte order。

该规则适用于所有固定宽度整数，包括版本号、高度、计数、配置位段、金额或其它后续 Proposal 明确为固定宽度的数值字段。若某一领域确需不同字节序，必须在对应 Proposal 中列为协议例外并说明原因；默认不得例外。

字节序不改变固定长度字节序列本身的内容。Hash、ID、公钥哈希、附件指纹和脚本字节序列按原始字节顺序编码，不做整数解释。

## 字节串编码

固定长度字节与可变长度字节串采用不同编码方式。

固定长度字节规则：

- `FixedBytes[N]` 直接写入 `N` 字节内容。
- 解码时必须读取精确 `N` 字节。
- 长度不足、长度过长或与字段定义不匹配时必须拒绝。
- Hash 和 ID 类型不得带长度前缀，除非对应高层结构明确把它们放入可变字节容器中。

可变长度字节串规则：

- 使用 canonical unsigned varint length prefix 编码字节长度。
- 长度前缀之后紧跟原始字节内容。
- 长度为 0 的字节串编码为长度前缀 0，后面不跟内容。
- 解码时必须验证长度前缀与实际内容长度一致。
- 字段自身的最大长度约束由对应高层 Proposal 定义，例如附言、标题、描述、附件 ID、锁定脚本或解锁脚本。

可变长度字节串只表达字节，不隐含文本、脚本或附件语义。

## 列表编码

列表必须先编码 item count，再按顺序编码每个 item 的规范化表示。

列表编码规则：

- `count` 使用 canonical unsigned varint 编码。
- `count` 表示后续 item 的数量，不表示字节总长度。
- 每个 item 必须使用该 item 类型自身的规范化编码。
- item 顺序属于列表语义的一部分，编码不得重新排序。
- 解码时必须确保实际 item 数量与 `count` 一致。
- 列表是否允许为空由对应高层 Proposal 定义。

对需要排序的集合，高层 Proposal 必须明确排序键和排序规则。本提案不把所有列表默认为集合，也不定义 Hash 树的节点排序或分组规则。

## 可选字段编码

可选字段必须显式编码 presence 标记。

编码规则：

- absent 使用单字节 `0x00`。
- present 使用单字节 `0x01`，随后编码字段值的规范化表示。
- presence 标记只能是 `0x00` 或 `0x01`，其它值必须拒绝。
- absent 时不得编码字段值。
- present 时必须编码完整字段值，不得省略。

该规则适用于后续 Proposal 中明确为 optional 的字段，例如附件 ID、附言、分叉绑定标识、可选有效期或其它可选扩展信息。字段是否可选由对应高层 Proposal 决定。

## 结构体序列化

结构体序列化必须由对应 Proposal 明确字段顺序和字段类型。结构体的规范化编码是按字段定义顺序依次拼接每个字段的规范化编码。

结构体编码规则：

- 字段顺序必须固定，不能按字段名、字段值或实现语言的反射顺序动态变化。
- 所有必填字段都必须出现。
- 可选字段必须按本提案的 presence 规则编码。
- 未知字段、重复字段、字段重排或隐式默认值不得参与规范化编码。
- 结构体版本字段如存在，应作为普通固定顺序字段编码；版本兼容策略由对应高层 Proposal 定义。
- 结构体编码不自动包含总长度，除非该结构体被作为可变长度字节串或高层 Proposal 明确要求带长度。

此规则可被后续区块头、交易头、输入项、输出项、附件 ID、签名消息和状态指纹结构引用，但本提案不定义这些结构的完整字段集合。

## 规范化编码要求

参与 Hash、ID、签名、状态指纹、校验根或持久化协议边界的数据必须使用规范化编码。

规范化编码要求：

- 同一逻辑值必须只有一种合法字节表示。
- 解码器必须拒绝非规范化但可被宽松解析的替代表达。
- 固定长度字段必须精确匹配长度，不能使用长度前缀替代。
- 固定宽度整数必须使用 big-endian byte order。
- 数值字段默认无符号，除非对应领域明确需要有符号值。
- 可变长度字节串必须使用 canonical unsigned varint length prefix。
- 列表必须先编码 item count，再编码每个 item。
- 可选字段必须使用 `0x00` 或 `0x01` presence 标记。
- Hash 输入不得混用非规范化对象表示、JSON、map 遍历顺序或实现语言私有序列化格式。
- 任何协议字段的最大长度、最大计数和溢出行为必须在对应高层 Proposal 或 Plan 中形成测试。

规范化编码的主要目的，是让 Hash 结果、签名消息和状态指纹在不同实现之间稳定一致。

## 测试向量要求

后续 Plan 实现基础编码时必须提供测试向量。测试向量应覆盖成功编码、成功解码和拒绝非规范化输入。

最低测试向量要求：

- 固定长度字节：正确长度通过，短 1 字节和长 1 字节拒绝。
- 固定宽度整数：big-endian 编码示例、最大值、零值和溢出拒绝。
- 无符号 varint：算法确认后必须覆盖零值、小值、边界值、最大允许值、非最短表达、溢出和未终止输入；算法确认前只能保留测试占位，不能固化测试字节。
- 可变长度字节串：空字节串、普通字节串、长度不匹配和超过字段最大长度。
- 列表：空列表、单元素列表、多元素列表、count 与实际 item 数量不匹配。
- 可选字段：`0x00` absent、`0x01` present、非法 presence 值、absent 后多余字段值。
- 结构体：字段顺序固定、字段重排拒绝、未知尾随数据默认拒绝；只有外层长度边界或容器协议已明确定义时，才允许按对应解码边界处理。

测试向量文件格式由后续 Plan 决定，但必须能表达输入字节、期望逻辑值、期望错误和对应规则来源。

## 未决问题

- 精确 varint 算法尚未由作者确认。确认前，后续 Plan 只能保留接口和测试占位；确认后，必须补齐边界值、非最短表达、溢出和未终止输入测试向量。
- 脚本源码/显示格式与链上 bytecode 的互转规则由后续 Script Proposal 定义；人类可读脚本文本不能替代链上规范化字节编码。
- signed integer 是否需要作为基础层通用编码仍待确认；脚本运行时 signed varint 必须由 Script Proposal 独立定义，不能由各模块自行发明。

## 对后续 Plan 的影响

后续 Plan 必须遵守以下影响：

- `pkg/types/` 的基础编码实现应以本文件为规则来源，提供固定长度字节、整数、字节串、列表、可选字段和结构体组合的最小工具。
- 后续区块、交易、状态、脚本和共识 Proposal 应引用本文件定义 Hash 输入和签名消息的规范化字节表达。
- 后续 Plan 不得在各模块中各自发明不同的整数、长度前缀、列表或 optional 编码规则。
- 在 varint 算法获作者确认前，Plan 可以保留接口与测试占位，但不得声称已固定最终协议算法，也不得生成依赖具体 varint 字节序列的测试向量。
- 脚本源码、人类可读地址、外部 JSON 或调试展示格式不得替代链上规范化编码；若需要互转，应在对应 Proposal 中另行定义。
- 本文件不授权实现完整交易结构、状态机、脚本 VM 或 Hash 树；这些内容必须由后续专门 Proposal 和 Plan 承接。
