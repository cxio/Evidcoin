# Cryptography And Hashing（密码学与哈希）

## 来源构想

本提案来源于以下构想与基础 Proposal：

- `docs/proposal/00.Project-Scope.md`：定义第一阶段只生成基础层共享规格，`pkg/crypto/` 负责 Hash 函数、地址哈希与签名接口抽象，不实现区块、交易、状态、脚本或共识逻辑。
- `docs/proposal/01.Types-And-Encoding.md`：定义规范化编码规则。所有参与 Hash、ID、签名、状态指纹和校验根计算的数据，必须只有一种合法字节表示。
- `docs/conception/blockchain.md`：提出区块头、`CheckRoot`、交易树、UTXO/UTCO 指纹、附件指纹、公钥哈希和铸凭哈希的算法分配，并强调长期历史安全主要由 Hash 约束保证。
- `docs/conception/附.交易.md`：提出 `TxID = SHA3-384(TxHeader)`、交易输入/输出树使用 `BLAKE3-256`、地址来源于公钥哈希，以及签名用于证明输出项拥有权。
- `docs/conception/5.信用结构.md`：提出 `Coin`、`Credit`、`Proof` 三类信元，附件 ID 中包含 `SHA3-512` 附件指纹，附件分片验证使用 `BLAKE3-256` 片组哈希。
- `docs/conception/1.共识-历史证明（PoH）.md`：提出铸凭哈希由签名数据计算，值小者胜出；铸凭哈希需兼顾大量参与者场景下的速度与信息熵。

本文件是基础层 Proposal 的密码学与 Hash 文档。后续区块、交易、状态、脚本、附件和共识 Proposal 可以引用本文件中的算法分配、输入规范与接口边界，但不得把尚未定义的高层结构提前固化到本文件中。

## 目标

本提案定义 Evidcoin 基础层可共享的密码学与 Hash 规则，为后续 `pkg/crypto/` 和相关模块实现提供稳定边界。

具体目标：

- 固定主要协议对象的 Hash 算法、输出长度与用途边界。
- 规定所有 Hash 输入必须使用 `01.Types-And-Encoding.md` 定义的规范化编码。
- 规定用途隔离原则，避免不同用途的数据被误用为同一 Hash 输入空间；具体 domain tag 是否进入协议 Hash 输入由对应 Proposal 确认。
- 定义地址哈希、公钥材料哈希和铸凭哈希的基础约束。
- 定义签名接口抽象边界，使上层模块可以使用后量子签名而不绑定具体依赖。
- 为后续 Plan 的测试向量、安全检查和包边界提供依据。

## 非目标

本提案不定义以下内容：

- 不定义完整区块结构、区块头字段顺序、区块验证流程或主链切换逻辑。
- 不定义完整交易结构、输入输出字段集合、交易状态机或签名消息结构。
- 不定义 PoH 完整流程、择优池同步、铸造者资格验证或铸凭交易格式。
- 不定义 Hash 树节点格式、验证路径格式、节点排序规则或树构造算法。
- 不定义脚本 VM、脚本指令、解锁脚本或锁定脚本执行语义。
- 不定义人类可读地址编码、地址前缀、Base58 细节或地址校验码算法。
- 不选择最终第三方库版本，不暴露具体 ML-DSA-65 实现依赖给上层模块。

这些内容应由后续专门 Proposal 承接。本文件只固化基础层的算法分配、Hash 输入约束和密码学接口边界。

## Hash 算法分配

Evidcoin 的基础 Hash 算法分配如下：

| 用途 | 算法 | 输出长度 | 说明 |
|------|------|----------|------|
| Block header | `SHA3-384` | 48B | 区块 ID 由区块头 Hash 得到，兼顾量子安全与数据量。 |
| CheckRoot | `SHA3-384` | 48B | 用于约束区块交易树根与 UTXO/UTCO 指纹等状态摘要。 |
| TxID | `SHA3-384` | 48B | 交易 ID 由交易头规范化编码计算，与区块头设计一致。 |
| Tree internal nodes | `BLAKE3-256` | 32B | 用于树内部节点、分支节点或根节点中需要高频计算的场景。 |
| Tree leaves | `SHA3-384` where specified by conception docs | 48B | 构想明确要求树末端叶子节点使用 `SHA3-384` 时，叶子与内部节点使用不同算法。 |
| Attachment fingerprint | `SHA3-512` | 64B | 对附件整体数据计算，服务于超长期完整性保护。 |
| Address hash | `SHA3-256(BLAKE2b-512(pubkey material))` | 32B | 用于账户地址、公钥哈希和复合公钥哈希的基础摘要。 |
| Mint hash | `BLAKE3-512` through explicit XOF output length | 64B | 用于铸凭哈希，必须显式输出 64 字节。 |

补充约束：

- UTXO/UTCO 指纹属于 `SHA3-384` 约束场景，后续状态 Proposal 应明确其叶子数据摘要、集合根和 `CheckRoot` 组合方式。
- 附件片组哈希属于 `BLAKE3-256` 树结构场景，与 `SHA3-512` 附件整体指纹用途不同。
- 交易输入集、输出集等树结构如沿用构想中的 `Hash256`，应解释为 `BLAKE3-256`，除非后续 Proposal 明确修订。
- `BLAKE3-512` 必须通过显式 XOF 输出长度得到 64B 结果，不得调用只返回默认 32B digest 的 API 后声称为 `BLAKE3-512`。

## Hash 输入规范

所有协议级 Hash 输入必须使用 `docs/proposal/01.Types-And-Encoding.md` 定义的规范化编码。

通用要求：

- 同一逻辑值必须只有一种合法 Hash 输入字节序列。
- 固定长度 Hash、ID、公钥哈希和附件指纹按原始字节顺序拼接，不添加长度前缀，除非它们被明确放入可变长度字节容器。
- 固定宽度整数使用 big-endian byte order。
- 可变长度字节串使用 canonical unsigned varint length prefix。
- 列表先编码 item count，再按固定顺序编码每个 item。
- 可选字段使用 `0x00` 或 `0x01` presence 标记。
- 结构体 Hash 输入必须由对应 Proposal 明确字段顺序和字段类型。
- Hash 输入不得使用 JSON、map 遍历顺序、反射序列化、平台字节序或实现语言私有二进制格式。

本提案不固定区块头、交易头、签名消息、PoH 演算数据或 Hash 树节点的完整输入格式。后续 Proposal 定义这些结构时，必须引用本节并给出字段顺序、类型、用途隔离方式和测试向量。

## 域分隔

不同协议用途的 Hash 必须具备明确的用途隔离语义，避免相同字节序列在不同上下文中产生可混淆的摘要意义。

域分隔原则：

- 每一种协议级 Hash 用途应有稳定的用途标签，例如区块头、交易 ID、校验根、地址哈希、附件指纹、铸凭哈希、树内部节点、树叶子节点等。
- 用途标签可以作为协议 Hash 输入的一部分参与计算，也可以由算法上下文、API 类型、测试向量标签或外层结构提供等价隔离；具体方式由对应 Proposal 确认。
- 若 domain tag 进入协议 Hash 输入，tag 本身必须使用规范化字节表达，建议使用短 ASCII 标识和版本信息；具体字节格式由对应 Proposal 固定。
- 不同算法不能替代用途隔离。即使两个用途使用不同算法，也应在测试向量中保留用途标签，避免实现层误接线。
- Hash 树内部节点、叶子节点、附件片组、交易集合和状态集合的用途隔离细节由后续 Hash 树或状态 Proposal 定义。

本文件只规定必须具备用途隔离语义，不直接修订构想层已经给出的 Hash 输入公式。最终 domain tag 字节表或等价隔离方式应在相关 Proposal 中确认，并纳入测试向量。

## 地址哈希

账户地址的基础哈希为：

```text
AddressHash = SHA3-256(BLAKE2b-512(pubkey material))
```

约束：

- `pubkey material` 必须是规范化的公钥材料字节序列，不能使用文本展示、公钥对象内存布局或第三方库私有序列化。
- 后续地址或签名 Proposal 必须明确 `pubkey material = key type / algorithm id / parameter set + canonical public key bytes` 的具体字段与顺序，避免多算法或多密钥格式下的地址语义混淆。
- 单签名地址对单个公钥材料执行地址哈希。
- 多重签名地址的复合公钥哈希应先对各参与公钥得到基础公钥哈希，再按后续交易或地址 Proposal 固定的 `M-Of-N` 规则排序、前置配比并重新计算复合哈希。
- 单签名地址与多重签名地址在外观上不应泄露类型差异；具体地址编码、前缀和校验码不在本提案定义。
- 地址哈希输出为 32B，量子安全强度低于 `SHA3-384` 场景。这是空间节省与长期安全之间的有意权衡，后续 Plan 应把该风险写入安全说明。

## 签名算法

基础层应提供签名能力抽象，而不是把交易、区块、PoH 或脚本语义写入签名接口。

接口边界：

- 签名抽象应表达 `Signer` 和 `Verifier` 能力，例如对规范化消息字节签名、验证签名和导出规范化公钥材料。
- 签名抽象必须允许 ML-DSA-65 作为后量子签名算法接入。
- 上层模块不得直接依赖具体 ML-DSA-65 第三方库、标准库候选实现或密钥对象内部结构。
- 具体依赖选择应封装在 `pkg/crypto/` 的适配层内，避免泄露到 `internal/blockchain/`、`internal/tx/`、`internal/script/` 或 `internal/consensus/`。
- 签名消息结构不在本提案定义。交易签名消息、链标识混入、PoH 铸凭演算数据和校验组签名数据必须由后续 Proposal 分别定义。
- 签名接口只处理已规范化的消息字节，不负责字段选择、权限语义、脚本验证或交易状态判断。

安全要求：

- 验证逻辑必须显式绑定算法标识或密钥类型，不能把不同签名算法的公钥和签名格式混用。
- 公钥材料用于地址哈希前必须有唯一规范化编码。
- 签名验证失败、消息编码失败、算法不匹配和密钥格式错误应作为不同错误类别供测试覆盖，但具体错误类型由后续 Plan 决定。

## 铸凭哈希

铸凭哈希用于 PoH 铸造竞争，构想要求值小者胜出，并要求在大量参与者场景下兼顾速度与信息熵。

基础规则：

- 铸凭哈希算法为 `BLAKE3-512`。
- 实现必须使用 BLAKE3 XOF 或等价接口显式输出 64 字节。
- 禁止调用默认 32 字节 digest API 后补零、重复、截断其它数据或以其它方式伪造 64 字节输出。
- 铸凭哈希输入必须是后续 PoH Proposal 定义的规范化签名数据或演算数据。
- 本提案不定义 `Mix`、`Stakes`、时间戳、评参区块、签名数据、择优凭证或完整 PoH 流程的编码。

后续 PoH Proposal 必须明确：

- 铸凭交易 ID、评参区块铸凭哈希、币权因子、时间因子和签名数据的字段顺序。
- 铸凭哈希的用途隔离方式，以及 domain tag 是否进入协议 Hash 输入。
- 字节序、整数宽度、溢出处理和规范化编码规则。
- 铸凭哈希按字节序列从小到大比较时的排序规则。

## 测试向量要求

后续 Plan 实现 `pkg/crypto/` 时必须提供跨实现可复现的测试向量。

最低测试向量要求：

- `SHA3-384`：覆盖空输入、短输入、结构化规范化输入，以及区块头/TxID/CheckRoot 用途标签示例。
- `SHA3-512`：覆盖附件整体数据的空输入、小输入和大输入分段喂入一致性。
- `BLAKE3-256`：覆盖树内部节点、树根或片组 Hash 的示例输入，确保输出为 32B。
- `BLAKE3-512`：覆盖显式 XOF 64B 输出，并加入防误用测试，确保默认 32B digest 不能通过。
- `AddressHash`：覆盖规范化公钥材料到 32B 地址哈希的示例；多签复合地址待后续地址或交易 Proposal 固定规则后补齐。
- 用途隔离：若 domain tag 进入协议 Hash 输入，同一 payload 在不同 domain tag 下必须得到不同测试结果；若采用等价隔离方式，测试向量必须能证明实现不会混用不同用途。
- 规范化编码：同一逻辑值的非规范化替代表达必须被拒绝，不能产生 Hash。
- 签名抽象：使用固定测试密钥或测试替身覆盖签名成功、验证成功、消息变更失败、公钥变更失败和算法不匹配失败。

测试向量文件应记录算法名、用途标签或 domain tag、输入字节、输出字节、规则来源和期望错误。若 varint 算法尚未最终确认，不得固化依赖具体 varint 字节的测试向量。

## 安全注意事项

- 长期历史安全主要依赖 Hash 约束。区块头、`CheckRoot`、`TxID`、UTXO/UTCO 指纹等核心摘要必须避免算法或输入格式歧义。
- `TxID` 的安全性会受到交易头内输入根、输出根等 `BLAKE3-256` 摘要的传递影响。这是构想中以性能换取树节点计算效率的有意权衡，后续交易 Proposal 应继续记录。
- `AddressHash` 输出 32B，量子安全余量低于 `SHA3-384` 输出。地址长期暴露时风险更高，钱包和交易层应鼓励避免地址重复暴露的使用习惯。
- 附件同时使用 `SHA3-512` 整体指纹和 `BLAKE3-256` 片组哈希，两者用途不同，不能相互替代。
- BLAKE3 默认 digest API 通常输出 32B；铸凭哈希必须显式请求 64B XOF 输出。
- 签名算法依赖选择必须可替换，避免上层协议绑定某个库的私有格式或不稳定 API。
- Hash 输入中不得包含未初始化内存、map 随机顺序、平台相关整数宽度、浮点数非规范表达或本地时间格式。
- 用途隔离缺失会增加跨协议重放、对象混淆和实现误接线风险，必须在测试中覆盖。

## 未决问题

- 最终 domain tag 字节表尚未固定；哪些 tag 进入协议 Hash 输入、哪些仅作为 API/测试层隔离，需要在对应 Proposal 中确认。
- `CheckRoot` 中交易树根、UTXO 指纹和 UTCO 指纹的精确字段顺序与是否包含 domain tag，需由区块链核心或状态 Proposal 定义。
- Hash 树节点格式、叶子前置序号、节点排序和验证路径格式，需由后续 Hash 树、交易或附件 Proposal 定义。
- 多重签名复合地址中公钥哈希排序规则、`M-Of-N` 编码和地址文本编码，需由后续地址或交易 Proposal 定义。
- ML-DSA-65 的具体实现来源需在实现阶段结合 Go 标准库和第三方库成熟度确认，但不得影响上层接口边界。
- canonical unsigned varint 算法尚未确认，涉及可变长度字段的测试向量需等待 `01.Types-And-Encoding.md` 的未决问题关闭后补齐。

## 对后续 Plan 的影响

后续 Plan 必须遵守以下影响：

- `pkg/crypto/` 应提供 `SHA3-384`、`SHA3-512`、`BLAKE3-256`、显式 XOF `BLAKE3-512`、`AddressHash` 和签名抽象的最小实现边界。
- `pkg/crypto/` 不得依赖 `internal/*` 包；高层模块可以依赖基础层 Hash 与签名接口。
- 所有 Hash API 应明确输出长度，避免调用方混淆 32B、48B 和 64B 类型。
- 所有协议级 Hash 调用应要求调用方传入已规范化编码的字节，或在对应高层模块内先完成规范化编码再调用 Hash。
- 后续区块 Proposal 应引用本文件定义 Block header、`CheckRoot` 和区块相关签名校验的算法边界。
- 后续交易 Proposal 应引用本文件定义 `TxID`、输入/输出树 Hash、地址哈希和签名抽象，但签名消息结构由交易 Proposal 自行定义。
- 后续状态 Proposal 应引用本文件定义 UTXO/UTCO 指纹的 `SHA3-384` 边界，但状态存储和集合指纹流程保持在状态层。
- 后续附件 Proposal 应引用本文件定义附件整体指纹和片组 Hash 的双 Hash 边界。
- 后续 PoH Proposal 应引用本文件定义铸凭哈希为显式 64B `BLAKE3-512`，但完整铸造竞争流程保持在共识层。
- 后续测试 Plan 必须包含算法分配、用途隔离、规范化输入、BLAKE3 XOF 长度和签名抽象的测试向量。
