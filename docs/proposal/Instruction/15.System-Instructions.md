# Proposal: System Instructions [164–169]（系统指令）

> **Prerequisite:** 本规范建立在 [Base Constraints](0.Base-Constraints.md) 文档之上，该文档定义了全局使用的指令格式、运行时模型与记号约定。


## 1. Overview（概览）

系统指令占用操作码区间 **[164–169]**（共 6 条，其中 167–168 保留）。它们提供超出常规脚本计算模型的系统级原语：实时时钟访问、区块链奖励校验、内置交易校验以及源码分段标记。

全部系统指令使用 `SYS_` 前缀，以区别于普通脚本指令，体现其在执行环境中的特权角色。

### Design Principles（设计原则）

系统指令用于衔接脚本沙箱执行模型与区块链运行需求。它们把复杂校验逻辑（奖励计算、交易签名验证）封装成单条指令，确保共识关键操作在所有节点上以统一方式实现，而不是由低层原语拼装，避免实现差异引发共识失败。


## 2. Instruction Definitions（指令定义）

### 2.1 Global Time Retrieval（全局时间获取）

```
164     SYS_TIME(1){}   Global time retrieval
```

- **Auxiliary params:** 1 byte——目标标识，指定要获取的时间组件。
- **Arguments:** 无。
- **Return value:** 依据目标标识，返回 Time 对象或 Int 值。
- 提供对客户端实时钟表的只读访问。返回值反映指令执行瞬间节点本地系统时间。
- **Entries:**
  - *(default)* ——完整 Time 对象。
  - `Stamp` ——Unix 毫秒级时间戳。
  - `Year` ——4 位年份（如 2026）。
  - `Month` ——月份（1–12）。
  - `YearDay` ——一年中的第几天（平年 1–365，闰年 1–366）。
  - `Day` ——月内第几天（1–31）。
  - `WeekDay` ——星期几（0–6，0 = Sunday）。
  - `Hour` ——小时（0–23）。
  - `Minute` ——分钟（0–59）。
  - `Second` ——秒（0–59）。
  - `Millisecond` ——毫秒分量（0–999）。
  - `Microsecond` ——微秒分量（0–999999）。
- 所有数字组件均为 Int 类型。
- **Examples:**
  ```
  SYS_TIME              // → Time 对象（默认）
  SYS_TIME[Stamp]       // → 1773755895000（Unix 毫秒）
  SYS_TIME[Year]        // → 2026
  SYS_TIME[WeekDay]     // → 4（Thursday）
  SYS_TIME[Millisecond] // → 547
  ```
- **Design note:** `SYS_TIME` 读取本地系统时钟，不同节点可能返回不同值。因此除非只做粗粒度比较（例如校验时间戳是否落在多分钟窗口内，节点间可一致判断），否则应仅用于 non-consensus 逻辑。

### 2.2 Award Verification（奖励校验）

```
165     SYS_AWARD(1,~)  Award verification
```

- **Auxiliary params:**
  1. 1 byte——奖励比例（如 `10` 表示 10%）。
  2. Varint——区块高度。
- **Arguments:** 无。
- **Return value:** 可兑付奖励金额（扣除留存部分后）。
- **Behavior:** 校验挖矿奖励兑付是否满足要求：
  1. 校验当前区块高度与附参配置区块高度的距离已合法（>29）。
  2. 统计后续48个区块Coinbase交易兑奖槽位里的确认数。
  3. 按奖励比例和确认数计算可兑付金额（**注**：确认数不足时存在扣除）。
- **Usage restriction:** 仅可用于 **Coinbase transaction** 的 output script。Coinbase transaction 受严格约束：
  - 奖励比例必须符合协议规则。
  - 同一 Coinbase transaction 中所有 output 的比例总和必须 = 100%。
  - 区块高度必须与Coinbase交易所属区块一致。
- **Chain segment inference:** 根据第二个辅助参数给出的区块高度，可快速推导校验链段终点高度（偏移 +29 到 +48 区块）。
- **Example:**
  ```
  SYS_AWARD[10, 50000]  // 校验区块 50000 的 10% 奖励 → 可兑付金额
                        // 花费时的当前区块高度至少 50030（29个区块已确认）
  ```

### 2.3 System Verification (Built-in Transaction Verification)（系统校验（内置交易校验））

```
166     SYS_CHKPASS     System verification
```

- **Auxiliary params:** 无。
- **Arguments:** 无（从执行环境隐式提取——特例）。
- **Return value:** 无。
- **Behavior:** 执行内建交易校验——属于类似 `PASS` 的 **gate instruction**。一旦校验失败，脚本立即终止（语义与 `PASS` 失败一致）。
- **Verification requirements:**
  - 接收地址必须为标准格式。
  - coin 与 credential 采用不同校验逻辑，但都由该单条指令触发。
  - coin 校验包含金额提取与余额校验。
  - credential 校验包含转移逻辑校验。
- **Implicit function signature:**
  ```
  SYS_CHKPASS(chk_type, sig_type, auth_flag, []sigs, []pubkeys, []pubkey_hashes)
  ```
  - `chk_type` ——验证类别：`0` 为币金，`1` 为凭信。
  - `sig_type` ——签名类型：`1` 为单签，`2` 为多签。
  - `auth_flag` ——授权标志，配置签名消息的约束。
  - `[]sigs` ——签名数组。
  - `[]pubkeys` ——公钥数组。
  - `[]pubkey_hashes` ——公钥哈希数组（仅多签模式使用）。
- **Design notes:**
  - 该指令不从数据栈消费参数。此为特例。
  - 首个参数 `chk_type` 从脚本的输出项配置字节中获取（低四位类型说明）。
  - 剩余5个参数从环境中提取（解锁时携带，导入环境）。
  - 这种设计可以把 unlock script 中的签名数据与交易 ID 计算解耦，后续即使移除签名也不影响链上完整性。
  - 链式哈希可在移除签名后继续提供必要安全保证。
  - 将单签/多签参数隐藏在可见脚本体之外可提升隐私——观察者无法仅凭 lock script 直接判断签名方案。
  - 本指令可存在于 GOTO/EMBED/SCRIPT 等指令引入的外部脚本中，没有限制。
- **Example:**
  ```
  SYS_CHKPASS           // 执行内建校验，失败即终止
  ```

### 2.4–2.5 Reserved（保留）

```
167     (Reserved, unused)
168     (Reserved, unused)
```

### 2.6 Source Code Null Point Marker（源码 NULL 点标记）

```
169     SYS_NULL    Source code null point marker
```

- **Auxiliary params:** 无。
- **Arguments:** 无。
- **Return value:** 无。
- 在当前指令流位置设置一个 **NULL point**。NULL 点作为 `SOURCE` 指令（opcode 135）的分段边界，用于提取特定代码区间。
- `SYS_NULL` 本身除建立标记外无运行时效果——不会修改数据栈、参数空间或其他执行状态。
- **Relationship to SOURCE:**
  - `SOURCE[2]`：从最近的前置 `SYS_NULL` 点提取已执行代码。
  - `SOURCE[-2]`：向后提取到最近的后继 `SYS_NULL` 点（后执行代码区段）。
- **Example:**
  ```
  SYS_NULL
  "Hai" {100} ADD
  ...
  SOURCE[2]             // 提取从 "Hai" 到当前 SOURCE（含 SOURCE）
  ```
  这里 `SYS_NULL` 建立边界，`SOURCE[2]` 提取边界与自身之间所有指令（不含 NULL 标记，含 SOURCE）。


## 3. Instruction Summary（指令总览）

| Opcode | Mnemonic | Gate? | Description |
|--------|----------|-------|-------------|
| 164 | SYS_TIME | No | 实时时钟访问 |
| 165 | SYS_AWARD | No | 挖矿奖励校验与计算 |
| 166 | SYS_CHKPASS | Yes | 内建交易校验（失败即终止） |
| 167–168 | — | — | 保留 |
| 169 | SYS_NULL | No | 源码 NULL 点标记 |

### Design Notes（设计说明）

- **Gate instructions:** `SYS_CHKPASS` 是系统内第二条 gate instruction（另一条是 `PASS`）。两者失败都会立即终止脚本。区别在于：`PASS` 只校验显式布尔参数，而 `SYS_CHKPASS` 使用隐式环境数据执行复杂多步骤校验。
- **SYS_CHKPASS privacy model:** 通过从环境提取签名相关数据而非从脚本直接消费，指令实现了关注点分离：lock script 定义“校验什么”，unlock script 提供“如何满足”。这种分离允许未来升级签名处理机制而不改动 lock script 格式。
- **SYS_NULL as a meta-instruction:** 与多数产生计算效果的指令不同，`SYS_NULL` 纯粹用于支撑 `SOURCE` 元编程能力。它体现脚本系统的自反性：脚本可以感知自身结构，并提取有意义的片段用于校验或委托。
