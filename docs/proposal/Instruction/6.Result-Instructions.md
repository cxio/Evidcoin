# Result Instructions（结果指令）

Status: Draft proposal
Source conception: `conception/Instruction/6.结果指令.md`

## Prerequisites（前置条件）

关于 stack model、argument zone、local frame 与 opcode layout，请参见 `0.Base-Constraints.md`。简要说明如下：

- **data stack** 用于承载中间值；**argument zone** 用于为下一条指令暂存输入；**local frame** 是按 block 隔离的作用域；**global frame** 是脚本级作用域。
- 每条指令由 `opcode + immediate + associated data` 组成；其中只有 opcode 是必选项。
- 脚本最终裁决由 **pass status** 决定——由 `PASS`（hard）与 `CHECK`（soft，last-one-wins）设置，并在脚本终止时读取。

## Overview（概述）

本指令族用于产生决定性、终止性或控制转移效果：校验 **pass status**（`PASS`、`CHECK`）；将控制转入另一笔交易的输出脚本，可选择隔离子环境（`GOTO`）或以内联方式扩展当前 frame（`EMBED`）；以及结束执行（`EXIT` 用于当前运行脚本，`RETURN` 用于函数型子块，`END` 仅用于公共校验终止）。

## Opcode range（操作码范围）

`[51 – 57]`，共 7 条指令。

## Instruction Catalog（指令目录）

### 51 — `PASS` — hard pass check

| Field | Value |
|---|---|
| Type | Pass status |
| Runtime args | a boolean |
| Stack effect | consumes one boolean |
| Returns | none |

**Description.** 当 boolean 为 true 时，pass status 设为 "passed"，并继续执行；当为 false 时，pass status 设为 "failed"，且脚本会**立即终止**。多个 `PASS` 的组合行为等价于逻辑 AND：必须全部成功。

**Implementation note.** `PASS` 失败时，可触发带外通知机制，便于外部观察者获知失败原因。

---

### 52 — `CHECK` — soft pass check

| Field | Value |
|---|---|
| Type | Pass status |
| Runtime args | a boolean |
| Stack effect | consumes one boolean |
| Returns | none |

**Description.** boolean 语义与 `PASS` 相同，但为 `false` 时**不会**终止脚本。pass status 会被后续每个 `CHECK` 覆盖，因此以最后一次 `CHECK` 为准。其组合行为近似逻辑 OR（后续成功可覆盖此前失败，反之亦然）。

**Design.** 脚本正常终止时，系统会检查 pass status 以给出最终裁决。

---

### 53 — `GOTO(~,48,~)` — jump into another transaction's output script (isolated)

| Field | Value |
|---|---|
| Type | Cross-script transfer |
| Immediates | (1) transaction year — varint，来源于 tx timestamp；(2) transaction ID — 48 bytes（SHA3-384）；(3) output index — varint，0-based |
| Runtime args | variadic — argument zone 中所有项都会传给 callee |
| Returns | none |

**Description.** 加载被引用的 transaction output 所附脚本，并在**独立子环境**中执行：

- callee 以独立的新 data stack 启动（由 caller 的 argument zone 初始化），并拥有自己的 argument zone 与 global frame。
- callee 与 caller 共享 `INPUT` / `OUTPUT` buffer。
- callee 与 caller 共享 pass status——其 `PASS` / `CHECK` 结果会直接影响顶层裁决。
- callee 本身也可以继续包含 `GOTO` 或 `EMBED` 调用。
- callee 的全部私有状态会在返回时丢弃。

```go
@POPS[5] GOTO[2026, 0xab93cf..., 1]
// 将栈顶 5 个元素拉入 argument zone，再跳转到 2026 年 tx 0xab93cf... 的第 2 个输出脚本执行，
// 并将这些元素作为 callee 的初始 stack。
```

**Design.** callee 到 caller 的唯一反馈通道是 pass status。若允许 callee 的 stack 残留回流到 caller，会形成不可控的数据注入向量。

---

### 54 — `EMBED(~,48,~)` — inline another transaction's output script

| Field | Value |
|---|---|
| Type | Cross-script transfer |
| Immediates | (1) transaction year（varint）；(2) transaction ID（48 bytes）；(3) output index（varint） |
| Runtime args | none |
| Returns | none |

**Description.** 加载被引用输出脚本，并在**当前 frame 内联执行**——即复用同一 data stack、argument zone、global frame 与 pass status。因此 callee 可压入后续由 caller 消费的值。

```go
EMBED[2026, 0xab93cf..., 0] FN_HASH256 DATA{...} ...
// 先内联执行被引用 tx 的第 1 个输出脚本，再对其产出的栈顶值进行哈希。
```

**Caution.** 由于 `EMBED` 会直接修改 caller 的 data stack，caller 必须信任被嵌入脚本。

---

### 55 — `EXIT` — terminate the script

| Field | Value |
|---|---|
| Type | Termination |
| Runtime args | optional — at most one value |
| Returns | the value, if any, is forwarded to the host (which may process it) |

**Description.** 立即停止执行。系统会检查 pass status 并给出裁决。通常用于 `IF` block 内，但在顶层同样合法。

---

### 56 — `RETURN` — return from a functional sub-block

| Field | Value |
|---|---|
| Type | Sub-block exit |
| Runtime args | optional single value |
| Returns | the value (or `nil` if absent) |

**Description.** 退出当前函数型 block 的本轮迭代——仅适用于 `MAP{}` 或 `FILTER{}`。它**不会**终止 `EACH{}` 循环；应使用 `BREAK`。

---

### 57 — `END` — terminate public-validation logic

| Field | Value |
|---|---|
| Type | Conditional termination |
| Runtime args | none |
| Returns | none |

**Description.** 仅在 public-validation node 上结束脚本。面向私有用途的 node 会忽略该指令并继续执行后续内容。对 public validator 而言，`END` 的行为等价于 `EXIT`。

```text
示例 —— 定向加密：
  - 交易方 A 将其 public key 发布到链上。
  - 交易方 B 基于 A 的 public key 派生共享密钥，加密 payload，
    并将密文与 B 的 public key 一并发布。
  - 交易方 A 基于 B 的 public key 重建共享密钥并解密。
解密逻辑只对 A 有意义，因此公共校验应提前结束——
放置位置应在公共校验之后、A 专用处理之前。
```

**Design.** 私有监听方要保持可达，通常仍需要开放的公共发布通道；因此其 handler script 往往会先*包含*公共校验逻辑，再通过 `END` 终止该部分，随后继续执行私有处理。

## Design notes（设计说明）

- **`PASS` vs. `CHECK`.** 二者表达两种不同控制流：`PASS` 是“此处必须成立”（短路 AND），`CHECK` 是“当前最终裁决”（last-write-wins）。两者可自由交错使用。
- **`GOTO` is isolated; `EMBED` is inlined.** `GOTO` 用于可验证委托，返回通道严格受限（仅 pass status）。`EMBED` 用于源码复用/组合，caller 有意将 callee 作为自身延续执行。
- **Cross-script identity.** `GOTO` 与 `EMBED` 均使用 `(year, txID, outputIndex)` 引用目标，使 script-locator 地址模型在各指令族中保持一致。
- **`END` is node-conditional.** 这是唯一一条效果取决于执行 node 类别的指令，可在同一脚本体内统一兼容公共校验与私有处理。
- **No "return value" from `GOTO`.** pass status 是唯一反馈信号——可防止 callee 向 caller stack 注入不受约束的数据。
