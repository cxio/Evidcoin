# Proposal: Result Instructions [51–57]（提案：结果指令 [51–57]）

> **Prerequisite:** 本规范建立在 [Base Constraints](0.Base-Constraints.md) 文档之上；该文档定义了全局采用的指令格式、运行时模型与记号约定。


## 1. Overview（概览）

结果指令占用 opcode 范围 **[51–57]**（共 7 条）。它们产生会直接影响脚本执行走向的**决定性结果**：通过/失败门控检查、跨脚本跳转、函数返回与脚本终止。这一类指令最终决定交易验证是成功还是失败。

### Design Principle（设计原则）

结果指令连接了“计算逻辑”与“验证结论”。其他指令主要处理数据，而结果指令把数据状态转化为可执行结果：一个 boolean 可变成 pass/fail 门控，一个地址可变成跨脚本跳转，一条终止指令可变成清晰可控的退出点。

### Pass-State Model（Pass-State 模型）

脚本引擎维护一个 **pass-state** 标记，用于表示当前验证结果：

- `PASS` 在成功时将其设为 **true**；失败时**终止**脚本（多个门控整体是逻辑 AND）。
- `CHECK` 将其设为 **true** 或 **false**，但不终止执行（后一个 CHECK 覆盖前一个，呈现逻辑 OR 语义）。
- 在脚本终止时（自然结束、`EXIT` 或 `END`），最终以 pass-state 判定验证结果。


## 2. Instruction Definitions（指令定义）

### 2.1 Pass Gate（通过门控）

```
51      PASS        Gate check (pass/fail)
```

- **Auxiliary params:** 无
- **Arguments:** 一个 boolean 值。
- **Return value:** 无。
- **Behavior:**
  - **true:** 将 pass-state 设为 PASS，继续执行。
  - **false:** 将 pass-state 设为 FAIL，脚本**立即终止**。
- 多个 `PASS` 构成逻辑 AND，必须全部通过。
- **Implementation note:** PASS 失败时，可触发通知机制向外部系统报告失败原因。
- **Example:**
  ```
  TOP FN_PUBHASH DATA{<expected_hash>} EQUAL PASS    // 门控 1：公钥哈希校验
  FN_CHECKSIG PASS                                   // 门控 2：签名校验
  ```

### 2.2 Check Gate（检查门控）

```
52      CHECK       Check (non-terminating gate)
```

- **Auxiliary params:** 无
- **Arguments:** 一个 boolean 值。
- **Return value:** 无。
- **Behavior:**
  - **true:** 将 pass-state 设为 PASS。
  - **false:** 将 pass-state 设为 FAIL，但执行**继续**（不终止）。
- 每次 `CHECK` 都会**覆盖**当前 pass-state，因此只有最后一次 `CHECK` 决定结果，整体呈现逻辑 OR 语义。
- **Design note:** 脚本正常终止时，系统依据 pass-state 判定验证结果。`CHECK` 适合尝试多种验证策略，只要其中一种通过（最终 CHECK 为 true）即可成功。

### 2.3 Independent Jump (GOTO)（独立跳转）

```
53      GOTO(~,48,~)    Independent jump
```

- **Auxiliary params:**
  1. Varint —— 交易年份（由交易时间戳推导）。
  2. 48 bytes —— transaction ID。
  3. Varint —— 输出索引（0-based）。
- **Arguments:** 可变数量。
- **Return value:** 无。
- **Behavior:**
  - 将执行重定向到指定交易输出的锁定脚本（介管）。目标脚本在**独立子环境**中运行：
  - 实参空间的内容会转移为目标脚本的**初始 data stack**。
  - 目标脚本拥有独立的 data stack、argument space 与 global variables。
  - import/export buffers 各自独立地与调用方**共享**。
  - 目标脚本的 pass-state 会回写到顶层调用者的 pass-state。
  - 目标脚本退出后，其私有环境数据全部丢弃。
- **Nesting rules:**
  - 主脚本层级最多调用 GOTO：**2** 次。
  - 运行时 GOTO 最大深度：**3**。
  - GOTO 目标脚本内部仍可包含 GOTO 或 EMBED。
- **Example:**
  ```
  @POPS[5] GOTO[2026, 0xab93cf..., 1]
  ```
  弹出栈顶 5 项到参数空间，跳转到 2026 年该交易的第 2 个输出脚本，并将这 5 项作为目标脚本初始栈内容。

### 2.4 Merged Embed (EMBED)（合并嵌入）

```
54      EMBED(~,48,~)   Merged embed
```

- **Auxiliary params:**
  1. Varint —— 交易年份。
  2. 48 bytes —— transaction ID。
  3. Varint —— 输出索引（0-based）。
- **Arguments:** 无。
- **Return value:** 无。
- **Behavior:** 取出指定交易输出的锁定脚本，并在**调用者环境**中执行 —— 共享 data stack、argument space、global variables。嵌入脚本可直接向调用者栈中压入数据。
  - import/export buffers 同样共享。
  - pass-state 直接作用于当前脚本。
- **Nesting rules:**
  - 运行时 EMBED 调用总数最多：**4** 次。
  - 最大深度：**0** —— 嵌入脚本内部不能再使用 EMBED 或 GOTO。
- **Example:**
  ```
  EMBED[2026, 0xab93cf..., 0] FN_HASH256 DATA{...} ...
  ```
  嵌入第 1 个输出脚本并执行（可向栈压值），然后继续后续哈希计算。

### GOTO vs. EMBED Comparison（GOTO 与 EMBED 对比）

| Aspect | GOTO | EMBED |
|--------|------|-------|
| Environment | Independent (new stack, argument space, globals) | Merged (shares caller's stack, argument space, globals) |
| Data transfer | Argument space → target's initial stack | Full environment sharing |
| Pass-state | Writes back to top-level caller | Directly modifies current script's state |
| Max calls | ≤ 2 at main level | ≤ 4 total across runtime |
| Max depth | ≤ 3 across runtime | 0 (no nesting within embedded script) |
| Buffer sharing | Shared | Shared |
| Residual data | Discarded on exit | Persists in shared environment |

### 2.5 Exit Script（退出脚本）

```
55      EXIT        Terminate script
```

- **Auxiliary params:** 无
- **Arguments:** 可选 —— 任意类型的单个值。
- **Return value:** 该参数值（返回给系统）。
- 立即终止脚本执行；系统随后依据 pass-state 判定验证结果。
- 若带参数，该值会返回给系统（可由外部 handler 处理）。
- 若无参数，则直接结束脚本。
- 常见于 `IF{}` 中用于条件提前退出，但也可在顶层使用。

### 2.6 Function Return（函数返回）

```
56      RETURN      Return from function block
```

- **Auxiliary params:** 无
- **Arguments:** 可选 —— 任意类型的单个值。
- **Return value:** 参数值；若无参数则为 `nil`。
- 退出当前**函数式块**（`MAP{}` 或 `FILTER{}`）的本次迭代。
- 仅允许在 `MAP{}` 与 `FILTER{}` tool instruction 块内使用；不适用于 `EACH{}`（`EACH{}` 使用 `BREAK`）。
- 返回值将作为该函数式块当前迭代的输出。

### 2.7 Public Validation End（公共验证结束）

```
57      END         End public validation
```

- **Auxiliary params:** 无
- **Arguments:** 无。
- **Return value:** 无。
- 仅用于公共验证节点终止脚本执行。私有参与节点通常忽略该指令并继续执行。
- 对公共验证者而言，其行为与 `EXIT` 一致 —— 都会评估 pass-state。
- **Use case — directed encryption:**
  1. A 方将其公钥发布到链上。
  2. B 方使用 A 的公钥生成共享密钥，随后加密数据并发布密文与 B 的公钥。
  3. A 方使用 B 的公钥重建共享密钥并解密数据。
  - 该密文仅对 A 方有意义，因此公共验证应在私有数据处理逻辑前结束。
- **Design note:** 私有监听方依赖公开广播将数据送达私有监控节点。私有处理通常先复用部分公共验证逻辑，再继续应用特定处理。
- **Script pattern:**
  ```
  // 公共验证段
  <signature checks>
  PASS
  END                     // 公共节点在此停止

  // 私有段（仅私有节点继续）
  <encrypted data processing>
  <application-specific logic>
  ```


## 3. Validation Flow Summary（验证流程摘要）

```
Script Start
    │
    ├─ PASS(true) ──► pass-state = PASS, continue
    ├─ PASS(false) ──► pass-state = FAIL, TERMINATE
    │
    ├─ CHECK(true) ──► pass-state = PASS, continue
    ├─ CHECK(false) ──► pass-state = FAIL, continue
    │
    ├─ GOTO ──► sub-environment → pass-state writes back
    ├─ EMBED ──► merged execution → pass-state directly affected
    │
    ├─ EXIT ──► TERMINATE, evaluate pass-state
    ├─ END ──► TERMINATE (public nodes only)
    ├─ RETURN ──► exit MAP/FILTER iteration
    │
    └─ Natural end ──► evaluate pass-state
```

无论通过哪种方式终止，最终验证结果都由终止时刻的 pass-state 决定。
