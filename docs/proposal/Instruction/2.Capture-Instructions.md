# Capture Instructions（截取指令）

Status: Draft proposal（提案草稿）
Source conception: `conception/Instruction/2.截取指令.md`

## Prerequisites（前置条件）

关于栈模型、本地/全局 frame、opcode 布局以及指令编码约定，请参见
[`0.Base-Constraints.md`](./0.Base-Constraints.md)。
下面先对本文会用到的概念做一个简要回顾：

- **Data stack** —— 任意指令结果的默认落点，也是运行时参数的默认来源。
- **Argument zone** —— 按“每条指令”临时存在的缓冲区，会覆盖 data stack，作为**next**指令的运行时参数来源。
- **Local frame** —— 仅追加（append-only）、块私有（block-private）的存储区，容量 128，索引范围为 `[-128, 127]`。
- **Capture instruction** —— 使用单符号前缀，对下一条指令结果的去向或其参数的来源进行**redirects**。

## Overview（概述）

Capture 指令是脚本语言的数据流“管线部件”。它们不会计算新值，而是重定向数据流：

- 拦截**following**指令原本会压入 data stack 的结果，并将其写入 argument zone（`@`）或追加到当前 local frame（`$`）；
- 从 local frame（`$[n]`）或 loop frame（`$X`）读取一个值，并放入 argument zone；
- 覆盖 next 指令的运行时参数来源，使其直接从 data stack 取操作数，而不是从 argument zone 取（`~`）。

由于这类指令被设计为与下一条指令紧密绑定，其 mnemonic 采用单个非字母数字符号，并且 capture 与目标指令之间无需空白分隔。

所有 capture 指令的一个关键属性是：通过它们获取到的值会进入 **argument zone**（或 local frame），**never**自动压入 data stack。

## Opcode Range（Opcode 范围）

`[19, 23]` —— 共 5 个 opcode。

## Instruction Catalog（指令目录）

```go
opcode  mnemonic    short description
-----------------------------------------------------------
19      @           参数截取（argument capture）
        // 栈效果：将下一条指令的结果重定向到 argument zone，
        //   而不是写入 data stack。
        // 源码层形式：
        //   @PEEKS[2]   读取栈底 2 个元素，并展开到 argument zone。
        //   @{1234}     将字面量 1234 直接放入 argument zone
        //               （否则它会自动压栈）。
        // 说明：可与 ~ 组合为 `@~TARGET`（见 opcode 23）。

20      $           本地 frame 截取（local-frame capture）
        // 栈效果：将下一条指令的结果追加到当前 local frame。
        // local frame 是在 0.Base-Constraints.md 中定义的
        // 仅追加、块私有存储（容量 128）。
        // 源码层形式：
        //   $TOP        引用栈顶元素，并追加到 local frame
        //               （浅拷贝）。
        //   $POPS[3]    弹出 3 个元素，并展开追加到 local frame
        //               （追加 3 个条目）。
        //   ${100}      将字面量 100 追加到 local frame。
        //   $SHIFT[3]   取栈顶 3 个元素，打包为单个 slice，
        //               再追加到 local frame（追加 1 个条目）。
        // 错误：
        //   - 追加超出容量 128 时触发 FAIL。
        // 说明：
        // local frame 仅对其所在语法块私有。
        // 同一块内的同级指令共享该 frame；嵌套块与父块使用各自独立的 frame。

21      $(1)        本地 frame 引用（读取）
        // 类型：与被引用 slot 相同
        // Immediate：1 字节有符号索引，范围 [-128, 127]。
        // 栈效果：将被引用的值放入 argument zone
        //   （不会自动压栈）。
        // 负索引从最近追加的条目开始计数
        // （$[-1] = 最近一次追加）。
        // 源码层形式：
        //   $[0]    引用 frame 中最先追加的值。
        //   $[-1]   引用最近追加的值。
        // 错误：
        //   - 索引超出已填充范围时触发 FAIL。
        // 实现说明：
        // 实现 MAY 支持符号引用：
        // 通过截取指令的 mnemonic 还原对应存储值
        // （例如 `${POP}` 读取由 $POP 存入的值）。
        // 该符号解析为 best-effort；当相同 mnemonic 被多次截取时，
        // 可能发生 shadowing。

22      $X(1)       循环 frame 变量引用
        // 类型：与被引用的循环变量相同
        // Immediate：1 字节选择器，范围 [0, 3]。
        // 栈效果：将被引用值放入 argument zone
        //   （与 @ 和 $[n] 一致）。
        // 循环 frame 变量（用于循环体内部，参见
        // ./7.Flow-Control.md）：
        //   selector  symbolic    meaning
        //   --------  ----------  -----------------------------------
        //   0         $Value      当前元素值
        //   1         $Index      当前数字索引（slice）
        //             $Key        当前字符串键（dict）
        //   2         $Slice      当前正在迭代的 slice
        //             $Dict       当前正在迭代的 dict
        //   3         $Size       成员总数
        // 名称区分大小写，且均以 `$` sigil 开头。
        // 说明：
        // 与块内局部的 $[n] 不同，$X 在循环体内的嵌套语法块之间可见，
        // 因为循环会建立一个逻辑上独立的 frame 层。

23      ~           直接栈参数路由（direct-stack argument routing）
        // 栈效果：强制下一条指令直接从 data stack 获取
        //   运行时参数，绕过 argument zone。
        // 组合规则：
        //   当与 capture（@ 或 $）组合时，顺序 MUST 为
        //   `<capture> ~ <target>`，即路由前缀位于
        //   capture 与其目标之间。
        // 源码层形式：
        //   ~EQUAL    弹出栈顶 2 个元素，比较是否相等，
        //             并自动压入布尔结果。
        //   @~EQUAL   同样比较，但布尔结果落入 argument zone。
        //   $~EQUAL   同样比较，但布尔结果被追加到 local frame。
        // 设计说明：
        // ~ 提供了显式的通道切换原语，扩展了合法的数据重排模式，
        // 无需强制脚本作者先将值经由 argument zone 中转。
```

> **Summary.** 在这 5 个 opcode 中，只有 `$`（local-frame capture）和 `~`
> （direct-stack routing）不是取值类：`$` 负责写入，`~` 负责改道。
> 其余 3 个（`@`、`$[n]`、`$X`）都会把值读入
> argument zone。

## Design Notes（设计说明）

- **Tight binding.** Capture 符号被设计为绑定到紧随其后的指令。编译器不要求 capture 与目标之间存在空白分隔（如 `@TOP`、`$POPS[3]`、`~EQUAL`）。
- **Argument zone, not data stack.** Capture 将数据写入 argument zone（或 local frame），其核心目的是不扰动 data stack。这使得 capture 驱动的数据流可以与面向栈的指令稳定组合：data stack 继续作为长生命周期工作区，而 argument zone 作为逐指令交接缓冲。
- **Composition order with `~`.** 当 `~` 与 capture（`@` 或 `$`）混用时，capture 在前，`~` 紧贴目标：`@~T`、`$~T`。该顺序与语义流程一致：先“截取 T 的结果”，同时 T 的操作数由栈直接供给。
- **Local-frame discipline.** local frame 是有界且仅追加的。脚本若需覆盖或轮转值，应通过显式读取（`$[n]`）与再次 capture 完成，而非原地修改。
