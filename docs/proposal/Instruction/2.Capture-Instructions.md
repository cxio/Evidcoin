# Capture Instructions（截取指令）

## 来源构想

- `docs/conception/Instruction/2.截取指令.md`：定义 opcode `19-23` 的截取语义。
- `docs/proposal/Instruction/0.Base-Constraints.md`：定义取参和返回值模型。

## 范围

截取指令改变后续目标指令返回值或取参来源，不单独表达业务状态。

应覆盖的指令族：

- `@`：将目标指令返回值放入实参区。
- `$`：将目标指令返回值放入当前局部域。
- `$[n]`：将目标返回值放入局部域指定位置。
- `$X`：访问或截取循环域变量。
- `~`：强制目标指令从数据栈取参。

## 规范要求

- 截取指令只作用于紧随其后的目标指令。
- 多前缀组合必须有固定解析顺序。
- `@~EQUAL` 等组合中，`@` 决定返回值去向，`~` 决定取参来源。
- 目标指令无返回值时，截取结果为空；是否失败由具体截取指令固定。
- 多返回值截取到实参区或局部域时必须按返回顺序自动展开。
- 局部域容量候选值为 128，越界访问必须失败。
- 循环变量只在循环体内有效，跨块访问只允许构想明确的 `$X` 场景。

## 风险

- 前缀组合解析不一致会导致脚本含义分裂。
- `~` 跳过实参区可能让脚本审计变难。
- 局部域截取如果允许名称别名，可能造成重名和兼容问题。

## 未决问题

- opcode `19-23` 与具体符号的一一映射。
- 无返回值目标被截取时是空操作还是失败。
- 局部域是否支持名称别名。
- 负索引规则是否适用于局部域。

## 对后续 Plan 的影响

- 解析器必须把截取前缀和目标指令绑定成一个执行单元。
- 测试必须覆盖前缀组合、局部域越界、循环变量作用域和 `~` 取参绕过。
