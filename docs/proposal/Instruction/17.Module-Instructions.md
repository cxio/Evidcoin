# Proposal: Module Instructions [225–250]（模块指令）

> **Prerequisite:** 本规范建立在 [Base Constraints](0.Base-Constraints.md) 文档之上，该文档定义了全局使用的指令格式、运行时模型与记号约定。


## 1. Overview（概览）

模块指令占用操作码区间 **[225–250]**（26 条）。模块提供 **categorized functionality**：执行模块指令，就会调用其构造方法，创建一个模块对象，然后通过通用的 `CALL` 指令（opcode 44）调用其开放的方法。

### Module Design Pattern（模块设计模式）

每条模块指令遵循统一模式：

- **Opcode** 标识模块本体，执行后创建模块对象，入栈。
- 模块对象通过 `CALL` 指令调用成员方法，可向模块构造函数传递实参。
- 模块成员函数返回多个值时，被封装为一个切片，不自动展开。

### Calling Convention（调用约定）

模块方法通过通用 `CALL` 指令（opcode 44）调用：

```
<module_obj> "<MethodName>" [args...] CALL[n]
```

等价地，也可以用截取指令先暂存至实参区，再执行调用：

```
@<module_obj>
@"<MethodName>"
@<arg>
CALL[1]     // 从实参区取指令的参数和<MethodName>的参数
// 注：
// 此处实无必要，主要用于先暂存不定数量的参数；
// CALL[n] 仍需用附参明确本次调用传递的参数数量。
```

> **Note:** 本节尚未完全定稿。下列成员列表为初步设计，后续会随系统成熟继续细化与扩展。


## 2. Instruction Definitions（指令定义）

### 2.1 Math Module（数学模块）

```
225     MO_MATH     Math operations module
```

- **Arguments:** 无。
- **Return value:** 数学模块对象（仅能通过 `CALL` 指令调用）。
- **Members:**

  | Name | Signature | Description |
  |------|-----------|-------------|
  | Abs | `(x float64) → float64` | 绝对值 |
  | Ceil | `(x float64) → float64` | 向上取整到最近整数 |
  | Floor | `(x float64) → float64` | 向下取整到最近整数 |
  | Pow | `(x, y float64) → float64` | 计算 `x` 的 `y` 次幂 |
  | Max | `(x, y float64) → float64` | 两值中较大者 |
  | Min | `(x, y float64) → float64` | 两值中较小者 |
  | Mod | `(x, y float64) → float64` | 浮点余数 |
  | ... | — | 其余成员待定 |

- **Examples:**
  ```
  MO_MATH "Abs" {-100} CALL[1]     //=> 100.0
  MO_MATH "Max" {-1} {99} CALL[2]  //=> 99.0
  ```
- **Design notes:**
  - Math 模块无需构造器参数——所有成员均为无状态函数。
  - 对齐 Go `math` package 的子集，优先覆盖链上计算常见需求。

### 2.2 Format Module（格式化模块）

```
226     MO_FMT     Data formatting module
```

- **Arguments:** 无。
- **Return value:** 格式化模块对象（仅能通过 `CALL` 指令调用）。
- **Members:**

  | Name | Signature | Description |
  |------|-----------|-------------|
  | Sprint | `(a ...any) → string` | 转换为字符串（无分隔符） |
  | Sprintf | `(format string, a ...any) → string` | 格式化字符串 |
  | Sprintln | `(a ...any) → string` | 转换为字符串（换行符结尾） |

- **Design notes:**
  - Format 模块无需构造器参数——所有成员均为无状态函数。
  - 对齐 Go `fmt` package 的子集，提供链上数据格式化的基本需求。
  - 对标 Go `fmt.Sprint`、`fmt.Sprintf`、`fmt.Sprintln`。

### 2.3 Reserved (227–249)（保留）

操作码 **227–249**（23 个槽位）保留给未来模块扩展，当前未分配，实现中不得使用。

未来模块候选包括但不限于：字符串工具、密码学原语、数据序列化、集合算法。


## 3. Instruction Definition — Standard Module Extension（指令定义——标准模块扩展）

### 3.1 Standard Module Reference（标准模块引用）

```
250     MO_XX(1){}  Standard module reference
```

- **Auxiliary params:** 1 byte——目标子模块索引（`[0–255]`）。
- **Associated data:** 无。`{}` 记法仅是子模块索引的友好别名，编译后仍保存 1 字节附参。
- **Arguments:** 由目标子模块决定（与构造普通模块相同，传递实参即可）。
- **Return value:** 目标子模块对象（后续通过 `CALL` 调用其成员方法）。
- **Design notes:**
  - 作为标准模块表的分发网关，最多支持 **256** 个子模块槽位。
  - `{}` 记法支持按子模块名称引用：`MO_XX{crypto}` 引入 crypto 标准模块。
  - 当需超过 256 个扩展模块时，使用扩展指令中的 `EXT_MO`（opcode 251），其提供 2-byte（64K）地址空间。
  - 256个标准模块的空间应已足够用于日常需求。
- **Example:**
  ```
  MO_XX{crypto}          // 引入 crypto 标准模块
  "Sign" <data> <key>
  @CALL[2]               // 调用 Sign 方法，结果暂存实参区
  ```


## 4. Design Notes（设计说明）

### Decoupled Method Invocation（解耦方法调用）

模块对象与方法调用通过通用 `CALL` 指令（opcode 44）解耦。这有以下优势：
- 模块的方法不需要硬编码参数数量，参数量由 `CALL` 指令的附参指出。
- 模块的方法定义数量没有了限制——不受 opcode 数量约束。
- 同一调用机制同时适用于模块和内置对象（如正则表达式、时间对象等）。

### Determinism Requirement（确定性要求）

所有模块成员函数在相同输入下都必须给出 **deterministic results**。这是共识关键约束——任何非确定性行为（如系统随机数、wall-clock 时间）都会导致节点间校验分叉。

### Extensibility（可扩展性）

当前已分配 opcode：
- `225` - MO_MATH（数学运算）
- `226` - MO_FMT（数据格式化）

保留区间（227–249）提供了充足增长空间。`MO_XX` 进一步扩展至 256 个子模块，当需更多时可使用 `EXT_MO`（opcode 251）提供 64K 的模块地址空间。
