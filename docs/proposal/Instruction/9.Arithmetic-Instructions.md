# Arithmetic Instructions（运算指令）

## 来源构想

- `docs/conception/Instruction/9.运算指令.md`：定义 opcode `80-103`。
- `docs/proposal/Instruction/0.Base-Constraints.md`：定义取参、返回和失败规则。

## 范围

运算指令包括表达式符号和命名指令：

- 表达式：`()`, `*`, `/`, `%`, `+`, `-`。
- 命名运算：`MUL`, `DIV`, `ADD`, `SUB`, `MOD`, `POW`, `LMOV`, `RMOV`, `AND`, `ANDX`, `OR`, `XOR`, `NEG`, `NOT`, `DIVMOD`, `REP`, `DEL`, `CLEAR`。

## 规范要求

- 表达式内部计算类型倾向为 Float，但这会带来确定性风险，必须在实现前确认。
- 命名指令应按输入类型保持类型或明确定义提升规则。
- 除零必须失败。
- 整数溢出、BigInt 长度超限和 Float NaN/Inf 必须固定处理。
- `ADD` 对 String、Bytes、Dict 的多态语义必须显式列入规则表。
- 位运算只应允许整数或字节类型。
- `REP`、`DEL`、`CLEAR` 涉及集合或栈副作用时必须说明是原地修改还是返回新值。

## 风险

- Float 运算跨平台一致性。
- `ADD` 多态过宽导致脚本审计困难。
- Dict 删除/清空副作用会影响共享引用。
- 表达式解析若允许嵌套普通指令，容易产生静态分析困难。

## 未决问题

- 公共验证是否允许 Float 运算。
- `POW` 对整数输入返回 Int、BigInt 还是 Float。
- `REP[0]` 的精确语义。
- `DEL/CLEAR` 作用于栈、Dict 还是通用集合的边界。
- 表达式最大嵌套深度和成本。

## 对后续 Plan 的影响

- 运算测试必须覆盖类型矩阵、溢出、除零、浮点边界、多态 ADD 和副作用。
