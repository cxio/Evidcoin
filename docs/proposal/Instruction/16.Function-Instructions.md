# Function Instructions（函数指令）

## 来源构想

- `docs/conception/Instruction/16.函数指令.md`：定义 opcode `170-224`。
- `docs/proposal/02.Cryptography-And-Hashing.md`：定义 Hash、地址和签名算法边界。
- `docs/proposal/03.Identifiers-And-Constants.md`：定义地址文本编码未决项。
- `docs/proposal/Instruction/0.Base-Constraints.md`：定义函数调用和成本边界。

## 范围

函数指令包括高频标准函数和自扩展函数：

- Base58/Base32/Base64 编码解码。
- Address 编码/解码。
- PubHash / MPubHash。
- CheckSig / MCheckSig。
- Hash224 / Hash256 / Hash384 / Hash512。
- Printf。
- `FN_X` 标准函数扩展引用。

## 规范要求

- Base 编码必须固定具体字母表、padding、大小写和错误规则。
- Address 函数必须等待地址文本编码 Proposal 固定前缀和 checksum。
- PubHash 必须使用规范化公钥材料。
- MPubHash 必须固定 `M-Of-N`、排序和重复处理。
- CheckSig/MCheckSig 必须使用交易 Proposal 定义的签名消息。
- Hash 函数输出长度必须与 `02.Cryptography-And-Hashing.md` 对齐。
- `FN_X` 必须有注册表、版本和成本规则，不能自由扩散。

## 风险

- 地址文本编码未定，过早实现会固化错误协议。
- Base58/Base32/Base64 变体差异很大。
- 签名函数若绕过交易签名消息，会造成重放风险。
- `FN_X` 无治理会造成实现碎片化。

## 未决问题

- 地址前缀、checksum 和 Base 编码。
- Hash224 是否存在协议级用途。
- 多签排序和重复规则。
- `FN_X` 注册表治理。
- Printf 是否允许公共验证执行。

## 对后续 Plan 的影响

- 函数测试必须使用固定算法测试向量，并拒绝未固定地址编码的最终向量。
- 签名函数必须复用 `pkg/crypto` 抽象。
