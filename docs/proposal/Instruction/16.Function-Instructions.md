# Function Instructions（函数指令）

Status: 草案提案
Source conception: `conception/Instruction/16.函数指令.md`

## Prerequisites（前置条件）

关于栈模型、本地帧、opcode 布局、立即数记号 `(n)` 与关联数据记号 `{}`，
参见 `0.Base-Constraints.md`。

相关概念回顾：

- 函数类 opcode 在语法形态上与普通 opcode 相同，但语义上表现为函数调用；
  为便于识别，其助记符统一使用 `FN_` 前缀。
- 大多数函数类 opcode 不带 immediate；输入来自运行时参数，并通过参数区
  或数据栈按引擎解析规则获取。
- 本区块预留了一个以单字节索引的自扩展槽位（`FN_X`，code 224），承载
  **standard function set**，可通过 1 字节索引寻址最多 256 个附加函数 opcode。

## Overview（概览）

本区块包含 **basic function set** 与 **standard function
set extension index**。基础集合覆盖编码（`Base58`、`Base32`、`Base64`、
地址编解码）、公钥哈希（单签 + 复合多签）、签名校验（单签 + 多签），以及
可参数化哈希族：输出位宽支持 224 / 256 / 384 / 512 bit，算法可在 `BLAKE3`
（默认）、`BLAKE2b`、`SHA-2`、`SHA-3` 中选择。基础范围最后包含一个格式化
打印辅助函数。code `224` 为标准函数集入口。

## Instruction Catalog — Basic Function Set（指令目录——基础函数集）

> 记号说明。“Form” 列用于表达函数式签名以便阅读；参数在运行时仍通过参数区
> 或数据栈提供，具体遵循引擎的参数解析规则。

```go
Code   Form
-----------------------------------------------------------
170    FN_BASE58 ( data Bytes|String )                      => String|Bytes
171    FN_BASE32 ( data Bytes|String )                      => String|Bytes
172    FN_BASE64 ( data Bytes|String )                      => String|Bytes
173    FN_ADDRESS( data Bytes|String, prefix String )       => String|Bytes
174    FN_PUBHASH( data Bytes )                             => Bytes
175    FN_MPUBHASH( pubHash []Bytes, m, n byte )            => Bytes
176    FN_CHECKSIG( chk_type, auth_flag byte,
                    pubKey, sig Bytes )                     => Bool
177    FN_MCHECKSIG( chk_type, auth_flag byte,
                     pubKeys, sigs, pubHashes []Bytes )     => Bool
178    FN_HASH224(1)( data Bytes )                          => Bytes (28B)
179    FN_HASH256(1)( data Bytes )                          => Bytes (32B)
180    FN_HASH384(1)( data Bytes )                          => Bytes (48B)
181    FN_HASH512(1)( data Bytes )                          => Bytes (64B)
182-222                                                       (unused)
223    FN_PRINTF( fmt String, val ...any )                  => nil
```

### 170 — `FN_BASE58`（Base58 编解码）

- **Type**: function, codec
- **Operands**: 1 runtime argument
- **Stack effect**: `Bytes -> String` 或 `String -> Bytes`
- **Description**: 将字节序列编码为 Base58 字符串，或将 Base58 字符串解码
  回字节序列。方向由参数的运行时类型自动判定。
- **Errors**: 解码时 Base58 输入不合法。

### 171 — `FN_BASE32`（Base32 编解码）

- **Type**: function, codec（RFC 4648，大写字母 + `234567`，无 padding）
- **Operands**: 1 runtime argument
- **Stack effect**: `Bytes -> String` 或 `String -> Bytes`
- **Description**: 执行 Base32 编码/解码。字符串形式不带 padding。
- **Errors**: 解码时 Base32 输入不合法。

### 172 — `FN_BASE64`（Base64 编解码）

- **Type**: function, codec（URL-friendly 字母表 `-_`，无 padding）
- **Operands**: 1 runtime argument
- **Stack effect**: `Bytes -> String` 或 `String -> Bytes`
- **Description**: 使用 URL-friendly 字母表且不带 padding 的 Base64 编码/
  解码。方向由参数类型自动判定。
- **Errors**: 解码时 Base64 输入不合法。

### 173 — `FN_ADDRESS`（地址编解码）

- **Type**: function, address codec
- **Operands**: 2 runtime arguments — `data`, `prefix`
- **Stack effect**: `Bytes, String -> String`（编码）或
  `String, String -> Bytes`（解码）
- **Description**: 将公钥哈希编码为账户地址（Base58 文本形式），或将账户地址
  解码回公钥哈希。文本地址主要用于 UI 展示与视觉识别；交易内部使用原始哈希。
  - **Encode flow**（Bytes -> String）：
    1. 将识别前缀追加到公钥哈希前：`[prefix][pkHash]`。
    2. 对结果执行双重哈希，取末尾 4 字节作为 checksum。
    3. 将 checksum 追加到公钥哈希后：`[pkHash][checksum]`。
    4. 编码为文本并在前方加上 prefix：`[prefix][encodedText]` => 账户地址。
  - **Decode flow**（String -> Bytes）：
    1. 去除识别前缀（长度已知），提取编码文本部分。
    2. 将文本解码为字节并拆分：`[pkHash][checksum#1]`。
    3. 将 prefix 重新前置到公钥哈希，并用同样的双重哈希重新得到
       `[checksum#2]`。
    4. 比较两个 checksum；一致则返回公钥哈希。
- **Errors**: 地址格式错误；checksum 不匹配；prefix 未知。

### 174 — `FN_PUBHASH`（单签公钥哈希）

- **Type**: function, public-key hashing（single-signature）
- **Operands**: 1 runtime argument — `pubKey` bytes
- **Stack effect**: `Bytes -> Bytes`
- **Description**: 从原始公钥导出单签公钥哈希。该过程是纯编码步骤；若要从
  账户地址恢复公钥哈希，应使用 `FN_ADDRESS` 的反向路径。
- **Errors**: 公钥字节格式错误。

### 175 — `FN_MPUBHASH`（复合多签公钥哈希）

- **Type**: function, composite public-key hashing（multisig）
- **Operands**: 3 runtime arguments — `pubHashes`, `m`, `n`
- **Stack effect**: `[]Bytes, byte, byte -> Bytes`
- **Description**: 计算 m-of-N 多签集合的复合公钥哈希。构造方式：
  - 按给定（确定性）顺序拼接输入的公钥哈希。
  - 在前面追加 2 字节 `m/n` 比例。
  - 对结果字节序列进行哈希，摘要即为复合哈希。
  注意：与 `FN_PUBHASH` 不同，这里的输入已是公钥哈希，而非原始公钥。
- **Errors**: `m > n`；哈希列表为空；哈希宽度不一致。

### 176 — `FN_CHECKSIG`（单签验证）

- **Type**: function, single-signature verification
- **Operands**: 4 runtime arguments — `chk_type`, `auth_flag`,
  `pubKey`, `sig`
- **Stack effect**: `byte, byte, Bytes, Bytes -> Bool`
- **Description**: 验证单个签名。
  - `chk_type`: `0` coin，`1` credit/proof。
  - `auth_flag`: 签名消息作用域/约束标记。
  - `pubKey`: 签名公钥。
  - `sig`: 签名字节。
  验证后，引擎还会在系统层记录本次结果（`true`/`false`），供后续 opcode
  （如 `SIGNED`）查询。
- **Errors**: 公钥/签名格式错误；未知 `chk_type`。

### 177 — `FN_MCHECKSIG`（多签验证）

- **Type**: function, multi-signature verification
- **Operands**: 5 runtime arguments — `chk_type`, `auth_flag`,
  `pubKeys`, `sigs`, `pubHashes`
- **Stack effect**: `byte, byte, []Bytes, []Bytes, []Bytes -> Bool`
- **Description**: 验证 m-of-N 多签：
  - `pubKeys` 与 `sigs` 按位置一一对应（每个签名对应其公钥）。
  - `pubHashes` 保存*未签名*参与者的公钥哈希，用于重建复合哈希并完成校验。
  完整验证算法见 `附.交易.md`（多签章节）。与 `FN_CHECKSIG` 一样，成功结果
  会记录到系统层。
- **Errors**: 参数形态不匹配；未达到阈值；复合哈希不匹配。

### 178 — `FN_HASH224(1)`（224-bit 参数化哈希）

- **Type**: function, parameterised hash（224-bit 输出）
- **Immediate**: 1 byte 算法选择器（在 `{}` 形式中可使用别名）
  - `0` — `BLAKE3`（默认）
  - `1` — `BLAKE2`（即 `BLAKE2b`）
  - `2` — `SHA-2`
  - `3` — `SHA-3`
- **Operands**: 1 runtime argument — 任意字节序列
- **Stack effect**: `Bytes -> Bytes`（28 bytes）

### 179 — `FN_HASH256(1)`（256-bit 参数化哈希）

- **Type**: function, parameterised hash（256-bit 输出）
- **Immediate**: 与 `FN_HASH224` 相同的算法选择器
- **Operands**: 1 runtime argument
- **Stack effect**: `Bytes -> Bytes`（32 bytes）
- **Examples**:
  ```go
  DATA{...} FN_HASH256{blake3}    // 使用 blake3.Sum256(...)
  DATA{...} FN_HASH256[0]         // 同上，默认选择器
  DATA{...} FN_HASH256            // 同上，最简写法
  ```
- **Note**: 在 64-bit x86 / ARM 上，`BLAKE2b` 通常比 `SHA-2` / `SHA-3`
  更快。

### 180 — `FN_HASH384(1)`（384-bit 参数化哈希）

- **Type**: function, parameterised hash（384-bit 输出）
- **Immediate**: 同一算法选择器
- **Operands**: 1 runtime argument
- **Stack effect**: `Bytes -> Bytes`（48 bytes）

### 181 — `FN_HASH512(1)`（512-bit 参数化哈希）

- **Type**: function, parameterised hash（512-bit 输出）
- **Immediate**: 同一算法选择器
- **Operands**: 1 runtime argument
- **Stack effect**: `Bytes -> Bytes`（64 bytes）

### 182–222（预留区间）

未使用。为基础集合后续扩展预留。

### 223 — `FN_PRINTF`（格式化打印）

- **Type**: function, formatted print
- **Operands**: 1 + variadic runtime arguments — `fmt`, `val ...any`
- **Stack effect**: `String, ... -> nil`
- **Description**: 向宿主控制台写入格式化消息（语义等同于 `fmt.Printf`）。
  与基础区块中的标准 `PRINT` opcode 不同，后者使用默认格式化。
- **Errors**: format/argument 不匹配不视为硬错误；输出为 best-effort。

## Instruction Catalog — Standard Function Set（指令目录——标准函数集）

```go
Code   Mnemonic       Form
-----------------------------------------------------------
224    FN_X(1){}               Standard-function reference
```

### 224 — `FN_X(1){}`（标准函数集分发）

- **Type**: function, dispatch into the standard function set
- **Immediate**: 1 byte 目标索引（花括号形式允许以函数符号名作为 immediate 的别名）
- **Operands**: 可变，由解析到的目标定义
- **Stack effect**: 取决于解析到的目标
- **Description**: 调用给定索引的标准函数。标准集合占用 1 字节地址空间（最多
  256 个条目）；每个条目都像普通函数 opcode 一样，拥有自己的参数个数与行为。
  标准集合条目仅由协议作者定义与发布。
- **Errors**: 索引未定义；解析目标的参数形态不匹配。

## Design Notes（设计说明）

- **Why a parameterised hash family.** 支持算法选择器可避免在 opcode 空间
  里固化单一算法；`BLAKE3` 作为默认值体现当前性能/安全折中，同时保留
  `SHA-2`/`SHA-3` 的兼容路径，并提供 `BLAKE2b` 的 x86/ARM 速度优势。
- **Composite multisig hash construction.** 在拼接前前置 `m/n` 比例，可将阈值
  绑定进摘要，使同一个哈希唯一标识 (`pubHashes`, `m`, `n`) 三元组，而非仅
  标识参与者集合。
- **`auth_flag` and consensus.** 该标志控制签名承诺的消息作用域；下游规范
  （交易模型、校验组）定义具体 bit 分配。
- **Standard set sizing.** `FN_X` 使用 1 字节索引，标准集合上限为 256 条目；
  若需继续扩展，必须使用扩展模块区间（`EXT_MO`，见 `18.Extension-Instructions.md`）。
