# Proposal: Function Instructions [170–209]（函数指令）

> **Prerequisite:** 本规范建立在 [Base Constraints](0.Base-Constraints.md) 文档之上，该文档定义了全局使用的指令格式、运行时模型与记号约定。


## 1. Overview（概览）

函数指令占用操作码区间 **[170–209]**（40 条）。它们带有 `FN_` 前缀，行为类似可调用函数：大多数从数据栈接收动态参数并直接返回结果，辅助参数很少或没有。

该区间分为两部分：

| Subrange | Name | Count | Description |
|----------|------|-------|-------------|
| 170–208 | Core Function Set | 39 | 用于编码、密码学、哈希与输出的内建函数 |
| 209 | Core Function Extension | 1 | 扩展函数定义的网关 |

### Design Principles（设计原则）

- **Function-like semantics:** 每条指令从数据栈（或参数空间）读取参数，并压入返回值。除 `FN_PRINTF` 外，无超出栈外的副作用。
- **Minimal auxiliary params:** 大多数函数指令辅助参数为 0 byte。少数使用 1 byte 选择算法或目标。
- **Bidirectional encoding:** Base 编码与 public-key-hash 指令会根据参数类型自动判定方向（encode / decode）。


## 2. Instruction Definitions — Core Function Set（指令定义——核心函数集）

### 2.1 Base58 Encode/Decode（Base58 编码/解码）

```
170     FN_BASE58   Base58 encode/decode
```

- **Arguments:** 一个值——编码时为 `Bytes`，解码时为 `String`。
- **Return value:** 编码返回 `String`，解码返回 `Bytes`。
- **Design notes:**
  - 方向由参数类型自动决定：`Bytes` → Base58 字符串；`String` → 从 Base58 解码为字节。
  - 使用 Bitcoin Base58 字母表（去除易混淆字符 `0`、`O`、`I`、`l`）。

### 2.2 Base32 Encode/Decode（Base32 编码/解码）

```
171     FN_BASE32   Base32 encode/decode
```

- **Arguments:** 一个值——编码时为 `Bytes`，解码时为 `String`。
- **Return value:** 编码返回 `String`，解码返回 `Bytes`。
- **Design notes:**
  - 遵循 RFC 4648 标准字母表（大写 `A–Z` + 数字 `2–7`）。
  - **No padding characters**——编码结果不附带 padding，解码也不要求 padding。

### 2.3 Base64 Encode/Decode（Base64 编码/解码）

```
172     FN_BASE64   Base64 encode/decode
```

- **Arguments:** 一个值——编码时为 `Bytes`，解码时为 `String`。
- **Return value:** 编码返回 `String`，解码返回 `Bytes`。
- **Design notes:**
  - 使用 **URL-safe** 字母表（`-`、`_` 替代 `+`、`/`）。
  - **No padding**——不会产生 `=`，解码也不要求 `=`。

### 2.4 Address Encoding（地址编码）

```
173     FN_ADDRESS  Encode public key hash to account address
```

- **Arguments:**
  1. `pbHash` (Bytes) ——公钥哈希。
  2. `prefix` (String) ——可读地址前缀。
- **Return value:** String——完整账户地址。
- **Encoding process:**
  1. 前缀拼接公钥哈希：`[prefix][pubkey_hash]`。
  2. 对结果做双重哈希，取**末尾 4 字节**作为 checksum。
  3. 将 checksum 追加到公钥哈希后：`[pubkey_hash][checksum]`。
  4. 对第 3 步字节序列做 Base58 编码，再加前缀：`[prefix][encoded_text]` → 账户地址。
- **Verification process (for reference):**
  1. 先按已知长度拆分 prefix 与编码文本。
  2. Base58 解码文本得字节，并拆为 `[pubkey_hash][checksum₁]`。
  3. 将 prefix 与 pubkey hash 拼接后双哈希，取末尾 4 字节得 `[checksum₂]`。
  4. 比较 `checksum₁` 与 `checksum₂`，一致即地址有效。
- **Example:**
  ```
  <pubkey_hash> "EC" FN_ADDRESS   // → "EC3K7mN..."
  ```

### 2.5 Public Key Hash / Address Decode（公钥哈希/地址解码）

```
174     FN_PUBHASH  Single-sig public key hash or address decode
```

- **Arguments:**
  1. `data` ——`Bytes`（原始公钥，做哈希）或 `String`（账户地址，做解码）。
  2. *(Optional)* `pfsize` (Int) ——地址前缀字节长度（解码地址时必需）。
- **Return value:** Bytes——公钥哈希。
- **Design notes:**
  - 方向由参数类型自动决定：`Bytes` → 计算单签公钥哈希；`String` → 解码地址并提取内嵌公钥哈希。
  - 解码地址时，指令**无法**判断原始哈希来自单签还是多签，只负责提取哈希字节。
  - 可选第二参数 `pfsize` 用于告知前缀长度，便于定位编码区起点。

### 2.6 Multi-Sig Public Key Hash（多签公钥组合哈希）

```
175     FN_MPUBHASH Create multi-sig composite public key hash
```

- **Arguments:**
  1. `pubHash` ([]Bytes) ——所有参与者的公钥哈希。
  2. `m` (Int) ——最少所需签名数。
  3. `n` (Int) ——参与者总数，必须满足 `m ≤ n`。
- **Return value:** Bytes——多签组合公钥哈希。
- **Design notes:**
  - **Creation only**——不支持从组合哈希反向解析出参与者哈希列表。
  - 组合哈希会以确定性方式编码参与者哈希集合与 `m-of-n` 阈值参数。

### 2.7 Single Signature Verification（单签校验）

```
176     FN_CHECKSIG Single signature verification
```

- **Arguments:**
  1. `flag` (Byte) ——认证标志，指示签名方案或哈希子集。
  2. `sig` (Bytes) ——待校验签名。
  3. `pubKey` (Bytes) ——用于校验的公钥。
- **Return value:** Bool——签名有效返回 `true`。
- **Example:**
  ```
  {0x01} <signature> <pubkey> FN_CHECKSIG PASS
  ```

### 2.8 Multi-Signature Verification（多签校验）

```
177     FN_MCHECKSIG    Multi-signature verification
```

- **Arguments:**
  1. `flag` (Byte) ——认证标志。
  2. `sigs` ([]Bytes) ——待校验签名集合。
  3. `pubKeys` ([]Bytes) ——公钥集合，与 `sigs` 位置一一对应。
  4. `pubHashs` ([]Bytes) ——未签名参与者的公钥哈希。
- **Return value:** Bool——若所有给定签名都能与对应公钥匹配，则为 `true`。
- **Design notes:**
  - 数据栈中的 `[]Bytes` 表示为 `[]any`，其中每个元素是 `Bytes`。
  - `pubKeys` 长度必须与 `sigs` 相同——每个签名按位置与对应公钥校验。
  - `pubHashs` 提供未签名参与者哈希，用于重建完整参与者集合并校验组合哈希。

### 2.9 Hash-224（224 位哈希）

```
178     FN_HASH224(1)   224-bit hash
```

- **Auxiliary params:** 1 byte——算法标识。
- **Arguments:** `data` (Bytes) ——待哈希输入。
- **Return value:** Bytes——28-byte 哈希摘要。
- **Algorithm identifiers:**

  | ID | Algorithm | Notes |
  |----|-----------|-------|
  | 0 (default) | SHA3-224 | FIPS-202 标准 |
  | 1 | SHA2-224 | 兼容遗留场景 |
  | 2 | BLAKE2s | 通过 BLAKE2s 输出 224-bit |

- **Example:**
  ```
  <data> FN_HASH224          // SHA3-224（默认）
  <data> FN_HASH224[1]       // SHA2-224
  ```

### 2.10 Hash-256（256 位哈希）

```
179     FN_HASH256(1)   256-bit hash
```

- **Auxiliary params:** 1 byte——算法标识。
- **Arguments:** `data` (Bytes) ——待哈希输入。
- **Return value:** Bytes——32-byte 哈希摘要。
- **Algorithm identifiers:**

  | ID | Algorithm | Notes |
  |----|-----------|-------|
  | 0 (default) | SHA3-256 | FIPS-202 标准 |
  | 1 | SHA2-256 | 兼容遗留场景 |
  | 2 | BLAKE2b-256 | 安全性优于 SHA2；在 64-bit x86/ARM 更快 |

- **Design notes:**
  - 在现代 64-bit 架构（x86-64、ARMv8）上，BLAKE2b 比 SHA2 具有更好的安全裕度和更高性能。

### 2.11 Hash-384（384 位哈希）

- **Auxiliary params:** 1 byte——算法标识。

- **Algorithm identifiers:**

- **Arguments:** `data` (Bytes) ——待哈希输入。
- **Return value:** Bytes——64-byte 哈希摘要。
- **Algorithm identifiers:**

### 2.13 Reserved (182–207)（保留）

操作码 **182–207**（26 个槽位）保留给未来核心函数扩展，当前未分配，实现中不得使用。

### 2.14 Formatted Print（格式化输出）

```
208     FN_PRINTF   Console print with formatting
```

- **Arguments:**
  1. `fmt` (String) ——格式字符串（遵循 `fmt.Printf` 约定）。
  2. `val` (...any) ——与格式占位符匹配的可变参数值。
- **Return value:** 无（`nil`）。
- **Design notes:**
  - 向控制台/日志输出格式化文本。它是函数指令中唯一带副作用的指令。
  - 与标准 `PRINT` 指令（基础指令集）不同，`PRINT` 仅输出单值且不支持格式化。
- **Example:**
  ```
  "Balance: %d EC" {1000} FN_PRINTF   // 输出: Balance: 1000 EC
  ```


## 3. Instruction Definition — Core Function Extension（指令定义——核心函数扩展）

### 3.1 Extended Function Reference（扩展函数引用）

```
209     FN_X(1){}   Extended function reference
```

- **Auxiliary params:** 1 byte——目标函数索引（`[0–255]`）。
- **Associated data:** 可选目标名称（使用 `{}` 记法时）。
- **Arguments:** 由目标扩展函数决定。
- **Return value:** 由目标扩展函数决定。
- **Design notes:**
  - 作为扩展函数表的分发网关。1-byte 辅助参数可寻址最多 **256** 个扩展函数槽位。
  - `{}` 记法支持按名称引用扩展函数以提升可读性：`FN_X{MyFunc}`。
  - 当需超过 256 个扩展函数时，使用扩展指令中的 `EX_FN`（opcode 250），其提供 2-byte（64K）地址空间。
- **Example:**
  ```
  <args...> FN_X[12]        // 调用索引 12 的扩展函数
  <args...> FN_X{Encrypt}   // 调用名为 "Encrypt" 的扩展函数
  ```


## 4. Design Notes（设计说明）

### Encoding Function Family（编码函数族）

三个 Base 编码指令（`FN_BASE58`、`FN_BASE32`、`FN_BASE64`）采用一致的双向设计：仅凭参数类型决定执行方向。这避免了额外的 encode/decode 成对操作码，使指令集更紧凑。

### Hash Function Family（哈希函数族）

四个哈希指令（`FN_HASH224` 到 `FN_HASH512`）在三类算法（SHA3、SHA2、BLAKE2）上提供统一接口。所有长度默认使用 SHA3，符合 FIPS-202，且具备更好的抗量子裕度。BLAKE2 作为高性能替代，适合非共识关键场景。

### Cryptographic Verification（密码学校验）

`FN_CHECKSIG` 与 `FN_MCHECKSIG` 是交易校验的核心。`flag` byte 支持未来扩展到更多签名方案，而不需要新增操作码。
