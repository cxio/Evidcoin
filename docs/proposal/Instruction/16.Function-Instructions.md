# Proposal: Function Instructions [170–224]（函数指令）

> **Prerequisite:** 本规范建立在 [Base Constraints](0.Base-Constraints.md) 文档之上，该文档定义了全局使用的指令格式、运行时模型与记号约定。


## 1. Overview（概览）

函数指令占用操作码区间 **[170–224]**（55 条）。它们带有 `FN_` 前缀，行为类似可调用函数：大多数从数据栈接收动态参数并直接返回结果，辅助参数很少或没有。

该区间分为两部分：

| Subrange | Name | Count | Description |
|----------|------|-------|-------------|
| 170–223 | Core Function Set | 54 | 用于编码、密码学、哈希与输出的内建函数 |
| 224 | Core Function Extension | 1 | 扩展函数定义的网关 |

### Design Principles（设计原则）

- **Function-like semantics:** 每条指令从数据栈（或参数空间）读取参数，并压入返回值。除 `FN_PRINTF` 外，无超出栈外的副作用。
- **Minimal auxiliary params:** 大多数函数指令辅助参数为 0 byte。少数使用 1 byte 选择算法或目标。
- **Bidirectional encoding:** Base 编码指令与 `FN_ADDRESS` 会根据参数类型自动判定方向（encode / decode）。


## 2. Instruction Definitions — Core Function Set（指令定义——核心函数集）

### 2.1 Base58 Encode/Decode（Base58 编码/解码）

```
170     FN_BASE58   Base58 encode/decode
```

- **Arguments:** 一个值——编码时为 `Bytes`，解码时为 `String`。
- **Return value:** 编码返回 `String`，解码返回 `Bytes`。
- **Design notes:**
  - 方向由参数类型自动决定：`Bytes` → Base58 字符串；`String` → 从 Base58 解码为字节。
  - 使用 Bitcoin Base58 字母表（去除易混淆字符 `0`、`O`、`I`、`l`）。

### 2.2 Base32 Encode/Decode（Base32 编码/解码）

```
171     FN_BASE32   Base32 encode/decode
```

- **Arguments:** 一个值——编码时为 `Bytes`，解码时为 `String`。
- **Return value:** 编码返回 `String`，解码返回 `Bytes`。
- **Design notes:**
  - 遵循 RFC 4648 标准字母表（大写 `A–Z` + 数字 `2–7`）。
  - **No padding characters**——编码结果不附带 padding，解码也不要求 padding。

### 2.3 Base64 Encode/Decode（Base64 编码/解码）

```
172     FN_BASE64   Base64 encode/decode
```

- **Arguments:** 一个值——编码时为 `Bytes`，解码时为 `String`。
- **Return value:** 编码返回 `String`，解码返回 `Bytes`。
- **Design notes:**
  - 使用 **URL-safe** 字母表（`-`、`_` 替代 `+`、`/`）。
  - **No padding**——不会产生 `=`，解码也不要求 `=`。

### 2.4 Address Encoding/Decoding（地址编码/解码）

```
173     FN_ADDRESS  Encode public key hash to account address or decode address to public key hash
```

- **Arguments:**
  1. `data` (Bytes|String) ——公钥哈希（编码时）或账户地址（解码时）。
  2. `prefix` (String) ——识别前缀。
- **Return value:** String|Bytes——账户地址（编码时）或公钥哈希（解码时）。
- **Design notes:**
  - 方向由参数类型自动判定：`Bytes` → 编码为地址字符串；`String` → 解码为公钥哈希。
  - 文本形式的账户地址主要用于 UI 展示和视觉识别。
- **Encoding process（输入 Bytes → 输出 String）:**
  1. 公钥哈希添加识别前缀：`[prefix][pubkey_hash]`。
  2. 对结果执行 2 次哈希运算，取**末尾 4 字节**作为 checksum。
  3. 将 checksum 追加到公钥哈希后：`[pubkey_hash][checksum]`。
  4. 对第 3 步字节序列做 Base58 编码，再加前缀：`[prefix][encoded_text]` → 账户地址。
- **Decoding process（输入 String → 输出 Bytes）:**
  1. 分离识别前缀（长度已知）和编码文本。
  2. Base58 解码文本得字节序列，并拆为 `[pubkey_hash][checksum₁]`。
  3. 将 prefix 与 pubkey hash 拼接后执行 2 次哈希运算，取末尾 4 字节得 `[checksum₂]`。
  4. 比较 `checksum₁` 与 `checksum₂`，一致即地址有效，返回公钥哈希。
- **Example:**
  ```
  <pubkey_hash> "EC" FN_ADDRESS     // → "EC3K7mN..."（编码）
  "EC3K7mN..." "EC" FN_ADDRESS      // → <pubkey_hash>（解码）
  ```

### 2.5 Public Key Hash Creation（公钥哈希创建）

```
174     FN_PUBHASH  Single-sig public key hash creation
```

- **Arguments:**
  1. `pubKey` (Bytes) ——公钥。
- **Return value:** Bytes——公钥哈希。
- **Design notes:**
  - 这是单纯的编码（哈希）操作，从公钥生成对应的公钥哈希。
  - 若需将账户地址解码为公钥哈希，请使用 `FN_ADDRESS` 的反向操作。
- **Example:**
  ```
  <pubkey> FN_PUBHASH   // → <pubkey_hash>
  ```

### 2.6 Multi-Sig Public Key Hash（多签公钥组合哈希）

```
175     FN_MPUBHASH Create multi-sig composite public key hash
```

- **Arguments:**
  1. `pubHash` ([]Bytes) ——所有参与者的公钥哈希。
  2. `m` (Int) ——最少所需签名数。
  3. `n` (Int) ——参与者总数，必须满足 `m ≤ n`。
- **Return value:** Bytes——多签组合公钥哈希。
- **Design notes:**
  - 类似 `FN_PUBHASH`，这也是纯创建操作。
  - 组合哈希会以确定性方式编码参与者哈希集合与 `m-of-n` 阈值参数。

### 2.7 Single Signature Verification（单签校验）

```
176     FN_CHECKSIG Single signature verification
```

- **Arguments:**
  1. `chk_type` (Byte) ——验证类型：`0` 为币金花费，`1` 为凭信转移。
  2. `auth_flag` (Byte) ——授权标识，配置签名消息的约束范围。
  3. `pubKey` (Bytes) ——用于校验的公钥。
  4. `sig` (Bytes) ——待校验签名数据。
- **Return value:** Bool——签名有效返回 `true`。
- **Design notes:**
  - 验证成功后，会在系统层标注验证结果（`true|false`）。
  - `chk_type` 仅区分两种情况：币金花费、凭信转移；高位字节保留供未来扩展。
- **Example:**
  ```
  {0} {0xFF} <pubkey> <signature> FN_CHECKSIG PASS
  ```

### 2.8 Multi-Signature Verification（多签校验）

```
177     FN_MCHECKSIG    Multi-signature verification
```

- **Arguments:**
  1. `chk_type` (Byte) ——验证类型：`0` 为币金花费，`1` 为凭信转移。
  2. `auth_flag` (Byte) ——授权标识，配置签名消息的约束范围。
  3. `pubKeys` ([]Bytes) ——签名公钥集合。
  4. `sigs` ([]Bytes) ——签名数据集合，与 `pubKeys` 成员一一对应。
  5. `pubHashs` ([]Bytes) ——未参与签名的剩余公钥哈希集合。
- **Return value:** Bool——若所有给定签名都能与对应公钥匹配，则为 `true`。
- **Design notes:**
  - `pubKeys` 长度必须与 `sigs` 相同——每个签名按位置与对应公钥校验。
  - `pubHashs` 提供未签名参与者哈希，用于重建完整参与者集合并校验组合哈希。
  - `m/N` 配比可由 `pubKeys` 大小（m）和 `pubHashs` 大小（N-m）计算得出。
  - 验证成功后，会在系统层标注验证结果（`true|false`）。
  - `chk_type` 设计同 `FN_CHECKSIG`，高位字节保留供未来扩展。
  - 验证流程详见「附.交易.md」多重签名章节。

### 2.9 Hash-224（224 位哈希）

```
178     FN_HASH224(1)   224-bit hash
```

- **Auxiliary params:** 1 byte——算法标识。
- **Arguments:** `data` (Bytes) ——待哈希输入。
- **Return value:** Bytes——28-byte 哈希摘要。
- **Algorithm identifiers:**

  | ID | Algorithm | Notes |
  |----|-----------|-------|
  | 0 (default) | BLAKE3 | 默认算法 |
  | 1 | BLAKE2 | 即 blake2b 算法 |
  | 2 | SHA2 | 兼容遗留场景 |
  | 3 | SHA3 | FIPS-202 标准 |

- **Example:**
  ```
  <data> FN_HASH224         // BLAKE3-224（默认）
  <data> FN_HASH224{blake3} // 采用 blake3.Sum224(...)
  <data> FN_HASH224[0]      // 同上，默认算法
  <data> FN_HASH224[1]      // blake2b.Sum224(...)
  ```

### 2.13 Reserved (182–222)（保留）

操作码 **182–222**（41 个槽位）保留给未来核心函数扩展，当前未分配，实现中不得使用。

### 2.14 Formatted Print（格式化输出）

```
223     FN_PRINTF   Console print with formatting
```

- **Arguments:**
  1. `fmt` (String) ——格式字符串（遵循 `fmt.Printf` 约定）。
  2. `val` (...any) ——与格式占位符匹配的可变参数值。
- **Return value:** 无（`nil`）。
- **Design notes:**
  - 向控制台/日志输出格式化文本。它是函数指令中唯一带副作用的指令。
  - 与标准 `PRINT` 指令（基础指令集）不同，`PRINT` 仅输出单值且不支持格式化。
- **Example:**
  ```
  "Balance: %d EC" {1000} FN_PRINTF   // 输出: Balance: 1000 EC
  ```


## 3. Instruction Definition — Core Function Extension（指令定义——核心函数扩展）

### 3.1 Extended Function Reference（扩展函数引用）

```
224     FN_X(1){}   Extended function reference
```

- **Auxiliary params:** 1 byte——目标函数索引（`[0–255]`）。
- **Associated data:** 可选目标名称（使用 `{}` 记法时）。
- **Arguments:** 由目标扩展函数决定。
- **Return value:** 由目标扩展函数决定。
- **Design notes:**
  - 作为扩展函数表的分发网关。1-byte 辅助参数可寻址最多 **256** 个扩展函数槽位。
  - `{}` 记法支持按名称引用扩展函数以提升可读性：`FN_X{MyFunc}`。
  - 标准函数集仅支持1字节空间，最多容纳256个函数指令，通常仅由官方定义和发布。
  - 当需超过 256 个扩展函数时，可使用扩展指令中的 `EXT_MO`（opcode 251）作为模块分类函数提供，其提供 2-byte（64K）地址空间。
- **Example:**
  ```
  <args...> FN_X[12]        // 调用索引 12 的扩展函数
  <args...> FN_X{Encrypt}   // 调用名为 "Encrypt" 的扩展函数
  ```


## 4. Design Notes（设计说明）

### Encoding Function Family（编码函数族）

三个 Base 编码指令（`FN_BASE58`、`FN_BASE32`、`FN_BASE64`）以及地址编码指令（`FN_ADDRESS`）采用一致的双向设计：仅凭参数类型决定执行方向。这避免了额外的 encode/decode 成对操作码，使指令集更紧凑。

### Hash Function Family（哈希函数族）

四个哈希指令（`FN_HASH224` 到 `FN_HASH512`）在多类算法（SHA3、SHA2、BLAKE2、BLAKE3）上提供统一接口。所有长度默认使用 SHA3，符合 FIPS-202，且具备更好的抗量子裕度。BLAKE2 和 BLAKE3 作为高性能替代，适合非共识关键场景。

### Public Key Hash Function Family（公钥哈希函数族）

`FN_PUBHASH` 与 `FN_MPUBHASH` 专司公钥哈希的创建，不支持反向解析。地址编码/解码由 `FN_ADDRESS` 统一处理，包括目标地址合法性校验。这样的分离使各函数职责清晰，避免过度的多向操作。

### Cryptographic Verification（密码学校验）

`FN_CHECKSIG` 与 `FN_MCHECKSIG` 是交易校验的核心。`flag` byte 支持未来扩展到更多签名方案，而不需要新增操作码。
