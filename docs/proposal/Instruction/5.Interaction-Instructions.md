# Interaction Instructions（交互指令）

Status: Draft proposal（提案草案）
Source conception（来源构想）: `conception/Instruction/5.交互指令.md`

## Prerequisites（前置条件）

关于 stack model、argument zone、local frame 与 opcode layout，请参见 `0.Base-Constraints.md`。简述如下：

- **data stack** 用于保存中间值；**argument zone** 用于为下一条指令暂存输入；**local frame** 与 **global frame** 用于保存作用域变量。
- 每条指令均为 `opcode + immediate + associated data`；其中只有 opcode 是必需项。
- 本指令族暴露两个 **I/O buffers**（FIFO 队列）：`INPUT` 与 `OUTPUT`。二者彼此独立，且都独立于 data stack / argument zone。

## Overview（概览）

交互指令族通过两个单向、FIFO 的 **I/O buffers** 将 script 与其宿主环境连接起来：`INPUT`（host → script）与 `OUTPUT`（script → host）。两个 buffer 相互隔离；生产者从尾部追加，消费者从头部读取。专用 opcode `BUFDUMP` 会将 `OUTPUT` buffer 刷新到已注册 handler，`PRINT` opcode 会向 host console 输出诊断信息。公共校验节点不会向 `INPUT` 注入数据，因此在公共节点上执行任何 `INPUT` 读取都会表现为隐式且干净的 `END`。

## Opcode range（操作码范围）

`[46 – 50]`，共 5 条指令（其中 1 个 slot 预留）。

## Instruction Catalog（指令目录）

### 46 — `INPUT(1)` — import from the INPUT buffer (blocking)（从 INPUT buffer 导入，阻塞）

| Field | Value |
|---|---|
| Type | Ingress |
| Immediate | 读取项数；`0` 表示“读取全部可用项” |
| Runtime args | none |
| Stack effect | 将可变数量的值压入目标区域（auto-spread） |

**Description.** 从 `INPUT` buffer 头部读取。若 buffer 为空或项目数少于请求值，script 将*正常退出*（不抛出错误）——当前 pass status 保持有效。

```go
@INPUT      // 将整个 buffer 导入 argument zone
INPUT[3]    // 导入前 3 项，并展开压入 data stack
@INPUT NOP  // 清空 buffer（丢弃内容）
```

**Design.**
- `INPUT` 失败是 *clean exit*，不是 failure。凡是已经通过的状态（`PASS` / 最近一次 `CHECK` 为 true）都保持通过。这使公共 validator 的行为可预测：它不会注入 `INPUT`，因此执行到 `INPUT` 时 script 仅以既有 pass status 终止，等价于隐式 `END`。
- 私有节点如果*需要* `INPUT`，必须在调用 script 前先把数据注入该 buffer。

---

### 47 — `OUTPUT` — export argument-zone contents to the OUTPUT buffer（将 argument zone 内容导出到 OUTPUT buffer）

| Field | Value |
|---|---|
| Type | Egress |
| Runtime args | variadic —— 追加 argument zone 中当前全部项 |
| Stack effect | 清空 argument zone；并追加到 `OUTPUT` 尾部 |

非阻塞。重复调用 `OUTPUT` 会持续累积到 buffer，直到 `BUFDUMP` 执行刷新。

```go
@POPS OUTPUT      // 导出整个 data stack
... OUTPUT        // argument zone 为空时，导出 0 项（no-op）
```

---

### 48 — `BUFDUMP(1)` — flush the OUTPUT buffer（刷新 OUTPUT buffer）

| Field | Value |
|---|---|
| Type | Egress / signal |
| Immediate | 用户自定义 tag 值（1 byte），用于标记 dump 点 |
| Runtime args | none |
| Stack effect | 对 data stack 无影响；清空 `OUTPUT` buffer |

**Description.** 以原子方式将当前 `OUTPUT` buffer 传递给已注册 handler，并清空该 buffer。非阻塞。tag 可让接收方识别数据属于哪个逻辑“dump 点”。

```go
@POPS[3] OUTPUT BUFDUMP
// 将栈顶 3 项弹入 argument zone，推送到 OUTPUT，然后执行刷新。
```

**Note.** handler 必须由外部注册。handler 属于第三方私有逻辑，仅需部署在消费该 dump 的节点上。

---

### 49 — *(reserved)*（预留）

该 slot 故意保留未用，以支持前向兼容。

---

### 50 — `PRINT` — print to the host console（打印到 host console）

| Field | Value |
|---|---|
| Type | Diagnostic |
| Runtime args | variadic —— argument zone 中全部项 |
| Stack effect | 清空 argument zone |

等价于在 host 侧以默认格式调用 `fmt.Println`。

```go
@TOP     PRINT   // 打印栈顶项
@POPS[3] PRINT   // 打印栈顶 3 项
```

## Design notes（设计说明）

- **Two buffers, one direction each.** `INPUT`（从 script 视角只读）与 `OUTPUT`（只写）的清晰分离，使 host/script 协议易于分析。
- **`INPUT` failure is "clean exit", not "script failure".** 这是有意做出的易用性设计，使*同一份 script bytes*既可运行于公共 validator（无输入，按既有 pass status 干净退出），也可运行于私有节点（预装输入后正常继续）。
- **`OUTPUT` is non-blocking and buffered.** 多次 `OUTPUT` 会先累积，直到 `BUFDUMP` 统一提交，使 script 可组装一条连贯消息。
- **`BUFDUMP` carries a tag**，接收方可据此路由或标记，而无需解析 payload 内容。
- **Interaction with `GOTO` external scripts.** `INPUT` / `OUTPUT` buffers *不是* caller 与 `GOTO` 目标 external script 之间的通道（见下文）。

## Appendix — Interaction with GOTO external scripts（附录：与 GOTO external scripts 的交互）

这两个 I/O buffers **不是** caller ↔ `GOTO` 通信通道。正确机制如下：

- **Caller → GOTO-script:** caller 通过 `GOTO` argument zone 传入初始数据，该数据会成为 callee 的初始 data stack。
- **GOTO-script → Caller:** 由于 callee script 在编辑时已固定且可知，caller 所需内容可以直接内联。callee 内部的 `PASS` / `CHECK` 结果会直接传播到 caller 的 pass status。

若允许 `INPUT` / `OUTPUT` 桥接到 `GOTO` callee，将显著增加脚本语义复杂度（跨环境数据注入、顺序、隔离）。当前协议有意禁止该行为。
