# Proposal: Interaction Instructions [46–50]（提案：交互指令 [46–50]）

> **Prerequisite:** 本规范建立在 [Base Constraints](0.Base-Constraints.md) 文档之上；该文档定义了全局采用的指令格式、运行时模型与记号约定。


## 1. Overview（概览）

交互指令占用 opcode 范围 **[46–50]**（共 5 条）。它们通过 **import/export buffers**（FIFO 队列）管理脚本执行环境与外部系统之间的数据交换，这些缓冲区即脚本的 I/O 通道。

该缓冲机制为脚本提供了与**链外**（off-chain）环境交互的受控接口：可从外部注入方接收数据，也可将数据导出至外部第三方 handler 处理。两个缓冲区彼此完全独立，不互通。

### Buffer Architecture（缓冲区架构）

```graph
External Source ──► [Import Buffer (FIFO)] ──► Script (INPUT)

Script (OUTPUT) ──► [Export Buffer (FIFO)] ──► BUFDUMP ──► External Handler
```

- **Import Buffer:** 从脚本视角是只读。数据由外部环境注入，`INPUT` 从中消费（dequeue）。
- **Export Buffer:** 从脚本视角是只写。数据由 `OUTPUT` 写入（enqueue），由 `BUFDUMP` 刷出到外部 handler。


## 2. Instruction Definitions（指令定义）

### 2.1 Import Data（导入数据）

```
46      INPUT(1)    Import from buffer
```

- **Auxiliary params:** 1 byte —— 要读取的条目数（`[0–255]`），`0` 表示读取全部可用数据。
- **Arguments:** 无。
- **Return value:** 可变数量 —— 条目会自动展开到目标空间。
- **Blocking behavior:** 若 import buffer 为空，或条目数少于请求值，执行将**失败并正常退出**（不是 error termination）。
- **Examples:**
  - `@INPUT` —— 导入缓冲区全部数据，并展开到实参空间。
  - `INPUT[3]` —— 从缓冲区头部导入 3 个条目，并展开到 data stack。
  - `@INPUT NOP` —— 消费并丢弃 import buffer 的全部内容。
- **Design note — graceful failure:** 导入失败被视为**正常退出**，并保留此前已达成的 pass-state。这一点很关键，因为：
  - 对永不提供导入数据的公共验证节点而言，`INPUT` 等价于隐式 `END` —— 脚本会优雅结束，不影响先前 `PASS` 的结果。
  - 这让脚本可自然分成“公共验证段”与“私有交互段”：仅当输入数据存在时，后者才会继续执行。

### 2.2 Export Data（导出数据）

```
47      OUTPUT      Export to buffer
```

- **Auxiliary params:** 无
- **Arguments:** 可变数量。
- **Return value:** 无。
- 将实参空间中的全部内容按 FIFO 顺序转移到 export buffer。非阻塞。
- 若实参空间为空，则导出 0 个条目。
- **Examples:**
  - `@POPS OUTPUT` —— 弹出全部栈项并导出到缓冲区。
  - `... OUTPUT` —— 若实参空间为空，则不会导出任何内容。

### 2.3 Buffer Dump (Flush Export)（缓冲区转储/刷出导出缓冲）

```
48      BUFDUMP(1)  Flush export buffer
```

- **Auxiliary params:** 1 byte —— 用户自定义 tag/identifier。
- **Arguments:** 无。
- **Return value:** 无。
- 将 export buffer 的内容刷出（并清空）到外部 handler。非阻塞。
- 辅助参数充当**位置标记**：用于标识该次 dump 点，使外部接收方可区分同一脚本内多次 dump。
- **External handler:** 若需要处理 `BUFDUMP` 数据，外部应注册第三方 handler 接收转储数据。handler 属于私有部署逻辑，仅有处理需求的节点才会部署。
- **Example:**
  ```
  @POPS[3] OUTPUT BUFDUMP
  ```
  弹出 3 个条目 → 导出到缓冲区 → 刷出到外部 handler。

### 2.4 Reserved（保留）

```
49      (Reserved, unused)
```

预留 opcode，供未来扩展使用。

### 2.5 Print (Debug Output)（打印/调试输出）

```
50      PRINT       Print to console
```

- **Auxiliary params:** 无
- **Arguments:** 可变数量。
- **Return value:** 无。
- 以默认格式将参数空间内容输出到控制台（等价于 `fmt.Println()`）。
- 主要用于调试与开发。
- **Examples:**
  - `@TOP PRINT` —— 打印栈顶项。
  - `@POPS[3] PRINT` —— 打印栈顶 3 项。


## 3. Buffer Independence（缓冲区独立性）

INPUT 与 OUTPUT 连接的是两个**各自独立的通道**，互不相通。它们专门服务于**链外（off-chain）事务**，而不用于 `GOTO`/`EMBED` 的跨脚本通信。

| Channel | Direction | Purpose |
|---------|-----------|----------|
| Import Buffer | External → Script | 外部环境向脚本注入数据（需事先注入：先注入，后执行脚本） |
| Export Buffer | Script → External | 脚本向外部 handler 推送数据（链外驱动） |

**BUFDUMP 监听是私有的**：handler 属于按需部署的第三方私有逻辑，只有有处理需求的节点才会部署，公共验证节点无需部署。


## 4. Interaction Pattern: Public vs. Private Validation（交互模式：公共验证与私有验证）

`INPUT` 的 graceful-failure 语义可支持“两阶段脚本模式”：

```
// Phase 1: Public validation (all nodes execute)
<signature verification>
<hash checks>
PASS

// Phase 2: Private interaction (only participating nodes)
INPUT[2]              // 公共节点在此优雅失败并结束
<private logic using imported data>
OUTPUT
BUFDUMP[1]
```

公共验证节点执行到 `INPUT` 时若无数据，会正常退出，并保留 `PASS` 状态。提供了导入数据的参与节点则继续执行私有逻辑。

该设计无需显式分支或条件编译，即可优雅地把共识关键验证与应用特定数据处理解耦。
