# Extension Instructions（扩展指令）

Status: Draft proposal（状态：草案提案）
Source conception: `conception/Instruction/18.扩展指令.md`（来源构想）

## Prerequisites（前置条件）

关于 stack model、local frame、opcode layout、immediate-operand 记法 `(n)`
以及 associated-data 记法 `{}`，请参见 `0.Base-Constraints.md`。

相关概念回顾：

- extension block 位于 opcode space 顶端，提供超出 base、function、
  module blocks 的通用扩展容量。
- extension opcodes 不属于 *base* instruction set，但它们是 engine
  灵活性边界中的关键组成部分。
- 最高的两个 opcode 值（`254` / `255`）为 **system-reserved**，
  完全不属于 instruction set。

## Overview（概述）

本区块定义两个 extension dispatch opcode：`EXT_MO` 与 `EXT_PRIV`。
其中，`EXT_MO` 提供一个 64K-slot 的 extension module address space
（可视为 block 17 标准 module 范围的超集/泄压阀）；`EXT_PRIV`
提供私有 extension dispatch，保留给调用方自定义需求，public
consensus path 对其不解释，也**不得**解释。`252` 号保留未使用。

## Opcode Range（Opcode 范围）

| Range     | Count | Notes                                                       |
|-----------|-------|-------------------------------------------------------------|
| `251..253`| 3     | 已启用的 extension opcodes（1 个未使用：`252`）             |
| `254..255`| 2     | System-reserved（不属于 instruction set）                  |
| **Total** | **3** | Block `[251..253]`                                          |

## Instruction Catalog（指令目录）

```go
Code   Mnemonic       Form
-----------------------------------------------------------
251    EXT_MO(2){}             扩展模块引用（64K 槽位）
252                            （未使用）
253    EXT_PRIV(2){}           私有扩展引用
254-255                        （system-reserved；不在指令集中）
```

### 251 — `EXT_MO(2){}`（扩展模块分派）

- **Type**: extension，module dispatch
- **Immediate**: 2 bytes — extension-module index
- **Operands**: 由解析后的目标 module constructor 定义
- **Stack effect**: `-> Module`
- **Description**: 通过 16-bit index 分派到 extension module，提供专用于
  extension modules 的 64K-slot module address space。其行为与 block 17
  中的 `MO_XX` 类似，但 index 范围大得多。extension modules 遵循与
  basic / standard modules 相同的 constructor + `CALL` 分派约定。
- **Errors**: extension-module index 未定义。

### 252（未使用）

未使用。保留给未来的 extension dispatch opcode。

### 253 — `EXT_PRIV(2){}`（私有扩展分派）

- **Type**: extension，private dispatch（仅 private nodes）
- **Immediate**: 2 bytes — private-extension index
- **Operands**: 由解析后的 private extension 定义
- **Stack effect**: 由实现定义
- **Description**: 提供 private extension dispatch 槽位，用于调用方特定需求，
  public validator nodes 不对其进行解释。public consensus path 在正常运行
  中 **not** 会触达该 opcode：该 opcode 合法出现的位置，仅在脚本显式 `END`
  之后，或在 `INPUT` 处触发的隐式 `END` 之后。**If `EXT_PRIV` appears before
  `END` / `INPUT`, the script is treated as a hard error and fails.** 在
  `END` 之后区域，private extension 可由发行方完全自定义。
- **Errors**: 在 `END` / `INPUT` 之前出现（hard script failure）；
  执行中的 private node 上 private-extension index 未定义。

### 254–255 — System-reserved（系统保留）

为未来系统用途保留。这两个 opcode 值被刻意设定为
**not** 属于 instruction set，且不得通过常规 extension 流程分配。
若要分配任一值，必须进行 hard upgrade 并同步修订规范。

## Design Notes（设计说明）

- **`EXT_MO` vs. `MO_XX`.** `MO_XX`（block 17）通过单字节寻址 256 个
  standard modules；当标准集合不足时，`EXT_MO` 提供 64K 的 extension
  module address space。此分离设计将高密度槽位保留在标准带内，同时让
  extension 在独立空间增长，避免挤占 base opcode 预算。
- **Public-vs-private gating for `EXT_PRIV`.** `END` 之前拒绝规则，是该
  opcode 可安全使用的契约：public validator nodes 可确认脚本中任何与
  consensus 相关的前缀都不含 private-extension 语义，而发行方在 `END`
  之后区域仍保有灵活的 private-side extension 能力。
- **Reservation discipline.** 将 `254`/`255` 保持在 instruction set 之外，
  可为潜在结构性变化（如 opcode-space escape sequences）保护两个连续的
  高位 opcode 值。
- **Open design surface.** 各 extension modules 与 private extensions 的
  具体语义有意后置；本区块仅定义 *dispatch* 契约，不定义被分派目标的内容。
