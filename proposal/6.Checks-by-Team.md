# Team-Based Verification（组队校验）

> **Design Proposal** — 本文档描述 Evidcoin 的组队交易校验架构，涵盖校验团队结构、角色分配、冗余与复核机制、组间协作、铸造集成、UTXO/UTCO 指纹、区块发布与安全保障。


## 1. Overview（概述）

传统区块链采用“单节点”模型：每个全节点独立校验每笔交易。随着交易量增长，这对参与者的硬件要求不断提高，并限制系统吞吐上限。

Evidcoin 用 **Team-Based Verification** 取代该模型——由一组节点（*verification team*）协同校验整块交易的合作式架构。交易天然相互独立，使团队成员并行分发既合理又高效。

核心设计原则：

| 原则 | 描述 |
|-----------|-------------|
| **Division over redundancy** | 全量去中心化冗余并非必要，责任分工即可满足安全性。 |
| **Micro-centralized teams** | 团队内部有层级分工，但团队之间以自由 P2P 关系相处。 |
| **Open membership** | 任意节点可自由加入任何团队；有能力的节点可同时参与多个团队。 |
| **Low barrier** | 创建校验团队仅需管理层具备中等硬件能力，不设置人为门槛。 |


## 2. Team Structure and Roles（组结构与角色分配）

一个校验组由三类职能角色与共享的 UTXO/UTCO 缓存服务构成。

```graph
┌─────────────────────────────────────────────────┐
│               Verification Team                 │
│                                                 │
│  ┌───────────────────────────────────────────┐  │
│  │         Management Layer（管理层）        │  │
│  │  ┌─────────────┐    ┌──────────────────┐  │  │
│  │  │ Broadcaster │    │    Dispatcher    │  │  │
│  │  │  （广播者） │    │    （调度员）    │  │  │
│  │  └─────────────┘    └──────────────────┘  │  │
│  └──────────┬───────────────────┬────────────┘  │
│             │                   │               │
│       ┌─────┴─────┐      ┌──────┴───────┐       │
│       │   Guards  │      │  Verifiers   │       │
│       │ （守卫者）│      │  （校验员）  │       │
│       └───────────┘      └──────────────┘       │
│                                                 │
│  ┌───────────────────────────────────────────┐  │
│  │    UTXO/UTCO Cache（UTXO/UTCO 缓存器）    │  │
│  └───────────────────────────────────────────┘  │
└─────────────────────────────────────────────────┘
```


### 2.1 Management Layer（管理层）

管理层包含两个子角色：**Broadcaster** 与 **Dispatcher**。

#### 2.1.1 Broadcaster（广播者）

职责：
- 接收合格铸造者的铸造申请并组织铸造流程
  （Coinbase 创建、CheckRoot 交付、签名收集、区块头组装）。
- 打包并向网络发布最终区块。同时向内部 UTXO/UTCO Cache 推送新交易的未花费/未转移输出项以更新缓存。
- 若其他团队铸造区块，则接收其发布的区块、验证之、补齐本地缺失交易，并继续转发。
- 直接验证少量差异交易（存在于接收区块但不在本地已验证集合中）以提升效率与安全性。
- 打包时需要处理双花问题，通常遵循先到先得的原则（但也可能交易费高者优先）。

#### 2.1.2 Dispatcher（调度员）

职责：
- 接收 Guards 的首领校验通过的交易并存入待校验池。
- 以可控冗余将交易分配给 Verifiers（每笔交易至少 2 名）。
- 评估校验结果，结果不一致时触发扩展复核。
- 记录成员工作表现以用于奖励分配。
- 实时同步已验证交易集合至 Broadcaster，以便快速打包或校验区块。

#### 2.1.3 Management Layer Connections（管理层连接关系）

| 方向 | 对端 |
|-----------|-------|
| **Intra-team** | Guards（接收交易）、Verifiers（分配任务并收集结果） |
| **Inter-team** | 其他校验团队的 Broadcaster |


### 2.2 Guards（守卫者）

Guards 是团队的守门人——外部交易的唯一入口。

职责：
- 接收来自其他团队 Guards 与 Verifiers 的交易。
- 对每笔入站交易执行 **Leader Verification**（见 §3）。
- 将首领校验通过的交易提交给 Dispatcher。
- 将接收的交易转发至其他团队 Guards 以加速全网广播。
- 维护 **blacklist**：记录通过首领校验但最终未通过完整校验的交易的首领输入。

管理层持续向 Guards 同步已提交交易的 ID，以防重复校验与提交。但该同步仅抑制重复提交，不影响 Guard 对外转播交易——即使交易已被其他 Guard 提交过，当前 Guard 仍可向外部团队转播，以保证全网广播效率。

全网快速广播遵循一个**随机抉择**策略：*按 50% 的概率，一半的交易先执行首领校验，通过后再转发，另一半直接转发*。

这可以一定程度缓解攻击的影响（如果有），同时也保有相对较高的传播效率。

#### Guard Connections（守卫者连接关系）

| 方向 | 对端 |
|-----------|-------|
| **Intra-team** | 管理层（提交已验证交易） |
| **Inter-team** | 接收来自：其他团队 Guards 与 Verifiers。发送至：其他团队 Guards。 |


### 2.3 Verifiers（校验员）

Verifiers 负责完整交易校验。

职责：
- 向 Dispatcher 请求校验任务。
- 执行完整校验（脚本执行、输入验证、金额检查等），并将通过/失败结果回报给 Dispatcher。
- 将已验证通过的交易发送给其他团队 Guards，加速网络传播。

实现说明：
- Verifiers 不知道分配的交易是常规校验还是复核重查。这种信息不对称可抑制串谋与恶意行为。
- 若交易计算量过大，Verifier 可拒绝并返回 *overloaded*。若同一交易收到多个此类回应，Dispatcher 可降低其优先级（冷处理）。

#### Verifier Connections（校验员连接关系）

| 方向 | 对端 |
|-----------|-------|
| **Intra-team** | 管理层（请求任务、提交结果） |
| **Inter-team** | 发送至：其他团队 Guards（提供已验证交易） |


### 2.4 UTXO and UTCO Cache（UTXO/UTCO 缓存器）

UTXO Cache 与 UTCO Cache 逻辑和实现相似，但作为**各自独立的服务**运行——因为校验节点的请求目标（币金或凭信）始终是明确的，两者无需合并。

两个缓存均维护各自的未花费/未转移输出集，并向团队成员提供快速查询能力。

```go
// UTXOCache 提供组内 UTXO 集的缓存查询服务。
type UTXOCache interface {
    // Lookup 查询指定输出点是否未花费，并返回其信息。
    Lookup(outpoint Outpoint) (*UTXOEntry, error)

    // Update 根据新打包区块的交易，标记或清理已花费输出项并添加新的输出。
    Update(tx *Transaction) error
}

// UTCOCache 提供组内 UTCO 集的缓存查询服务，逻辑与 UTXOCache 类似。
type UTCOCache interface {
    Lookup(outpoint Outpoint) (*UTCOEntry, error)
    Update(tx *Transaction) error
}
```

关键行为：
- **UTXO Cache**：在管理层**打包区块后**（或验证其它组发布的区块后），将新交易的未花费币金推送至缓存。
- **UTCO Cache**：在管理层**打包区块后**（或验证其它组发布的区块后），将新交易的未转移输出项推送至缓存。

#### External Script Reference Cache（外部脚本引用缓存）

交易中存在对外部脚本的引用（`GOTO`、`EMBED`、`SCRIPT` 指令），应当实现一个独立的脚本引用缓存。需考虑及时清理和实时更新。

```go
// ScriptCache 提供外部脚本引用的缓存查询服务（GOTO/EMBED/SCRIPT 指令）。
type ScriptCache interface {
    // Lookup 按脚本引用地址查询缓存的脚本字节码。
    Lookup(ref ScriptRef) ([]byte, error)

    // Store 缓存外部脚本，供后续复用。
    Store(ref ScriptRef, script []byte) error

    // Evict 清理长时间未用或不再需要的脚本缓存项。
    Evict(ref ScriptRef) error
}
```

## 3. Leader Verification（首领校验）

首领校验是 Guards 对入站交易执行的轻量预检。仅校验 **first input**（“leader input”）；若通过则转入完整校验。

### 3.1 Purpose（目的）

首领校验使交易在无需完整校验成本的情况下快速传播。通过仅校验一个输入，Guards 可迅速判断交易是否可能合法并近实时转发。

### 3.2 Constraints（约束条件）

为限制滥用（如首领输入有效但后续输入无效的洪泛攻击）：

1. **The leader input must be a Coin type**（币金），且在所有 coin 输入中具有最高 coin-age（币权）。
2. **Blacklisting**：若交易通过首领校验但最终未通过完整校验，该首领输入引用被加入临时黑名单。
3. **Freeze duration**：黑名单首领输入冻结 **24 hours**，期间以该输入为首领的交易会被 Guard 拒绝。

这使洪泛攻击成本高昂：攻击者必须消耗有效且高币权的 UTXO，一旦被检测到，每个都会冻结 24 小时。


## 4. Security Mechanisms（安全保障机制）

### 4.1 Redundant Verification（冗余校验）

每笔交易分配给多个 Verifiers（冗余 ≥ 2，团队可配置）。结果合并如下：

| 结果 | 处理 |
|--------|---------|
| All Verifiers report valid | 交易被接受为有效 |
| At least one reports invalid | 交易进入 Extended Review |

### 4.2 Extended Review（扩展复核）

当校验结果不一致时，Dispatcher 升级到表现更好的 Verifiers（基于历史绩效选择）。复核同样采用冗余。

**Level 1 Review（一级复核）:**

| 结果 | 处理 |
|--------|---------|
| Zero error reports | 交易被接受为有效 |
| More than half report invalid | 交易被拒绝为无效 |
| Less than half report invalid | 升级到 Level 2 |

**Level 2 Review（二级复核）:**

| 结果 | 处理 |
|--------|---------|
| Any error report | 交易被拒绝为无效 |
| All report valid | 交易被接受为有效 |

通过扩展复核拒绝的交易不会被永久丢弃。若另一校验团队验证并将其包含在发布区块中，接收团队会在区块同步时重新校验。

### 4.3 Inter-Team Feedback（组间反馈）

Verifiers 直接将已验证交易发送给其他团队 Guards。若接收 Guard 所属团队随后判定该交易无效，Guard 会通知原始 Verifier，后者可将反馈通知其管理层，然后管理层启动扩展复核。该跨团队反馈回路可显著增强整体校验的可靠性。

管理层维护每笔交易的 Guard 投递记录，然后才能将错误信息反馈到其递送来源 Verifiers。


## 5. Transaction Inclusion Priority（交易收录优先级）

管理层在组装区块时遵循以下优先级规则：

| 优先级 | 判据 |
|----------|-----------|
| **Highest** | 有凭信提前销毁的交易 |
| **Secondary** | 币权（币龄*币量）销毁更多的交易 |
| **Tertiary** | 有更高交易费的交易 |

> **注：**
> 这是一个共约而不是协议（强制规则）。


## 5A. Non-Includable Transactions（不收录的交易）

管理层在打包区块时，必须排除以下两类交易：

| 排除规则 | 描述 |
|----------|------|
| **Intra-transaction TxID prefix collision** | 同一笔交易内，任意两个输入项的 TxID 前 20 字节相同。 |
| **UTXO/UTCO leaf-layer TxID prefix collision** | 新交易进入 UTXO/UTCO 集时，其 TxID 前 8 字节与同一叶子层内已有交易的前 8 字节碰撞。 |

这两条规则确保 TxID 引用始终有效，是预防性安全措施——虽然碰撞概率极低，但为了长期安全性仍然排除。


## 6. Minting Integration（铸造集成）

### 6.1 Minter Identity（铸造者身份）

仅择优池（Best Pool）成员可签名区块。铸造竞争向所有角色开放——管理层、Guards、Verifiers，甚至不参与校验工作的 spectators（围观者）。

管理层不知道哪些节点具备铸造资格（铸造预选与交易校验无关）。铸造者可同时参与多个团队，理论上可能签多个区块。

### 6.2 Lowest-Revenue Principle（低收益原则）

为防止铸造者为获利而签名多个竞争区块：

> **Rule**: When the same minter signs multiple blocks at the same height, only the block with the **lowest revenue** is considered valid.

该规则有两重作用：
1. **消除铸造者多签激励**——多签只会导致收益取最低，因此没有动机。
2. **抑制校验组重新打包区块**——即使校验组新收到高交易费的交易，重新打包也无法提升铸造者收益，从而消除拖延出块的动机。

### 6.3 Time-Skew Delay（错时延迟）

当出块时间到达后，适合该区块收录的交易会被暂停转播，直到区块已经被确定下来。

这从交易传播层面斩断铸造者等待更多交易以获取更多交易费的念想，与低收益原则形成协同：低收益原则从收益端消除动机，错时延迟从交易供给端切断来源，二者共同确保区块按时出块。

### 6.4 Information Separation（信息分离）

铸造流程刻意在铸造者与管理层之间进行信息隔离，形成相互制衡：

- **minter** 构造 Coinbase 交易，但不知道完整区块的 `CheckRoot`（交易哈希树根与 UTXO/UTCO 指纹的组合哈希）。
- **Management Layer** 知道 `CheckRoot`，但无法伪造铸造者签名。

该隔离促成双方的合作，使得铸造竞争参与者拥有宽泛的受众面。

### 6.5 Minting Workflow（铸造流程）

```graph
Minter                         Management Layer
  │                                   │
  │  1. Submit Best Pool proof        │
  ├──────────────────────────────────>│
  │                                   │
  │  2. Return: fee total, reward     │
  │     addresses, minting amount,    │
  │     service reward addresses...   │
  │<──────────────────────────────────┤
  │                                   │
  │  3. Construct & submit Coinbase   │
  ├──────────────────────────────────>│
  │                                   │  Verify Coinbase,
  │                                   │  package block,
  │                                   │  compute hash tree
  │  4. Return: Coinbase Merkle path, │
  │     TreeRoot, UTXO fingerprint,   │
  │     UTCO fingerprint, CheckRoot   │
  │<──────────────────────────────────┤
  │                                   │
  │  Verify Coinbase is included      │
  │  (via Merkle path)                │
  │                                   │
  │  5. Sign CheckRoot, submit        │
  ├──────────────────────────────────>│
  │                                   │  Verify signature,
  │                                   │  assemble block header,
  │                                   │  publish block
  │  6. Block published               │
  │<──────────────────────────────────┤
```

说明：
- 区块头由管理层创建，使其对版本控制与结构决策更有自主权。
- 铸造者在签名前使用返回的 Merkle 路径验证其 Coinbase 交易是否被包含。
- 若铸造者指定的公共服务奖励地址被管理层拒绝，铸造者将无法获得签名数据，可转而尝试其他团队。

### 6.6 Spectator Participation（围观者参与）

铸造竞争与校验工作相互独立。为最大化铸造候选池（通过更广泛竞争提升安全性），校验团队允许非校验节点——**spectators**——加入成为铸造候选者。

这在事实上不可避免：如果 spectator 赢得择优池竞争，必须能进入团队执行铸造流程。

管理层向 spectator 铸造者提供可选信息：
- 建议的公共服务奖励地址
- Coinbase 交易的奖励槽标记

以上仅为建议，最终由铸造者裁量。

### 6.7 Anti-Selfish-Minting: Stakes Threshold（防自私铸造：币权阈值）

铸造者理论上可在不依赖团队协助的情况下生成空块（仅含 Coinbase 交易），因为只需前一区块 ID 与当前 UTXO 和 UTCO 的指纹。

To deter this:

> **Rule**: If a candidate block's total Stakes destruction exceeds the primary block's
> Stakes by a factor of **3×**, the candidate block wins.

这确保自私铸造的主块必须至少校验约 1/3 的可用交易，否则会被候选块取代。对单个 spectator 或个人 Verifier 而言，实现 1/3 交易量校验并不现实。

| 术语 | 定义 |
|------|------------|
| **Primary block** | 由择优池排名最靠前（如第一名）候选者铸造的区块 |
| **Candidate block** | 由排名次于主块铸造者的候选者铸造的区块（冗余安全） |

> **注：**
> 如果择优池第一名因为下线而没有参与铸造，则第二名候选者铸造的区块就是主块。
> 这是一个相对的概念。

### 6.8 Minimum Minter Overhead（铸造者最低开销原则）

不参与校验工作的铸造者很可能是围观者，硬件性能有限。设计充分考虑这一点，铸造者仅需完成以下最低工作：

| 步骤 | 操作 |
|------|------|
| 1 | 签名择优凭证 → 向管理层证明铸造资格 |
| 2 | 构造合规的 Coinbase 交易 → 提交给管理层 |
| 3 | 确认 Coinbase 未被篡改（通过 Merkle 路径验证），签署区块 → 提交给管理层 |

> **注：**
> 区块签署者也称为 *minting-credential signer*（铸凭者）。如果铸凭者同时也是校验组的工作成员，则拥有双重收益：铸造分成与工作报酬。


## 7. Block Publication（区块发布）

区块发布为提升网络效率而优化，分三阶段进行：

### 7.1 Stage 1: Block Proof Broadcast（广播区块证明）

首先广播最小证明数据：
- Coinbase 交易
- Coinbase Merkle 验证路径（连接 Coinbase 与 TreeRoot 的哈希序列）
- 铸造者签名
- 区块头

这使接收团队无需下载全部交易即可立即验证区块真实性。

### 7.2 Stage 2: Block Summary（发布区块概要）

发布完整交易 ID 列表，并进行优化：
- 每个交易 ID **truncated to the first 16 bytes**，按序传输。
- 64K 交易的区块：`64K × 16 = 1 MB` 数据。
- 接收者将截断 ID 与本地已验证交易集对比。
- 缺失项将触发请求完整交易数据。

> **注：**
> 若有歧义项（哈希前缀碰撞，极少发生），会收到额外补充数据以消除歧义。

### 7.3 Stage 3: Transaction Synchronization（同步交易数据）

同步区块内所有交易的完整数据。由于多数团队已验证大部分交易，仅需获取少量缺失交易。接收者随后：
1. 重建交易哈希树。
2. 将计算得到的树根与区块头 `CheckRoot` 对比验证。
3. 接受或拒绝区块。


## 8. UTXO/UTCO Fingerprint（UTXO/UTCO 指纹）

### 8.1 Structure（结构）

UTXO/UTCO 集是区块链所有未花费/未转移输出的集合。UTXO 指纹是对整个 UTXO 集的 **4 级分层哈希树**，UTCO 指纹与其采用相同的结构和计算方式。

4 个层级依次为：

```
Level 1 (root): Year       ─── 交易时间戳对应的 UTC 年度（顶层）
Level 2:        TxID[8]    ─── TxID 第 8 字节的值
Level 3:        TxID[13]   ─── TxID 第 13 字节的值
Level 4 :       TxID[18]   ─── TxID 第 18 字节的值
Leaf-node ...              ─── 叶子节点：Hash512(TxID ‖ FlagOutputs)
```

> **注**：三级子分层使用 TxID 字节序列的 `[8, 13, 18]`（而非 `[0, 1, 2]`），以获得更好的随机性分布，避免 ID 塑造者人为影响分层平衡。

叶子节点将 TxID 与其所有输出的花费状态位组合后哈希：

```go
// FlagOutputs 表示一笔交易的输出项花费状态。
type FlagOutputs struct {
    Count     int    // 标记位字节数
    FlagBytes []byte // 每位对应一个输出项，1 = 未花费/未转移，0 = 已无效
}

// 叶子节点哈希：Hash512(TxID ‖ FlagOutputs)
```

```graph
┌──────────────────────────────────────────────┐
│            UTXO/UTCO Fingerprint             │
│               (top-level hash)               │
└──────────────────┬───────────────────────────┘
                   │
      ┌──────────┬─┴──────────┬───────────┐
    Year₁      Year₂        Year₃        ...
      │
    ┌─┴───────────────┬─────────────────┐
[TxID[8]=0x3a]  [TxID[8]=0x7f]         ...
      │
    ┌─┴───────────────┬─────────────────┐
[TxID[13]=…]   [TxID[13]=…]          ...
      │
    ┌─┴───────────────┬─────────────────┐
[TxID[18]=…]   [TxID[18]=…]          ...
    │
Leaf_A: Hash512(TxID_A ‖ FlagOutputs), Leaf_B: Hash512(TxID_B ‖ FlagOutputs), ...
```

这是一个**宽成员的哈希校验树**，4 层分级可显著减少每次输出状态变化时需重算的节点数量；顶层年度虽然无限增长，但粒度足够大，可以接受。

### 8.2 Computation Efficiency（计算效率）

当某笔交易的输出被花费或转移时，只需更新受影响路径上的 4 个节点：

1. 重算该交易的 **Leaf**（更新 `FlagOutputs` 中对应位后重新 Hash512）。
2. 沿路径向上，重算父级 **TxID[18]** 分组哈希。
3. 重算父级 **TxID[13]** 分组哈希。
4. 重算父级 **TxID[8]** 分组哈希。
5. 重算父级 **Year** 哈希。
6. 重算顶层 **Fingerprint**。

树中绝大多数中间节点甚至年度哈希保持不变，使增量更新高效可行。

> **提示：**
> UTXO/UTCO 指纹无需实时更新，而是在汇总后一次性计算（并行）。

### 8.3 UTXO/UTCO Cache Set（UTXO/UTCO 缓存集）

UTXO/UTCO 指纹结构也可扩展为**缓存集**，在叶子层下额外创建对交易输出数据的检索子集。

由于分层分组隔离，到达叶子层时每组的交易数量已很少，可用微型集合实现快速检索：

```go
// UTXOCacheLeaf 是 UTXO 缓存集在叶子层的检索结构。
// key 为交易 ID 的前 8 字节（int64），value 为该交易的可用输出集。
type UTXOCacheLeaf map[int64]Outputs
```

UTXO/UTCO 缓存集与指纹结构同构，额外提供对交易输出项数据的快速检索服务，是 §2.4 中缓存器的底层存储结构。

> **注**：缓存集仅存储已验证合法的输出项，交易数据本身的合法性验证由独立流程完成。

### 8.4 Multi-Chain Protection（多链保护）

UTXO/UTCO 双指纹与区块 ID 链结合，形成区块链历史的**多路耦合保护**机制：

- **Block ID chain**：每个区块头包含 `PrevBlock`，顺序链接区块历史。
- **UTXO fingerprint chain**：每个区块的 `CheckRoot` 纳入 UTXO 指纹约束，汇总历史交易的未花费状态。
- **UTCO fingerprint chain**：每个区块的 `CheckRoot` 纳入 UTCO 指纹约束，汇总历史交易的未转移凭信状态。

攻击者必须同时伪造区块 ID 链与累积 UTXO/UTCO 状态，才能篡改历史，使历史重构在实践中不可行。

### 8.5 Chain Constraint（链式约束）

关键定义：
- **Current block**：当前正在校验与组装、即将创建的区块。
- **Current UTXO set**：当前区块交易所依赖的 UTXO 集，**尚未减去**当前区块自身收录交易的花费。

当前区块的 UTXO 指纹由 *current UTXO set* 计算而来（即前一区块交易后的结果集）。因无需等待新交易收录变化，计算时间充裕，并形成递推的链式约束：

```graph
UTXO Set(N) ──fingerprint──► Block(N).CheckRoot
    │
    │  去掉 Block(N) 的花费，加上 Block(N) 的新输出与 Coinbase
    ▼
UTXO Set(N+1) ──fingerprint──► Block(N+1).CheckRoot
```

**Reverse verification（逆向推导）**：以区块 101 为例，当前 UTXO 集即为第 100 号区块的结果集：

1. 当前 UTXO 集减去 100 号区块的新交易输出项和 Coinbase 铸币，加回 100 号区块的输入项，即得 **100 号区块的 current UTXO set**。
2. 计算该集合的指纹，应与 100 号区块头内的 UTXO 指纹相符。
3. 如此循环迭代，可从最新 UTXO 集逆向验证区块链至任意历史位置。

UTCO 集可执行类似的逆向验证，依循的是凭信转移的逻辑（转移计次递减、输出项变更规范等）。

### 8.6 New Node Bootstrap（新节点引导）

得益于 UTXO/UTCO 指纹，新加入节点仅需获取：
1. 区块头链（对于较久远的历史，可借助年块链缩减数据体积）
2. 末端局部完整区块
3. 当前 UTXO 和 UTCO 集

该最小数据集即可使节点高置信验证目标主链的合法性，大幅降低新节点入门门槛，提升区块链系统整体效率。

> **提示**：当前 UTXO 集可通过 UTXO 指纹验证，而 UTXO 指纹可通过区块头链来证实。UTCO 同理。


## 9. Optimal Team Size（最优规模）

团队越大未必越好。以 6 分钟出块、每块最多 64K（65536）笔交易估算：

- 吞吐：约 182 笔/秒
- 预计足够规模：**50–60 nodes**

Verifiers 自由领取任务——能力强者处理更多、弱者处理更少。该自适应负载分配自然平衡团队工作。


## 10. Hash Verification Tree with Ordered Leaves（含序哈希校验树）

用于区块中交易的哈希校验树支持 **ordered leaf nodes**：每个叶子（交易 ID）包含 3 字节序号前缀，从而实现：

| 能力 | 描述 |
|------------|-------------|
| **Random sampling** | 无需下载全部交易即可抽检树中任意位置 |
| **Positional extraction** | 按区块内位置定位并验证特定交易 |
| **Sync optimization** | 在节点同步时识别缺失交易的位置（不仅是 ID） |

该结构也用于附件分片哈希树，可对大文件附件进行局部验证。


## 11. Known Limitations（已知不足）

| 限制 | 描述 |
|------------|-------------|
| **Trust in Management Layer** | 铸造者必须将收益发送至管理层指定的地址，需一定信任。 |
| **Self-minting Management** | 若管理层自身为铸造者，缺乏机制阻止其保留全部收益，仅能通过团队间充分竞争来缓解。 |
| **Evaluation subjectivity** | 公共服务奖励评估是局部且概率性的，非全局可验证（见 Services 提案 §4）。 |

> **注：**
> 管理层铸造者保留全部收益时，其公共服务奖励依然受“后续铸造者认可”的约束。


## 12. Inter-Team Topology（组间拓扑）

```graph
  ┌──────────┐         ┌──────────┐         ┌──────────┐
  │  Team A  │         │  Team B  │         │  Team C  │
  │          │         │          │         │          │
  │ Broad.<──┼────────>┼──>Broad. │<───────>┼──>Broad. │
  │          │         │          │         │          │
  │ Guard <──┼────┐    │  Guard <─┼───┐     │  Guard   │
  │          │    │    │          │   │     │          │
  │ Verif.───┼────┼───>┼──>Guard  │   │     │          │
  │          │    │    │          │   │     │          │
  │          │    │    │  Verif.──┼───┼────>┼──>Guard  │
  └──────────┘    │    └──────────┘   │     └──────────┘
                  │                   │
                  └───────────────────┘
```

- **Broadcaster ↔ Broadcaster**：团队之间区块交换的全互联。
- **Verifier → Guard**：跨团队投递已验证交易。
- **Guard ← Guard/Verifier**：跨团队接收入站交易。
- **Guard ↔ Guard**：是交易快速广播的关键主通道。

每个团队是半开放的微中心：内部分层、外部 P2P。
