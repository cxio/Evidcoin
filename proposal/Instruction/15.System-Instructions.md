# Proposal: System Instructions [164–169]

> **Prerequisite:** This specification builds upon the [Base Constraints](0.Base-Constraints.md) document, which defines the instruction format, runtime model, and notation conventions used throughout.


## 1. Overview

System instructions occupy opcode range **[164–169]** (6 instructions, with opcodes 167–168 reserved). They provide system-level primitives that operate outside the normal script computation model: real-time clock access, blockchain-specific award verification, built-in transaction validation, and source code segmentation markers.

All system instructions use the `SYS_` prefix to distinguish them from regular script instructions, reflecting their privileged role in the execution environment.

### Design Principles

System instructions bridge the gap between the script's sandboxed execution model and the blockchain's operational requirements. They encapsulate complex verification logic (award calculations, transaction signature validation) into single instructions, ensuring that consensus-critical operations are implemented uniformly across all nodes rather than being composed from lower-level primitives where implementation differences could cause consensus failures.


## 2. Instruction Definitions

### 2.1 Global Time Retrieval

```
164     SYS_TIME(1){}   Global time retrieval
```

- **Auxiliary params:** 1 byte — target identifier specifying which time component to retrieve.
- **Arguments:** None.
- **Return value:** A Time object or an Int value, depending on the target identifier.
- Provides read-only access to the client's real-time clock. The returned value reflects the node's local system time at the moment of instruction execution.
- **Entries:**
  - *(default)* — The complete Time object.
  - `Stamp` — Unix timestamp in seconds.
  - `Year` — 4-digit year (e.g., 2026).
  - `Month` — Month of the year (1–12).
  - `YearDay` — Day of the year (1–365 or 1–366 for leap years).
  - `Day` — Day of the month (1–31).
  - `WeekDay` — Day of the week (0–6, where 0 = Sunday).
  - `Hour` — Hour of the day (0–23).
  - `Minute` — Minute of the hour (0–59).
  - `Second` — Second of the minute (0–59).
  - `Millisecond` — Millisecond component (0–999).
  - `Microsecond` — Microsecond component (0–999999).
- All numeric component values are of type Int.
- **Examples:**
  ```
  SYS_TIME              // → Time object (default)
  SYS_TIME[Stamp]       // → 1741132800 (Unix seconds)
  SYS_TIME[Year]        // → 2026
  SYS_TIME[WeekDay]     // → 4 (Thursday)
  SYS_TIME[Millisecond] // → 547
  ```
- **Design note:** Since `SYS_TIME` reads the local system clock, different nodes will return different values. Like the random number instructions, this should only be used in non-consensus logic unless the time value is used for coarse comparisons (e.g., checking whether a timestamp falls within a multi-minute window that all nodes would agree upon).

### 2.2 Award Verification

```
165     SYS_AWARD(1,~)  Award verification
```

- **Auxiliary params:**
  1. 1 byte — award rate (e.g., `10` represents 10%).
  2. Varint — block height (must match the block height of the script being executed).
- **Arguments:** None.
- **Return value:** The redeemable award amount (after deducting the retention portion).
- **Behavior:** Checks whether the redemption of a mining award meets the required conditions:
  1. Verifies that the specified block height is valid and matches the script's execution block.
  2. Confirms that the required number of block confirmations has been reached.
  3. Calculates the redeemable amount based on the award rate, deducting any retention.
- **Usage restriction:** Only usable in **Coinbase transaction** output scripts. Coinbase transactions are subject to strict regulation:
  - The award rate must conform to protocol rules.
  - All outputs in a Coinbase transaction must have rates that total ≤ 100%.
  - The block height must match the script's block.
- **Chain segment inference:** From the block height specified in the second auxiliary parameter, the end-block height of the verification chain segment can be quickly inferred (offset of +25 to +48 blocks).
- **Example:**
  ```
  SYS_AWARD[10, 50000]  // verify 10% award at block height 50000 → redeemable amount
  ```

### 2.3 System Verification (Built-in Transaction Verification)

```
166     SYS_CHKPASS     System verification
```

- **Auxiliary params:** None.
- **Arguments:** None (extracted implicitly from the execution environment — this is a special case).
- **Return value:** None.
- **Behavior:** Performs built-in transaction verification — a **gate instruction** similar to `PASS`. If verification fails, script execution terminates immediately (same semantics as a failed `PASS`).
- **Verification requirements:**
  - The receiver address must be in standard format.
  - Coins and credentials follow different verification logic, but both are initiated by this single instruction.
  - Coin verification includes amount extraction and balance validation.
  - Credential verification includes transfer logic validation.
- **Implicit function signature:**
  ```
  SYS_CHKPASS(sig_type, auth_flag, []sigs, []pubkeys, []pubkey_hashes)
  ```
  - `sig_type` — Signature type: `1` for single signature, `2` for multi-signature.
  - `auth_flag` — Authorization flag.
  - `[]sigs` — Array of signatures.
  - `[]pubkeys` — Array of public keys.
  - `[]pubkey_hashes` — Array of public key hashes (used only in multi-signature mode).
- **Design notes:**
  - The instruction extracts its parameters from the environment rather than consuming explicit stack arguments. This separates the unlock script's signature data from the transaction ID computation, allowing signatures to be removed later without affecting chain integrity.
  - Chain hashing provides the necessary security guarantees even after signature removal.
  - Keeping signature and multi-signature arguments out of the visible script body enhances privacy — observers cannot trivially determine the signing scheme from the lock script alone.
- **Example:**
  ```
  SYS_CHKPASS           // perform built-in verification, terminate on failure
  ```

### 2.4–2.5 Reserved

```
167     (Reserved, unused)
168     (Reserved, unused)
```

### 2.6 Source Code Null Point Marker

```
169     SYS_NULL    Source code null point marker
```

- **Auxiliary params:** None.
- **Arguments:** None.
- **Return value:** None.
- Sets a **NULL point** at the current position in the instruction stream. NULL points serve as segment boundaries for the `SOURCE` instruction (opcode 135), enabling extraction of specific code regions.
- `SYS_NULL` itself produces no runtime effect beyond establishing the marker — it does not modify the data stack, argument space, or any execution state.
- **Relationship to SOURCE:**
  - `SOURCE[2]` extracts pre-executed code from the nearest preceding `SYS_NULL` point.
  - `SOURCE[-2]` extracts post-execution code to the nearest following `SYS_NULL` point.
- **Example:**
  ```
  SYS_NULL
  "Hai" {100} ADD
  ...
  SOURCE[2]             // extracts from "Hai" up to and including this SOURCE
  ```
  The `SYS_NULL` establishes a boundary, and `SOURCE[2]` extracts all instructions between that boundary and itself (excluding the NULL marker, including SOURCE).


## 3. Instruction Summary

| Opcode | Mnemonic | Gate? | Description |
|--------|----------|-------|-------------|
| 164 | SYS_TIME | No | Real-time clock access |
| 165 | SYS_AWARD | No | Mining award verification and calculation |
| 166 | SYS_CHKPASS | Yes | Built-in transaction verification (terminates on failure) |
| 167–168 | — | — | Reserved |
| 169 | SYS_NULL | No | Source code null point marker |

### Design Notes

- **Gate instructions:** `SYS_CHKPASS` is the second gate instruction in the system (alongside `PASS`). Both cause immediate script termination on failure. The difference is that `PASS` evaluates an explicit boolean argument, while `SYS_CHKPASS` performs a complex, multi-step verification using implicitly extracted environment data.
- **SYS_CHKPASS privacy model:** By extracting signature-related data from the environment rather than consuming it from the script, the instruction enables a separation of concerns: the lock script defines *what* must be verified, while the unlock script provides *how* to satisfy that verification. This separation allows future protocol upgrades to modify signature handling without changing lock script formats.
- **SYS_NULL as a meta-instruction:** Unlike most instructions that produce computational effects, `SYS_NULL` exists purely to support the `SOURCE` meta-programming facility. It demonstrates the script system's capacity for self-reflection — scripts can reason about their own structure and extract meaningful segments for verification or delegation purposes.
