# Proposal: Environment Instructions [128–137]

> **Prerequisite:** This specification builds upon the [Base Constraints](0.Base-Constraints.md) document, which defines the instruction format, runtime model, and notation conventions used throughout.


## 1. Overview

Environment instructions occupy opcode range **[128–137]** (10 instructions, with opcode 137 reserved). They provide read access to contextual information from the nested execution environment hierarchy, as well as global variable management and script source extraction facilities.

### Environment Hierarchy

The script runtime operates within a five-level nested domain hierarchy, where each outer domain serves as the parent of the inner one:

```
System Domain  ──►  Transaction Domain  ──►  Validation Domain  ──►  Script Domain  ──►  Block Domain
```

1. **System Domain** — Node runtime environment. Provides system-level information such as real-time clock, current chain height, and block timing.
2. **Transaction Domain** — The transaction being validated, encompassing its multiple inputs, outputs, and the source outputs referenced by those inputs.
3. **Validation Domain** — The current output's validation context, including state information about `GOTO`/`EMBED`-imported script verification.
4. **Script Domain** — The script's own execution environment, including top-level scripts and `GOTO`/`EMBED` sub-scripts. The data stack and argument space belong to this domain. Sub-scripts invoked via `GOTO` have their own independent data stack and argument space.
5. **Block Domain** — Syntactic block-level local scope. Block domains are mutually isolated — they cannot access each other's local variables.

### Design Principles

Most environment instructions require no explicit arguments — they retrieve information based on an auxiliary parameter that identifies the target entry. The exception is `SETVAR`, which takes a value argument for assignment.

Environment information exposed by these instructions should also be made available to middleware layers, enabling external systems to maintain awareness of script execution state.


## 2. Instruction Definitions

### 2.1 Environment Variable Retrieval

```
128     ENV(1){}    Environment variable retrieval
```

- **Auxiliary params:** 1 byte — target identifier specifying which environment entry to retrieve.
- **Arguments:** None.
- **Return value:** The value of the target environment entry.
- **System domain entries:**
  - `Timestamp` — Current real time in seconds (Unix timestamp).
  - `ChainHeight` — Current blockchain height as determined by the transaction timestamp.
  - `BlockTime` — Current block timestamp (ideal/computed value).
- **Transaction domain entries:**
  - `Height` — Block height of the transaction.
  - `TxTime` — Transaction timestamp.
  - `TxID` — Transaction identifier.
  - **Note:** When the script is referenced as an input source, these entries target the **source transaction's block**, not the pending transaction.
- **Validation domain entries:**
  - `InGoto` — Bool indicating whether execution is inside a `GOTO` target script.
  - `InEmbed` — Bool indicating whether execution is inside an `EMBED` target script.
  - `InScript` — Bool indicating whether execution is inside a `SCRIPT` sub-script.
  - `InCode` — Bool indicating whether execution is inside a `CODE` block.
  - `Gotos` — Count of `GOTO` calls made so far.
  - `Embeds` — Count of `EMBED` calls made so far.
  - `Source0` — Main script source (lock script without the unlock segment).
  - `Source1` – `Source9` — External script sources (from `GOTO`/`EMBED` targets).
  - `Sources` — Count of external scripts currently loaded (always < 10). Computed at runtime.
- **Examples:**
  ```
  ENV[Timestamp]        // retrieve current system time
  ENV[TxID]             // retrieve the transaction identifier
  ENV[InGoto]           // check if currently inside a GOTO target
  ENV[Sources]          // get count of loaded external scripts
  ```

### 2.2 Input Item Retrieval

```
129     IN(1){}     Input item retrieval
```

- **Auxiliary params:** 1 byte — target identifier specifying which input entry to retrieve.
- **Arguments:** None.
- **Return value:** The value of the specified input entry.
- **Domain:** Validation domain. Retrieves information about the future spend's input part — the input that references the current output being validated.
- **Entries:**
  - `Index` — Input index within the transaction.
  - `Amount` — Amount value carried by this input.
  - `Account` — Public key hash of the input (equivalent to the `Receiver` field from the source output: `INOUT:Receiver`).
  - `Remarks` — Remark data attached to the input.
  - `Sigs` — Number of signatures provided.
  - `CanSigs` — Number of possible signatures in a multisig arrangement.
  - `SigType` — Signature type identifier.
  - `Script` — Input script content (includes the unlock segment).
  - `Source` — Input script source (without the unlock segment).
- **Examples:**
  ```
  IN[Amount]            // retrieve the input amount
  IN[Account]           // retrieve the input's public key hash
  IN[Sigs]              // get the signature count
  ```

### 2.3 Output Item Retrieval

```
130     OUT(~,1){}  Output item retrieval
```

- **Auxiliary params:**
  1. Varint — output index (0-based) within the pending transaction.
  2. 1 byte — target identifier specifying which output entry to retrieve.
- **Arguments:** None.
- **Return value:** The value of the specified output entry.
- **Domain:** Transaction domain. Targets the current pending transaction's output set.
- **Entries:**
  - `Amount` — Output amount.
  - `Receiver` — Receiver address (public key hash).
  - `Remarks` — Remark data.
  - `Creator` — Creator identifier.
  - `Title` — Title field.
  - `Description` — Description field.
  - `Content` — Content field.
  - `Count` — Credential transfer count.
  - `Attachment` — Attachment identifier.
  - `Source` — Output script bytes.
- **Examples:**
  ```
  OUT[0, Amount]        // retrieve amount from the first output
  OUT[2, Receiver]      // retrieve receiver address from the third output
  OUT[1, Attachment]    // retrieve attachment ID from the second output
  ```

### 2.4 Source Output Item Retrieval

```
131     INOUT(~,1){}    Source output item retrieval
```

- **Auxiliary params:**
  1. Varint — output index (0-based) within the input's source transaction output set.
  2. 1 byte — target identifier specifying which entry to retrieve.
- **Arguments:** None.
- **Return value:** The value of the specified source output entry.
- **Domain:** Transaction domain. Retrieves sibling entries from the input's source transaction output set — the outputs that existed alongside the current input's source output.
- **Entries:** Same as `OUT` (Amount, Receiver, Remarks, Creator, Title, Description, Content, Count, Attachment, Source), plus:
  - `Timestamp` — Source transaction creation time.
- **Design note:** This instruction is **resource-intensive**. It requires loading the input source transaction's complete sibling output set, which may involve additional storage lookups. Use judiciously.
- **Examples:**
  ```
  INOUT[0, Amount]      // retrieve amount from the source transaction's first output
  INOUT[1, Timestamp]   // retrieve creation time of the source transaction via its second output
  ```

### 2.5 Source Script Info Retrieval

```
132     XFROM(1){}  Source script info retrieval
```

- **Auxiliary params:** 1 byte — target identifier specifying which source script entry to retrieve.
- **Arguments:** None.
- **Return value:** The value of the specified entry.
- **Domain:** Transaction domain. For external scripts loaded via `GOTO` or `EMBED`, retrieves information about the source transaction that provided the script.
- **Entries:**
  - `InSize` — Number of inputs in the source transaction.
  - `InAmount` — Total input amount of the source transaction.
  - `OutSize` — Number of outputs in the source transaction.
  - `OutAmount` — Total output amount of the source transaction.
  - `Timestamp` — Source transaction timestamp.
  - `Amount` — Amount of the specific output referenced.
  - `Account` — Account (public key hash) associated with the referenced output.
  - `PayType` — Payment type: `coin`, `credential`, or `attestation`.
  - `Source` — Instruction sequence from the start of the source script up to the jump point.
- **Examples:**
  ```
  XFROM[PayType]        // check the payment type of the source script's transaction
  XFROM[InSize]         // get the input count of the source transaction
  XFROM[Amount]         // get the amount of the referenced output
  ```

### 2.6 Global Variable Retrieval

```
133     VAR(1)      Global variable retrieval
```

- **Auxiliary params:** 1 byte — variable position index `[0–255]`.
- **Arguments:** None.
- **Return value:** The value stored at the specified global variable position.
- **Domain:** Script domain. Unlike block-local scope (which is append-only), global variables support assignment and overwrite via `SETVAR`.
- **Sharing semantics:** Global variables are **not shared** across `GOTO` boundaries (each GOTO target has its own script domain), but `EMBED` targets **share** the caller's global variable space.
- **Example:**
  ```
  VAR[0]                // retrieve global variable at position 0
  VAR[100]              // retrieve global variable at position 100
  ```

### 2.7 Global Variable Assignment

```
134     SETVAR(1)   Global variable assignment
```

- **Auxiliary params:** 1 byte — variable position index `[0–255]`.
- **Arguments:** A single value of any type.
- **Return value:** None.
- **Domain:** Script domain. Stores the argument value into the specified global variable slot, overwriting any previous value.
- **Example:**
  ```
  {42} SETVAR[0]        // store integer 42 into global variable slot 0
  "hello" SETVAR[5]     // store string "hello" into global variable slot 5
  ```

### 2.8 Script Source Extraction

```
135     SOURCE(1)   Script source extraction
```

- **Auxiliary params:** 1 byte — extraction identifier specifying which segment of the script source to extract.
- **Arguments:** None.
- **Return value:** The extracted script source segment (as a byte sequence or instruction representation).
- **Domain:** Script domain.
- **Extraction identifiers:**
  - `0` — Current block's complete code.
  - `1` — Pre-executed code up to and including the `SOURCE` instruction itself.
  - `11` — Pre-executed code up to but excluding the `SOURCE` instruction.
  - `2` — Pre-executed code from the nearest `SYS_NULL` point (excluding the NULL marker, including SOURCE).
  - `12` — Pre-executed code from the nearest `SYS_NULL` point (excluding both the NULL marker and SOURCE).
  - `-1` — Post-execution code from the current position to the end of the block.
  - `-2` — Post-execution code from the current position to the nearest `SYS_NULL` point.
- **NULL points:** Marked by the `SYS_NULL` instruction (opcode 169). These serve as segment boundaries for `SOURCE` extraction.
- **Design note:** Only the top-level `SOURCE` instruction (not within a `GOTO`/`EMBED` target) can extract the complete script including the unlock segment.
- **Examples:**
  ```
  SOURCE[0]             // extract current block's complete code
  SOURCE[1]             // extract all code executed so far, including this SOURCE
  SOURCE[-1]            // extract remaining code to block end
  ```

### 2.9 Multi-Signature Position Check

```
136     MULSIG(1)   Multi-signature position check
```

- **Auxiliary params:** 1 byte — target position (0-based) in the multisig public key hash list.
- **Arguments:** None.
- **Return value:** Bool — `true` if the specified position in the multisig arrangement has a valid corresponding entry.
- **Domain:** Validation domain. Checks whether a specific position within the multisig public key hash list is occupied.
- **Visibility:** This instruction is visible within `GOTO` target scripts, allowing delegated scripts to inspect the multisig configuration of the calling context.
- **Example:**
  ```
  MULSIG[0]             // check if position 0 in multisig list is present
  MULSIG[2] PASS        // require position 2 to be present, or fail
  ```

### 2.10 Reserved

```
137     (Reserved, unused)
```


## 3. Domain Access Summary

| Instruction | Domain | Description |
|-------------|--------|-------------|
| ENV | System / Transaction / Validation | General environment variable retrieval |
| IN | Validation | Input item info for the future spend |
| OUT | Transaction | Pending transaction output entries |
| INOUT | Transaction | Source transaction sibling output entries |
| XFROM | Transaction | Source script transaction info (for GOTO/EMBED) |
| VAR | Script | Global variable read |
| SETVAR | Script | Global variable write |
| SOURCE | Script | Script source segment extraction |
| MULSIG | Validation | Multi-signature position check |

### Design Notes

- **ENV** consolidates access to three domains (System, Transaction, Validation) through a single instruction with domain-specific entry identifiers. This keeps the opcode space compact while providing broad environment visibility.
- **INOUT** is intentionally flagged as resource-intensive. Script authors should prefer `OUT` or `IN` when the required information is available from the current transaction context, and only use `INOUT` when sibling output data from the source transaction is genuinely needed.
- **Global variables** (VAR/SETVAR) provide mutable state within a script execution, complementing the append-only local scope system. The GOTO isolation boundary prevents unintended state leakage between independent script executions.
- **SOURCE extraction** enables meta-programming patterns where scripts can inspect and reason about their own instruction sequences, which is particularly useful for verification and code-signing scenarios.
