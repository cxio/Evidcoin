# Proposal: Result Instructions [50–56]

> **Prerequisite:** This specification builds upon the [Base Constraints](0.Base-Constraints.md) document, which defines the instruction format, runtime model, and notation conventions used throughout.


## 1. Overview

Result instructions occupy opcode range **[50–56]** (7 instructions). They produce **decisive outcomes** that affect the script's overall execution trajectory: pass/fail gate checks, cross-script jumps, function returns, and script termination. These are the instructions that determine whether a transaction's validation succeeds or fails.

### Design Principle

The result instruction category bridges the gap between computational logic and validation outcomes. While other instructions manipulate data, result instructions translate data states into actionable results — a boolean becomes a pass/fail gate, an address becomes a cross-script jump, and a termination instruction becomes a clean exit point.

### Pass-State Model

The script engine maintains a **pass-state** flag that represents the current validation outcome:

- `PASS` sets it to **true** on success, or **terminates** the script on failure (logical AND across multiple gates).
- `CHECK` sets it to **true** or **false** without terminating (last CHECK wins — logical OR semantics).
- At script termination (natural end, `EXIT`, or `END`), the pass-state determines validation success.


## 2. Instruction Definitions

### 2.1 Pass Gate

```
50      PASS        Gate check (pass/fail)
```

- **Auxiliary params:** None
- **Arguments:** A boolean value.
- **Return value:** None.
- **Behavior:**
  - **true:** Sets the pass-state to PASS. Execution continues.
  - **false:** Sets the pass-state to FAIL. Script execution **terminates immediately**.
- Multiple `PASS` instructions implement logical AND — every gate must succeed.
- **Implementation note:** A failed PASS may trigger a notification mechanism to inform external systems about the failure reason.
- **Example:**
  ```
  TOP FN_PUBHASH DATA{<expected_hash>} EQUAL PASS    // Gate 1: public key hash check
  FN_CHECKSIG PASS                                     // Gate 2: signature verification
  ```

### 2.2 Check Gate

```
51      CHECK       Check (non-terminating gate)
```

- **Auxiliary params:** None
- **Arguments:** A boolean value.
- **Return value:** None.
- **Behavior:**
  - **true:** Sets the pass-state to PASS.
  - **false:** Sets the pass-state to FAIL. Execution **continues** (does not terminate).
- The pass-state is **overwritten** by each subsequent `CHECK`, meaning only the last `CHECK` determines the outcome — effectively logical OR semantics.
- **Design note:** At normal script termination, the system evaluates the pass-state to determine validation success. `CHECK` allows scripts to attempt multiple verification strategies, succeeding if any one of them passes (when the last CHECK is true).

### 2.3 Independent Jump (GOTO)

```
52      GOTO(~,64,~)    Independent jump
```

- **Auxiliary params:**
  1. Varint — transaction year (derived from the transaction timestamp).
  2. 64 bytes — transaction ID.
  3. Varint — output index (0-based).
- **Arguments:** Variable count.
- **Return value:** None.
- **Behavior:** Redirects execution to the lock script of the specified transaction output. The target script runs in an **independent sub-environment**:
  - Argument space contents are transferred as the target script's **initial data stack**.
  - The target script has its own data stack, argument space, and global variables.
  - Import/export buffers are **shared** with the caller.
  - The target script's pass-state writes back to the top-level caller's pass-state.
  - All private environment data is discarded upon the target script's exit.
- **Nesting rules:**
  - Maximum GOTO calls at the main script level: **2**.
  - Maximum GOTO depth across the runtime: **3**.
  - GOTO targets may themselves contain GOTO instructions.
- **Example:**
  ```
  @POPS[5] GOTO[2025, 0xab93cf..., 1]
  ```
  Pop top 5 items into argument space, jump to the 2025-era transaction's second output script, transferring those 5 items as the target's initial stack.

### 2.4 Merged Embed (EMBED)

```
53      EMBED(~,64,~)   Merged embed
```

- **Auxiliary params:**
  1. Varint — transaction year.
  2. 64 bytes — transaction ID.
  3. Varint — output index (0-based).
- **Arguments:** None.
- **Return value:** None.
- **Behavior:** Fetches the lock script from the specified transaction output and executes it within the **caller's environment** — sharing the data stack, argument space, and global variables. The embedded script can push data onto the caller's stack.
  - Import/export buffers are also shared.
  - Pass-state directly affects the current script.
- **Nesting rules:**
  - Maximum total EMBED calls across the runtime: **4**.
  - Maximum depth: **0** — an embedded script cannot itself use EMBED or GOTO.
- **Example:**
  ```
  EMBED[2025, 0xab93cf..., 0] FN_HASH256 DATA{...} ...
  ```
  Embed the first output script, execute it (which may push values onto the stack), then continue with hash computation.

### GOTO vs. EMBED Comparison

| Aspect | GOTO | EMBED |
|--------|------|-------|
| Environment | Independent (new stack, argument space, globals) | Merged (shares caller's stack, argument space, globals) |
| Data transfer | Argument space → target's initial stack | Full environment sharing |
| Pass-state | Writes back to top-level caller | Directly modifies current script's state |
| Max calls | ≤ 2 at main level | ≤ 4 total across runtime |
| Max depth | ≤ 3 across runtime | 0 (no nesting within embedded script) |
| Buffer sharing | Shared | Shared |
| Residual data | Discarded on exit | Persists in shared environment |

### 2.5 Exit Script

```
54      EXIT        Terminate script
```

- **Auxiliary params:** None
- **Arguments:** Optional — a single value of any type.
- **Return value:** The argument value (returned to the system).
- Terminates script execution immediately. The system evaluates the pass-state to determine validation success.
- If an argument is present, it is returned to the system (may be processed by external handlers).
- If no argument is present, the script simply ends.
- Typically appears inside `IF{}` blocks for conditional early termination, but can be used at the top level.

### 2.6 Function Return

```
55      RETURN      Return from function block
```

- **Auxiliary params:** None
- **Arguments:** Optional — a single value of any type.
- **Return value:** The argument value; `nil` if no argument.
- Exits the current iteration of a **functional block** (`MAP{}` or `FILTER{}`).
- Only valid within `MAP{}` and `FILTER{}` tool instruction blocks — not for `EACH{}` loops (which use `BREAK`).
- Returns a value that becomes the output for the current iteration of the functional block.

### 2.7 Public Validation End

```
56      END         End public validation
```

- **Auxiliary params:** None
- **Arguments:** None.
- **Return value:** None.
- Terminates script execution **for public validation nodes only**. Private/participating nodes ignore this instruction and continue execution.
- For public validators, this behaves identically to `EXIT` — the pass-state is evaluated.
- **Use case — directed encryption:**
  1. Party A publishes their public key on-chain.
  2. Party B creates a shared secret using A's public key, encrypts data, and publishes both the encrypted data and B's public key.
  3. Party A reconstructs the shared secret using B's public key and decrypts the data.
  - The encrypted data is only meaningful to Party A, so public validation should end before the private data-processing logic.
- **Design note:** Private listeners rely on open public broadcasts to reach private monitoring nodes. Private processing typically includes a subset of the public validation logic, followed by application-specific operations.
- **Script pattern:**
  ```
  // Public validation section
  <signature checks>
  PASS
  END                     // Public nodes stop here

  // Private section (only private nodes continue)
  <encrypted data processing>
  <application-specific logic>
  ```


## 3. Validation Flow Summary

```
Script Start
    │
    ├─ PASS(true) ──► pass-state = PASS, continue
    ├─ PASS(false) ──► pass-state = FAIL, TERMINATE
    │
    ├─ CHECK(true) ──► pass-state = PASS, continue
    ├─ CHECK(false) ──► pass-state = FAIL, continue
    │
    ├─ GOTO ──► sub-environment → pass-state writes back
    ├─ EMBED ──► merged execution → pass-state directly affected
    │
    ├─ EXIT ──► TERMINATE, evaluate pass-state
    ├─ END ──► TERMINATE (public nodes only)
    ├─ RETURN ──► exit MAP/FILTER iteration
    │
    └─ Natural end ──► evaluate pass-state
```

The final validation result is always determined by the pass-state at the point of termination, regardless of how termination occurs.
