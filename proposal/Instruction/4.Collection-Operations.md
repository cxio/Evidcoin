# Proposal: Collection Operations [36–45]

> **Prerequisite:** This specification builds upon the [Base Constraints](0.Base-Constraints.md) document, which defines the instruction format, runtime model, and notation conventions used throughout.


## 1. Overview

Collection operations occupy opcode range **[36–45]** (10 instructions). They provide manipulation primitives for the two iterable collection types in the script system: **slices** (ordered, indexed sequences) and **dicts** (string-keyed maps with insertion-order iteration).

These instructions operate on collection values that reside on the data stack or in the argument space. Since scripts are read-only during execution, any instruction that logically modifies a collection (e.g., `SET`, `EXTEND`, `REVERSE`) must produce a **new** collection or operate on a copy, never mutating the original script data.

### Collection Types

| Type | Description | Key Type | Iteration Order |
|------|------------|----------|----------------|
| Slice | Ordered sequence of typed values | Integer index | Sequential (0, 1, 2, ...) |
| Dict | String-keyed associative map | String key | **Insertion order** (deterministic) |

> **Critical design decision:** Dict iteration uses **insertion-order semantics** (not Go's native random-order map). This is essential for blockchain determinism — all nodes must iterate dict entries in identical order.


## 2. Instruction Definitions

### 2.1 Sub-Slice

```
36      SLICE(1)    Create sub-slice
```

- **Auxiliary params:** 1 byte — sub-slice length (`[0–255]`). `0` means all remaining from start position.
- **Arguments:**
  1. Target slice.
  2. Start index (0-based). Supports negative indexing from the end.
- **Return value:** A sub-slice (reference into the original slice, not a deep copy).
- **Strict mode:** If position or count overflows the target, execution fails.
- **Examples (given `[1, 2, 3, 4, 5, 6, 7, 8]`):**
  - `{2} SLICE[3]` → `[3, 4, 5]`
  - `{-4} SLICE[2]` → `[5, 6]`
  - `{-3} SLICE` → `[6, 7, 8]` (all from position -3)
  - `{8} SLICE` → `[]` (empty slice, position at end)
- **Note:** The returned sub-slice references the original data. Use `COPY` (tool instruction) for an independent deep copy.

### 2.2 Reverse

```
37      REVERSE     Reverse a slice
```

- **Auxiliary params:** None
- **Arguments:** Target slice.
- **Return value:** A new slice with members in reversed order.
- **Example:**
  - Given `[1, 2, 3, 4, 5]`: `REVERSE` → `[5, 4, 3, 2, 1]`

### 2.3 Merge Slices

```
38      MERGE       Merge multiple slices
```

- **Auxiliary params:** None
- **Arguments:** Variable count — multiple slices.
- **Return value:** A new slice containing all members from input slices, concatenated in argument order.
- **Example:**
  ```
  {1, 2, 3}
  {10, 20}
  @POP @POP MERGE  // → [10, 20, 1, 2, 3]
  ```
  Note: `@POP` captures in FIFO order, so `{10, 20}` (top) is captured first, then `{1, 2, 3}`.

### 2.4 Extend Slice

```
39      EXTEND      Extend a slice with additional members
```

- **Auxiliary params:** None
- **Arguments:** Variable count — the first argument is the target slice, subsequent arguments are individual values to append.
- **Return value:** A new slice with the additional members appended.
- **Example:**
  ```
  {1, 2, 3}
  {10} {20}
  @POPS[3] EXTEND  // → [1, 2, 3, 10, 20]
  ```
  The first argument (bottom of the three popped items) is the target slice; `10` and `20` are appended.

### 2.5 Pack (Compact Binary Encoding)

```
40      PACK        Pack slice into bytes
```

- **Auxiliary params:** None
- **Arguments:** A slice.
- **Return value:** A byte sequence (`Bytes`).
- Decodes each slice member to its raw binary representation and concatenates them tightly (no padding, no separators).
- Members may be of mixed types — each is decoded to its base binary form and appended.
- **Example:**
  - `POPS[3] PACK` — pop top 3 items, package as slice, then pack into contiguous bytes.

### 2.6 Spread (Expand Slice)

```
41      SPREAD      Spread slice members
```

- **Auxiliary params:** None
- **Arguments:** A slice.
- **Return value:** Variable count — the individual members of the slice.
- The inverse of slice bundling. Extracts each member from the slice and places them individually into the target space, maintaining original order.
- **Examples:**
  - `$SPREAD` — spread slice members into local scope (each becomes a separate entry).
  - `@SPREAD` — spread into argument space.
  - `SPREAD` — spread onto data stack (each member pushed individually, maintaining order).
- **Tip:** Sequential `@~SPREAD` can expand multiple stacked slices into the argument space one by one.

### 2.7 Set Dict Entry

```
42      SET         Set dict key-value pair(s)
```

- **Auxiliary params:** None
- **Arguments:**
  1. Target dict (`map[string]any`).
  2. Key: a single string, or a string slice (`[]string`).
  3. Value: a single value, or a value slice.
- **Return value:** The modified dict (same reference, not a copy — operates on the dict's internal mutable state).
- **Behavior:**
  - **Single key + single value:** Sets one key-value pair.
  - **Key slice + value slice:** Sets pairs by positional correspondence (key[i] → value[i]).
  - **Key slice + single value:** Sets all keys to the same value.
- **Design note:** Dicts are not the primary collection type in the script system. They exist as a convenience, constructible from two slices via conversion instructions.

### 2.8 Get Dict Entry

```
43      ITEM        Get dict value(s) by key
```

- **Auxiliary params:** None
- **Arguments:**
  1. Target dict.
  2. Key: a single string, or a string slice.
- **Return value:** The corresponding value(s). If a key does not exist, returns `nil` (default zero value). If the key is a slice, returns a value slice.
- **Examples:**
  - `"AAA" ITEM` — get the value for key "AAA".
  - `{"AA", "BB"} ITEM` — get values for keys "AA" and "BB" as a slice.

### 2.9 Get Slice Member by Index

```
44      INDEX       Get slice member(s) by index
```

- **Auxiliary params:** None
- **Arguments:**
  1. Target slice.
  2. Index: a single integer, or an integer slice. Supports negative indexing.
- **Return value:** The value(s) at the specified position(s). If the index argument is a slice, returns a value slice.
- **Examples:**
  - `{1} INDEX` — get the second member (0-based).
  - `{1, 3} INDEX` — get the second and fourth members as a slice.
- **Design note:** Primarily intended for accessing individual elements or non-contiguous positions. Use `SLICE` for contiguous sub-sequences.

### 2.10 Collection Size

```
45      SIZE        Get collection size
```

- **Auxiliary params:** None
- **Arguments:** A collection (slice or dict).
- **Return value:** An integer — the number of members/entries.


## 3. Operational Semantics

### 3.1 Mutability Model

Since scripts are read-only, collection instructions that appear to "modify" a collection follow these rules:

| Operation | Creates New Collection? | Notes |
|-----------|----------------------|-------|
| SLICE | No (reference/sub-slice) | Shares underlying data with original |
| REVERSE | Yes | New slice with reversed order |
| MERGE | Yes | New slice from multiple sources |
| EXTEND | Yes | New slice with appended members |
| PACK | Yes | New byte sequence |
| SPREAD | N/A | Decomposes, no new collection |
| SET | No (in-place on dict) | Mutates the dict value directly |

**SET** is the exception: it operates on the dict's internal state. Since dicts are runtime-created values (not part of the read-only script bytecode), this in-place mutation is safe.

### 3.2 Dict Iteration Order

Dict implementations **must** use insertion-order-preserving data structures. This is a blockchain consensus requirement — all validating nodes must produce identical iteration sequences for the same dict. Go's native `map` type does **not** satisfy this requirement and must not be used directly; an ordered map implementation is required.

### 3.3 Strict Bounds Checking

`SLICE`, `INDEX`, and any positional access enforce strict bounds. Out-of-range positions cause execution failure, not silent default behavior. This matches the strict philosophy applied to stack operations.
