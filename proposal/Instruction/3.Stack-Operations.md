# Proposal: Stack Operations [25–35]

> **Prerequisite:** This specification builds upon the [Base Constraints](0.Base-Constraints.md) document, which defines the instruction format, runtime model, and notation conventions used throughout.


## 1. Overview

Stack operations occupy opcode range **[25–35]** (11 instructions). They provide direct manipulation of the data stack: pushing, popping, referencing, cloning, and viewing entries. These instructions form the mechanical foundation of the stack machine, enabling data to be positioned, duplicated, and transferred between the stack and other runtime spaces (argument space, local scope) via capture prefixes.

### Design Principle

Stack operations are divided into two behavioral categories based on how they return multiple values:

- **Auto-expanding instructions** (opcodes 27–32): When returning multiple items, each item is placed individually into the target space (stack, argument space, or local scope). The items maintain their original order.
- **Slice-returning instructions** (opcodes 33–35): Multiple items are bundled into a single slice value. The slice itself is one entry in the target space.

This distinction is critical for correct data flow management.

### Argument Consumption Pattern

Several stack operations accept positional arguments from the argument space (e.g., `PEEK`, `PEEKS`, `VIEW`). These follow the standard retrieval rules: argument space first, data stack fallback. When the position itself comes from the stack, be aware that the position value is consumed (popped) before the referenced items are accessed.


## 2. Instruction Definitions

### 2.1 No Operation

```
25      NOP         No operation
```

- **Auxiliary params:** None
- **Arguments:** Variable count (including zero).
- **Return value:** None
- Performs no computation, but **reads and discards** all values in the argument space, effectively clearing it.
- **Examples:**
  - `@SHIFT[] NOP` — extract all stack entries to argument space, then discard them (clears the entire stack).
  - `@SHIFT NOP` — same; zero-value auxiliary parameter brackets may be omitted.
  - `@POP NOP` — pop and discard the top stack item.
- **Design note:** The variable argument count includes zero, so NOP never needs to access the data stack. It is purely a sink for the argument space.

### 2.2 Push to Stack

```
26      PUSH        Push to data stack
```

- **Auxiliary params:** None
- **Arguments:** Variable count.
- **Return value:** None
- Transfers all argument space members onto the data stack in their original FIFO order.
- If the argument space is empty, this is a no-op.
- **Examples:**
  - `$[0] $[1] $[2] PUSH` — push three local scope values onto the data stack.
  - `@PEEK @INPUT PUSH` — capture the stack-bottom item and import buffer data, then push both onto the stack.

### 2.3 Pop Single Item

```
27      POP         Pop top item
```

- **Auxiliary params:** None
- **Arguments:** None (operates on stack top directly).
- **Return value:** The top stack item.
- Removes and returns the topmost stack entry.
- Typically used with a capture prefix; without one, the value re-enters the stack (pointless).
- **Examples:**
  - `@POP` — pop top item into argument space.
  - `$POP` — pop top item into local scope.
  - `POP` — meaningless (pops and re-pushes the same value).

### 2.4 Pop Multiple Items

```
28      POPS(1)     Pop multiple top items
```

- **Auxiliary params:** 1 byte — number of items to pop (`[0–255]`). `0` means all.
- **Arguments:** None.
- **Return value:** Items in original order (auto-expanding).
- Removes the specified number of items from the stack top. If the count exceeds the current stack height, execution **fails** (strict mode).
- **Examples:**
  - `@POPS[4]` — pop top 4 items, expand into argument space.
  - `$POPS` — pop all stack items, expand into local scope.
  - `POPS[4]` — meaningless (pops and re-pushes in the same order).

### 2.5 Reference Top Item

```
29      TOP         Reference top item
```

- **Auxiliary params:** None
- **Arguments:** None.
- **Return value:** The top stack item (by reference / shallow copy).
- Does **not** remove the item from the stack. Useful both with and without capture prefixes.
- **Examples:**
  - `@TOP` — reference top item into argument space (shallow copy).
  - `$TOP` — reference top item into local scope.
  - `TOP` — duplicate the top item (pushes a copy onto the stack).

### 2.6 Reference Multiple Top Items

```
30      TOPS(1)     Reference multiple top items
```

- **Auxiliary params:** 1 byte — number of items (`[0–255]`). `0` means all.
- **Arguments:** None.
- **Return value:** Items in original order (auto-expanding).
- References (shallow copies) the specified number of items from the stack top without removing them. If the count exceeds stack height, execution fails.
- **Examples:**
  - `$TOPS[3]` — reference top 3 items, expand into local scope.
  - `@TOPS[]` or `@TOPS` — reference all stack items into argument space.
- **Distinction from REP:** `TOPS[n]` references the top N distinct items (shallow copy each). `REP[n]` (arithmetic instruction) duplicates a **single** top item N times.

### 2.7 Peek at Arbitrary Position

```
31      PEEK        Reference item at arbitrary position
```

- **Auxiliary params:** None
- **Arguments:** Optional — target position (default `0`, i.e., stack bottom). Supports negative indexing from the top.
- **Return value:** The item at the specified position (by reference).
- Position indexing starts from the stack bottom (`0`). Negative values count from the stack top (`-1` = top item).
- **Examples:**
  - `PEEK` — reference the bottom-most item, push copy to stack top.
  - `@{0} PEEK` — same, with explicit position.
  - `@PEEK` — reference bottom item into argument space.
  - `@{-1} PEEK` — reference top item (equivalent to `TOP`).

### 2.8 Peek at Arbitrary Position Range

```
32      PEEKS(1)    Reference items from arbitrary position
```

- **Auxiliary params:** 1 byte — number of items. `0` means all from the start position onward.
- **Arguments:** Optional — start position (default `0`). Supports negative indexing.
- **Return value:** Items in stack order (auto-expanding).
- If the position is out of range or the requested count exceeds available items, execution fails.
- **Examples:**
  - `@PEEKS[5]` — reference bottom 5 items, expand into argument space.
  - `@{1} @PEEKS[2]` — starting from position 1, reference 2 items into argument space.
  - `PEEKS[5]` — reference bottom 5 items, expand onto stack top (clone bottom segment to top).

### 2.9 Shift Out as Slice

```
33      SHIFT(1)    Shift out top items (as slice)
```

- **Auxiliary params:** 1 byte — number of items (`[0–255]`). `0` means all.
- **Arguments:** None.
- **Return value:** A single **slice** containing the removed items.
- Removes items from the stack top and packages them into one slice value. If the count exceeds stack height, execution fails.
- **Examples:**
  - `@SHIFT[4]` — remove top 4 items, package as slice into argument space.
  - `$SHIFT[4]` — same, into local scope (adds 1 entry).
  - `SHIFT[0]` or `SHIFT` — remove all stack items, package as one slice, push onto stack.

### 2.10 Clone as Slice

```
34      CLONE(1)    Clone top items (as slice)
```

- **Auxiliary params:** 1 byte — number of items (`[0–255]`). `0` means all.
- **Arguments:** None.
- **Return value:** A single **slice** containing shallow copies of the items.
- Does not remove items from the stack. Packages copies into one slice value.
- **Examples:**
  - `@CLONE[3]` — clone top 3 items as slice into argument space.
  - `$CLONE[3]` — clone top 3 into local scope (1 entry).
  - `CLONE[3]` — clone top 3 as slice, push onto stack.
  - `CLONE` — clone all stack items as one slice, push onto stack.

### 2.11 View as Slice (Arbitrary Position)

```
35      VIEW(1)     View items from arbitrary position (as slice)
```

- **Auxiliary params:** 1 byte — number of items. `0` means all from the start position onward.
- **Arguments:** Optional — start position (default `0`). Supports negative indexing.
- **Return value:** A single **Any slice** containing references to the items.
- Does not remove items. Packages references into a `[]any` slice.
- If position is out of range or count exceeds available items, execution fails.
- **Examples:**
  - `@VIEW[5]` — view bottom 5 items as slice into argument space.
  - `VIEW[5]` — view bottom 5 items as slice, push onto stack.
  - `@{-5} VIEW` — view top 5 items as slice (equivalent to `CLONE[5]`).


## 3. Behavioral Summary

### Auto-Expanding vs. Slice-Returning

| Instruction | Removes from Stack | Return Style | Count Param |
|------------|-------------------|-------------|-------------|
| POP | Yes | Single item | — |
| POPS(n) | Yes | Auto-expand | Auxiliary (0=all) |
| TOP | No | Single item | — |
| TOPS(n) | No | Auto-expand | Auxiliary (0=all) |
| PEEK | No | Single item | Argument (position) |
| PEEKS(n) | No | Auto-expand | Auxiliary + Argument |
| SHIFT(n) | Yes | Single slice | Auxiliary (0=all) |
| CLONE(n) | No | Single slice | Auxiliary (0=all) |
| VIEW(n) | No | Single slice | Auxiliary + Argument |

### Strict Mode

All count-based operations enforce strict bounds checking. Requesting more items than the stack contains results in execution failure (not silent truncation). This prevents subtle bugs in script logic.

### Position Indexing Convention

For position-based instructions (`PEEK`, `PEEKS`, `VIEW`):
- **Positive values** count from the stack **bottom** (0 = first pushed item).
- **Negative values** count from the stack **top** (-1 = most recently pushed item).
