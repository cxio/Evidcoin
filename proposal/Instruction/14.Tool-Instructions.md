# Proposal: Tool Instructions [138–163]

> **Prerequisite:** This specification builds upon the [Base Constraints](0.Base-Constraints.md) document, which defines the instruction format, runtime model, and notation conventions used throughout.


## 1. Overview

Tool instructions occupy opcode range **[138–163]** (26 instructions, with opcodes 149–150 unused and 156–163 reserved for future quantum-safe cryptographic support). They provide commonly-used functional operations including script evaluation, collection manipulation, pattern matching, string processing, random number generation, numeric sequences, higher-order iteration, and extension points for embedded code.

### Design Principles

Tool instructions represent higher-level operations that combine multiple primitive steps into single, optimized instructions. Several of these instructions (`MAP{}`, `FILTER{}`) create **independent script domains** with private data stacks and argument spaces, enabling functional programming patterns without polluting the caller's execution state.

### Reserved Area

Opcodes **[156–163]** (8 slots) are reserved for future quantum-safe cryptographic primitives. As quantum computing advances threaten current cryptographic algorithms, these slots provide upgrade paths for post-quantum signature verification, key exchange, and hash functions without requiring changes to the instruction set structure.


## 2. Instruction Definitions

### 2.1 Script Evaluation

```
138     EVAL        Execute script
```

- **Arguments:** A Script value (typed — must be created by `CODE` or `SCRIPT` instructions, not arbitrary bytes).
- **Return value:** A slice containing any data stack remnants from the evaluated script.
- **Behavior:** Treats the argument as a script instruction sequence and executes it in an **independent execution domain** (environment), fully isolated from the caller's data stack, argument space, and global variables.
- After execution completes, any values remaining on the evaluated script's data stack are collected and returned to the caller as a single slice.
- `EVAL` can only interact with the caller through these returned data stack remnants — they appear as ordinary entries on the caller's stack.
- **Design note:** `EVAL` is **not intended for public validation** logic. It primarily serves private domain environments where dynamic script execution is needed. The strict typing requirement (Script type only) prevents arbitrary code injection.
- **Example:**
  ```
  CODE{ {1} {2} ADD }
  EVAL                    // → [3] (a slice containing the result)
  ```

### 2.2 Shallow Copy

```
139     COPY        Shallow copy of a slice
```

- **Arguments:** A slice.
- **Return value:** A new slice with shallow copies of each member.
- Member values are copied by value for primitive types. Sub-slices and dicts within the slice are **not** recursively copied — the new slice holds references to the same inner collections.
- **Example:**
  ```
  {1, 2, {3, 4}}
  COPY                    // → [1, 2, [3, 4]] (inner [3, 4] is shared reference)
  ```

### 2.3 Deep Copy

```
140     DCOPY       Deep copy of a slice
```

- **Arguments:** A slice.
- **Return value:** A new slice with recursively deep-copied members.
- Sub-slices are recursively copied until a non-slice type is reached. Members typed as `any` are also inspected and copied if they contain slice or dict values.
- **Example:**
  ```
  {1, 2, {3, {4, 5}}}
  DCOPY                   // → [1, 2, [3, [4, 5]]] (all nested structures are independent copies)
  ```

### 2.4 Dict Key-Value Extraction

```
141     KEYVAL(1)   Dict key-value extraction
```

- **Auxiliary params:** 1 byte — extraction type:
  - `0` — Extract both keys and values as two separate slices.
  - `1` — Extract keys only.
  - `2` — Extract values only.
- **Arguments:** A Dict.
- **Return value:** 1 or 2 slices depending on the extraction type.
- When extraction type is `0`, two slices are returned with 1:1 positional correspondence (keys[i] corresponds to values[i]). Both slices auto-expand into the receiver domain.
- **Examples:**
  ```
  KEYVAL[0]             // → keys_slice, values_slice (two slices, 1:1 correspondence)
  KEYVAL[1]             // → keys_slice only
  KEYVAL[2]             // → values_slice only
  ```

### 2.5 Regex Match

```
142     MATCH(1)    Regex match
```

- **Auxiliary params:** 1 byte — match mode:
  - `_` (default) — Test for match, returns Bool.
  - `g` — FindAll: returns all non-overlapping matches.
  - `G` — FindAllSubmatch: returns all matches with subgroup captures.
  - `o` — Find: returns the first single match.
  - `s` — FindSubmatch: returns the first match with subgroup captures.
- **Arguments:**
  1. Target string or byte sequence.
  2. A compiled regular expression (`*RegExp` type).
- **Return value:** Bool, a single value, or a slice — depending on the match mode.
- **Examples:**
  ```
  "hello world" RE{h\w+} MATCH       // → true (default test mode)
  "abc 123 def 456" RE{\d+} MATCH[g]  // → ["123", "456"]
  "2025-03-05" RE{(\d+)-(\d+)-(\d+)} MATCH[s]  // → ["2025-03-05", "2025", "03", "05"]
  ```

### 2.6 Substring Extraction

```
143     SUBSTR(~)   Substring extraction
```

- **Auxiliary params:** Varint — rune count (number of characters to extract).
- **Arguments:**
  1. A UTF-8 encoded string.
  2. Start character position (0-based). Supports negative values for indexing from the end.
- **Return value:** The extracted substring.
- Operates on **rune boundaries**, not byte offsets, ensuring correct handling of multi-byte UTF-8 characters.
- If the string has insufficient length to satisfy the extraction, returns an empty string (treated as an error condition).
- **Examples:**
  ```
  "hello world" {6} SUBSTR[5]     // → "world"
  "hello" {-3} SUBSTR[2]          // → "ll"
  ```

### 2.7 String Replacement

```
144     REPLACE(1)  String replacement
```

- **Auxiliary params:** 1 byte — replacement count. `0` means replace all occurrences.
- **Arguments:**
  1. Target string.
  2. Match pattern — a plain substring or a compiled regular expression (`*RegExp`).
  3. Replacement string.
- **Return value:** A new string with the specified replacements applied.
- When the match pattern is a regular expression, **all occurrences** are always replaced regardless of the replacement count parameter.
- **Examples:**
  ```
  "aabbcc" "bb" "XX" REPLACE[1]      // → "aaXXcc"
  "aabbccbb" "bb" "XX" REPLACE[0]    // → "aaXXccXX" (replace all)
  "abc123def" RE{\d+} "NUM" REPLACE   // → "abcNUMdef" (regex always replaces all)
  ```

### 2.8 Secure Random Number

```
145     RANDOM      Create secure random number
```

- **Arguments:** Optional — an upper bound (Int or BigInt, must be > 0). Default upper bound: `int64` maximum value.
- **Return value:** A cryptographically secure random number in the range `[0, upper_bound)`.
- Uses `crypto/rand` for generation.
- **WARNING:** Different nodes will produce **different values** for the same script execution. This instruction must only be used in **non-consensus logic** (private domain, post-`END` sections, or `EVAL` contexts). Using it in consensus-critical validation will cause network disagreement.
- **Example:**
  ```
  {100} RANDOM          // → secure random integer in [0, 100)
  RANDOM                // → secure random integer in [0, int64_max)
  ```

### 2.9 Fast Random Number

```
146     QRAND       Fast (insecure) random number
```

- **Arguments:** Optional — an Int upper bound (must be > 0).
- **Return value:** A pseudo-random number.
- Prioritizes speed over cryptographic security. Suitable for efficiency-priority scenarios where the quality of randomness is not critical.
- Same consensus warning as `RANDOM` applies — use only in non-consensus logic.
- **Example:**
  ```
  {1000} QRAND         // → fast random integer in [0, 1000)
  ```

### 2.10 Shuffle Slice

```
147     SLRAND      Shuffle slice member order
```

- **Arguments:** A slice.
- **Return value:** A new slice with members in randomly reordered positions.
- The seed is cryptographically secure, but the shuffling algorithm's randomness quality is **not guaranteed** to meet cryptographic standards.
- Same consensus warning applies — use only in non-consensus logic.
- **Example:**
  ```
  {1, 2, 3, 4, 5}
  SLRAND                // → e.g., [3, 1, 5, 2, 4] (random permutation)
  ```

### 2.11 Float Comparison with Epsilon

```
148     CMPFLO(1)   Float comparison with epsilon
```

- **Auxiliary params:** 1 byte — comparison type:
  - `0` — Equal (within epsilon tolerance).
  - `-1` — Less than or equal (with epsilon tolerance).
  - `1` — Greater than or equal (with epsilon tolerance).
- **Arguments:** Three values:
  1. First numeric value.
  2. Second numeric value.
  3. Epsilon (tolerance threshold).
- **Return value:** Bool.
- Designed for floating-point equality testing where standard comparison instructions (`EQ`, `LT`, etc.) are insufficient due to precision limitations.
- **Example:**
  ```
  {0.1} {0.3} ADD {0.4} {0.0001} CMPFLO[0]   // → true (0.30000...04 ≈ 0.4 within epsilon)
  ```

### 2.12–2.13 Reserved

```
149     (Reserved, unused)
150     (Reserved, unused)
```

### 2.14 Numeric Sequence

```
151     RANGE(1)    Create numeric sequence
```

- **Auxiliary params:** 1 byte — sequence length (maximum 255). A value of `0` creates an empty slice.
- **Arguments:**
  1. Start value (Int or Float).
  2. Step value (same type as start).
- **Return value:** A numeric slice of the specified length.
- The result type follows the type of the start value argument.
- **Examples:**
  ```
  {10} {1} RANGE[5]     // → [10, 11, 12, 13, 14]
  {0.0} {0.5} RANGE[4]  // → [0.0, 0.5, 1.0, 1.5]
  {100} {-10} RANGE[3]  // → [100, 90, 80]
  {0} {1} RANGE[0]      // → [] (empty slice)
  ```
- **Design note:** The maximum length is 255 (1-byte auxiliary parameter). For larger collections, generate multiple ranges and merge them with `MERGE`.

### 2.15 Map Iteration

```
152     MAP{}(~)    Map iteration
```

- **Auxiliary params:** Varint — sub-block byte length.
- **Arguments:** The first argument is the target collection (slice or dict). Any additional arguments become the initial members of the sub-block's private data stack.
- **Return value:** A slice containing the collected results from all iterations.
- **Behavior:** Iterates over each member of the target collection, executing the sub-block once per member. Each iteration's return value (via `RETURN`) is collected into the result slice.
- **Execution domain:** The sub-block has an **independent script domain** with its own data stack and argument space. However, successive iterations **share** the same data stack and argument space — state persists across iterations.
- **Loop variables** (accessible via `$X` capture instructions):
  - `$Value` (index 0) — current element value.
  - `$Index` / `$Key` (index 1) — current index (slice) or key (dict).
  - `$Slice` / `$Dict` (index 2) — the collection itself.
  - `$Size` (index 3) — collection size.
- If an iteration returns `nil`, that result is **ignored** (not added to the result slice). This provides a built-in filtering capability.
- **Dict iteration order:** When the target is a dict, iteration follows **insertion order**, consistent with the ordered map design (see [Collection Operations](4.Collection-Operations.md)).
- **Example:**
  ```
  {1, 2, 3, 4, 5}
  MAP{
       $Value {10} MUL
       RETURN              // return current value × 10
  }
  // → [10, 20, 30, 40, 50]
  ```

### 2.16 Filter Iteration

```
153     FILTER{}(~) Member filtering
```

- **Auxiliary params:** Varint — sub-block byte length.
- **Arguments:** The first argument is the target collection (slice or dict). Any additional arguments become the initial members of the sub-block's private data stack.
- **Return value:** A collection containing only the members for which the sub-block returned `true`.
- **Behavior:** Same execution semantics as `MAP{}`, but the sub-block is expected to return a boolean value. Members whose iteration returns `true` are included in the result; others are excluded.
- When the target is a **dict**, `FILTER` returns a new dict containing only the qualifying key-value pairs.
- The return type matches the target collection type (slice in → slice out, dict in → dict out).
- **Example:**
  ```
  {1, 2, 3, 4, 5, 6}
  FILTER{
       $Value {2} MOD {0} EQ
       RETURN              // return true for even numbers
  }
  // → [2, 4, 6]
  ```

### 2.17 Embedded Go Extension

```
154     EGO{}(~)    Embedded Go extension
```

- **Auxiliary params:** Varint — content byte length.
- **Associated data:** Sub-block content treated as raw byte data (Go source code).
- **Arguments:** None.
- **Return value:** Determined by the embedded code.
- **Restrictions:**
  - Only effective in **attestation identification scripts**. Validation nodes executing coin or credential scripts will **ignore** this instruction entirely (treated as a no-op).
  - Can appear in lock scripts for `OUTPUT` transfer purposes — the embedded code travels with the script but is only executed in attestation contexts.
- **Execution:** When applicable, the embedded Go code is executed by an interpreter (e.g., Yaegi) within a sandboxed environment.
- **Design note:** This extension point enables attestation scripts to leverage Go's full expressiveness for complex identification logic that would be impractical to express in stack-based instructions alone.

### 2.18 Embedded Shell Extension

```
155     SHELL{}(~)  Embedded shell extension
```

- **Auxiliary params:** Varint — content byte length.
- **Associated data:** Sub-block content treated as raw byte data (shell script). The first line serves as the interpreter directive (shebang line, e.g., `#!/bin/bash`).
- **Arguments:** None.
- **Return value:** Determined by the embedded script.
- **Restrictions:** Same as `EGO` — only effective in attestation identification scripts. Ignored by validation nodes in coin/credential scripts.
- **Design note:** Provides an alternative to `EGO` for cases where shell scripting is more appropriate or where external command-line tools need to be invoked.

### 2.19–2.26 Reserved for Quantum Safety

```
156–163 (Reserved) — Future quantum-safe cryptographic support
```

Eight opcode slots reserved for post-quantum cryptographic primitives. These may include:

- Quantum-resistant signature verification algorithms.
- Post-quantum key encapsulation mechanisms.
- Quantum-safe hash functions.

The specific instructions will be defined when quantum computing threats necessitate migration from current cryptographic algorithms.

### Design Notes

- **Consensus safety:** Three instructions (`RANDOM`, `QRAND`, `SLRAND`) produce non-deterministic results across nodes. They are explicitly restricted to non-consensus logic. Script authors must ensure these instructions appear only in private sections (after `END`) or within `EVAL` isolation contexts.
- **MAP/FILTER domain model:** Both instructions create independent script domains for their sub-blocks, but iterations within each block share state. This design enables accumulator patterns (where iteration N depends on iteration N-1) while keeping the functional block isolated from the caller.
- **Extension instructions** (`EGO`, `SHELL`) are deliberately restricted to attestation scripts. This prevents arbitrary code execution in financial transaction validation while enabling rich programmability for attestation identification — a domain where custom logic requirements are more varied and less security-critical than coin/credential transfers.
