# Proposal: Interaction Instructions [46–49]

> **Prerequisite:** This specification builds upon the [Base Constraints](0.Base-Constraints.md) document, which defines the instruction format, runtime model, and notation conventions used throughout.


## 1. Overview

Interaction instructions occupy opcode range **[46–49]** (4 instructions). They manage data exchange between the script execution environment and external systems through **import/export buffers** — FIFO queues that serve as the script's I/O channels.

The buffer system provides a controlled interface for scripts to receive input from and send output to the broader validation context, including cross-script communication via `GOTO`/`EMBED` and external third-party handler integration.

### Buffer Architecture

```
External Source ──► [Import Buffer (FIFO)] ──► Script (INPUT)
                                                    │
Script (OUTPUT) ──► [Export Buffer (FIFO)] ──► BUFDUMP ──► External Handler
```

- **Import Buffer:** Read-only from the script's perspective. Data is consumed (dequeued) by `INPUT`. The buffer is shared across `GOTO`/`EMBED` boundaries.
- **Export Buffer:** Write-only from the script's perspective. Data is enqueued by `OUTPUT` and flushed by `BUFDUMP`. Also shared across `GOTO`/`EMBED` boundaries.


## 2. Instruction Definitions

### 2.1 Import Data

```
46      INPUT(1)    Import from buffer
```

- **Auxiliary params:** 1 byte — number of items to read (`[0–255]`). `0` means all available.
- **Arguments:** None.
- **Return value:** Variable count — items are auto-expanded into the target space.
- **Blocking behavior:** If the import buffer is empty or contains fewer items than requested, execution **fails and exits normally** (not an error termination).
- **Examples:**
  - `@INPUT` — import all buffer data, expand into argument space.
  - `INPUT[3]` — import 3 items from buffer head, expand onto data stack.
  - `@INPUT NOP` — consume and discard all import buffer contents.
- **Design note — graceful failure:** Import failure is treated as a **normal exit**, preserving any pass-state already achieved. This is critical because:
  - For public validation nodes that never provide import data, `INPUT` serves as an implicit `END` — the script terminates gracefully without affecting prior `PASS` results.
  - This enables scripts to have a "public validation section" followed by a "private interaction section" that only executes when input data is available.

### 2.2 Export Data

```
47      OUTPUT      Export to buffer
```

- **Auxiliary params:** None
- **Arguments:** Variable count.
- **Return value:** None.
- Transfers all argument space contents to the export buffer, preserving FIFO order. Non-blocking.
- If the argument space is empty, zero items are exported.
- **Examples:**
  - `@POPS OUTPUT` — pop all stack items, export to buffer.
  - `... OUTPUT` — if argument space is empty, exports nothing.

### 2.3 Buffer Dump (Flush Export)

```
48      BUFDUMP(1)  Flush export buffer
```

- **Auxiliary params:** 1 byte — a user-defined tag/identifier.
- **Arguments:** None.
- **Return value:** None.
- Flushes (clears) the export buffer content to an external handler. Non-blocking.
- The auxiliary parameter serves as a **positional marker** — a tag that identifies this dump point, allowing external receivers to distinguish between multiple dump operations in the same script.
- **External handler requirement:** A third-party handler must be registered to receive dumped data. Handlers are private deployment logic — only nodes that need specific processing will deploy them.
- **Example:**
  ```
  @POPS[3] OUTPUT BUFDUMP
  ```
  Pop 3 items → export to buffer → flush to external handler.

### 2.4 Print (Debug Output)

```
49      PRINT       Print to console
```

- **Auxiliary params:** None
- **Arguments:** Variable count.
- **Return value:** None.
- Outputs argument space contents to the console using default formatting (equivalent to `fmt.Println()`).
- Intended for debugging and development purposes.
- **Examples:**
  - `@TOP PRINT` — print the top stack item.
  - `@POPS[3] PRINT` — print the top 3 stack items.


## 3. Buffer Sharing Semantics

Import and export buffers are shared across script boundaries created by `GOTO` and `EMBED`:

| Aspect | GOTO | EMBED |
|--------|------|-------|
| Import buffer | Shared with caller | Shared with caller |
| Export buffer | Shared with caller | Shared with caller |
| Data stack | Independent (new) | Shared with caller |
| Argument space | Transfer → initial stack | Shared with caller |

This means a `GOTO` target script can `INPUT` data that was placed in the buffer by the caller, and can `OUTPUT` data that the caller (or another `GOTO` target) will later `INPUT` or that will be flushed via `BUFDUMP`.


## 4. Interaction Pattern: Public vs. Private Validation

The `INPUT` instruction's graceful-failure behavior enables a two-phase script pattern:

```
// Phase 1: Public validation (all nodes execute)
<signature verification>
<hash checks>
PASS

// Phase 2: Private interaction (only participating nodes)
INPUT[2]              // Fails gracefully on public nodes → script ends here
<private logic using imported data>
OUTPUT
BUFDUMP[1]
```

Public validation nodes reach `INPUT`, find no data, and exit normally with the `PASS` state preserved. Participating nodes that provide import data continue into the private logic section.

This design elegantly separates consensus-critical validation from application-specific data processing without requiring explicit branching or conditional compilation.
