# Proposal: Capture Instructions [20-24]

> **Prerequisite:** This specification builds upon the [Base Constraints](0.Base-Constraints.md) document, which defines the instruction format, runtime model, and notation conventions used throughout.


## 1. Overview

Capture instructions occupy opcode range **[20-24]** (5 instructions). They serve as **prefix modifiers** that intercept, redirect, or re-source the data flow around other instructions. Rather than operating on the data stack themselves, capture instructions alter *how* the immediately following instruction interacts with the runtime environment.

This category is fundamental to the script system's expressiveness. By decoupling instructions from the rigid push/pop pattern of a pure stack machine, capture instructions enable flexible data routing through the argument space, local scope, and loop variables.

### Design Principle

Capture instruction mnemonics use special symbols (`@`, `~`, `$`) rather than words, allowing them to visually attach to their target instruction without mandatory whitespace separation. This compact notation reflects their role as lightweight modifiers rather than standalone operations.

### Key Distinction

Of the five capture instructions:
- **Three are value-retrieving** (`@`, `$[n]`, `$X[n]`): they place values into the argument space.
- **One is value-storing** (`$`): it redirects output to local scope.
- **One is source-switching** (`~`): it forces stack-direct argument consumption.


## 2. Instruction Definitions

### 2.1 Argument Capture

```
20      @           Argument capture
```

- **Auxiliary params:** None
- **Associated data:** None
- **Behavior:** Intercepts the return value of the **immediately following** instruction and places it into the argument space (FIFO queue) instead of onto the data stack.
- **Examples:**
  - `@PEEKS[2]` — peek at the bottom 2 stack items, expand them into the argument space.
  - `@{1234}` — place integer 1234 into argument space (it would normally auto-push to stack).
- **Semantics:**
  - If the following instruction returns multiple values (e.g., `PEEKS` returning N items), all values are appended to the argument space in order.
  - The capture applies to exactly one following instruction — it does not propagate.

### 2.2 Direct Stack Consumption

```
21      ~           Direct stack consumption
```

- **Auxiliary params:** None
- **Associated data:** None
- **Behavior:** Directs the immediately following instruction to source its arguments **directly from the data stack**, bypassing the argument space entirely, even if the argument space contains values.
- This is an explicit channel-switching directive.
- **Ordering with other prefixes:** If the target instruction also has a capture prefix (`@` or `$`), the `~` must appear **after** the capture prefix and **before** the target instruction:
  ```
  @~EQUAL    — take operands from stack, capture result to argument space
  $~EQUAL    — take operands from stack, store result to local scope
  ```
- **Examples:**
  - `~EQUAL` — take top two stack values for equality comparison (ignoring argument space).
  - `@~EQUAL` — same comparison, but result goes to argument space.
  - `$~EQUAL` — same comparison, but result goes to local scope.
- **Design note:** Supporting explicit stack-direct consumption adds flexibility for data manipulation, particularly when the argument space is occupied with values intended for a later instruction.

### 2.3 Local Scope Store

```
22      $           Local scope store
```

- **Auxiliary params:** None
- **Associated data:** None
- **Behavior:** Intercepts the return value of the immediately following instruction and **appends** it to the current block's local scope, instead of pushing to the data stack.
- Local scope is an **append-only** static storage area with a capacity of 128 entries per syntactic block.
- **Examples:**
  - `$TOP` — reference the stack top item and append it to local scope (reference copy).
  - `$POPS[3]` — pop top 3 items from stack, expand all three into local scope (adds 3 entries).
  - `${100}` — place value 100 directly into local scope.
  - `$SHIFT[3]` — take top 3 stack items packed as a single slice, append to local scope (adds 1 entry).
- **Scope isolation:** Local scope is strictly block-private. Different blocks (including sibling blocks and nested blocks) cannot access each other's local scopes. Only instructions at the same syntactic level share a local scope.

### 2.4 Local Scope Retrieve

```
23      $(1)        Local scope retrieve (by reference)
```

- **Auxiliary params:** 1 byte — member index (signed int8, range `[-128, 127]`).
- **Associated data:** None
- **Behavior:** Retrieves a value from the current block's local scope by index and places it into the **argument space**.
- Negative indices count from the end: `$[-1]` is the last appended entry.
- Must be used in conjunction with prior `$` (store) operations.
- **Examples:**
  - `$[0]` — retrieve the first value added to local scope.
  - `$[-1]` — retrieve the most recently added value.
- **Possible extension:** Name-based retrieval (e.g., `${POP}` to reference the value stored by `$POP`) may be supported, though this introduces name-collision concerns when the same instruction name is used for multiple stores.

### 2.5 Loop Variable Reference

```
24      $X(1)       Loop variable reference
```

- **Auxiliary params:** 1 byte — variable identifier `[0-3]`.
- **Associated data:** None
- **Behavior:** Within a loop body (`EACH{}`), retrieves a loop-context variable and places it into the **argument space**. Unlike local scope (`$[n]`), loop variable references can **cross syntactic block boundaries** within the enclosing loop.
- **Variable identifiers:**

  | Index | Named Reference | Description |
  |-------|----------------|-------------|
  | 0 | `$Value` | Current iteration element value |
  | 1 | `$Index` / `$Key` | Current iteration key — integer index for slices, string key for dicts |
  | 2 | `$Slice` / `$Dict` | The collection being iterated over |
  | 3 | `$Size` | Collection size (number of members) |

- Named references are **case-sensitive** and prefixed with `$`.
- The iterator target determines which name aliases apply: slice iteration uses `$Index` and `$Slice`; dict iteration uses `$Key` and `$Dict`.
- **Cross-block access:** Unlike `$[n]` which is block-private, `$Value`, `$Index`, etc. are accessible from any depth within the enclosing `EACH{}` block, including nested `IF{}`, `BLOCK{}`, etc.


## 3. Prefix Ordering Rules

When multiple capture prefixes are combined, they must follow this strict order:

```
[capture: @ or $] [source-switch: ~] <target instruction>
```

| Combination | Meaning |
|-------------|---------|
| `@INSTR` | Capture result of INSTR -> argument space |
| `$INSTR` | Capture result of INSTR -> local scope |
| `~INSTR` | Force INSTR to read arguments from stack (bypass argument space) |
| `@~INSTR` | Force stack arguments + capture result to argument space |
| `$~INSTR` | Force stack arguments + capture result to local scope |

Invalid combinations:
- `~@INSTR` — wrong order (source-switch before capture).
- `@$INSTR` — conflicting destinations (cannot capture to both argument space and local scope).


## 4. Data Flow Summary

```
                   ┌─────────────┐
                   │  Data Stack  │ <-- default push target
                   └──────┬──────┘
                          │ fallback source
                          v
┌──────────────┐   ┌─────────────┐   ┌──────────────┐
│ Local Scope  │   │  Argument   │   │   Target     │
│   $[n]       │<--│   Space     │-->│  Instruction │
│   $X[n]      │   │  (FIFO)     │   │              │
└──────────────┘   └─────────────┘   └──────────────┘
       ^                  ^
       │                  │
    $ prefix           @ prefix
    (store)           (capture)

    ~ prefix: forces target to bypass argument space -> read from stack directly
```

The three retrieval instructions (`@`, `$[n]`, `$X[n]`) all deposit their values into the argument space, making them available for the next instruction that consumes arguments. The store instruction (`$`) redirects to local scope. The source-switch (`~`) overrides the argument retrieval priority.
