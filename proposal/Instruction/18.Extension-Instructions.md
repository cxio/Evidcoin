# Proposal: Extension Instructions [250–253]（扩展指令）

> **Prerequisite:** 本规范建立在 [Base Constraints](0.Base-Constraints.md) 文档之上，该文档定义了全局使用的指令格式、运行时模型与记号约定。


## 1. Overview（概览）

扩展指令占用操作码区间 **[250–253]**（4 条）。它们提供通用扩展机制，用于在基础指令集、函数集、模块集之外继续增加功能。每条扩展指令都使用 **2-byte** 辅助参数，每种扩展类型最多可寻址 **65,536**（64K）个目标槽位。

定义了四类扩展：

| Opcode | Instruction | Category | Purpose |
|--------|-------------|----------|---------|
| 250 | EX_FN | Function | 扩展函数分发（超出 FN_X 的 256 槽位） |
| 251 | EX_MO | Module | 扩展模块分发（超出 MO_X 的 256 槽位） |
| 252 | EX_INST | General | 自包含扩展指令单元 |
| 253 | EX_PRIV | Private | 第三方私有扩展 |

> **Note:** 操作码 **254** 与 **255** 为系统保留值，不属于指令集，绝不可出现在脚本 bytecode 中。

> **Note:** 这些指令用于未来扩展，详细语义仍需后续设计；当前规范先定义地址框架与目标使用场景。


## 2. Instruction Definitions（指令定义）

### 2.1 Function-Type Extension（函数型扩展）

```
250     EX_FN(2){}  Function-type extension
```

- **Auxiliary params:** 2 bytes——目标函数索引（`[0–65535]`）。
- **Associated data:** 可选目标名称（使用 `{}` 记法时）。
- **Arguments:** 由目标扩展函数决定。
- **Return value:** 由目标扩展函数决定。
- **Design notes:**
  - 将函数地址空间扩展到 `FN_X`（opcode 209）256 槽位之外。`FN_X` + `EX_FN` 合计可支持 256 + 65,536 = **65,792** 个扩展函数。
  - 2-byte 索引采用 big-endian 字节序编码。
  - 扩展函数沿用核心函数调用约定：参数来自数据栈，结果压回数据栈。
- **Example:**
  ```
  <args...> EX_FN[300]          // 调用索引 300 的扩展函数
  <args...> EX_FN{CustomHash}   // 调用名为 "CustomHash" 的扩展函数
  ```

### 2.2 Module-Type Extension（模块型扩展）

```
251     EX_MO(2){}  Module-type extension
```

- **Auxiliary params:** 2 bytes——目标模块索引（`[0–65535]`）。
- **Associated data:** 可选“目标名称+成员名”（使用 `{}` 记法时）。
- **Arguments:** 由目标扩展模块成员函数决定。
- **Return value:** 由目标扩展模块成员函数决定。
- **Design notes:**
  - 将模块地址空间扩展到 `MO_X`（opcode 249）256 子模块之外。每个扩展模块维护自身成员函数表。
  - 沿用相同模块模式：模块内成员仍通过 1-byte 进行选择。
  - `{}` 记法支持点式成员访问，与模块约定一致。
- **Example:**
  ```
  <args...> EX_MO{Storage.Put}   // 调用扩展模块 "Storage" 的 Put 成员
  <args...> EX_MO[512]           // 调用索引 512 的扩展模块
  ```

### 2.3 General-Type Extension（通用型扩展）

```
252     EX_INST(2){}    General-type extension
```

- **Auxiliary params:** 2 bytes——目标指令索引（`[0–65535]`）。
- **Associated data:** 可选目标名称（使用 `{}` 记法时）。
- **Arguments:** 由目标扩展指令决定。
- **Return value:** 由目标扩展指令决定。
- **Design notes:**
  - 这是不适合函数或模块范式时的兜底扩展入口。
  - 扩展指令是 **self-contained logic units**——可自行定义辅助参数、数据格式与执行语义。
  - 2-byte 地址空间为通用扩展提供 64K 槽位。
- **Example:**
  ```
  <args...> EX_INST[1024]        // 调用索引 1024 的扩展指令
  <args...> EX_INST{Snapshot}    // 调用名为 "Snapshot" 的扩展指令
  ```

### 2.4 Third-Party Private Extension（第三方私有扩展）

```
253     EX_PRIV(2){}    Third-party private extension
```

- **Auxiliary params:** 2 bytes——目标扩展索引（`[0–65535]`）。
- **Associated data:** 可选目标名称（使用 `{}` 记法时）。
- **Arguments:** 由第三方扩展决定。
- **Return value:** 由第三方扩展决定。
- **Design notes:**
  - 专用于 **private, non-consensus** 功能。公共校验节点**不会处理**这类指令——在共识校验中会被跳过或忽略。
  - 适用于私有应用、企业部署、实验性功能等不参与公共链共识的场景。
  - 第三方可在该命名空间内独立开发与注册扩展，而不影响公共网络。
  - 含 `EX_PRIV` 的脚本在网络上依然有效，但其私有指令不会产生共识可见效果。
- **Example:**
  ```
  <args...> EX_PRIV{AuditLog}   // 私有审计日志（校验节点忽略）
  <args...> EX_PRIV[100]        // 索引 100 的私有扩展
  ```


## 3. Design Notes（设计说明）

### Address Space Summary（地址空间总览）

| Mechanism | Aux Size | Slots | Typical Use |
|-----------|----------|-------|-------------|
| `FN_X` (opcode 209) | 1 byte | 256 | 常用扩展函数 |
| `EX_FN` (opcode 250) | 2 bytes | 65,536 | 大规模函数扩展 |
| `MO_X` (opcode 249) | 1 byte | 256 sub-modules × 256 members | 常用扩展模块 |
| `EX_MO` (opcode 251) | 2 bytes | 65,536 modules × 256 members | 大规模模块扩展 |
| `EX_INST` (opcode 252) | 2 bytes | 65,536 | 通用扩展 |
| `EX_PRIV` (opcode 253) | 2 bytes | 65,536 | 私有/非共识扩展 |

### Consensus Boundary（共识边界）

`EX_INST`（参与共识）与 `EX_PRIV`（非共识）的区分是基础边界。公共校验节点必须执行 `EX_FN`、`EX_MO`、`EX_INST` 并将其效果纳入状态迁移；`EX_PRIV` 明确排除在共识之外，用于私链或应用层功能，而不增加公共网络负担。

### System-Reserved Opcodes（系统保留操作码）

操作码 **254** 与 **255** 为系统永久保留。它们作为哨兵值，不得出现在合法脚本 bytecode 中。实现应将其出现视为 bytecode 格式错误。
