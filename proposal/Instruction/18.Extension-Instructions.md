# Proposal: Extension Instructions [250–253]

> **Prerequisite:** This specification builds upon the [Base Constraints](0.Base-Constraints.md) document, which defines the instruction format, runtime model, and notation conventions used throughout.


## 1. Overview

Extension instructions occupy opcode range **[250–253]** (4 instructions). They provide a generic extension mechanism for adding functionality beyond the base instruction set, the function set, and the module set. Each extension instruction uses a **2-byte** auxiliary parameter, enabling an address space of up to **65,536** (64K) target slots per extension type.

Four extension categories are defined:

| Opcode | Instruction | Category | Purpose |
|--------|-------------|----------|---------|
| 250 | EX_FN | Function | Extended function dispatch (beyond FN_X's 256 slots) |
| 251 | EX_MO | Module | Extended module dispatch (beyond MO_X's 256 slots) |
| 252 | EX_INST | General | Self-contained extended instruction units |
| 253 | EX_PRIV | Private | Third-party private extensions |

> **Note:** Opcodes **254** and **255** are system-reserved and are not part of the instruction set. They must never be used in script bytecode.

> **Note:** These instructions are reserved for future expansion. Their detailed semantics require further design work; the current specification establishes the addressing framework and intended use cases.


## 2. Instruction Definitions

### 2.1 Function-Type Extension

```
250     EX_FN(2){}  Function-type extension
```

- **Auxiliary params:** 2 bytes — target function index (`[0–65535]`).
- **Associated data:** Optional target name (when using `{}` notation).
- **Arguments:** Determined by the target extended function.
- **Return value:** Determined by the target extended function.
- **Design notes:**
  - Extends the function address space beyond the 256 slots provided by `FN_X` (opcode 209). Together, `FN_X` and `EX_FN` provide a combined capacity of 256 + 65,536 = **65,792** extended functions.
  - The 2-byte index is encoded in big-endian byte order.
  - Extended functions follow the same calling convention as core function instructions: arguments from the data stack, result pushed to the data stack.
- **Example:**
  ```
  <args...> EX_FN[300]          // call extended function at index 300
  <args...> EX_FN{CustomHash}   // call extended function named "CustomHash"
  ```

### 2.2 Module-Type Extension

```
251     EX_MO(2){}  Module-type extension
```

- **Auxiliary params:** 2 bytes — target module index (`[0–65535]`).
- **Associated data:** Optional target name with member (when using `{}` notation).
- **Arguments:** Determined by the target extended module's member function.
- **Return value:** Determined by the target extended module's member function.
- **Design notes:**
  - Extends the module address space beyond the 256 sub-module slots provided by `MO_X` (opcode 249). Each extended module maintains its own member function table.
  - Follows the same module pattern: 1-byte member selection within each extended module.
  - The `{}` notation supports dotted member access, consistent with module conventions.
- **Example:**
  ```
  <args...> EX_MO{Storage.Put}   // call Put member of extended module "Storage"
  <args...> EX_MO[512]           // call extended module at index 512
  ```

### 2.3 General-Type Extension

```
252     EX_INST(2){}    General-type extension
```

- **Auxiliary params:** 2 bytes — target instruction index (`[0–65535]`).
- **Associated data:** Optional target name (when using `{}` notation).
- **Arguments:** Determined by the target extended instruction.
- **Return value:** Determined by the target extended instruction.
- **Design notes:**
  - A catch-all extension point for instructions that do not fit the function or module paradigms.
  - Extended instructions are **self-contained logic units** — they may define their own auxiliary parameters, data formats, and execution semantics.
  - The 2-byte address space provides 64K slots for general-purpose extensions.
- **Example:**
  ```
  <args...> EX_INST[1024]        // call extended instruction at index 1024
  <args...> EX_INST{Snapshot}    // call extended instruction named "Snapshot"
  ```

### 2.4 Third-Party Private Extension

```
253     EX_PRIV(2){}    Third-party private extension
```

- **Auxiliary params:** 2 bytes — target extension index (`[0–65535]`).
- **Associated data:** Optional target name (when using `{}` notation).
- **Arguments:** Determined by the third-party extension.
- **Return value:** Determined by the third-party extension.
- **Design notes:**
  - Dedicated to **private, non-consensus** functionality. Public validation nodes **do not process** these instructions — they are skipped or ignored during consensus validation.
  - Intended for private applications, enterprise deployments, or experimental features that operate outside the public blockchain consensus.
  - Third parties may independently develop and register extensions within this namespace without affecting the public network.
  - Scripts containing `EX_PRIV` instructions are valid on the network but the private instructions produce no consensus-visible effects.
- **Example:**
  ```
  <args...> EX_PRIV{AuditLog}   // private audit logging (ignored by validators)
  <args...> EX_PRIV[100]        // private extension at index 100
  ```


## 3. Design Notes

### Address Space Summary

| Mechanism | Aux Size | Slots | Typical Use |
|-----------|----------|-------|-------------|
| `FN_X` (opcode 209) | 1 byte | 256 | Common extended functions |
| `EX_FN` (opcode 250) | 2 bytes | 65,536 | Large-scale function extensions |
| `MO_X` (opcode 249) | 1 byte | 256 sub-modules × 256 members | Common extended modules |
| `EX_MO` (opcode 251) | 2 bytes | 65,536 modules × 256 members | Large-scale module extensions |
| `EX_INST` (opcode 252) | 2 bytes | 65,536 | General-purpose extensions |
| `EX_PRIV` (opcode 253) | 2 bytes | 65,536 | Private/non-consensus extensions |

### Consensus Boundary

The distinction between `EX_INST` (consensus-participating) and `EX_PRIV` (non-consensus) is fundamental. Public validation nodes must execute `EX_FN`, `EX_MO`, and `EX_INST` instructions and include their effects in state transitions. `EX_PRIV` instructions are explicitly excluded from consensus — they enable private-chain or application-layer functionality without burdening the public network.

### System-Reserved Opcodes

Opcodes **254** and **255** are permanently reserved by the system. They serve as sentinel values and must never appear in valid script bytecode. Implementations should treat their presence as a bytecode format error.
