# Proposal: Pattern Instructions [116–127]（提案：模式指令）

> **Prerequisite:** 本规范基于 [Base Constraints](0.Base-Constraints.md) 文档，后者定义了贯穿全文的指令格式、运行时模型与符号约定。


## 1. Overview（概述）

模式指令占用 opcode 范围 **[116–127]**（共 12 条）。它们提供 **pattern matching on scripts**：用于验证目标指令序列是否符合特定结构模式（formal verification）。在匹配过程中，还可从目标中提取信息供后续处理。

这是 Evidcoin 脚本系统中最复杂的一类指令。模式匹配使脚本能够在运行时检查并校验其他脚本的结构，这对以下场景至关重要：

- **Script template verification** —— 确认锁定脚本是否符合已知标准模板（如 P2PKH、多签）。
- **Structural introspection** —— 在不执行脚本的前提下，提取其中嵌入的公钥、哈希值或参数。
- **Policy enforcement** —— 在接受脚本前，验证其是否遵循治理规则。

### Core Concepts（核心概念）

**MODEL block:** 所有模式匹配都在 `MODEL{}` block（opcode 116）内完成。该 block 会开启专用解析子环境。块内的模式指令描述目标应具备的结构，执行引擎按位置对目标指令序列进行匹配。

**Matching methods:** 模式系统支持以下 8 种互补匹配策略：

| Method | Description |
|--------|-------------|
| Position matching | 按位置将模式指令与目标指令一一匹配 |
| Instruction matching | 模式中的普通指令要求目标同位置出现同一指令（值指令同时隐含类型） |
| Auxiliary param matching | 在指令匹配成立时，auxiliary params 参与相等性比较 |
| Data matching | 在指令匹配成立时，associated data 参与相等性比较 |
| Type matching | 仅匹配目标数据类型而不关心具体值（`?!` / `??`） |
| Range limiting | 将 Int 或 Float 值限制在指定范围内（`!{}`） |
| Regex matching | 对文本/字节序列应用正则表达式（`RE{}`） |
| Segment matching | 匹配可变长度的连续指令片段（`_()`, `...`） |

### Terminology（术语）

- **Target script:** 被检查的指令序列（`MODEL{}` 的参数）。
- **Pattern:** `MODEL{}` block 内部用于描述预期结构的指令序列。
- **Static instruction:** 指令的完整字节表示：opcode、auxiliary params、associated data 三者合并后的整体。匹配单位即为此。
- **Extraction:** 从已匹配目标指令中捕获指定部分，作为返回值输出的过程。


## 2. Instruction Definitions（指令定义）

### 2.1 Model Block — MODEL（模型块）

```
116     MODEL{}(~)  Pattern matching block
```

- **Auxiliary params:** Varint —— 子块代码长度。该 varint 的 **high bit** 作为提取标志位：置位时，MODEL 除匹配结果外还会返回提取值。这也将子块最大长度限制为 **32 KB**（15-bit 长度，2^15 = 32,768 bytes）。
- **Arguments:** 一个值 —— 目标序列（`Script` 或 `Bytes` 类型）。
- **Return value:**
  - **Without extraction**（high bit = 0）：`Bool` —— 匹配成功为 `true`，否则 `false`。
  - **With extraction**（high bit = 1）：`(Slice, Bool)` —— 返回按提取顺序收集的切片与布尔匹配结果。若匹配失败，Slice 为空。
- MODEL block 会开启 **parsing sub-environment**，引擎并行推进 pattern 与 target，逐位置比较。
- MODEL 内的 **ordinary instructions** 会被解释为 **strict equality matchers**：目标在对应位置必须是完全相同指令（同 opcode、同 auxiliary params、同 associated data）。
- **Nested MODEL:** 若目标脚本本身含有 MODEL block，其内容在匹配中被视为 **ordinary associated data**，引擎不会递归进入。这样可避免无限匹配深度并保持可预测性能。
- **Examples:**
  ```
  // 简单结构检查：目标是否匹配 "DUP HASH EQUAL PASS"？
  @source
  MODEL{
       DUP HASH EQUAL PASS
  }
  // 返回：Bool（目标与 DUP HASH EQUAL PASS 完全一致则为 true）
  ```
  ```
  // 启用提取的结构检查（high bit 置位）：
  @source
  MODEL{
       DATA[20]{?} #[32]     // 匹配 20 字节 DATA，并提取其数据内容
       DUP HASH EQUAL PASS
  }
  // 返回：(Slice, Bool) —— Slice 包含提取出的 20-byte 数据
  ```
- **Design notes:**
  - 32 KB 子块上限对模式定义已很宽裕，即使复杂多签模式通常也仅数百字节。
  - 通过 high bit 承载 extraction flag，无需单独新增指令或模式切换，MODEL 保持紧凑。
  - MODEL block 可用于锁定脚本中的模板校验，也可用于校验逻辑里检查交易输出脚本结构。

### 2.2 Instruction Wildcard — _（单指令通配）

```
117     _           Instruction wildcard
```

- **Auxiliary params:** 无。
- **Arguments:** 无（仅模式指令）。
- **Return value:** 无（匹配指令）。
- 在当前目标位置匹配 **any single instruction**。目标可为任意 opcode、任意 auxiliary params、任意 associated data。但该位置 **must** 有指令存在，`_` 不能越过目标末尾匹配。
- “static instruction”包含 opcode、auxiliary params 与 associated data 的整体，`_` 匹配的就是这个整体。
- **Examples:**
  ```
  MODEL{
       _              // 第 1 条可为任意指令
       HASH           // 第 2 条必须是 HASH
       _              // 第 3 条可为任意指令
       PASS           // 第 4 条必须是 PASS
  }
  ```
- **Design note:** `_` 是最基础的通配表达，含义是“此处内容不关心，但必须有一条指令”。

### 2.3 Instruction Segment Wildcard — _()（定长片段通配）

```
118     _(~)        Instruction segment wildcard
```

- **Auxiliary params:** Varint —— 要匹配的总指令数（含当前位置）。
- **Arguments:** 无（仅模式指令）。
- **Return value:** 无（匹配指令）。
- 从当前位置开始匹配长度恰好为 N 的连续指令片段。它是 `_` 的 **plural form**，并且在片段长度已知时是 `...` 的 **efficient version**。
- 片段仅按 **same-level relationships** 计数。结构块指令（如 `IF{}`、`EACH{}`、`MODEL{}`）按 **single instruction** 计，其子代码块属于 associated data，不单独计数。
- **Examples:**
  ```
  MODEL{
       _[3]           // 匹配任意 3 条指令
       PASS           // 第 4 条必须是 PASS
  }
  ```
  ```
  // _[1] 等价于 _
  MODEL{
       _[1] HASH _[1] PASS
  }
  ```
- **Design note:** 当可在模式编译期确定要跳过的指令数时，强烈建议使用 `_()` 而非 `...`，因为它无需 backtracking；引擎可直接将目标游标前移 N 条。

### 2.4 Optional Instruction Sequence — ?{}（可选序列）

```
119     ?{}(~)      Optional instruction sequence
```

- **Auxiliary params:** Varint —— 序列字节长度（包裹的指令序列）。
- **Arguments:** 无（仅模式指令）。
- **Return value:** 无（匹配指令）。
- 包裹一个“可完整匹配或完全不存在”的指令序列：若存在则必须 **exactly** 匹配（含 auxiliary params 与 associated data 的全等）；否则允许整体缺失。这是 all-or-nothing 的可选匹配。
- 若当前位置可匹配完整包裹序列，则目标游标前进到序列之后；若首条指令不匹配，则该可选序列被 **skipped**，目标游标保持不动，后续模式继续从同一目标位置匹配。
- **Single-instruction shorthand:** 当可选序列仅含一条指令时，可省略花括号。`?POPS[3]` 等价于 `?{POPS[3]}`。
- **Examples:**
  ```
  MODEL{
       ?{@TOP PRINT}    // 可选匹配 @TOP 后接 PRINT
       DUP HASH         // 然后必须匹配 DUP HASH
       PASS
  }
  ```
  ```
  MODEL{
       ?POPS[3]         // 可选匹配 aux param 为 3 的 POPS
       PASS
  }
  ```
  ```
  MODEL{
       ?{1, 3, 5}       // 可选匹配值集合 {1, 3, 5}
       EQUAL PASS
  }
  ```
- **Design notes:**
  - 可选序列执行 **strict equality**：opcode、auxiliary params、associated data 必须全部一致。若需“可选且灵活”的匹配，请结合 wildcard indicator（opcode 120）。
  - 该可选序列采用 **non-backtracking**：若首条匹配但后续失败，则整段可选匹配失败，不会部分推进目标游标。

### 2.5 Wildcard Indicator — ?()（通配指示器）

```
120     ?(1)        Wildcard indicator
```

- **Auxiliary params:** 1 byte —— bitmask，标记其后续一条指令的哪些部分可通配。
- **Arguments:** 无（仅模式指令）。
- **Return value:** 无（匹配指令）。
- 该指令会修改模式中 **immediately following** 指令的匹配行为。没有 wildcard indicator 时，默认按全等匹配；有 wildcard indicator 时，被标记部分可接受任意值。
- **Bitmask definitions:**

  | Bit | Hex | Meaning |
  |-----|-----|---------|
  | 0 | `0x01` | 第 1 个 auxiliary parameter 可任意 |
  | 1 | `0x02` | 第 2 个 auxiliary parameter 可任意 |
  | 2 | `0x04` | 第 3 个 auxiliary parameter 可任意 |
  | 3 | `0x08` | 第 4 个 auxiliary parameter 可任意 |
  | 4 | `0x10` | 第 5 个 auxiliary parameter 可任意 |
  | 5 | `0x20` | associated data 可任意 |
  | 6 | `0x40` | 指令本体可选（可不存在） |
  | 7 | `0x80` | 对 associated data 做哈希匹配（按 hash 比较而非逐字节相等） |

- **Shorthand notation:** 为提高可读性，支持以下语法糖：
  - `DATA[20]{?}` → `?[0x20] DATA[20]` —— DATA 长度参数固定为 20-byte，但数据内容可任意。
  - `DATA[?]{?}` → `?[0x21] DATA` —— DATA 指令的辅助参数与数据内容都可任意。
  - `?DATA` → `?[0x40] DATA` —— DATA 指令可选；未明确的其他部分默认可任意。
- **Deep match vs. single match for block instructions:**
  - **Deep match (no wildcard indicator):** 当 block instruction（如 `IF{}`、`BLOCK{}`）在模式中且前面没有 wildcard indicator 时，引擎会 **enters** 子块，并对子块内容执行递进式位置匹配。
  - **Single match (with wildcard indicator):** 当 block instruction 带 wildcard indicator 时，其子块被视为 **associated data**；引擎把它当作单个不透明单元匹配，不进入子块内部。
- **Hash match mode**（bit 7，`0x80`）：不再逐字节比较 associated data，而是对目标 associated data 计算 hash，并与模式提供的 hash 进行比较。适用于按指纹匹配大数据。
- **Examples:**
  ```
  MODEL{
       ?[0x20] DATA[20]     // DATA 长度 20 字节，内容任意
       DUP HASH EQUAL PASS
  }
  ```
  ```
  MODEL{
       ?[0x21] DATA         // DATA 长度与内容均任意
       PASS
  }
  ```
  ```
  MODEL{
       ?[0x40] PRINT        // PRINT 为可选（可出现也可不出现）
       PASS
  }
  ```
  ```
  // Hash match：通过哈希指纹验证 DATA 内容
  MODEL{
       ?[0x80] DATA{0x4f262ae8...}    // 匹配内容哈希等于给定值的 DATA
       PASS
  }
  ```
- **Design notes:**
  - bitmask 编码紧凑（单字节）且表达力强，“任意数据”或“可选指令”等常见模式都只需一个 bit。
  - Hash matching（bit 7）特别适合校验大型嵌入脚本或数据载荷，可显著缩小模式体积。
  - bits 0–4 分别对应 auxiliary parameters 1–5。多数指令仅有 2–3 个辅助参数，bits 3–4 不常用但为复杂指令保留。

### 2.6 Instruction Extraction — #()（指令提取）

```
121     #(1)        Instruction extraction
```

- **Auxiliary params:** 1 byte —— bitmask，指定从“前一条已匹配的目标指令”提取哪些部分。
- **Arguments:** 无（仅模式指令）。
- **Return value:** 本指令本身不直接返回值；提取结果会累计到 MODEL 内部 extraction queue，最终作为 MODEL 返回值中的 `Slice` 组件。
- 必须 **immediately follow** 一条带 wildcard indicator（`?` opcode 120）的指令。提取对象是 **most recently matched** 的目标指令。
- **Bitmask definitions:**

  | Bit | Hex | Meaning |
  |-----|-----|---------|
  | 0 | `0x01` | 提取第 1 个 auxiliary parameter |
  | 1 | `0x02` | 提取第 2 个 auxiliary parameter |
  | 2 | `0x04` | 提取第 3 个 auxiliary parameter |
  | 3 | `0x08` | 提取第 4 个 auxiliary parameter |
  | 4 | `0x10` | 提取第 5 个 auxiliary parameter |
  | 5 | `0x20` | 提取 associated data |
  | 6 | `0x40` | 仅提取 opcode（作为 Byte 值） |
  | 7 | `0x80` | 提取完整指令（opcode + aux params + data，作为 Bytes） |

- 当 **multiple bits** 同时置位时，对应部分会按 **in order**（bit 0 到 bit 7）展开并写入内部 extraction queue。
- **Examples:**
  ```
  // 从 DATA 指令中提取 20-byte 数据内容
  MODEL{
       ?[0x20] DATA[20] #[0x20]     // 数据内容通配，然后提取数据
       DUP HASH EQUAL PASS
  }
  // 返回：(Slice{<20-byte data>}, true)
  ```
  ```
  // 同时提取辅助参数与 associated data
  MODEL{
       ?[0x21] DATA #[0x21]         // 两者均通配，并同时提取
       PASS
  }
  // 返回：(Slice{<aux param value>, <data bytes>}, true)
  ```
  ```
  // 提取完整指令
  MODEL{
       _ #[0x80]                    // 匹配任意指令并整体提取
       PASS
  }
  // 返回：(Slice{<complete instruction bytes>}, true)
  ```
- **Design notes:**
  - extraction bitmask 有意与 wildcard indicator bitmask（bits 0–5, 7）保持镜像一致。差异仅在 bit 6：在 wildcard indicator 中表示“optional”，在 extraction 中表示“extract opcode”。
  - 提取不会影响匹配流程；它只是对已匹配位置进行被动记录。
  - extraction queue 按 MODEL 中的出现顺序记录，输出顺序确定且可预测。

### 2.7 Type Match and Optional — ?!() / ??()（类型匹配与可选）

```
122     ?!(1){Type} Type match | optional
```

- **Auxiliary params:** 1 byte —— 类型标识。其 **high bit** 区分两种模式：
  - high bit **clear**（前缀 `!`）：**Match test** —— 目标指令必须产出指定类型值；不满足则 MODEL **fails immediately**（返回 `false`）。
  - high bit **set**（前缀 `?`）：**Optional test** —— 检查目标指令是否产出指定类型，但 **never fails**。若不匹配，则从同一目标位置继续后续匹配（即仅跳过本测试）。
- **Arguments:** 无（仅模式指令）。
- **Return value:** 无（匹配指令）。
- **Supported types**（取类型标识低 7 bits）：

  | ID | Type | Description |
  |----|------|-------------|
  | 0 | Bool | 布尔值 |
  | 1 | Byte | 单字节（0–255） |
  | 2 | Rune | Unicode code point |
  | 3 | Int | 有符号整数 |
  | 4 | BigInt | 任意精度整数 |
  | 5 | Float | 64-bit 浮点数 |
  | 6 | String | UTF-8 字符串 |
  | 7 | Bytes | 字节序列 |
  | 8 | RegExp | 正则表达式 |
  | 9 | Time | 时间戳 |
  | 10 | Script | Script/code 值 |
  | 11 | Dict | 字典（有序 map） |
  | 12 | Number | 任意数值类型（Byte、Rune、Int、BigInt、Float） |
  | 13 | Expr | 表达式值 |
  | 14 | Model | Pattern/model 值 |
  | 15 | Values | 值集合（slice） |

- 类型由目标指令推断：值指令天然携带类型；其他指令则取其返回值类型。
- **Examples:**
  ```
  MODEL{
       ?![7]              // 目标必须产出 Bytes；否则 MODEL 失败
       _                  // 下一条可为任意指令
       PASS
  }
  ```
  ```
  MODEL{
       ??[6]              // 可选匹配 String 指令（high bit 置位：6 + 128 = 134）
       HASH EQUAL PASS    // 无论是否匹配都继续
  }
  ```
- **Design notes:**
  - 在单一 opcode 内提供 match 与 optional 双模式，可减少 opcode 消耗并覆盖两类需求。
  - 类型匹配比指令匹配更粗粒度：它关注产出值的 **what**，而非产出过程的 **how**。当模式允许“任何能产出指定类型的指令”时非常有用。
  - `Number` 元类型（ID 12）是便捷别名，用于匹配任意五种数值类型。

### 2.8 Integer Range Match — !{}（整数区间匹配）

```
123     !{}(~,~)    Integer value range match
```

- **Auxiliary params:**
  - Varint 1 —— 下界（含）。
  - Varint 2 —— 上界（不含）。
- **Arguments:** 无（仅模式指令）。
- **Return value:** 无（匹配指令）。
- 将目标指令产出的 Int 值与半开区间 `[lower, upper)` 比较。目标必须是产出 Int 的指令。若越界，MODEL **fails**（返回 `false`）。
- 不支持 BigInt 区间匹配 —— 上下界与目标值都必须落在标准整数精度内。
- **Examples:**
  ```
  MODEL{
       !{0, 1024}         // 目标整数必须满足：0 ≤ n < 1024
       PASS
  }
  ```
  ```
  MODEL{
       !{1, 256}          // 类 byte 的有效范围：1 ≤ n < 256
       _[2]
       EQUAL PASS
  }
  ```
- **Design note:** 采用半开区间 `[lower, upper)`，与 Evidcoin 脚本系统的统一数学约定一致（与 `WITHIN`，opcode 111 保持一致），可减少 off-by-one 歧义并便于组合。

### 2.9 Float Range Match — !{}（Float 区间匹配）

```
124     !{}(8,8,4)  Float value range match
```

- **Auxiliary params:**
  - 8 bytes —— 下界（含，IEEE 754 float64）。
  - 8 bytes —— 上界（不含，IEEE 754 float64）。
  - 4 bytes —— 相等 epsilon（IEEE 754 float32）。用于下界“含等于”比较的容差：`|target - lower| ≤ epsilon` 视为相等。
- **Arguments:** 无（仅模式指令）。
- **Return value:** 无（匹配指令）。
- 将目标指令产出的 Float 值与 `[lower, upper)` 比较，并在下界相等测试中使用 epsilon 精度。目标必须是产出 Float 的指令。若越界，MODEL **fails**。
- epsilon 参数用于处理浮点相等天然不精确的问题。epsilon 为 `0` 时，下界比较为精确（bitwise equality）；epsilon 非 `0` 时，下界邻域内的值也视为满足。
- **Examples:**
  ```
  MODEL{
       !{0.0, 1.0, 0}       // 0.0 ≤ x < 1.0，下界精确比较
       PASS
  }
  // 匹配 [0.0, 1.0) 内任意 float，且 0 比较为精确零比较
  ```
  ```
  MODEL{
       !{0.5, 100.0, 0.001}  // 0.5 ≤ x < 100.0，epsilon = 0.001
       PASS
  }
  // [0.499, 0.501] 内的值都可通过下界测试
  ```
- **Design notes:**
  - 固定 8+8+4 bytes（共 20 bytes）布局是有意选择，因为浮点边界无法像 varint 一样有效压缩。
  - epsilon 使用 float32（4 bytes）而非 float64，是因为 epsilon 通常较小，不需要 double precision，可节省每条指令 4 bytes。
  - 本指令与整数区间匹配（opcode 123）共用 `!{}` 助记符，具体由 opcode 编号区分。

### 2.10 Regex Search — RE{}（正则搜索）

```
125     RE{}(1,1)   Regex search within MODEL
```

- **Auxiliary params:**
  - 1 byte —— 搜索模式：
    - `g` —— `FindAll`：查找 regex 在目标中的所有不重叠匹配。
    - `G` —— `FindAllSubmatch`：查找所有匹配并包含 submatch groups。
    - `s` —— `FindSubmatch`：查找首个匹配并包含 submatch groups。
  - 1 byte —— regex 内容长度（最大 255 bytes）。
- **Associated data:** regex pattern 字符串（最多 255 bytes）。
- **Arguments:** 无（仅模式指令）。
- **Return value:** 不直接返回值 —— 匹配结果内部保存，通过 `&` 提取指令（opcode 126）访问。
- 在目标指令的 associated data（或值）的 **文本或字节表示** 上执行正则搜索。regex 引擎遵循标准 RE2/Go `regexp` 语义。
- regex pattern 直接以内联 associated data 形式存放，长度由第 2 个辅助参数给出。
- **Examples:**
  ```
  MODEL{
       RE{/^[a-f0-9]{40}$/s}    // 匹配 40 位十六进制字符串，FindSubmatch 模式
       &[0]                     // 提取完整匹配（index 0）
       PASS
  }
  ```
  ```
  MODEL{
       RE{/(\d+)\.(\d+)/G}      // 查找所有 "N.N" 模式并提取子组
       &[0]                     // 提取第一组匹配结果
       PASS
  }
  ```
- **Design notes:**
  - 255-byte 的 regex 长度上限足以覆盖常见校验模式，同时避免过度复杂 regex 带来的性能问题。
  - 搜索模式用单字节编码，紧凑且足够覆盖常见 regex 搜索场景（`g`、`G`、`s`）。
  - RE2/Go 语义保证线性时间匹配，可避免灾难性回溯，这对区块链脚本执行安全性非常关键。

### 2.11 Regex Extraction — &()（正则提取）

```
126     &(1)        Regex extraction
```

- **Auxiliary params:** 1 byte —— 要提取的匹配/子匹配索引（0–255）。
- **Arguments:** 无（仅模式指令）。
- **Return value:** 不直接返回值 —— 提取值将追加到 MODEL 内部 extraction queue。
- 必须跟随其关联的 `RE{}` 指令。按索引从 regex 匹配结果中提取特定成员。
- **Index semantics:**
  - 对 `FindSubmatch`（`s` 模式）：index 0 为完整匹配，index 1 为第一个子组，index 2 为第二个子组，依此类推。
  - 对 `FindAll`（`g` 模式）：index 0 为第一个匹配，index 1 为第二个匹配，依此类推。
  - 对 `FindAllSubmatch`（`G` 模式）：结果会扁平化，index 0 为第一条完整匹配，index 1 为第一条匹配的第一个子组，依此类推。
- 若索引越界或 regex 无匹配结果，则提取值为 `nil`。
- 最多可寻址 256 个结果（index 0–255）。
- **Examples:**
  ```
  MODEL{
       RE{/^(\w+):(\d+)$/s}     // 匹配 "word:number" 模式
       &[1]                       // 提取 word 部分（子组 1）
       &[2]                       // 提取 number 部分（子组 2）
       PASS
  }
  // extraction queue: [<word>, <number>]
  ```
- **Design note:** `&` 助记符参考了常见脚本语言中对 regex capture group 的引用方式。

### 2.12 Arbitrary Segment Wildcard — ...（任意片段通配）

```
127     ...         Arbitrary instruction segment wildcard
```

- **Auxiliary params:** 无。
- **Arguments:** 无（仅模式指令）。
- **Return value:** 无（匹配指令）。
- 在当前位置匹配 **any contiguous sequence** 指令，包含 **zero-length**（即可匹配空序列）。这是模式系统中最宽松的通配符。
- 每个 `...` 仅作用于 **current nesting level**。block instruction 视为单一实体，`...` 不会下钻到子块内部。
- **Non-greedy semantics:** `...` 会匹配能使后续模式成功的**最短**连续序列，引擎尽量少推进目标游标。
- **Examples:**
  ```
  MODEL{
       ...                // 跳过任意前缀指令
       HASH EQUAL PASS    // 随后要求末尾出现 HASH EQUAL PASS
  }
  ```
  ```
  MODEL{
       DUP ...            // 以 DUP 开始，中间任意
       PASS               // 以 PASS 结束
  }
  ```
  ```
  MODEL{
       ... HASH ... PASS  // HASH 与 PASS 需按顺序出现，前后及中间任意
  }
  ```
- **Design notes:**
  - **WARNING: Resource-intensive.** `...` 需要 trial matching（backtracking）来寻找最短可行匹配。最坏情况下复杂度为 O(n × m)（n 为剩余目标长度，m 为剩余模式长度）。**Prefer `_()` with an explicit instruction count whenever possible.**
  - 选择 non-greedy（最短匹配）而非 greedy（最长匹配），是因为在模式校验场景中，通常希望尽早定位到后续锚点指令可匹配的位置，而非尽可能吞掉更多指令。
  - 单个 MODEL block 中多个 `...` 会叠加性能成本。建议尽量减少使用，并在两侧放置具体指令作为约束。


## 3. Matching Semantics Summary（匹配语义汇总）

### 3.1 Positional Matching Algorithm（按位匹配算法）

MODEL block 引擎维护两个游标：pattern 游标与 target 游标。匹配流程如下：

1. 对普通指令与精确匹配指令，**Advance both cursors**，保持同步前进。
2. **Wildcard instructions**（`_`、`_()`、`...`）在消耗一个模式位置的同时，按规则推进目标游标。
3. **Modifier instructions**（`?()`、`#()`、`&()`）只消耗一个模式位置，不推进目标游标 —— 它们用于修改或观察当前位置匹配行为。
4. **Optional sequences**（`?{}`、`??`）在当前位置尝试匹配；失败时目标游标不变。
5. **Range and type checks**（`!{}`、`?!`）在当前位置检查目标指令的值或类型；失败则 MODEL 直接返回 `false`。
6. 当两个游标同时到达各自序列末尾时，匹配 **succeeds**。
7. 出现以下情形时匹配 **fails**：
   - 必需匹配项（指令全等、类型检查、区间检查）不成立。
   - pattern 已结束但 target 仍有剩余指令（除非 pattern 以 `...` 结尾）。
   - target 已结束但 pattern 仍有非可选指令。

### 3.2 Block Instruction Matching（块指令匹配）

带 `{}` 子块的 block instruction（如 `IF{}`、`EACH{}`、`BLOCK{}`）遵循两条规则：

- **Without wildcard indicator (deep match):** 引擎进入 pattern 与 target 的子块，对子块内容递归执行按位匹配。
- **With wildcard indicator (single match):** 整个 block instruction（含子块）视为单个不透明指令，子块内容按 wildcard indicator bitmask 规则作为 associated data 匹配。

### 3.3 Extraction Queue（提取队列）

提取（`#` 与 `&`）会在匹配过程中把值按顺序累积到队列。启用 extraction flag 时，该队列会作为 MODEL 返回值中的 `Slice`。队列顺序按模式从左到右的提取发生顺序确定。


## 4. Design Notes（设计说明）

### 4.1 Performance Considerations（性能考量）

模式匹配天然比普通指令执行更昂贵，因为它需要双游标遍历，并且在 `...` 场景下可能触发 backtracking。可通过以下方式缓解：

- **Prefer `_()` over `...`** —— 显式长度通配可完全避免回溯。
- **Minimize `...` usage** —— 每个 `...` 都是潜在回溯点。
- 尽量把具体锚点指令提前，帮助在不匹配目标上快速失败。
- 32 KB 子块上限也为模式复杂度提供了隐式边界。

### 4.2 Safety Properties（安全属性）

- **No script mutation:** 模式匹配纯观察，不修改目标脚本，符合脚本不可变约束。
- **No recursive MODEL interpretation:** 对目标中的嵌套 MODEL 一律按数据处理，避免无界递归。
- **Bounded regex execution:** RE2/Go 语义保证 regex 线性时间匹配，且 255-byte 模式上限限制了 regex 复杂度。
- **Deterministic results:** 所有匹配操作都具确定性 —— 同一 pattern 与 target 必得同一结果。

### 4.3 Relationship to Other Instructions（与其他指令关系）

| Instruction | Relationship |
|-------------|-------------|
| `SOURCE` (opcode 56) | 提取当前 block 的指令序列为 Script 值，常用于为 MODEL 提供 target 参数 |
| `HASH` (opcodes 170+) | 可对提取数据做 hash，用于后续校验 |
| `EQUAL` (opcode 104) | 提取值可与期望值通过比较指令进行对比 |
| `PASS` / `CHECK` (opcodes 50–51) | MODEL 的布尔结果可直接接入校验指令 |
