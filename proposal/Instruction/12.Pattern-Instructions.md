# Proposal: Pattern Instructions [116–127]

> **Prerequisite:** This specification builds upon the [Base Constraints](0.Base-Constraints.md) document, which defines the instruction format, runtime model, and notation conventions used throughout.


## 1. Overview

Pattern instructions occupy opcode range **[116–127]** (12 instructions). They provide **pattern matching on scripts** — the ability to verify whether a target instruction sequence conforms to a specific structural pattern (formal verification). During matching, information can be extracted from the target for further processing.

This is the most complex instruction category in the Evidcoin script system. Pattern matching enables scripts to inspect and validate the structure of other scripts at runtime, which is essential for:

- **Script template verification** — confirming that a lock script conforms to a known standard pattern (e.g., P2PKH, multi-signature).
- **Structural introspection** — extracting embedded public keys, hash values, or parameters from scripts without executing them.
- **Policy enforcement** — validating that scripts adhere to governance rules before accepting them.

### Core Concepts

**MODEL block:** All pattern matching occurs within a `MODEL{}` block (opcode 116), which opens a dedicated parsing sub-environment. Inside this block, pattern instructions describe the expected structure of the target, and the engine performs positional matching against the target instruction sequence.

**Matching methods:** The pattern system supports eight complementary matching strategies:

| Method | Description |
|--------|-------------|
| Position matching | Pattern instructions are matched against target instructions by position |
| Instruction matching | An ordinary instruction in the pattern requires the same instruction at the target position (value instructions imply type) |
| Auxiliary param matching | Under instruction match, auxiliary parameters participate in equality comparison |
| Data matching | Under instruction match, associated data participates in equality comparison |
| Type matching | Match target data type regardless of value (`?!` / `??`) |
| Range limiting | Constrain integer or float values to a specified range (`!{}`) |
| Regex matching | Apply regular expressions to text/byte sequences (`RE{}`) |
| Segment matching | Match contiguous instruction segments of variable length (`_()`, `...`) |

### Terminology

- **Target script:** The instruction sequence being examined (the argument to `MODEL{}`).
- **Pattern:** The instruction sequence inside the `MODEL{}` block that describes the expected structure.
- **Static instruction:** An instruction's complete byte representation — opcode, auxiliary parameters, and associated data combined. This is the unit of matching.
- **Extraction:** The process of capturing specific parts of matched target instructions for use as return values.


## 2. Instruction Definitions

### 2.1 Model Block — MODEL

```
116     MODEL{}(~)  Pattern matching block
```

- **Auxiliary params:** Varint — sub-block code length. The **high bit** of this varint serves as an extraction flag: when set, the MODEL block returns extracted values in addition to the match result. This limits the sub-block to a maximum of **32 KB** (15-bit length, 2^15 = 32,768 bytes).
- **Arguments:** One value — the target sequence (`Script` or `Bytes` type).
- **Return value:**
  - **Without extraction** (high bit = 0): `Bool` — `true` if the pattern matches the target, `false` otherwise.
  - **With extraction** (high bit = 1): `(Slice, Bool)` — a slice containing all extracted values (in extraction order) and a boolean match result. If the match fails, the slice is empty.
- The MODEL block opens a **parsing sub-environment** in which the engine steps through the pattern and the target in parallel, comparing position by position.
- **Ordinary instructions** inside MODEL are interpreted as **strict equality matchers** — the target must contain the exact same instruction (same opcode, same auxiliary parameters, same associated data) at the corresponding position.
- **Nested MODEL:** If the target script itself contains a MODEL block, that MODEL's content is treated as **ordinary associated data** during matching — the engine does not recurse into it. This prevents infinite matching depth and maintains predictable performance.
- **Examples:**
  ```
  // Simple structure check: does the target match "DUP HASH EQUAL PASS"?
  @source
  MODEL{
       DUP HASH EQUAL PASS
  }
  // Returns: Bool (true if target is exactly DUP HASH EQUAL PASS)
  ```
  ```
  // Structure check with extraction (high bit set):
  @source
  MODEL{
       DATA[20]{?} #[32]     // match DATA with 20 bytes, extract the data content
       DUP HASH EQUAL PASS
  }
  // Returns: (Slice, Bool) — slice contains the extracted 20-byte data
  ```
- **Design notes:**
  - The 32 KB sub-block limit is generous for pattern definitions — even complex multi-signature patterns rarely exceed a few hundred bytes.
  - The extraction flag in the high bit avoids the need for a separate instruction or mode switch, keeping the MODEL instruction compact.
  - MODEL blocks may appear within lock scripts for template validation, or within validation logic that inspects transaction output scripts.

### 2.2 Instruction Wildcard — _

```
117     _           Instruction wildcard
```

- **Auxiliary params:** None.
- **Arguments:** None (pattern-only instruction).
- **Return value:** None (matching directive).
- Matches **any single instruction** at the current target position. The target instruction may be any opcode, with any auxiliary parameters and any associated data. However, the target position **must** contain an instruction — `_` does not match past the end of the target.
- A "static instruction" encompasses the opcode, its auxiliary parameters, and its associated data as a single unit. `_` matches this entire unit.
- **Examples:**
  ```
  MODEL{
       _              // first instruction can be anything
       HASH           // second instruction must be HASH
       _              // third instruction can be anything
       PASS           // fourth must be PASS
  }
  ```
- **Design note:** `_` is the simplest wildcard — it expresses "I don't care what this instruction is, but something must be here."

### 2.3 Instruction Segment Wildcard — _()

```
118     _(~)        Instruction segment wildcard
```

- **Auxiliary params:** Varint — the total number of instructions to match (including the current position).
- **Arguments:** None (pattern-only instruction).
- **Return value:** None (matching directive).
- Matches a contiguous segment of exactly N instructions starting from the current target position. This is the **plural form** of `_` and the **efficient version** of `...` when the segment length is known.
- The segment refers to **same-level relationships** only. A structural block instruction (e.g., `IF{}`, `EACH{}`, `MODEL{}`) counts as a **single instruction** — its sub-code block is part of its associated data and is not counted separately.
- **Examples:**
  ```
  MODEL{
       _[3]           // match any 3 instructions
       PASS           // fourth must be PASS
  }
  ```
  ```
  // _[1] is equivalent to _
  MODEL{
       _[1] HASH _[1] PASS
  }
  ```
- **Design note:** When the number of instructions to skip is known at pattern-compile time, `_()` is strongly preferred over `...` because it avoids backtracking. The engine can simply advance the target position by N instructions without trial matching.

### 2.4 Optional Instruction Sequence — ?{}

```
119     ?{}(~)      Optional instruction sequence
```

- **Auxiliary params:** Varint — sequence byte length (of the enclosed instructions).
- **Arguments:** None (pattern-only instruction).
- **Return value:** None (matching directive).
- Encloses an instruction sequence that must match the target segment **exactly** (full equality including auxiliary parameters and associated data) **or not be present at all**. This is an all-or-nothing optional match.
- If the target at the current position matches the entire enclosed sequence, the target position advances past all matched instructions. If the first instruction does not match, the optional sequence is **skipped** and the target position remains unchanged — subsequent pattern instructions continue matching from the same target position.
- **Single-instruction shorthand:** When the optional sequence contains only one instruction, the braces may be omitted. `?POPS[3]` is equivalent to `?{POPS[3]}`.
- **Examples:**
  ```
  MODEL{
       ?{@TOP PRINT}    // optionally match @TOP followed by PRINT
       DUP HASH          // then require DUP HASH
       PASS
  }
  ```
  ```
  MODEL{
       ?POPS[3]          // optionally match POPS with aux param 3
       PASS
  }
  ```
  ```
  MODEL{
       ?{1, 3, 5}        // optionally match the value set {1, 3, 5}
       EQUAL PASS
  }
  ```
- **Design notes:**
  - The optional sequence performs **strict equality** — all opcodes, auxiliary parameters, and associated data must match exactly. For flexible matching within an optional sequence, combine with wildcard indicators (`?` opcode 120).
  - The matching is **non-backtracking** for the optional sequence: if the first instruction matches but a subsequent one does not, the entire optional sequence fails (the target position is not partially advanced).

### 2.5 Wildcard Indicator — ?()

```
120     ?(1)        Wildcard indicator
```

- **Auxiliary params:** 1 byte — a bitmask identifying which parts of the **following** instruction may be wildcarded.
- **Arguments:** None (pattern-only instruction).
- **Return value:** None (matching directive).
- Modifies the matching behavior of the **immediately following** instruction in the pattern. Without a wildcard indicator, an instruction in the pattern requires exact equality. With a wildcard indicator, designated parts of the instruction are relaxed to accept any value.
- **Bitmask definitions:**

  | Bit | Hex | Meaning |
  |-----|-----|---------|
  | 0 | `0x01` | 1st auxiliary parameter may be arbitrary |
  | 1 | `0x02` | 2nd auxiliary parameter may be arbitrary |
  | 2 | `0x04` | 3rd auxiliary parameter may be arbitrary |
  | 3 | `0x08` | 4th auxiliary parameter may be arbitrary |
  | 4 | `0x10` | 5th auxiliary parameter may be arbitrary |
  | 5 | `0x20` | Associated data may be arbitrary |
  | 6 | `0x40` | Instruction itself is optional (may be absent) |
  | 7 | `0x80` | Hash match on associated data (compare by hash rather than byte equality) |

- **Shorthand notation:** For readability, the following syntactic sugar is supported:
  - `DATA[20]{?}` → `?[0x20] DATA[20]` — DATA with exactly 20-byte length parameter, but data content arbitrary.
  - `DATA[?]{?}` → `?[0x21] DATA` — DATA instruction with both auxiliary parameter and data arbitrary.
  - `?DATA` → `?[0x40] DATA` — DATA instruction is optional; all unspecified parts are implicitly arbitrary.
- **Deep match vs. single match for block instructions:**
  - **Deep match (no wildcard indicator):** When a block instruction (e.g., `IF{}`, `BLOCK{}`) appears in the pattern without a preceding wildcard indicator, the engine **enters** its sub-block and performs progressive positional matching against the target's sub-block contents.
  - **Single match (with wildcard indicator):** When a block instruction has a wildcard indicator, its sub-block is treated as **associated data** — the engine matches it as a single opaque unit without entering it.
- **Hash match mode** (bit 7, `0x80`): Instead of comparing the associated data byte-by-byte, the engine computes a hash of the target's associated data and compares it against the hash value provided in the pattern. This enables matching large data payloads by their fingerprint.
- **Examples:**
  ```
  MODEL{
       ?[0x20] DATA[20]     // DATA with 20 bytes, content arbitrary
       DUP HASH EQUAL PASS
  }
  ```
  ```
  MODEL{
       ?[0x21] DATA         // DATA with arbitrary length and content
       PASS
  }
  ```
  ```
  MODEL{
       ?[0x40] PRINT        // PRINT is optional (may or may not be present)
       PASS
  }
  ```
  ```
  // Hash match: verify DATA content by its hash fingerprint
  MODEL{
       ?[0x80] DATA{0x4f262ae8...}    // match DATA whose content hash equals given value
       PASS
  }
  ```
- **Design notes:**
  - The bitmask encoding is compact (single byte) and expressive — common patterns like "arbitrary data" or "optional instruction" each require only one bit.
  - Hash matching (bit 7) is particularly useful for verifying large embedded scripts or data payloads without including the full content in the pattern, reducing pattern size significantly.
  - Bits 0–4 correspond to auxiliary parameters 1–5. Most instructions have at most 2–3 auxiliary parameters, so bits 3–4 are rarely used but available for complex instructions.

### 2.6 Instruction Extraction — #()

```
121     #(1)        Instruction extraction
```

- **Auxiliary params:** 1 byte — a bitmask identifying which parts of the previously matched target instruction to extract.
- **Arguments:** None (pattern-only instruction).
- **Return value:** None directly — extracted values are accumulated into MODEL's internal extraction queue, which becomes the `Slice` component of MODEL's return value.
- Must **immediately follow** an instruction that was preceded by a wildcard indicator (`?` opcode 120). It extracts parts from the **most recently matched** target instruction.
- **Bitmask definitions:**

  | Bit | Hex | Meaning |
  |-----|-----|---------|
  | 0 | `0x01` | Extract 1st auxiliary parameter |
  | 1 | `0x02` | Extract 2nd auxiliary parameter |
  | 2 | `0x04` | Extract 3rd auxiliary parameter |
  | 3 | `0x08` | Extract 4th auxiliary parameter |
  | 4 | `0x10` | Extract 5th auxiliary parameter |
  | 5 | `0x20` | Extract associated data |
  | 6 | `0x40` | Extract opcode alone (as Byte value) |
  | 7 | `0x80` | Extract complete instruction (opcode + aux params + data as Bytes) |

- When **multiple bits** are set, the corresponding parts are extracted and expanded **in order** (bit 0 first, bit 7 last) into the internal extraction queue.
- **Examples:**
  ```
  // Extract the 20-byte data content from a DATA instruction
  MODEL{
       ?[0x20] DATA[20] #[0x20]     // wildcard data, then extract data
       DUP HASH EQUAL PASS
  }
  // Returns: (Slice{<20-byte data>}, true)
  ```
  ```
  // Extract both the auxiliary parameter and the associated data
  MODEL{
       ?[0x21] DATA #[0x21]         // wildcard both, extract both
       PASS
  }
  // Returns: (Slice{<aux param value>, <data bytes>}, true)
  ```
  ```
  // Extract the complete instruction
  MODEL{
       _ #[0x80]                     // match any instruction, extract it entirely
       PASS
  }
  // Returns: (Slice{<complete instruction bytes>}, true)
  ```
- **Design notes:**
  - The extraction bitmask intentionally mirrors the wildcard indicator bitmask (bits 0–5, 7) for consistency. Bit 6 differs: in the wildcard indicator it means "optional," while in extraction it means "extract opcode."
  - Extraction does not affect the matching process — it is a passive observation step that records information from already-matched positions.
  - The extraction queue is ordered by encounter sequence within the MODEL block, providing deterministic output ordering.

### 2.7 Type Match and Optional — ?!() / ??()

```
122     ?!(1){Type} Type match | optional
```

- **Auxiliary params:** 1 byte — type identifier. The **high bit** distinguishes between two modes:
  - High bit **clear** (prefix `!`): **Match test** — the target instruction must produce a value of the specified type. If it does not, the MODEL block **fails immediately** (returns `false`).
  - High bit **set** (prefix `?`): **Optional test** — checks whether the target instruction produces a value of the specified type, but **never fails**. If the type does not match, matching continues from the same target position (the test is simply skipped).
- **Arguments:** None (pattern-only instruction).
- **Return value:** None (matching directive).
- **Supported types** (lower 7 bits of the type identifier):

  | ID | Type | Description |
  |----|------|-------------|
  | 0 | Bool | Boolean value |
  | 1 | Byte | Single byte (0–255) |
  | 2 | Rune | Unicode code point |
  | 3 | Int | Signed integer |
  | 4 | BigInt | Arbitrary-precision integer |
  | 5 | Float | 64-bit floating point |
  | 6 | String | UTF-8 string |
  | 7 | Bytes | Byte sequence |
  | 8 | RegExp | Regular expression |
  | 9 | Time | Timestamp |
  | 10 | Script | Script/code value |
  | 11 | Dict | Dictionary (ordered map) |
  | 12 | Number | Any numeric type (Byte, Rune, Int, BigInt, Float) |
  | 13 | Expr | Expression value |
  | 14 | Model | Pattern/model value |
  | 15 | Values | Value set (slice) |

- The type is inferred from the target instruction. Value instructions inherently encode their type; for other instructions, the type of their return value is used.
- **Examples:**
  ```
  MODEL{
       ?![7]              // target must produce Bytes — fail MODEL if not
       _                  // next instruction can be anything
       PASS
  }
  ```
  ```
  MODEL{
       ??[6]              // optionally match a String instruction (high bit set: 6 + 128 = 134)
       HASH EQUAL PASS    // continue regardless
  }
  ```
- **Design notes:**
  - The dual-mode design (match vs. optional) in a single opcode reduces opcode consumption while covering both use cases.
  - Type matching is coarser than instruction matching — it cares about **what** value is produced, not **how** it is produced. This is useful when the pattern accepts any instruction that produces a value of the required type.
  - The `Number` meta-type (ID 12) is a convenience alias that matches any of the five numeric types.

### 2.8 Integer Range Match — !{}

```
123     !{}(~,~)    Integer value range match
```

- **Auxiliary params:**
  - Varint 1 — lower bound (inclusive).
  - Varint 2 — upper bound (exclusive).
- **Arguments:** None (pattern-only instruction).
- **Return value:** None (matching directive).
- Matches the target instruction's integer value against the half-open interval `[lower, upper)`. The target must be an integer-producing instruction (Int type). If the value falls outside the range, the MODEL block **fails** (returns `false`).
- **BigInt range matching is not supported** — both bounds and the target value must fit within standard integer precision.
- **Examples:**
  ```
  MODEL{
       !{0, 1024}         // target integer must satisfy: 0 ≤ n < 1024
       PASS
  }
  ```
  ```
  MODEL{
       !{1, 256}          // valid byte-like range: 1 ≤ n < 256
       _[2]
       EQUAL PASS
  }
  ```
- **Design note:** The half-open interval `[lower, upper)` is the standard mathematical convention used throughout the Evidcoin script system (consistent with `WITHIN`, opcode 111). This convention avoids off-by-one ambiguity and composes cleanly.

### 2.9 Float Range Match — !{}

```
124     !{}(8,8,4)  Float value range match
```

- **Auxiliary params:**
  - 8 bytes — lower bound (inclusive, IEEE 754 float64).
  - 8 bytes — upper bound (exclusive, IEEE 754 float64).
  - 4 bytes — equality epsilon (IEEE 754 float32). Defines the tolerance for the lower-bound inclusive comparison: `|target - lower| ≤ epsilon` is treated as equal.
- **Arguments:** None (pattern-only instruction).
- **Return value:** None (matching directive).
- Matches the target instruction's float value against the range `[lower, upper)` with epsilon-based precision for the lower bound equality test. The target must be a float-producing instruction. If the value falls outside the range, the MODEL block **fails**.
- The epsilon parameter addresses the inherent imprecision of floating-point equality. When epsilon is `0`, the lower bound comparison is exact (bitwise equality). When epsilon is non-zero, values within the epsilon neighborhood of the lower bound are accepted.
- **Examples:**
  ```
  MODEL{
       !{0.0, 1.0, 0}       // 0.0 ≤ x < 1.0, exact lower bound
       PASS
  }
  // Matches any float in [0.0, 1.0) with exact zero comparison
  ```
  ```
  MODEL{
       !{0.5, 100.0, 0.001}  // 0.5 ≤ x < 100.0, epsilon = 0.001
       PASS
  }
  // Values in [0.499, 0.501] satisfy the lower bound test
  ```
- **Design notes:**
  - The 8+8+4 byte fixed layout (20 bytes total) is chosen because floating-point bounds cannot be meaningfully compressed via varint encoding.
  - The epsilon is stored as float32 (4 bytes) rather than float64 because epsilon values are typically small and do not require full double precision. This saves 4 bytes per instruction.
  - This instruction shares the `!{}` mnemonic with the integer range match (opcode 123). The engine disambiguates by opcode number.

### 2.10 Regex Search — RE{}

```
125     RE{}(1,1)   Regex search within MODEL
```

- **Auxiliary params:**
  - 1 byte — search mode:
    - `g` — `FindAll`: find all non-overlapping matches of the regex in the target.
    - `G` — `FindAllSubmatch`: find all matches including submatch groups.
    - `s` — `FindSubmatch`: find the first match including submatch groups.
  - 1 byte — regex content length (maximum 255 bytes).
- **Associated data:** The regex pattern string (up to 255 bytes).
- **Arguments:** None (pattern-only instruction).
- **Return value:** None directly — match results are stored internally and accessed via the `&` extraction instruction (opcode 126).
- Performs a regular expression search on the **text or byte representation** of the target instruction's associated data (or value). The regex engine follows standard RE2/Go `regexp` semantics.
- The regex pattern is embedded directly as associated data, with its length specified by the second auxiliary parameter.
- **Examples:**
  ```
  MODEL{
       RE{/^[a-f0-9]{40}$/s}    // match a 40-character hex string, FindSubmatch mode
       &[0]                       // extract the full match (index 0)
       PASS
  }
  ```
  ```
  MODEL{
       RE{/(\d+)\.(\d+)/G}      // find all "N.N" patterns with submatches
       &[0]                       // extract first match group
       PASS
  }
  ```
- **Design notes:**
  - The 255-byte regex length limit is sufficient for validation patterns but prevents excessively complex regex patterns that could cause performance issues.
  - Search mode is a single byte for compactness. The three modes (`g`, `G`, `s`) cover the standard regex search use cases.
  - RE2/Go regex semantics guarantee linear-time matching, preventing catastrophic backtracking — an important safety property for blockchain script execution.

### 2.11 Regex Extraction — &()

```
126     &(1)        Regex extraction
```

- **Auxiliary params:** 1 byte — the index of the match/submatch to extract (0–255).
- **Arguments:** None (pattern-only instruction).
- **Return value:** None directly — the extracted value is added to MODEL's internal extraction queue.
- Must **follow** the `RE{}` instruction it is associated with. Extracts a specific member from the regex match results by index.
- **Index semantics:**
  - For `FindSubmatch` (`s` mode): index 0 is the full match, index 1 is the first submatch group, index 2 is the second, etc.
  - For `FindAll` (`g` mode): index 0 is the first match, index 1 is the second match, etc.
  - For `FindAllSubmatch` (`G` mode): results are flattened — index 0 is the full first match, index 1 is the first submatch of the first match, etc.
- If the index is out of bounds or the regex produced no matches, the extracted value is `nil`.
- **Maximum 256 results** can be addressed (index 0–255).
- **Examples:**
  ```
  MODEL{
       RE{/^(\w+):(\d+)$/s}     // match "word:number" pattern
       &[1]                       // extract the word part (submatch 1)
       &[2]                       // extract the number part (submatch 2)
       PASS
  }
  // Extraction queue: [<word>, <number>]
  ```
- **Design note:** The `&` mnemonic is chosen by analogy with regex capture group references in common scripting languages.

### 2.12 Arbitrary Segment Wildcard — ...

```
127     ...         Arbitrary instruction segment wildcard
```

- **Auxiliary params:** None.
- **Arguments:** None (pattern-only instruction).
- **Return value:** None (matching directive).
- Matches **any contiguous sequence** of instructions at the current target position, including a **zero-length** sequence (matching nothing). This is the most permissive wildcard in the pattern system.
- Each `...` operates at the **current nesting level** only. Block instructions are treated as single entities — `...` does not descend into sub-blocks.
- **Non-greedy semantics:** `...` matches the **minimal** contiguous sequence that allows the remaining pattern to complete successfully. The engine advances the target position as little as possible.
- **Examples:**
  ```
  MODEL{
       ...                // skip any prefix instructions
       HASH EQUAL PASS    // then require HASH EQUAL PASS at the end
  }
  ```
  ```
  MODEL{
       DUP ...            // starts with DUP, then anything in between
       PASS               // ends with PASS
  }
  ```
  ```
  MODEL{
       ... HASH ... PASS  // HASH and PASS must appear in order, anything around them
  }
  ```
- **Design notes:**
  - **WARNING: Resource-intensive.** The `...` wildcard requires the engine to perform trial matching (backtracking) to find the minimal match. In the worst case, this is O(n × m) where n is the remaining target length and m is the remaining pattern length. **Prefer `_()` with an explicit instruction count whenever possible.**
  - Non-greedy (minimal match) semantics are chosen over greedy (maximal match) because in pattern validation, the intent is typically to find the earliest point where subsequent pattern instructions can match, not to consume as much as possible.
  - Multiple `...` wildcards in a single MODEL block compound the performance cost. Pattern authors should strive to minimize their use and constrain them with specific instructions on both sides.


## 3. Matching Semantics Summary

### 3.1 Positional Matching Algorithm

The MODEL block engine maintains two cursors: one for the pattern and one for the target. Matching proceeds as follows:

1. **Advance both cursors** in lockstep for ordinary instructions and exact-match pattern instructions.
2. **Wildcard instructions** (`_`, `_()`, `...`) advance the target cursor by the appropriate amount while consuming one pattern position.
3. **Modifier instructions** (`?()`, `#()`, `&()`) consume one pattern position without advancing the target cursor — they modify or observe the match at the current position.
4. **Optional sequences** (`?{}`, `??`) attempt matching at the current target position. On failure, the target cursor remains unchanged.
5. **Range and type checks** (`!{}`, `?!`) test the target instruction's value or type at the current position. On failure, the MODEL block returns `false`.
6. The match **succeeds** when both cursors reach the end of their respective sequences simultaneously.
7. The match **fails** when:
   - A required match (instruction equality, type check, range check) does not hold.
   - The pattern is exhausted but target instructions remain (unless the pattern ends with `...`).
   - The target is exhausted but non-optional pattern instructions remain.

### 3.2 Block Instruction Matching

Block instructions (those with `{}` sub-blocks, such as `IF{}`, `EACH{}`, `BLOCK{}`) are matched according to two rules:

- **Without wildcard indicator (deep match):** The engine enters the sub-block of both the pattern and the target, performing recursive positional matching on the sub-block contents.
- **With wildcard indicator (single match):** The entire block instruction — including its sub-block — is treated as a single opaque instruction. The sub-block content is matched as associated data according to the wildcard indicator's bitmask.

### 3.3 Extraction Queue

Extraction (`#` and `&` instructions) accumulates values into an ordered queue during the matching process. This queue becomes the `Slice` component of MODEL's return value when the extraction flag is set. Values appear in the queue in the order they were extracted (left-to-right through the pattern).


## 4. Design Notes

### 4.1 Performance Considerations

Pattern matching is inherently more expensive than normal instruction execution because it involves dual-cursor traversal and potentially backtracking (for `...`). To mitigate this:

- **Prefer `_()` over `...`** — explicit-length wildcards avoid backtracking entirely.
- **Minimize `...` usage** — each `...` introduces a potential backtracking point.
- **Place specific anchors** (known instructions) as early as possible in the pattern to fail fast on non-matching targets.
- The 32 KB sub-block limit provides an implicit bound on pattern complexity.

### 4.2 Safety Properties

- **No script mutation:** Pattern matching is purely observational — it reads the target script but never modifies it, consistent with the script immutability constraint.
- **No recursive MODEL interpretation:** Nested MODEL blocks in the target are treated as data, preventing unbounded recursion.
- **Bounded regex execution:** RE2/Go semantics guarantee linear-time regex matching, and the 255-byte pattern limit constrains regex complexity.
- **Deterministic results:** All matching operations are deterministic — the same pattern and target always produce the same result.

### 4.3 Relationship to Other Instructions

| Instruction | Relationship |
|-------------|-------------|
| `SOURCE` (opcode 56) | Extracts the current block's instruction sequence as a Script value — often used to obtain the target argument for MODEL |
| `HASH` (opcodes 170+) | Hash functions may be used on extracted data for further verification |
| `EQUAL` (opcode 104) | Extracted values can be compared with expected values using standard comparison instructions |
| `PASS` / `CHECK` (opcodes 50–51) | MODEL's boolean result feeds directly into validation instructions |
