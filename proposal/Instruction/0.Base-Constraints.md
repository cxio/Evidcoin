# Proposal: Base Constraints for the Script Instruction Set

This document defines the foundational constraints and structural conventions governing all instructions in the Evidcoin script system. Every instruction category specification implicitly incorporates the constraints described herein.


## 1. Script Immutability

Scripts are **read-only** during execution. The instruction stream must not be modified at any point during evaluation. Instructions that perform assignment-like operations (e.g., local scope storage, global variable writes) operate on separate runtime data structures, never on the script bytecode itself.

Where an instruction logically requires mutation of its operand (e.g., value construction from inline data), the engine must work on a copy to preserve the original script content.


## 2. Core Runtime Concepts

### 2.1 Data Stack

A LIFO structure holding runtime values produced and consumed by instructions.

- **Maximum height:** 256 entries.
- **Maximum item size:** 1,024 bytes (1 KB).
- All value instructions automatically push their result onto the data stack unless intercepted by a capture prefix (`@`, `$`).

### 2.2 Argument Space

A FIFO queue for staging operands before instruction consumption.

- When an instruction requires arguments, the engine checks the argument space first. Only if the argument space is empty does it fall back to the data stack.
- Arguments are consumed **atomically** — the instruction either takes all required arguments from the argument space or none (falling back entirely to the stack).
- This mechanism decouples operand ordering from stack position, which is the key innovation over traditional pure-stack script languages.

### 2.3 Local Scope

A block-private, append-only storage area.

- **Maximum capacity:** 128 entries per scope.
- Each syntactic block (`IF{}`, `EACH{}`, `BLOCK{}`, etc.) creates an independent local scope.
- **No inheritance** between scopes — sibling and nested blocks cannot access each other's local scopes.
- Values are added via the `$` (local store) capture prefix and read via `$[n]` (local retrieve).

### 2.4 Global Scope

A script-domain-wide shared variable space.

- **256 slots**, indexed `[0-255]`.
- Accessed via `VAR` / `SETVAR` instructions.
- Shared across all blocks within the same script execution, including `EMBED` targets.
- **Not shared** across `GOTO` boundaries (each GOTO creates an independent script domain).

### 2.5 Opcode

A single-byte unique identifier for each instruction.

- Range: `[0-253]` available for the instruction set (254 total usable opcodes).
- Opcodes `254` and `255` are reserved for system use and are not part of the instruction set.

### 2.6 Auxiliary Parameters

Compile-time constant parameters embedded directly after the opcode byte.

- Represent static properties of the instruction itself (e.g., block length, algorithm selector, index).
- Fixed-length (1, 2, or 64 bytes) or variable-length integer (varint), as specified per instruction.
- When an instruction has multiple auxiliary parameters, they appear consecutively in declaration order.

### 2.7 Associated Data

Payload data carried by the instruction, following any auxiliary parameters.

- Used for larger data blocks: string content, byte sequences, embedded code blocks, etc.
- Length is typically specified by a preceding auxiliary parameter.
- Instructions carrying associated data are denoted with `{}` in their mnemonic (e.g., `DATA{}`, `CODE{}`).

### 2.8 Arguments

Runtime values consumed by the instruction during execution.

- Sourced from the argument space (priority) or data stack (fallback).
- **Not** part of the instruction's byte representation in the script.
- Each instruction specifies its argument requirement: `0` (none), `n` (fixed count), or `-1` (variable/optional).


## 3. Instruction Byte Layout

Every instruction in the compiled bytecode conforms to this sequential layout:

```
[Opcode: 1 byte] [Auxiliary Params: 0-N bytes] [Associated Data: 0-N bytes]
```

- The **opcode** is always exactly 1 byte.
- **Auxiliary parameters** follow immediately, their count and sizes determined by the opcode's specification.
- **Associated data** follows the auxiliary parameters, its length typically encoded in a preceding auxiliary parameter.
- **Arguments** exist only at runtime — they are never serialized into the bytecode.


## 4. Instruction Documentation Format

Instruction specifications use the following notation conventions throughout all proposal documents:

### 4.1 Header Line Format

```
<opcode>    <mnemonic>[{}][(aux_params)]    <brief description>
```

- **Opcode**: Decimal integer (`0`-`253`).
- **Mnemonic**: The instruction name in uppercase (e.g., `DATA`, `GOTO`, `IF`).
- **`{}`**: Present if the instruction carries associated data.
- **`(aux_params)`**: Parenthesized auxiliary parameter sizes, comma-separated. `~` denotes varint. `64` denotes a fixed 64-byte field. Multiple parameters appear as `(~,64,~)`.
- **Brief description**: Short summary of the instruction's purpose.

### 4.2 Detail Block

Below the header line, indented comment lines (`//`) provide:

- **Type**: The result type pushed onto the stack (if any).
- **Auxiliary parameter descriptions**: Purpose and encoding of each parameter.
- **Associated data description**: Content format and semantics.
- **Argument requirements**: What runtime operands are consumed.
- **Examples**: Concrete usage with `// Example:` prefix.
- **Design notes**: Rationale or implementation hints with `// Design:` prefix.

### 4.3 Syntax Conventions

| Notation | Meaning |
|----------|---------|
| `INSTR{content}` | Instruction with associated data (curly braces) |
| `INSTR[value]` | Instruction with auxiliary parameter expressed as index/value (square brackets) |
| `INSTR(n)` | Auxiliary parameter specification: `n` bytes fixed |
| `INSTR(~)` | Auxiliary parameter: variable-length integer (varint) |
| `INSTR(~,64,~)` | Multiple auxiliary parameters in order |


## 5. Instruction Set Segmentation

The 254 usable opcodes are divided into four contiguous segments:

| Segment | Opcode Range | Count | Purpose |
|---------|-------------|-------|---------|
| Base Instructions | `[0-169]` | 170 | Core language: values, stack ops, flow control, arithmetic, comparison, logic, pattern matching, environment, tools, system calls |
| Function Instructions | `[170-209]` | 40 | Built-in functions: encoding, hashing, signature verification, plus 1 self-extension slot (`FN_X`) |
| Module Instructions | `[210-249]` | 40 | Categorized modules: math, time, regex, plus 1 self-extension slot (`MO_X`) |
| Extension Instructions | `[250-253]` | 4 | Generic extension points: `EX_FN`, `EX_MO`, `EX_INST`, `EX_PRIV`, each addressable to 64K via 2-byte sub-opcode |

### Unlock Script Restriction

Unlock scripts (placed in transaction inputs) are restricted to opcodes `[0-49]` only. This limits them to value instructions, capture instructions, stack operations, collection operations, and interaction instructions — preventing unlock scripts from performing flow control, cryptographic operations, or system calls directly.


## 6. Resource Limits Summary

| Resource | Limit |
|----------|-------|
| Data stack height | 256 entries |
| Stack item size | 1,024 bytes (1 KB) |
| Lock script size | < 1,024 bytes (1 KB) |
| Unlock script size | < 4,096 bytes (4 KB) |
| Local scope capacity | 128 entries per block |
| Global variable slots | 256 |
| Maximum transaction size | 65,535 bytes |
| Unlock script opcode range | `[0-49]` |
