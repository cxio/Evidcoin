# Proposal: Function Instructions [170–209]

> **Prerequisite:** This specification builds upon the [Base Constraints](0.Base-Constraints.md) document, which defines the instruction format, runtime model, and notation conventions used throughout.


## 1. Overview

Function instructions occupy opcode range **[170–209]** (40 instructions). They carry the `FN_` prefix and behave like callable functions: most accept dynamic arguments from the data stack and return results directly, with few or no auxiliary parameters.

This range is divided into two parts:

| Subrange | Name | Count | Description |
|----------|------|-------|-------------|
| 170–208 | Core Function Set | 39 | Built-in functions for encoding, cryptography, hashing, and output |
| 209 | Core Function Extension | 1 | Gateway to extended function definitions |

### Design Principles

- **Function-like semantics:** Each instruction reads its arguments from the data stack (or argument space) and pushes a return value. No side effects beyond the stack, except `FN_PRINTF`.
- **Minimal auxiliary params:** Most function instructions have zero auxiliary bytes. The few that do use a single byte for algorithm or target selection.
- **Bidirectional encoding:** Base-encoding and public-key-hash instructions auto-detect direction (encode vs. decode) based on argument type.


## 2. Instruction Definitions — Core Function Set

### 2.1 Base58 Encode/Decode

```
170     FN_BASE58   Base58 encode/decode
```

- **Arguments:** One value — `Bytes` for encoding, `String` for decoding.
- **Return value:** `String` (if encoding) or `Bytes` (if decoding).
- **Design notes:**
  - Direction is auto-determined by argument type: `Bytes` → encode to Base58 string; `String` → decode from Base58 to bytes.
  - Uses the Bitcoin Base58 alphabet (no ambiguous characters: `0`, `O`, `I`, `l` excluded).

### 2.2 Base32 Encode/Decode

```
171     FN_BASE32   Base32 encode/decode
```

- **Arguments:** One value — `Bytes` for encoding, `String` for decoding.
- **Return value:** `String` (if encoding) or `Bytes` (if decoding).
- **Design notes:**
  - Conforms to RFC 4648 with the standard uppercase alphabet (`A–Z`) plus digits `2–7`.
  - **No padding characters** — padding is omitted from encoded output and not expected during decoding.

### 2.3 Base64 Encode/Decode

```
172     FN_BASE64   Base64 encode/decode
```

- **Arguments:** One value — `Bytes` for encoding, `String` for decoding.
- **Return value:** `String` (if encoding) or `Bytes` (if decoding).
- **Design notes:**
  - Uses the **URL-safe** alphabet (`-` and `_` instead of `+` and `/`).
  - **No padding** — `=` characters are neither produced nor required.

### 2.4 Address Encoding

```
173     FN_ADDRESS  Encode public key hash to account address
```

- **Arguments:**
  1. `pbHash` (Bytes) — the public key hash.
  2. `prefix` (String) — the human-readable address prefix.
- **Return value:** String — the full account address.
- **Encoding process:**
  1. Prepend prefix to the public key hash: `[prefix][pubkey_hash]`.
  2. Double-hash the result; take the **last 4 bytes** as the checksum.
  3. Append the checksum to the public key hash: `[pubkey_hash][checksum]`.
  4. Base58-encode the byte sequence from step 3, then prepend the prefix: `[prefix][encoded_text]` → account address.
- **Verification process (for reference):**
  1. Separate the known-length prefix from the encoded text portion.
  2. Base58-decode the text to bytes; split into `[pubkey_hash][checksum₁]`.
  3. Prepend prefix to the pubkey hash, double-hash, take last 4 bytes as `[checksum₂]`.
  4. Compare `checksum₁` with `checksum₂` — match confirms address validity.
- **Example:**
  ```
  <pubkey_hash> "EC" FN_ADDRESS   // → "EC3K7mN..."
  ```

### 2.5 Public Key Hash / Address Decode

```
174     FN_PUBHASH  Single-sig public key hash or address decode
```

- **Arguments:**
  1. `data` — `Bytes` (raw public key) for hashing, or `String` (account address) for decoding.
  2. *(Optional)* `pfsize` (Int) — address prefix length in bytes (required when decoding).
- **Return value:** Bytes — the public key hash.
- **Design notes:**
  - Direction is auto-determined by argument type: `Bytes` → compute single-signature public key hash; `String` → decode account address to extract the embedded public key hash.
  - When decoding an address, the instruction **cannot** distinguish whether the original hash was single-sig or multi-sig — it simply extracts the hash bytes.
  - The optional second argument `pfsize` specifies the prefix length so the decoder knows where the encoded portion begins.

### 2.6 Multi-Sig Public Key Hash

```
175     FN_MPUBHASH Create multi-sig composite public key hash
```

- **Arguments:**
  1. `pubHash` ([]Bytes) — public key hashes of all participants.
  2. `m` (Int) — minimum number of required signatures.
  3. `n` (Int) — total number of participants. Must satisfy `m ≤ n`.
- **Return value:** Bytes — the composite multi-signature public key hash.
- **Design notes:**
  - **Creation only** — there is no reverse decoding from a composite hash back to individual participant hashes.
  - The composite hash deterministically encodes the set of participant hashes together with the `m`-of-`n` threshold parameters.

### 2.7 Single Signature Verification

```
176     FN_CHECKSIG Single signature verification
```

- **Arguments:**
  1. `flag` (Byte) — authentication flag indicating the signature scheme or hash subset.
  2. `sig` (Bytes) — the signature to verify.
  3. `pubKey` (Bytes) — the public key against which to verify.
- **Return value:** Bool — `true` if the signature is valid.
- **Example:**
  ```
  {0x01} <signature> <pubkey> FN_CHECKSIG PASS
  ```

### 2.8 Multi-Signature Verification

```
177     FN_MCHECKSIG    Multi-signature verification
```

- **Arguments:**
  1. `flag` (Byte) — authentication flag.
  2. `sigs` ([]Bytes) — set of signatures to verify.
  3. `pubKeys` ([]Bytes) — set of public keys, in 1:1 correspondence with `sigs`.
  4. `pubHashs` ([]Bytes) — remaining public key hashes (participants who did not sign).
- **Return value:** Bool — `true` if all provided signatures are valid against their corresponding public keys.
- **Design notes:**
  - `[]Bytes` is represented as `[]any` on the data stack, where each element is a `Bytes` value.
  - The `pubKeys` slice must have the same length as `sigs` — each signature is verified against its positionally corresponding public key.
  - `pubHashs` provides the hashes of non-signing participants, enabling reconstruction of the full participant set for composite hash validation.

### 2.9 Hash-224

```
178     FN_HASH224(1)   224-bit hash
```

- **Auxiliary params:** 1 byte — algorithm identifier.
- **Arguments:** `data` (Bytes) — the input to hash.
- **Return value:** Bytes — 28-byte hash digest.
- **Algorithm identifiers:**

  | ID | Algorithm | Notes |
  |----|-----------|-------|
  | 0 (default) | SHA3-224 | FIPS-202 standard |
  | 1 | SHA2-224 | Legacy compatibility |
  | 2 | BLAKE2s | 224-bit output via BLAKE2s |

- **Example:**
  ```
  <data> FN_HASH224          // SHA3-224 (default)
  <data> FN_HASH224[1]       // SHA2-224
  ```

### 2.10 Hash-256

```
179     FN_HASH256(1)   256-bit hash
```

- **Auxiliary params:** 1 byte — algorithm identifier.
- **Arguments:** `data` (Bytes) — the input to hash.
- **Return value:** Bytes — 32-byte hash digest.
- **Algorithm identifiers:**

  | ID | Algorithm | Notes |
  |----|-----------|-------|
  | 0 (default) | SHA3-256 | FIPS-202 standard |
  | 1 | SHA2-256 | Legacy compatibility |
  | 2 | BLAKE2b-256 | Superior security vs. SHA2; faster on 64-bit x86/ARM |

- **Design notes:**
  - BLAKE2b provides better security margins than SHA2 while being faster on modern 64-bit architectures (x86-64, ARMv8).

### 2.11 Hash-384

```
180     FN_HASH384(1)   384-bit hash
```

- **Auxiliary params:** 1 byte — algorithm identifier.
- **Arguments:** `data` (Bytes) — the input to hash.
- **Return value:** Bytes — 48-byte hash digest.
- **Algorithm identifiers:**

  | ID | Algorithm | Notes |
  |----|-----------|-------|
  | 0 (default) | SHA3-384 | FIPS-202 standard |
  | 1 | SHA2-384 | Legacy compatibility |
  | 2 | BLAKE2b-384 | BLAKE2b with 384-bit output |

### 2.12 Hash-512

```
181     FN_HASH512(1)   512-bit hash
```

- **Auxiliary params:** 1 byte — algorithm identifier.
- **Arguments:** `data` (Bytes) — the input to hash.
- **Return value:** Bytes — 64-byte hash digest.
- **Algorithm identifiers:**

  | ID | Algorithm | Notes |
  |----|-----------|-------|
  | 0 (default) | SHA3-512 | FIPS-202 standard |
  | 1 | SHA2-512 | Legacy compatibility |
  | 2 | BLAKE2b-512 | Full BLAKE2b output |

### 2.13 Reserved (182–207)

Opcodes **182–207** (26 slots) are reserved for future core function additions. They are not assigned and must not be used by implementations.

### 2.14 Formatted Print

```
208     FN_PRINTF   Console print with formatting
```

- **Arguments:**
  1. `fmt` (String) — format string (follows `fmt.Printf` conventions).
  2. `val` (...any) — variadic values matching the format specifiers.
- **Return value:** None (`nil`).
- **Design notes:**
  - Outputs formatted text to the console/log. This is the only function instruction with a side effect.
  - Distinct from the standard `PRINT` instruction (in base instructions), which outputs a single value without formatting.
- **Example:**
  ```
  "Balance: %d EC" {1000} FN_PRINTF   // prints: Balance: 1000 EC
  ```


## 3. Instruction Definition — Core Function Extension

### 3.1 Extended Function Reference

```
209     FN_X(1){}   Extended function reference
```

- **Auxiliary params:** 1 byte — target function index (`[0–255]`).
- **Associated data:** Optional target name (when using `{}` notation).
- **Arguments:** Determined by the target extended function.
- **Return value:** Determined by the target extended function.
- **Design notes:**
  - Acts as a dispatch gateway to an extended function table. The 1-byte auxiliary parameter addresses up to **256** extended function slots.
  - The `{}` notation allows referencing extended functions by name for readability: `FN_X{MyFunc}`.
  - For addressing beyond 256 extended functions, use `EX_FN` (opcode 250) from the extension instructions, which provides a 2-byte (64K) address space.
- **Example:**
  ```
  <args...> FN_X[12]        // call extended function at index 12
  <args...> FN_X{Encrypt}   // call extended function named "Encrypt"
  ```


## 4. Design Notes

### Encoding Function Family

The three base-encoding instructions (`FN_BASE58`, `FN_BASE32`, `FN_BASE64`) share a consistent bidirectional design: the argument type alone determines the operation direction. This eliminates the need for separate encode/decode opcodes and keeps the instruction set compact.

### Hash Function Family

The four hash instructions (`FN_HASH224` through `FN_HASH512`) provide a unified interface across three algorithm families (SHA3, SHA2, BLAKE2). SHA3 is the default for all lengths, aligning with FIPS-202 and providing quantum-resistance margins. BLAKE2 variants are offered as high-performance alternatives for non-consensus-critical contexts.

### Cryptographic Verification

`FN_CHECKSIG` and `FN_MCHECKSIG` are central to transaction validation. The `flag` byte allows future expansion to support different signature schemes without adding new opcodes.
