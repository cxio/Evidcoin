# Proposal: Module Instructions [210–249]

> **Prerequisite:** This specification builds upon the [Base Constraints](0.Base-Constraints.md) document, which defines the instruction format, runtime model, and notation conventions used throughout.


## 1. Overview

Module instructions occupy opcode range **[210–249]** (40 instructions). Modules introduce **categorized functionality** — each module groups a family of related operations under a single opcode, with a 1-byte auxiliary parameter selecting the specific member function.

### Module Design Pattern

Every module instruction follows a consistent pattern:

- **Opcode** identifies the module itself.
- **Auxiliary param** (1 byte) indexes a member function within the module, supporting up to 256 members per module.
- If a module needs to create an **object representing itself**, the first member (`Create`, index 0) serves as the constructor. Subsequent member functions take the created object as their first parameter.

### Notation Conventions

Module members can be referenced in two equivalent notations:

| Notation | Example | Description |
|----------|---------|-------------|
| Brace syntax | `MO_MATH{Abs}` | Member name inside curly braces |
| Dot syntax | `MO_MATH.Abs` | Dot-separated module and member name |

Both notations are interchangeable in script source; they compile to the same bytecode.

> **Note:** This section is not yet fully defined. The member lists below represent initial designs; further detailed definition and extension are expected as the system matures.


## 2. Instruction Definitions

### 2.1 Math Module

```
210     MO_MATH(1){}    Math operations module
```

- **Auxiliary params:** 1 byte — member function identifier.
- **Members:**

  | Name | Signature | Description |
  |------|-----------|-------------|
  | Abs | `(x float64) → float64` | Absolute value |
  | Ceil | `(x float64) → float64` | Round up to nearest integer |
  | Floor | `(x float64) → float64` | Round down to nearest integer |
  | Pow | `(x, y float64) → float64` | Raise `x` to the power of `y` |
  | Max | `(x, y float64) → float64` | Larger of two values |
  | Min | `(x, y float64) → float64` | Smaller of two values |
  | Mod | `(x, y float64) → float64` | Floating-point remainder |
  | ... | — | Additional members TBD |

- **Examples:**
  ```
  {-3.14} MO_MATH{Abs}       // → 3.14
  {2.0} {10.0} MO_MATH{Pow}  // → 1024.0
  {3.7} MO_MATH.Ceil          // → 4.0
  ```
- **Design notes:**
  - The Math module does not have a `Create` constructor — all members are stateless functions.
  - Mirrors a subset of Go's `math` package, selected for common on-chain computation needs.

### 2.2 Time Module

```
211     MO_TIME(1){}    Time module
```

- **Auxiliary params:** 1 byte — member function identifier.
- **Members:**

  | Name | Signature | Description |
  |------|-----------|-------------|
  | Create | `(sec int64) → Time` | Construct a Time object from UNIX timestamp (seconds) |
  | Now | `() → Time` | Current time (deterministic in consensus context) |
  | Add | `(tm Time, d int64) → Time` | Add duration (milliseconds) to a time |
  | Clock | `(tm Time) → (hour, min, sec int)` | Extract hour, minute, second components |
  | Date | `(tm Time) → (year, month, day int)` | Extract year, month, day components |
  | ... | — | Additional members TBD |

- **Examples:**
  ```
  {1608883593} MO_TIME{Create}     // construct Time from UNIX timestamp
  MO_TIME.Now                       // get current time
  <tm> {3600000} MO_TIME{Add}      // add 1 hour (in milliseconds)
  <tm> MO_TIME.Clock                // → hour, min, sec (three values)
  ```
- **Design notes:**
  - `Create` (index 0) is the constructor — it produces a `Time` object that other members operate on.
  - Multi-return members like `Clock` and `Date` push multiple values onto the data stack.
  - In consensus execution, `Now` must return a deterministic value (e.g., the block timestamp), not the wall-clock time of the executing node.

### 2.3 Regular Expression Module

```
212     MO_RE(1){}      Regular expression module
```

- **Auxiliary params:** 1 byte — member function identifier.
- **Members:**

  | Name | Signature | Description |
  |------|-----------|-------------|
  | Create | `(str string) → *Regexp` | Compile a regex pattern string into a Regexp object |
  | Find | `(re *Regexp, b []byte) → []byte` | Find first match in byte slice |
  | FindAll | `(re *Regexp, b []byte, n int) → [][]byte` | Find all matches (up to `n`; `-1` = unlimited) |
  | Split | `(re *Regexp, s string, n int) → []string` | Split string by pattern (up to `n` parts; `-1` = unlimited) |
  | String | `(re *Regexp) → string` | Return the original pattern string |
  | ... | — | Additional members TBD |

- **Examples:**
  ```
  "^[A-Z]{2}\\d+" MO_RE{Create}         // compile regex
  <re> <data> MO_RE.Find                  // find first match
  <re> <data> {-1} MO_RE{FindAll}        // find all matches
  <re> "a-b-c-d" {-1} MO_RE{Split}      // → ["a", "b", "c", "d"]
  ```
- **Design notes:**
  - Uses the RE2 engine (linear-time guarantees, no backtracking), consistent with Go's `regexp` package. This prevents denial-of-service via pathological patterns.
  - `Create` compiles the pattern; if the pattern is invalid, execution fails immediately.

### 2.4 Reserved (213–248)

Opcodes **213–248** (36 slots) are reserved for future module additions. They are not assigned and must not be used by implementations.

Potential candidates for future modules include (but are not limited to): string utilities, cryptographic primitives, data serialization, and collection algorithms.


## 3. Instruction Definition — Standard Module Extension

### 3.1 Extended Module Reference

```
249     MO_X(1){}   Extended module reference
```

- **Auxiliary params:** 1 byte — target sub-module index (`[0–255]`).
- **Associated data:** Optional target name with member (when using `{}` notation).
- **Arguments:** Determined by the target sub-module's member function.
- **Return value:** Determined by the target sub-module's member function.
- **Design notes:**
  - Acts as a dispatch gateway to an extended module table, supporting up to **256** sub-module slots.
  - Each sub-module manages its own member functions using its own 1-byte auxiliary parameter, following the same module pattern as the built-in modules.
  - The `{}` notation supports dotted member access: `MO_X{Exp.Create}` references the `Create` member of the extended module `Exp`.
  - For addressing beyond 256 extended modules, use `EX_MO` (opcode 251) from the extension instructions, which provides a 2-byte (64K) address space.
- **Example:**
  ```
  <args...> MO_X{Crypto.Sign}    // call Sign member of extended module "Crypto"
  <args...> MO_X[5]              // call extended module at index 5 (default member)
  ```


## 4. Design Notes

### Object-Oriented Pattern

Modules that produce objects (via `Create`) effectively introduce a lightweight object-oriented pattern into the stack-based script system. The object is passed as the first argument to subsequent member calls, mimicking method invocation without requiring a class system.

### Determinism Requirement

All module member functions must produce **deterministic results** given identical inputs. This is a consensus-critical constraint — non-deterministic behavior (e.g., using system randomness, wall-clock time) would cause validation divergence across nodes.

### Extensibility

The reserved range (213–248) provides ample room for growth. The `MO_X` extension gateway further expands capacity to 256 additional sub-modules, each with up to 256 members — a total of 65,536 extended member functions before resorting to the `EX_MO` instruction.
