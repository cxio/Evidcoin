# Proposal: Value Instructions [0-19]

> **Prerequisite:** This specification builds upon the [Base Constraints](0.Base-Constraints.md) document, which defines the instruction format, runtime model, and notation conventions used throughout.


## 1. Overview

Value instructions occupy opcode range **[0-19]** (20 instructions). They represent literal values that, upon execution, automatically push themselves onto the data stack. These are the fundamental data-producing instructions of the script system.

Value instructions serve as the primary mechanism for introducing data into the execution environment — constants, typed literals, inline data, code blocks, and external script references all enter through this category.

### Design Principle

Each value instruction is self-describing: the opcode determines the type, the auxiliary parameters (if any) specify sizing, and the associated data carries the payload. There is no ambiguity about what a value instruction produces.

Numeric literals are enclosed in curly braces `{}` (rather than appearing as bare numbers) to maintain visual consistency with the script's "instruction sequence" paradigm, where every element has clear boundaries.


## 2. Instruction Definitions

### 2.1 Nil Value

```
0       NIL         Null value: nil
```

- **Type:** Nil
- **Auxiliary params:** None
- **Associated data:** None
- The opcode byte alone represents this value. Pushes a `nil` onto the data stack.
- Used as a generic absence-of-value marker.

### 2.2 Boolean Values

```
1       TRUE        Boolean true
2       FALSE       Boolean false
```

- **Type:** Bool
- **Auxiliary params:** None
- **Associated data:** None
- The opcode byte alone encodes the boolean value. Runtime representation is `true` or `false`.
- Example bytecode: `0x01` -> `true`, `0x02` -> `false`.

### 2.3 Byte Value

```
3       ''          Single byte: byte [0, 255]
```

- **Type:** Byte
- **Auxiliary params:** None
- **Associated data:** 1 byte — the uint8 value.
- An alias for uint8, suitable for small positive integer arithmetic.
- Syntax uses single quotes around a character literal.
- **Examples:**
  - `'A'` — character A (value 65). Compiled: `0x03 0x41`.
  - `'\n'` — escape sequence for newline (value 10).
  - `'\x41'` — hexadecimal escape for A.

### 2.4 Rune Value

```
4       ''          Rune (Unicode code point): rune [0, 0x10FFFF]
```

- **Type:** Rune
- **Auxiliary params:** None
- **Associated data:** 4 bytes — the uint32 Unicode code point.
- Represents a single Unicode code point.
- **Examples:**
  - `'嗨'` — Chinese character in single quotes.
  - `'\u4e16'` — Unicode escape form (`\uhhhh` or `\Uhhhhhhhh`).
  - `'\u0041'` — character A stored as Rune type (4 bytes, distinct from Byte).
- **Disambiguation:** The compiler determines whether a single-quoted literal is Byte (opcode 3) or Rune (opcode 4) based on the code point range. Values <= 255 use Byte by default; Rune is used when the `\u` / `\U` notation is explicit or the character exceeds the byte range.

### 2.5 Integer Value

```
5       {}          Integer: int64 [-1<<63, 1<<63 - 1]
```

- **Type:** Int
- **Auxiliary params:** None
- **Associated data:** Variable-length integer (varint encoding).
- Standard signed 64-bit integer.
- **Examples:**
  - `{12345}` — integer value 12345.
  - `{123_45}` — same value; single underscores are permitted as visual separators.

### 2.6 Big Integer Value

```
6       {}(1)       Big integer: big.Int
```

- **Type:** BigInt
- **Auxiliary params:** 1 byte — specifies the byte length of the following big integer data.
- **Associated data:** Big integer value in big.Int encoding.
- Maximum size: 255 bytes (limited by the 1-byte length prefix).
- **Examples:**
  - `{0x8000ffff_8000ffff_8001}` — exceeds int64 range, parsed as BigInt.
- Typically written in hexadecimal; underscores permitted as separators.

### 2.7 Float Value

```
7       {}          Float: float64
```

- **Type:** Float
- **Auxiliary params:** None
- **Associated data:** 8 bytes — IEEE 754 float64.
- **Examples:**
  - `{3.14159}` — floating-point literal.
  - `{2.16e10}` — scientific notation.
  - `{123.}` — trailing dot distinguishes from integer `{123}`.
- **Disambiguation:** The presence of a decimal point (`.`) or exponent notation (`e`/`E`) distinguishes Float from Int.

### 2.8 String Values

```
8       ""(1)       Short string: string [0, 256) bytes
9       ""(2)       Long string: string [256, 65536) bytes
```

- **Type:** String
- **Auxiliary params:** 1 byte (opcode 8) or 2 bytes (opcode 9) — the string's byte length.
- **Associated data:** UTF-8 encoded string content.
- The compiler automatically selects the appropriate opcode based on content length.
- Supports Go-style escape sequences within double quotes.
- Backtick syntax (`` `...` ``) provides raw string literals with no escape processing.
- **Examples:**
  - `"Hello world"` — simple double-quoted string.
  - `"Hello\nWorld"` — string with escape sequence.
  - `` `Hello\nworld` `` — raw string; `\n` is literal two characters, not newline.

### 2.9 Value Set (Typed Array Literal)

```
10      {...}(~)    Value set: [...]
```

- **Type:** Typed slice (e.g., `[]Int`, `[]Float`, `[]String`)
- **Auxiliary params:** Varint — total byte length of the member value sequence.
- **Associated data:** Sequence of value instructions, each in its own opcode-encoded format.
- A convenience constructor for creating homogeneous typed arrays inline.
- **Type uniformity requirement:** All members must be of the same type, or at least **promotable** to a common type. Promotion is performed at runtime:
  - `Byte -> Rune -> Int -> BigInt`
  - `Byte -> Rune -> Int -> Float`
  - Float and String are terminal — they cannot be promoted further.
- **Examples:**
  - `{TRUE, FALSE}` — `[]Bool`
  - `{'A', 'B'}` — `[]Byte`
  - `{'你', '好'}` — `[]Rune`
  - `{2, 4, 8}` — `[]Int`
  - `{100, }` — single-member set `[100]`; trailing comma required.
  - `{3.14, 5}` — `[]Float` (Int `5` promoted to Float).
  - `{"Hello", "World"}` — `[]String`
- **Error case:** `{}` — empty braces without type context is an error.
- **Design note:** Type promotion happens at runtime rather than compile time. This is a special case within value instructions — not an independent type, but a convenient inline construction mechanism.

### 2.10 Byte Data (Binary Blobs)

```
11      DATA{}(1)   Small data: []byte [0, 256) bytes
12      DATA{}(2)   Large data: []byte [256, 65536) bytes
```

- **Type:** Bytes
- **Auxiliary params:** 1 byte (opcode 11) or 2 bytes (opcode 12) — the data length.
- **Associated data:** Raw byte content.
- Written in hexadecimal without `0x` prefix. The compiler automatically selects the opcode based on data length.
- **Examples:**
  - `DATA{46af3fb4}` — 4 bytes, auto-calculated length.
  - `DATA[8]{46af3fb4}` — explicit 8-byte length; trailing positions zero-padded.
- **Compiled bytecode example:** `DATA{414243}` (content "ABC") ->
  - `0x0B` (opcode 11), `0x03` (length 3), `0x41 0x42 0x43` (data).

### 2.11 Regular Expression

```
13      /.../(1)    Regular expression: RE2
```

- **Type:** RegExp
- **Auxiliary params:** 1 byte — content byte length (maximum 255).
- **Associated data:** UTF-8 encoded RE2 regular expression.
- Uses RE2 syntax ([google/re2](https://github.com/google/re2/wiki/Syntax)), which guarantees linear-time matching.
- Written between forward slashes.
- **Examples:**
  - `/(?i)[a-z0-9]+/` — case-insensitive ASCII alphanumeric match.
  - `/(?i)Zip-(\d+)/` — case-insensitive with capture group.
- **Size limit:** 255 bytes maximum for the expression text.

### 2.12 Date/Time Value

```
14      DATE{}      Time object
```

- **Type:** Time
- **Auxiliary params:** None
- **Associated data:** Signed variable-length integer — UNIX timestamp in milliseconds.
- Curly-brace syntax accepts RFC 3339 formatted date-time strings (compiled to timestamp).
- Square-bracket syntax accepts raw UNIX millisecond timestamps.
- **Examples:**
  - `DATE{2020-11-24T15:04:05Z}` — RFC 3339 UTC.
  - `DATE{2020-11-24T15:04:05.456+08:00}` — RFC 3339 with timezone.
  - `DATE[1606227845000]` — raw UNIX timestamp in milliseconds.

### 2.13 Reserved

```
15-16   (Reserved, unused)
```

Two opcodes held in reserve for future value types.

### 2.14 Code Block (Inline Instruction Sequence)

```
17      CODE{}(~)   Code/instruction sequence
```

- **Type:** Script
- **Auxiliary params:** Varint — byte length of the enclosed instruction sequence.
- **Associated data:** Compiled instruction sequence (opcodes + auxiliary params + associated data).
- The enclosed instructions are **inert data** — they are not executed upon encounter. The CODE block is pushed onto the stack as a Script-typed value.
- Can be used with `EVAL` for dynamic execution, or provided to third-party extensions as executable logic.
- **Example:**
  ```
  CODE{
       TOP FN_HASH256 DATA{<hashResult>} EQUAL PASS
  }
  ```
  A hash-barrier verification sequence stored as data.
- **Compiled bytecode:** `0x11` (opcode 17), followed by varint length, followed by the compiled inner sequence.

### 2.15 Script Reference (External)

```
18      SCRIPT(~,64,~)  Script reference
```

- **Type:** Script
- **Auxiliary params:**
  1. Varint — transaction year (computed from the transaction timestamp).
  2. 64 bytes — transaction ID.
  3. Varint — output index (0-based).
- **Associated data:** None
- Fetches the lock script from the specified transaction output and pushes it onto the stack as a Script-typed value.
- Unlike `GOTO`/`EMBED`, this instruction does **not** execute the referenced script — it merely loads it as data.
- **Example:**
  ```
  SCRIPT[2025, 0xab93cf..., 0] FN_HASH256 ...
  ```
  Load the script from 2025-era transaction's first output, then hash it.
- **Use case:** Referencing attestation transaction identification scripts for verification or comparison.

### 2.16 Generic Value Accessor

```
19      VALUE{}(1)      Retrieve a named/indexed value
```

- **Type:** Any
- **Auxiliary params:** 1 byte — value index.
- **Associated data:** None (when using index form) or value name (when using name form).
- A general-purpose value retrieval instruction that overlaps partially with `ENV{}` but provides a broader access mechanism.
- **Examples:**
  - `VALUE{someName}` — retrieve by name.
  - `VALUE[0]` — retrieve by index.
- **Design note:** Separated out to allow certain essential values to be available within unlock scripts (opcodes [0-49]). Also positioned for potential future use in quantum-computing preparatory pre-evaluation.


## 3. Type System Summary

The value instructions introduce the following type hierarchy:

| Type | Opcode | Size | Promotable To |
|------|--------|------|---------------|
| Nil | 0 | 0 bytes | — |
| Bool | 1, 2 | 1 byte (opcode only) | — |
| Byte | 3 | 1 byte | Rune, Int, BigInt, Float |
| Rune | 4 | 4 bytes | Int, BigInt, Float |
| Int | 5 | 1-10 bytes (varint) | BigInt, Float |
| BigInt | 6 | 2-256 bytes | — |
| Float | 7 | 8 bytes | — |
| String | 8, 9 | 2-65537 bytes | — |
| Bytes | 11, 12 | 2-65537 bytes | — |
| RegExp | 13 | 2-256 bytes | — |
| Time | 14 | 1-10 bytes (varint) | — |
| Script | 17, 18 | variable | — |
| Any | 19 | variable | — |

### Promotion Rules

Type promotion applies within value sets (opcode 10) and is evaluated at runtime:

```
Byte -> Rune -> Int -> BigInt
Byte -> Rune -> Int -> Float
```

Float and String are terminal types with no further promotion path. BigInt and Float are sibling promotions from Int — a value set cannot contain both BigInt and Float members (they promote along divergent paths).


## 4. Compilation Notes

- The compiler resolves literal syntax to the appropriate opcode automatically (e.g., `'A'` -> Byte or Rune based on code point; `{123}` -> Int, `{123.}` -> Float).
- Auxiliary parameter values (string lengths, data lengths, varint-encoded integers) are computed at compile time.
- Value set member type compatibility is verified at compile time where possible, with runtime promotion for cross-type scenarios.
